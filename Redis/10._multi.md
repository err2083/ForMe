# Redis 트랜잭션의 매커니즘과 ACID 특성

## 1. Redis 트랜잭션 개요

Redis 트랜잭션은 여러 개의 명령어를 하나의 논리적 단위로 묶어서 실행하는 기능입니다. 트랜잭션 내의 모든 명령은 순차적으로 처리되며, 실행 도중 다른 클라이언트의 명령이 끼어들 수 없음을 보장합니다.

* **기본 목적**: 여러 명령을 원자적으로 실행하여 데이터의 일관성을 유지하고 동시성을 제어하기 위해 사용합니다. (예: 은행 계좌 이체 시 A 계좌 출금과 B 계좌 입금의 동시 실행)
* **특징**: Redis는 단일 스레드로 동작하기 때문에, 트랜잭션이 실행되는 동안 해당 클라이언트는 서버를 독점하는 것과 같은 격리성을 제공받습니다.

## 2. 트랜잭션 동작 흐름 (MULTI & EXEC)

Redis 트랜잭션은 명령어를 즉시 실행하지 않고 큐(Queue)에 쌓아두었다가 한 번에 실행하는 방식을 취합니다.

1. **트랜잭션 시작 (MULTI)**: 클라이언트가 `MULTI` 명령을 보내면 해당 연결은 트랜잭션 모드로 돌입합니다.
2. **명령 큐잉 (Queuing)**: 이후 입력하는 모든 명령은 즉시 실행되지 않고 서버의 커넥션별 큐에 저장됩니다. 이때 각 명령의 응답으로 `QUEUED`라는 상태 값을 받게 됩니다.
3. **일괄 실행 (EXEC)**: `EXEC` 명령을 호출하는 순간, 큐에 쌓여 있던 모든 명령이 처음부터 순서대로(FIFO) 실행됩니다.
4. **결과 반환**: 모든 명령의 실행 결과는 배열 형태로 한 번에 반환됩니다.

## 3. 주요 명령어 상세

* **MULTI**: 트랜잭션을 시작하라는 신호입니다. 이후 클라이언트는 일반 모드에서 트랜잭션 모드로 전환됩니다.
* **EXEC**: 큐에 쌓인 모든 명령을 순차적으로 실행하고 트랜잭션을 종료합니다. 실행 중에는 단일 스레드 특성상 다른 클라이언트의 개입이 차단됩니다.
* **DISCARD**: 트랜잭션을 취소하고자 할 때 사용합니다. 큐에 쌓인 모든 명령을 폐기하고 클라이언트를 일반 모드로 복구시킵니다.
* **WATCH (낙관적 잠금)**: 특정 키의 변경 사항을 감지합니다. `WATCH`한 키가 트랜잭션 실행 전(EXEC 호출 전)에 다른 클라이언트에 의해 변경되었다면, 해당 트랜잭션은 실패(Null 반환)하게 됩니다. 조건부 로직 구현에 사용됩니다.

## 4. 에러 처리 및 전통적 DB와의 차이점

Redis 트랜잭션은 우리가 흔히 아는 RDBMS(MySQL 등)의 트랜잭션과는 결정적인 차이가 있습니다.

* **롤백(Rollback) 미지원**: 실행 중 일부 명령이 실패(런타임 오류)하더라도 이미 실행된 이전 명령을 취소하거나 전체를 되돌리지 않습니다. 실패한 명령을 제외한 나머지 명령은 그대로 적용됩니다.
* **오류 유형에 따른 동작**:
  * **문법 오류 (Syntax Error)**: 명령어를 큐에 넣는 시점에 문법이 잘못되었다면 `EXEC` 호출 시 전체 트랜잭션이 거부되고 아무 명령도 실행되지 않습니다.
  * **실행 오류 (Runtime Error)**: 명령 실행 시점(예: String 키에 List 명령 사용)에 발생하는 오류는 해당 명령만 실패 처리하고 나머지는 계속 진행됩니다.

## 5. Redis 트랜잭션의 ACID 분석

### 1) 원자성 (Atomicity) - 부분적 지원

* 전통적인 의미의 'All or Nothing'과는 다릅니다.
* 명령 일괄 실행은 보장되지만, 중간에 실패하더라도 이전 명령의 결과가 유지되는 '롤백 없는 원자성'을 가집니다. 문법 오류 시에만 전체가 취소됩니다.

### 2) 일관성 (Consistency) - 애플리케이션 책임

* Redis는 데이터 타입 검증 정도의 일관성만 제공합니다.
* '잔액이 음수가 되면 안 된다'와 같은 비즈니스 로직상의 일관성은 Redis가 아닌 애플리케이션 레벨에서 `WATCH` 등을 활용해 직접 구현해야 합니다.

### 3) 격리성 (Isolation) - 완전 지원

* Redis의 가장 강력한 특징입니다. 단일 스레드 구조 덕분에 `EXEC`가 실행되는 동안 다른 명령이 끼어들 여지가 전혀 없습니다.
* 모든 트랜잭션은 직렬화되어 순차적으로 실행되는 것과 같은 효과를 가집니다.

### 4) 지속성 (Durability) - 설정 의존

* 트랜잭션 자체의 기능이라기보다는 Redis의 영속성(Persistence) 설정(AOF/RDB)에 따라 결정됩니다.
* `AOF` 설정이 `Always`라면 완벽한 지속성을 가지며, `Everysec`라면 최대 1초치의 데이터 손실 가능성이 있습니다.

## 6. 실무적 관점

* Redis 트랜잭션은 실제로는 명령 큐잉과 일괄 실행(Pipelining과 유사한 이점)에 더 집중된 형태입니다.
* 원자적 롤백이 보장되지 않기 때문에 복잡한 비즈니스 트랜잭션보다는 성능 최적화나 간단한 동시성 제어 용도로 고려됩니다.
