# Redis Hash 타입의 구조와 실무 활용

## 1. Redis Hash 개요

Redis Hash는 하나의 키 내부에 여러 개의 **필드(Field)와 값(Value)의 쌍**을 저장하는 컬렉션입니다.

* **구조적 특징**: 프로그래밍 언어의 해시맵(HashMap), 딕셔너리(Dictionary), 혹은 객체(Object)와 매우 유사한 구조입니다.
* **RDB 비유**: 관계형 데이터베이스(RDB)의 **하나의 행(Row)**과 비슷하며, 각 필드는 컬럼(Column)에 대응됩니다.
* **데이터 타입**: 필드 이름과 값은 모두 문자열(String)이며, Binary Safe 특성을 가져 이미지, JSON 등 어떤 이진 데이터든 저장할 수 있습니다.
* **활용도**: 객체 표현에 최적화되어 있어 사용자 정보(이름, 이메일, 나이 등)나 상품 정보 등을 관리할 때 주로 사용됩니다.

## 2. 내부 저장 구조 및 인코딩 (Memory Optimization)

Redis는 데이터의 크기에 따라 두 가지 인코딩 방식을 자동으로 전환하여 메모리 효율을 극대화합니다.

### 1) ListPack (리스트팩)

* **특징**: 메모리 효율적인 압축 인코딩 방식입니다. 필드와 값을 연속된 메모리 공간에 순차적으로 저장합니다. (Redis 7.0 이전의 ziplist를 대체함)
* **사용 조건**:
  * 해시 필드 개수가 설정값(기본 512개) 이하일 때
  * 모든 필드 이름과 값의 크기가 설정값(기본 64바이트) 미만일 때
* **성능**: 순차 구조이므로 탐색 시 O(N)이지만, 데이터 자체가 작아 실제 속도는 매우 빠릅니다.

### 2) Hash Table (해시 테이블)

* **특징**: 일반적인 해시 테이블 구조로, 필드 이름을 해시 함수로 계산하여 버킷에 저장합니다.
* **사용 조건**: 위 ListPack 조건을 하나라도 벗어나면 자동으로 전환됩니다.
* **성능**: 메모리 사용량은 더 많지만, 필드 접근 속도가 **O(1)**로 고정되어 대용량 데이터에서도 매우 빠릅니다.

## 3. Hash vs 개별 String 키 비교

동일한 데이터를 저장할 때 여러 개의 String 키를 사용하는 것보다 하나의 Hash를 사용하는 것이 **메모리 측면에서 더 효율적**입니다.

* **이유**: Redis에서 각 키는 고유한 메타데이터를 가집니다. 여러 String 키로 나누면 키마다 메타데이터가 붙지만, Hash를 쓰면 하나의 키에 대한 메타데이터만 관리하면 되기 때문입니다.

## 4. 핵심 명령어 상세

### 1) 기본 설정 및 조회

* **HSET key field value [field value...]**: 하나 또는 여러 개의 필드와 값을 설정합니다.
  * 새로 추가된 필드 개수를 반환하며, 기존 필드 수정 시에는 0을 반환합니다.
  * Redis 4.0 이상부터는 여러 필드를 한 번에 설정할 수 있어 `HMSET`을 대체합니다.
* **HGET key field**: 특정 필드의 값을 조회합니다. 존재하지 않으면 `nil`을 반환합니다.

### 2) 다중 작업 (Multi-field Operations)

* **HMGET key field [field...]**: 여러 필드의 값을 한 번에 조회하여 배열로 반환합니다.
  * 요청한 필드 순서대로 값이 반환됩니다.
  * **성능 이점**: 네트워크 왕복 횟수(RTT)를 줄여 전체적인 처리 성능을 비약적으로 향상시킵니다.
* **HMSET**: 과거 다중 필드 설정에 사용되었으나 현재는 `HSET`으로 통합되어 **Deprecated(권장되지 않음)** 되었습니다. 하위 호환성을 위해서만 존재합니다.

### 3) 필드 증가 (Atomic Increment)

* **HINCRBY key field increment**: 필드의 값을 정수형으로 해석하여 지정한 크기만큼 증가시킵니다. (64비트 정수 범위)
* **HINCRBYFLOAT key field increment**: 필드의 값을 부동소수점(실수)으로 해석하여 증가시킵니다.
* **특징**: **원자적(Atomic) 연산**이므로 레이스 컨디션(경쟁 상태) 없이 안전하게 카운터를 관리할 수 있습니다.

### 4) 정보 조회 및 확인

* **HEXISTS key field**: 특정 필드가 존재하는지 확인합니다 (존재 시 1, 미존재 시 0). 시간 복잡도 O(1).
* **HKEYS key**: 해당 해시의 **모든 필드 이름**을 배열로 반환합니다.
* **HVALS key**: 해당 해시의 **모든 값**을 배열로 반환합니다.
* **주의**: `HKEYS`와 `HVALS`는 O(N) 연산이며 반환 순서가 보장되지 않습니다. 필드가 매우 많은 대용량 해시에서 사용 시 성능 저하를 일으킬 수 있으므로 주의해야 합니다.

## 5. 실무 가이드 및 성능 고려사항

* **객체당 필드 수**: 일반적으로 해시는 하나의 객체를 표현하므로 필드 수가 10~20개 내외인 경우가 많습니다. 이론상 최대 42억 개(2^32-1)까지 저장 가능하지만, 메모리 제한과 성능을 고려해 적절히 관리해야 합니다.
* **부분 업데이트**: 객체 전체를 다시 쓰지 않고 특정 필드(예: 사용자의 나이만 변경)만 개별적으로 접근하여 업데이트할 수 있어 효율적입니다.
* **일관성**: `HSET`이나 `HMGET` 등을 활용해 여러 데이터를 한 번에 처리함으로써 데이터 일관성을 유지하고 네트워크 자원을 최적화하는 것이 중요합니다.
