# Redis 키 관리 및 효율적인 조회 기법

## 1. 키 조회 명령어 비교: KEYS vs SCAN

Redis 서버 내의 키를 검색할 때 사용하는 두 가지 대표적인 명령어입니다. 성능과 안정성 측면에서 큰 차이가 있으므로 반드시 구분하여 사용해야 합니다.

### 1) KEYS (패턴 매칭 방식)

* **특징**: 특정 패턴(와일드카드)에 맞는 모든 키를 한 번에 조회합니다.
* **패턴 예시**:
  * `KEYS *`: 모든 키 반환.
  * `KEYS user*`: 'user'로 시작하는 모든 키 반환.
  * `KEYS user[ab]*`: 'user' 뒤에 a 또는 b가 오는 키와 매칭.
  * `KEYS user?`: 'user' 뒤에 임의의 한 문자가 오는 키와 매칭.
* **위험성**:
  * **O(N) 복잡도**: 전체 키 개수에 비례하여 시간을 소모하며, 모든 키를 순회하며 비교합니다.
  * **서버 블로킹**: Redis는 싱글 스레드로 동작하므로, `KEYS`가 실행되는 동안 다른 모든 요청(읽기/쓰기)이 중단됩니다.
  * **실무 사례**: 대규모 서비스(예: 쿠팡)에서 이 명령어를 잘못 사용하여 일시적인 서비스 장애가 발생한 사례가 뉴스에 보도되기도 했습니다.
* **가이드**: **프로덕션(운영) 환경에서는 절대 사용 금지**입니다. 개발 환경이나 로컬 테스트 용도로만 사용해야 합니다.

### 2) SCAN (커서 기반 방식)

* **특징**: 커서(Cursor)를 사용하여 키 목록을 일부씩 나누어 조회하는 안전한 대안입니다.
* **작동 원리**:
  * `SCAN 0`으로 시작하여 일부 키 배열과 **다음 커서 값**을 반환받습니다.
  * 반환된 커서 값을 다음 `SCAN` 명령에 넣어 호출을 반복합니다.
  * 커서가 다시 **0**으로 돌아오면 전체 순회가 완료된 것입니다.
* **장점**:
  * **Non-blocking**: 한 번의 호출이 짧게 끝나므로 호출 사이사이에 다른 명령어가 처리될 수 있어 서비스 장애를 방지합니다.
  * **안전성**: 호출당 시간 복잡도가 낮아 프로덕션 환경에서 권장됩니다.
* **주의사항**:
  * 순회 도중 키가 추가/삭제되거나 해시 테이블 리사이징이 발생하면 **중복된 키가 반환**될 수 있으므로 클라이언트 측의 중복 처리 로직이 필요합니다.

## 2. 키 존재 확인: EXISTS

특정 키가 현재 데이터베이스에 존재하는지 확인하는 명령어입니다.

* **기본 사용**: `EXISTS key`
  * 존재하면 **1**, 존재하지 않으면 **0**을 반환합니다.
  * 시간 복잡도는 **O(1)**로 매우 빠릅니다.
* **다중 확인 (3.0.3+버전)**: `EXISTS key1 key2 ...`
  * 여러 개의 키를 인자로 주면 존재하는 키의 총 개수를 반환합니다.
* **GET과의 차이점**:
  * `GET`을 써서 결과가 `nil`인지 확인할 수도 있지만, `EXISTS`가 의도가 훨씬 명확합니다.
  * **효율성**: 값 전체를 가져오는 `GET`보다 존재 여부(0/1)만 전달하는 `EXISTS`가 네트워크 자원을 훨씬 적게 소모합니다.
  * **호환성**: String이 아닌 타입(List, Set 등)의 키에 `GET`을 쓰면 에러가 발생하지만, `EXISTS`는 타입에 상관없이 안전하게 확인 가능합니다.

## 4. 기타 유용한 키 관리 명령어

강의에서 언급된 필수적인 보조 명령어들입니다.

* **TYPE**: 키의 데이터 타입(String, List 등)을 확인합니다.
* **RENAME**: 키의 이름을 변경합니다.
* **RANDOMKEY**: 데이터베이스에서 랜덤한 키 하나를 반환합니다.

## 5. 실무 가이드 요약

* **방어력 80% 전략**: 모든 명령어를 외우기보다 실무에서 필수적인 80%의 지식을 완벽히 익히는 것이 효율적입니다.
* **네트워크 최적화**: 단순히 기능이 동작하는 것을 넘어, 전송되는 데이터 양을 최소화하는 명령어를 선택하는 습관이 중요합니다.
