# Redis Pub/Sub의 매커니즘과 실무 활용

## 1. Redis Pub/Sub 개요

Redis는 단순한 캐싱 툴을 넘어, 메시지 큐(MSQ)나 분산 환경에서의 이벤트 버스로 활용될 수 있는 **발행/구독(Publisher/Subscriber) 패턴**을 제공합니다.

* **기본 동작**: 발행자(Publisher)가 특정 채널에 메시지를 발행하면, 해당 채널을 구독하고 있는 모든 구독자(Subscriber)에게 메시지가 실시간으로 전달됩니다.
* **철학**: 메시징 시스템 스펙트럼에서 가장 단순한 형태를 지향하며, 복잡한 라우팅이나 영속성 없이 실시간 전파에만 집중합니다.
* **Fire and Forget**: 메시지를 전송한 후 수신 여부를 확인하지 않고 잊어버리는 경량화된 시스템입니다.

## 2. 주요 특징: 느슨한 결합 (Loose Coupling)

Pub/Sub 패턴은 발행자와 구독자 간의 의존성을 제거하여 유연한 시스템 아키텍처(EDA, MSA) 구현을 가능하게 합니다.

* **공간적 분리**: 발행자는 누가 구독하는지 알 필요가 없고, 구독자 역시 누가 발행하는지 알 필요가 없습니다. 오직 '채널 이름' 하나로만 연결됩니다.
* **시간적 분리 (제약 사항)**: 일반적인 메시지 큐와 달리 Redis Pub/Sub은 메시지를 저장하지 않습니다. 따라서 발행 시점에 구독자가 연결되어 있지 않으면 해당 메시지는 유실됩니다. 즉, 서비스가 제대로 동작하려면 발행자와 구독자가 동시에 활성화되어 있어야 합니다.

## 3. 핵심 기능 및 명령어

### 1) 구독 (Subscription)

* **SUBSCRIBE [channel]**: 하나 이상의 채널을 구독합니다. 명령 실행 시 클라이언트는 구독 전용 모드로 전환됩니다.
* **UNSUBSCRIBE [channel]**: 특정 채널의 구독을 해지합니다.
* **PSUBSCRIBE [pattern]**: 와일드카드 패턴을 사용하여 여러 채널을 한 번에 구독합니다.
  * `*`: 0개 이상의 문자 매칭
  * `?`: 정확히 1개의 문자 매칭
* **주의사항**: 일반 구독과 패턴 구독은 독립적으로 관리됩니다. 만약 동일한 채널을 두 방식으로 모두 구독했다면, 메시지 발생 시 중복해서 수신하게 됩니다.

### 2) 발행 (Publishing)

* **PUBLISH [channel] [message]**: 특정 채널에 메시지를 발행하며, 구독 중인 모든 클라이언트에게 실시간으로 브로드캐스팅합니다.

## 4. 내부 구현 구조

Redis는 효율적인 메시지 전달을 위해 두 가지 핵심 데이터 구조를 사용합니다.

* **pubsub_channels (Hash Table)**:
  * 키는 '채널 이름', 값은 해당 채널을 구독하는 '클라이언트 리스트'입니다.
  * 채널을 찾는 데 O(1), 구독자 리스트를 순회하며 메시지를 복사·전달하는 데 O(N)의 시간이 소요됩니다.
* **pubsub_patterns (Linked List)**:
  * 패턴 문자열과 구독자 리스트를 노드로 가지는 연결 리스트입니다.
  * 메시지 발행 시 리스트를 처음부터 끝까지 순회하며 패턴 매칭을 수행하므로, 일반 구독보다 자원 소모가 큰 비싼 연산입니다. 실무에서는 꼭 필요한 경우가 아니면 패턴 구독은 지양하는 것이 좋습니다.

## 5. 메시지 전달 보장 수준 (Quality of Service)

Redis Pub/Sub은 성능 최적화를 위해 전달 보장 수준을 최소화했습니다.

* **At-Most-Once (최대 1번)**: 메시지는 0번 또는 1번 전달됩니다. 유실 가능성이 있으며 리트라이 정책이 없습니다.
* **비교**:
  * **Redis Pub/Sub**: Fire and Forget (유실 가능, 초경량)
  * **RabbitMQ**: At-Least-Once (ACK 메커니즘을 통한 최소 1번 전달 보장)
  * **Kafka**: Exactly-Once (트랜잭션을 통한 정확히 1번 전달 보장)

## 6. 인프라 환경별 동작 방식

### 1) 클러스터(Cluster) 환경

* **브로드캐스트 방식**: 구독자가 어느 노드에 연결되어 있는지 알 수 없기 때문에, 발행된 메시지는 모든 노드에 전파되어야 합니다.
* **클러스터 버스**: 노드 간 통신을 위한 별도의 TCP 연결(기본 포트 + 10000)과 바이너리 프로토콜(Gossip)을 통해 메시지를 전송합니다.
* **문제점**: 노드 수가 많아질수록 네트워크 대역폭 사용량이 N배로 증폭되어 병목 현상이 발생할 수 있습니다.
* **Shard Pub/Sub (Redis 7.0+)**: 채널을 특정 해시 슬롯에 할당하여 해당 슬롯을 담당하는 노드 내에서만 메시지를 처리하도록 최적화된 방식(`SSUBSCRIBE`, `SPUBLISH`)입니다.

### 2) 마스터-레플리카(Replication) 환경

* **복제 불가**: 일반적인 키와 달리 Pub/Sub 메시지는 마스터에서 레플리카로 복제되지 않습니다.
* **설계 철학**: Pub/Sub은 '지금 이 순간'의 이벤트에만 집중하므로, 안정성을 위해 과거 데이터를 기록하거나 복제할 필요가 없다고 판단하기 때문입니다. 따라서 모든 Pub/Sub 클라이언트는 반드시 마스터 노드에 연결되어야 합니다.

## 7. 실무 가이드 및 요약

* Redis Pub/Sub은 매우 가볍고 빠르지만 **영속성이 전혀 없으므로**, 유실되어도 무방한 실시간 알림, 일시적인 이벤트 전파 등에만 사용해야 합니다.
* 중요도가 높고 유실되면 안 되는 메시지라면 Redis Streams나 RabbitMQ, Kafka와 같은 전문 메시징 시스템을 고려해야 합니다.
* 구독자가 없는 채널은 메모리를 거의 점유하지 않으므로 매우 효율적인 시스템 리소스 관리가 가능합니다.
