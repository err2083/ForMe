# Redis 영속성(Persistence)과 키 관리 전략

## 1. Redis 영속성의 개념과 필요성

Redis는 모든 데이터를 메모리(RAM)에 저장하여 마이크로초 단위의 빠른 응답을 제공하지만, 메모리는 전원이 꺼지면 데이터가 사라지는 **휘발성**을 가집니다.

* **필요한 상황**:
  * **서버 업데이트/재시작**: 기능 배포, 보안 패치, Redis 버전 업그레이드 시 데이터 유지 필요.
  * **장애 상황**: 하드웨어 장애, 전원 문제, 네트워크 장애, 프로세스 강제 종료 시 데이터 복구 필요.
  * **비즈니스 데이터**: 세션 정보(로그인, 장바구니), 결제 정보, 포인트 잔액 등 손실 시 사용자 경험이 크게 나빠지는 데이터 관리.
* **영속성이 불필요한 경우**: 단순 캐시 용도로만 사용하여 원본 데이터가 DB에 따로 있고, 사라져도 다시 채울 수 있는 경우.

## 2. RDB (Redis Database) - 스냅샷 방식

특정 시점의 메모리 데이터 전체를 하나의 바이너리 파일(`dump.rdb`)로 저장하는 방식입니다.

### 1) 저장 방식

* **SAVE**: 동기 방식으로 저장하며, 작업이 완료될 때까지 Redis 서버가 모든 클라이언트 요청을 차단합니다. 운영 환경에서는 권장되지 않습니다.
* **BGSAVE (Background Save)**: 백그라운드 방식으로, `fork()` 시스템 콜을 통해 자식 프로세스를 생성하여 스냅샷을 만듭니다. 부모 프로세스는 계속 요청을 처리합니다.
* **자동 저장 (save 설정)**: 특정 시간 동안 일정 횟수 이상의 데이터 변경이 발생하면 자동으로 `BGSAVE`를 실행합니다.

### 2) 기술적 특징: Copy-on-Write (COW)

* `fork()` 직후 부모와 자식 프로세스는 동일한 메모리 페이지를 공유합니다.
* 부모 프로세스가 데이터를 변경할 때만 해당 메모리 페이지만 복사하므로 메모리를 효율적으로 사용합니다.
* **주의**: 최악의 경우(모든 데이터 변경 시) 전체 메모리만큼의 여유 공간이 필요할 수 있습니다.

### 3) 장단점

* **장점**: 단일 파일로 관리가 쉽고 백업/이전이 용이하며, 바이너리 형식이어서 복구 속도가 매우 빠릅니다.
* **단점**: 마지막 스냅샷 이후 발생한 데이터는 손실될 수 있습니다. (예: 5분 주기 저장 시 최대 5분치 데이터 손실 가능)

## 3. AOF (Append Only File) - 로그 방식

데이터를 변경하는 모든 명령을 로그 파일에 순차적으로 기록하는 방식입니다.

### 1) 작동 원리

* 모든 쓰기 명령(SET, DEL 등)을 파일 끝에 추가합니다. (읽기 명령은 기록 제외)
* 서버 재시작 시 로그 파일을 처음부터 끝까지 재실행하여 상태를 복구합니다.

### 2) 디스크 동기화 전략 (appendfsync)

* **always**: 매 명령마다 즉시 디스크에 저장. 가장 안전하지만 성능이 가장 낮음.
* **everysec (권장)**: 1초마다 버퍼를 디스크로 저장. 성능과 안전성의 균형이 좋아 기본 설정으로 권장됨.
* **no**: OS가 적절한 때에 저장하도록 맡김. 성능은 가장 좋지만 최대 30초가량의 데이터 손실 위험이 있음.

### 3) AOF 재작성 (Rewrite / BGREWRITEAOF)

* 시간이 지남에 따라 파일이 비대해지는 것을 막기 위해, 현재 메모리 상태를 만드는 최적의 명령으로 파일을 다시 씁니다.
* 중복된 명령을 제거하여 파일 크기를 획기적으로 줄이며, 백그라운드 자식 프로세스가 수행합니다.

### 4) 장단점

* **장점**: 데이터 손실이 거의 없으며(1초 이내), 텍스트 형식이어서 사람이 읽고 편집할 수 있습니다. (실수로 실행한 `FLUSHALL` 명령을 로그에서 지우고 복구 가능)
* **단점**: RDB보다 파일 크기가 크고, 모든 명령을 재실행해야 하므로 복구 속도가 상대적으로 느립니다.

## 4. 하이브리드(Hybrid) 방식 (Redis 4.0+)

RDB의 빠른 로딩과 AOF의 안정성을 결합한 방식이며 현재 가장 권장되는 방식입니다.

* **구조**: 하나의 파일 내부에 **앞부분은 RDB(바이너리 스냅샷)**, **뒷부분은 AOF(텍스트 로그)** 형식이 공존합니다.
* **재작성 과정**:
    1. 자식 프로세스가 현재 메모리 상태를 RDB 형식으로 파일 앞부분에 씁니다.
    2. 재작성 중에 들어온 새로운 명령들은 별도의 버퍼에 저장했다가 뒷부분에 AOF 형식으로 추가합니다.
* **복구**: 앞부분은 RDB 로더로 빠르게 로드하고, 뒷부분의 소수 명령만 AOF 파서로 재실행하므로 매우 빠르고 안전합니다.

## 5. 시나리오별 권장 방식

| 시나리오 | 권장 방식 |
| :--- | :--- |
| **순수 캐싱 용도** | 영속성 비활성화 |
| **정기적 백업 중시 (약간의 손실 허용)** | RDB 방식 |
| **데이터 손실 최소화 필요** | AOF 방식 |
| **빠른 복구와 높은 안정성 동시 필요** | **하이브리드 방식 (일반적인 운영 환경)** |
| **극도의 안정성 (이중 백업)** | RDB + AOF 동시 활성화 |
