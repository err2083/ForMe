# Redis List 타입의 구조와 실무 활용

## 1. Redis List 개요 및 특징

Redis List는 문자열 값들의 **연결 리스트(Linked List)** 구조로, 데이터의 순서를 유지하며 양 끝에서의 작업에 최적화되어 있습니다 [1], [2].

* **기본 정의**: 단순한 배열과 달리 양방향 연결 리스트로 구현되어 있어 Head(왼쪽)와 Tail(오른쪽)에서의 접근이 매우 빠릅니다 [1], [2].
* **성능 최적화**: 리스트의 양 끝에서 일어나는 삽입과 삭제 작업은 리스트에 포함된 요소의 개수(10개 혹은 100만 개 등)와 상관없이 항상 **O(1)의 시간 복잡도**를 가집니다 [1], [2], [3].
* **중복 허용**: Set과 달리 동일한 값의 중복 저장을 허용합니다 [4].
* **저장 용량**: 하나의 리스트에 최대 **2^32 - 1개(약 42억 개)**의 요소를 저장할 수 있으며, 실제로는 서버의 가용 메모리 용량에 따라 제한됩니다 [1], [4].

## 2. 내부 저장 구조의 진화

Redis는 메모리 효율성과 성능을 동시에 잡기 위해 데이터 규모에 따라 인코딩 방식을 변경합니다.

* **7.0 이전 버전**:
  * 데이터가 작을 때는 압축된 형태인 **ziplist**를 사용합니다 [5].
  * 데이터가 크거나 많아지면 일반적인 **linkedlist**로 전환합니다 [5].
* **7.0 이후 버전 (Quicklist)**:
  * **하이브리드 구조**: 여러 개의 ziplist를 linkedlist로 연결한 **quicklist**를 기본으로 사용합니다 [5].
  * 각 블록 내부는 ziplist의 압축 효율을 활용하고, 블록 사이는 linkedlist의 빠른 삽입/삭제 성능을 활용하여 메모리 효율과 성능을 모두 확보했습니다 [5], [4].

## 3. 핵심 명령어 상세

### 1) 데이터 추가 (Push)

* **LPUSH / RPUSH**: 각각 Head(왼쪽)와 Tail(오른쪽)에 요소를 추가합니다 [6], [7], [3].
* **다중 추가**: 한 번의 명령으로 여러 값을 순차적으로 넣을 수 있으며, 명령 실행 후 리스트의 전체 길이를 반환합니다 [7], [3].
* **자동 생성**: 대상 키가 존재하지 않으면 빈 리스트를 자동으로 생성한 후 요소를 추가합니다 [8].

### 2) 데이터 추출 (Pop)

* **LPOP / RPOP**: 각각 왼쪽 혹은 오른쪽 끝의 요소를 제거하고 그 값을 반환합니다 [9], [8].
* **Count 옵션 (6.2+)**: 한 번에 여러 개의 요소를 꺼낼 수 있으며, 요청한 개수보다 실제 요소가 적으면 남은 모든 요소를 반환합니다 [10].
* **자동 삭제**: 리스트의 마지막 요소를 꺼내 비어 있게 되면 Redis는 해당 키를 메모리에서 자동으로 삭제합니다 [10], [11].

### 3) 조회 및 범위 검색

* **LRANGE key start stop**: 지정된 인덱스 범위의 요소들을 조회합니다 (0부터 시작, 음수 인덱스 지원) [12], [11].
  * `0 -1`: 리스트의 처음부터 끝까지 전체를 조회합니다 [13].
  * 성능: O(S+N) 복잡도를 가지며, 양 끝 조속 조희는 빠르나 중간 부분 조회는 성능 저하가 발생할 수 있습니다 [13], [14].
* **LLEN**: 리스트의 길이를 반환합니다. 길이를 메타데이터로 별도 저장하므로 요소 수에 상관없이 **O(1)**로 즉시 반환됩니다 [15], [14], [16].
* **LINDEX**: 특정 인덱스의 값을 조회합니다. 인덱스 위치까지 순회가 필요하므로 **O(N)**의 시간 복잡도를 가집니다 [1], [15], [16], [17].

## 4. 실무 활용 패턴 및 전략

### 1) 기본 자료구조 구현 [18], [19], [20]

* **큐(Queue)**: FIFO(선입선출) 구조. `RPUSH`로 넣고 `LPOP`으로 꺼내거나, `LPUSH`로 넣고 `RPOP`으로 꺼내 구현합니다.
* **스택(Stack)**: LIFO(후입선출) 구조. 한쪽 끝에서만 넣고 빼는 방식(`LPUSH` + `LPOP` 등)으로 구현합니다.

### 2) 블록킹 팝 (Blocking POP) - 생산자/소비자 패턴

리스트가 비어 있을 때 CPU를 낭비하는 폴링(Polling) 방식 대신 효율적인 이벤트 처리를 지원합니다.

* **BLPOP / BRPOP**: 리스트에 데이터가 없을 경우 즉시 `nil`을 반환하지 않고, 새로운 데이터가 들어올 때까지 대기합니다 [20], [21].
* **타임아웃**: 대기 시간을 설정할 수 있으며, `0`으로 설정 시 무한정 대기합니다 [21].
* **우선순위 큐**: 여러 개의 리스트를 동시에 모니터링할 수 있으며, 왼쪽에 명시한 키의 우선순위가 더 높게 처리됩니다 (BLPOP list1 list2 list3)

## 5. 성능 및 설계 고려사항

* 리스트는 양 끝단 작업에 매우 강력하지만, **중간 요소를 자주 수정하거나 조회**해야 하는 상황에는 적합하지 않으므로 다른 자료구조를 고려해야 합니다 [14], [17].
* 작업 큐로 활용 시 `LLEN`을 통해 현재 대기 중인 작업량을 실시간으로 모니터링하기 용이합니다 [16].
