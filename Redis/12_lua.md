# Redis Lua 스크립팅의 이해와 고급 활용

## 1. Lua 스크립팅 개요 및 필요성

Redis Lua 스크립팅은 Redis 서버 내부에 내장된 Lua 인터프리터를 통해 사용자가 작성한 스크립트를 직접 실행하는 고급 기능입니다.

* **고급 패턴**: Redis 활용 중 가장 고급화된 단계로, 일반적인 명령어 조합으로는 불가능한 복잡한 로직을 처리합니다.
* **원자성 보장 (ACID)**: 앞서 배운 파이프라인이나 트랜잭션이 완벽히 보장하지 못하는 원자성을 제공합니다. 스크립트 실행 중에는 다른 클라이언트가 끼어들 수 없어 '중간 상태'가 노출되지 않습니다.
* **복잡한 로직의 서버 측 실행**: 조건문(if), 반복문(for/while), 복잡한 계산 등을 애플리케이션 레벨이 아닌 Redis 서버 내에서 직접 실행할 수 있습니다.
* **네트워크 절감 (RTT 감소)**: 애플리케이션과 Redis 사이를 여러 번 오가며 데이터를 읽고 써야 하는 로직을 단 한 번의 호출로 끝낼 수 있어 네트워크 왕복 시간을 획기적으로 줄입니다.
* **실무 예시 (재고 관리)**: '재고를 확인(GET)하고, 충분하면 차감(DECR)'하는 로직을 애플리케이션에서 수행하면 그 사이에 다른 클라이언트가 개입할 수 있으나, Lua 스크립트를 사용하면 모든 과정이 원자적으로 처리됩니다.

## 2. Lua 언어의 특징 및 선정 이유

Redis가 내부 스크립팅 언어로 Lua를 선택한 기술적 배경과 특징입니다.

* **특징**: 매우 가볍고 빠르며, 메모리 사용량이 적습니다. 문법이 간단하여 배우기 쉽고 내장하기 용이합니다.
* **안정성 (Sandbox)**: 스크립트가 파일 시스템에 접근하거나 네트워크 연결을 직접 만들 수 없도록 제한된 샌드박스 환경을 제공하여 Redis 서버의 안정성을 해치지 않습니다.
* **동적 타입 언어**: 변수 선언 시 타입을 지정하지 않으며, 값에 따라 자동으로 결정됩니다.
* **변수 스코프**: `local` 키워드를 사용하여 지역 변수를 선언하는 것이 원칙입니다. `local`이 없으면 전역 변수로 취급되어 다른 스크립트와 충돌할 위험이 있으므로 지양해야 합니다.
* **데이터 타입**: Nil, Boolean, Number, String, Table, Function 등을 지원합니다. (Redis 연동 시 주로 Number, String, Table 사용)
* **참/거짓 판단**: Lua에서는 `false`와 `nil`만 거짓으로 판단합니다. 숫자 `0`이나 빈 문자열(`""`)은 참(True)으로 간주하므로 주의가 필요합니다.

## 3. 실행 메커니즘: EVAL 명령어

스크립트를 실행할 때는 `EVAL` 명령어를 사용하며, Redis는 이를 파싱하여 인터프리터를 통해 실행합니다.

* **명령어 구조**: `EVAL [스크립트 문자열] [키의 개수] [키1 키2...] [값1 값2...]`
* **키 개수(numkeys)의 의미**: Redis 클러스터 환경에서 해당 스크립트가 어느 노드에서 실행될지 결정하기 위함입니다. 관련된 모든 키는 반드시 동일한 해시 슬롯(Hash Slot)에 속해야 합니다.
* **데이터 접근**: 스크립트 내에서 `KEYS` 배열로 키에 접근하고, `ARGV` 배열로 값에 접근합니다. 인덱스는 1번부터 시작합니다 (예: `KEYS[1]`, `ARGV[1]`).
* **격리성 유지**: 스크립트가 실행되는 동안 Redis의 이벤트 루프는 멈추고 다른 클라이언트의 명령은 대기 상태가 됩니다. 작업이 완료된 후에야 다음 명령이 처리됩니다.

## 4. 반환값 처리 및 타입 변환

Lua 스크립트의 실행 결과는 Redis 프로토콜로 자동 변환되어 반환됩니다.

* **변환 규칙**:
  * Lua Number -> Redis Integer
  * Lua String -> Redis Bulk String
  * Lua Table -> Redis Array
* **Boolean 처리의 특수성**: Lua의 `true`는 Redis에서 `1`로 변환되고, `false`는 `nil`로 변환됩니다.

## 5. 성능 최적화 전략 (EVALSHA)

스크립트의 크기가 커질수록 발생하는 네트워크 대역폭 낭비와 서버의 파싱 부하를 줄이기 위한 기법입니다.

* **캐싱 원리**: 스크립트 자체를 매번 전송하는 대신, 서버에 한 번 로드하여 캐싱하고 이후에는 40글자의 SHA1 해시값만 전송하여 실행합니다.
* **명령어**:
  * `SCRIPT LOAD`: 스크립트를 서버 메모리에 로드하고 SHA1 해시를 반환합니다.
  * `EVALSHA`: 반환된 해시값을 이용해 스크립트를 실행합니다.
* **폴백(Fallback) 패턴**: `EVALSHA` 시도 시 스크립트가 캐시에 없으면(`NOSCRIPT` 에러), 다시 `SCRIPT LOAD`를 하거나 전체 스크립트를 담은 `EVAL`을 보내는 방식으로 구현합니다. (서버 재시작 시 캐시가 휘발되기 때문)

## 6. 스크립트 작성 시 주의사항 및 최적화 기법

* **Redis 명령 최소화**: 스크립트 내부에서도 Redis 명령 호출은 비용이 발생합니다. 예를 들어 `EXISTS` 확인 후 `GET`을 하는 대신, 바로 `GET`을 호출하여 결과가 `nil`인지 확인하는 것이 더 효율적입니다.
* **일괄 처리 명령 활용**: 여러 키를 다룰 때는 루프를 돌며 개별 접근하는 것보다 `MGET` 등을 활용하는 것이 좋습니다.
* **테이블 효율성**: Lua 테이블은 메모리를 많이 소모하고 가비지 컬렉션(GC) 부하를 늘립니다. 필요한 만큼만 사용하고, 가능하면 정적으로 크기를 다루거나 재사용해야 합니다.
* **조기 반환 (Early Return)**: 특정 조건(예: 재고 부족)이 맞지 않으면 즉시 `return`하여 불필요한 계산과 이후의 명령 실행을 방지해야 합니다. 이는 평균 실행 시간을 줄이고 데이터 일관성을 유지하는 데 도움이 됩니다.
* **직렬화 최적화**: 애플리케이션 레벨에서의 직렬화/역직렬화 과정은 자원 소모가 크므로, 이를 최소화하거나 간소화하는 것이 성능 향상에 유리합니다.
