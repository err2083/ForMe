# 레디스 캐싱 전략 및 아키텍처 설계

## 1. 캐싱(Caching)의 기본 개념과 도입 목적

캐싱은 자주 사용하는 데이터를 데이터베이스보다 빠른 저장소(메모리 기반의 Redis 등)에 보관하여 시스템의 전반적인 성능을 향상시키는 기법입니다.

* **도입 목적**:
  * **데이터베이스 부하 감소**: 디스크 I/O가 발생하는 DB의 부담을 줄여 단일 장애점(SPOA) 발생 가능성을 낮춥니다.
  * **응답 속도 개선**: 메모리 기반 조회로 사용자에게 밀리초 단위의 빠른 응답을 제공합니다.
* **핵심 과제 (Trade-off)**:
  * **일관성(Consistency)**: 캐시 데이터와 실제 DB 데이터 간의 일치를 유지하는 것입니다.
  * **성능(Performance)**: 높은 일관성을 위해 캐시를 자주 갱신하면 성능이 떨어지고, 성능을 위해 갱신을 늦추면 데이터 불일치가 발생합니다.

## 2. 주요 캐싱 전략 패턴 (4가지)

### 1) Cache-Aside (지연 로딩, Lazy Loading)

애플리케이션이 캐시를 직접 제어하며 필요할 때만 데이터를 캐시에 올리는 방식입니다. 가장 널리 사용되는 기본 패턴입니다.

* **읽기 흐름**:
    1. 애플리케이션이 캐시 확인.
    2. **Cache Hit**: 캐시에서 즉시 데이터 반환.
    3. **Cache Miss**: DB 조회 -> 결과를 캐시에 저장(SET) -> 데이터 반환.
* **쓰기 흐름**: DB에 데이터를 먼저 쓰고 기존 캐시를 삭제(Invalidate)합니다. 이후 읽기 요청 시 새로운 데이터가 캐시에 채워집니다.
* **장점**: 구현이 쉽고 이해하기 직관적입니다. 필요한 데이터만 캐시하므로 메모리 효율적이며, Redis가 장애가 나도 DB를 통해 서비스 유지가 가능합니다.
* **단점**: 첫 요청 시 캐시 미스로 인해 응답이 느릴 수 있고, 캐시 삭제 전 읽기 시 오래된 데이터를 볼 가능성이 있습니다. 모든 읽기 로직에 캐시 확인 코드가 들어가 중복이 발생할 수 있습니다.

### 2) Write-Through (동기 쓰기)

데이터베이스에 데이터를 쓸 때 캐시와 데이터베이스를 동시에 업데이트하는 방식입니다.

* **동작 방식**: 모든 쓰기 요청이 캐시를 거쳐 DB로 전달됩니다. 쓰기가 완료되어야 성공으로 반환됩니다.
* **장점**: 캐시와 DB가 항상 일치하여 데이터 불일치 시간이 없습니다. 쓴 데이터를 바로 읽을 때 캐시 미스가 없어 읽기 성능이 매우 좋습니다.
* **단점**: 매 쓰기마다 두 번의 작업(캐시+DB)이 필요해 쓰기 지연 시간이 증가합니다. 자주 읽지 않는 데이터도 캐시에 저장되어 메모리가 낭비될 수 있습니다.
* **적합한 사례**: 금융 데이터, 재고 정보 등 강한 일관성이 필요한 시스템.

### 3) Write-Behind / Write-Back (비동기 쓰기)

쓰기 작업을 캐시에만 먼저 수행하고, 실제 DB 업데이트는 비동기로 처리하는 방식입니다.

* **동작 방식**: 캐시에 쓰고 즉시 완료 반환 -> 백그라운드 워커나 큐(Queue)가 일정 주기로 DB에 일괄(Batch) 업데이트.
* **장점**: 쓰기 속도가 압도적으로 빠르며 DB 부하를 획기적으로 줄입니다. 트래픽 폭주 시 큐가 완충 작용을 하여 시스템을 보호합니다.
* **단점**: DB 반영 전 캐시 장애 시 데이터 손실 위험이 큽니다. 캐시와 DB 간의 일시적 불일치 시간이 발생하며 구현이 복잡합니다.
* **적합한 사례**: 로그 데이터, 조회수 통계 등 데이터 유실을 일부 감수하더라도 높은 쓰기 성능이 필요한 경우.

### 4) Read-Through (자동 로딩)

애플리케이션이 캐시 레이어만 상대하고, 캐시가 직접 DB로부터 데이터를 로딩하는 방식입니다.

* **동작 방식**: 캐시 미스 발생 시 캐시 레이어가 스스로 DB에서 데이터를 가져와 캐싱한 뒤 반환합니다.
* **장점**: 애플리케이션 코드에서 캐시 로직이 분리되어 비즈니스 로직만 신경 쓸 수 있습니다. 중앙에서 일관된 캐시 정책 관리가 가능합니다.
* **단점**: Redis 자체 기능이 아니므로 별도의 캐시 레이어 구현이 필요하며 초기 구축이 복잡합니다.

## 3. 캐시 스탬피드 (Cache Stampede) 현상

캐시가 만료되는 순간(TTL 만료) 대규모 트래픽이 몰려 모든 요청이 동시에 DB로 전달되어 DB 과부하를 일으키는 현상입니다. 'Thundering Herd' 문제라고도 불립니다.

### 해결 방안

1. **PER(Probabilistic Early Recomputation) 알고리즘**:
   * 캐시가 만료되기 직전에 확률적인 계산(수학적 공식 기반)을 통해 미리 캐시를 갱신하는 방식입니다.
   * TTL과 설정된 베타(Beta) 값을 비교하여 만료 전 업데이트를 유도함으로써 캐시 히트율을 최대한 유지합니다.
2. **분산 락 (Distributed Lock / SetNX)**:
   * 캐시 미스 발생 시 첫 번째 요청만 락을 획득하여 DB를 조회하고 캐시를 채우게 합니다.
   * 나머지 요청은 락이 풀릴 때까지 대기하거나 잠시 후 캐시를 재확인하게 하여 DB 몰림을 방지합니다.
3. **백그라운드 갱신 (Cron)**:
   * 캐시 만료 시간을 두지 않거나 매우 길게 잡고, 별도의 크론탭(Batch)이 주기적으로 DB 값을 읽어 캐시를 갱신합니다.
   * 구현이 단순하지만 DB와 캐시 사이의 실시간 일관성은 다소 떨어질 수 있습니다.

## 4. 실무 설계 가이드

* **동시성 고려**: 짧은 순간에 수만 개의 요청이 몰릴 경우 DB 커넥션 풀 고갈을 막기 위해 락 기반 제어나 확률적 갱신 기법 도입이 필수적입니다.
* **80/20 법칙**: 전체 기능의 사소한 세부 사항에 집착하기보다, 대규모 트래픽 상황에서 시스템의 안정성을 보장할 수 있는 핵심적인 캐시 전략(80%의 성능 보장)을 먼저 구축하는 것이 중요합니다.
* **프레임워크 활용**: Spring Cache나 Django Cache Framework 등에서 지원하는 Read-Through 패턴 등을 활용하면 유지보수 효율을 높일 수 있습니다.
