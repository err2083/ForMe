# Redis Stream의 매커니즘과 메시지 큐 활용 전략

## 1. Redis Stream 개요

Redis Stream은 Redis 5.0에서 추가된 데이터 타입으로, 시간 순서대로 메시지를 쌓아두는 '메세지 로그' 형태의 자료구조입니다.

* **핵심 가치**: 기존 Pub/Sub 모델의 한계인 '메시지 유실' 문제를 해결하고 영속성을 보장합니다.
* **메시지 브로커 기능**: 카프카(Kafka)나 RabbitMQ와 같은 전문 메시지 브로커의 핵심 기능을 Redis 내부에서 제공하여 별도의 시스템 구축 없이 메시지 큐 기능을 구현할 수 있습니다.
* **구조**: 메시지 내용은 필드(Field)와 값(Value)의 쌍으로 구성되어 있어 Redis Hash와 유사한 형태를 가집니다. (예: `sensor-id: 1234`, `temperature: 25.5`)

## 2. Redis Stream의 주요 특징

* **메시지 보존**: 일반적인 큐와 달리 메시지를 소비(Consume)해도 즉시 삭제되지 않고 보존됩니다.
* **다중 소비자**: 여러 소비자가 동일한 메시지를 독립적으로 읽을 수 있는 구조를 지원합니다.
* **시간순 정렬**: 고유한 ID를 기반으로 메시지가 시간 순서대로 엄격하게 정렬됩니다.
* **재처리 가능**: 과거 특정 시점의 메시지부터 다시 읽기가 가능하여 장애 복구에 유리합니다.

## 3. List 자료구조와의 차이점 및 한계 극복

* **List의 한계**: `RPOP`이나 `LPOP`을 사용하면 소비 즉시 메시지가 삭제됩니다. 소비자가 처리 중에 장애(OOM 등)로 죽으면 메시지가 유실되며 복구할 방법이 없습니다.
* **Stream의 장점**: 메시지를 소비해도 데이터가 남아있으며, 'XACK'를 통한 명시적 처리 완료 확인(Acknowledgement) 메커니즘을 제공합니다. 또한 ID 기반의 범위 조회가 가능합니다.

## 4. 메시지 ID 구조와 생성 규칙

* **ID 포맷**: `타임스탬프-시퀀스번호` 형식입니다 (예: `1704123456789-0`).
  * **타임스탬프**: 밀리초(ms) 단위의 시간 값입니다.
  * **시퀀스 번호**: 동일한 밀리초 내에 여러 메시지가 들어올 경우 순서를 구분합니다.
* **증가성 보장**: 아이디 값은 반드시 이전보다 커야 합니다. 이전보다 작은 아이디는 추가할 수 없으며, 이는 시간순 정렬을 유지하기 위한 필수 제약입니다.

## 5. 핵심 명령어 및 데이터 읽기/쓰기

### 1) 메시지 추가 (XADD)

* **명령어**: `XADD stream * field value`
* **자동 ID**: `*` 기호를 넣으면 타임스탬프 기반의 고유 아이디가 자동으로 생성됩니다.
* **크기 제한 (MAXLEN)**: `MAXLEN 1000`과 같은 옵션을 주어 스트림의 크기를 제한할 수 있습니다. 설정된 개수를 넘으면 가장 오래된 메시지부터 삭제됩니다.

### 2) 메시지 읽기 (XREAD)

* **명령어**: `XREAD STREAMS stream 0` (0 입력 시 처음부터 전부 조회)
* **특정 ID 이후**: 특정 아이디를 지정하면 해당 아이디를 제외한 그 이후의 메시지만 받아옵니다.
* **블로킹 모드 (BLOCK)**: `XREAD BLOCK 5000 STREAMS stream $`
  * 새로운 메시지가 올 때까지 지정한 시간(ms) 동안 대기합니다. `0` 설정 시 무한 대기합니다.
  * **장점**: 클라이언트가 주기적으로 데이터를 묻는 '폴링(Polling)' 방식의 리소스 낭비를 방지합니다.

### 3) 범위 조회 (XRANGE)

* **명령어**: `XRANGE stream - +` (처음부터 끝까지 조회)
* **역순 조회**: `XREVRANGE`를 통해 최신 데이터부터 거꾸로 읽어올 수 있습니다.

## 6. 컨슈머 그룹 (Consumer Group)의 동작 원리

스트림의 가장 핵심적인 기능으로, 여러 워커가 메시지를 분산 처리할 수 있게 합니다.

* **메시지 분배**: 그룹 내의 소비자들에게 메시지를 나누어 전달합니다 (메시지 1은 A에게, 메시지 2는 B에게). 각 메시지는 그룹 내에서 단 한 명의 소비자에게만 전달됩니다.
* **독립적 처리**: 하나의 스트림에 여러 개의 그룹을 만들 수 있습니다. 알림 서비스 그룹, 분석 서비스 그룹 등이 각각 독립적으로 모든 메시지를 처리할 수 있습니다.
* **Last Delivered ID**: 그룹별로 마지막으로 전달한 메시지 아이디를 관리하여, 새로운 소비자가 합류해도 이미 다른 소비자에게 전달된 과거 메시지는 받지 않도록 제어합니다.

## 7. 메시지 생명주기와 PEL (Pending Entries List)

메시지가 확실히 처리되었는지 보장하기 위한 내부 관리 매커니즘입니다.

1. **메시지 요청**: 소비자가 `XREADGROUP`으로 메시지를 가져갑니다.
2. **PEL 기록**: Redis는 어떤 소비자가 어떤 메시지를 가져갔는지 'Pending Entries List(대기 목록)'에 기록합니다. 이때 메시지는 '처리 중' 상태가 됩니다.
3. **처리 완료 (XACK)**: 소비자가 비즈니스 로직을 마친 후 `XACK`를 보내면 Redis는 PEL에서 해당 메시지를 삭제합니다.
4. **장애 대응 (XCLAIM)**: 만약 소비자가 `XACK`를 보내지 못하고 죽으면 메시지는 PEL에 남습니다. 다른 소비자가 `XPENDING`으로 이를 확인하고 `XCLAIM`을 통해 소유권을 가져와 대신 재처리할 수 있습니다. 이를 통해 '최소 한 번(At-least-once)' 전달을 보장합니다.

## 8. 운영 주의사항 및 실무 가이드

* **메모리 관리**: 메시지가 계속 쌓이면 메모리가 증가하여 서버가 중단될 수 있습니다. 별도 배치 작업을 통해 데이터를 삭제하거나 아카이브 정책을 세워야 합니다.
* **XACK 누락 주의**: 처리가 완료된 후 반드시 `XACK`를 보내야 합니다. 누락 시 PEL이 무한히 커져 메모리 장애의 원인이 됩니다.
* **멱등성(Idempotency) 보장**: 네트워크 문제나 재처리 과정에서 동일한 메시지를 두 번 받을 수 있습니다. 주문 ID 등을 체크하여 중복 처리가 발생하지 않도록 로직을 설계해야 합니다.

## 9. Pub/Sub vs Stream 선택 기준

* **Pub/Sub**: 메시지를 저장하지 않음, 실시간 브로드캐스트, 유실 허용 가능, 가볍고 빠름 (예: 채팅방 메시지 전파).
* **Stream**: 메시지를 저장함, 컨슈머 그룹 분산 처리, 유실 허용 안 됨, 재처리 필요 (예: 주문/결제 이벤트 처리).
