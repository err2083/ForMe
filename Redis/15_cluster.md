# Redis 클러스터의 딥다이브 아키텍처와 운영 매커니즘

## 1. Redis 클러스터 개요 및 필요성

Redis 클러스터는 여러 개의 Redis 인스턴스를 하나의 논리적인 데이터베이스처럼 묶어서 사용하는 기능입니다. 단순한 사이드 프로젝트를 넘어 운영 환경(Production)에서 안정성을 보장하기 위해 필수적으로 사용됩니다.

* **수평 확장 (Scaling Out)**: 단일 인스턴스의 메모리와 처리 능력 한계를 극복하여 노드 추가만으로 용량과 처리량을 선형적으로 확장할 수 있습니다.
* **고가용성 (High Availability)**: 자동 장애 조치(Failover) 기능을 통해 특정 노드가 중단되어도 서비스를 지속할 수 있습니다.
* **데이터 분산**: 데이터를 여러 노드에 자동으로 분산 저장하여 메모리 제한 문제를 해결하고 읽기/쓰기 부하를 분산시킵니다.

## 2. 클러스터 아키텍처와 통신 구조

### 1) 노드 구성 (Minimum Production Setup)

* **최소 구성**: 클러스터는 최소 **3개의 마스터 노드**로 구성됩니다.
* **복제 구조**: 각 마스터는 하나 이상의 레플리카(Replica)를 가지며, 마스터 장애 시 레플리카가 마스터로 자동 승격됩니다.
* **표준 구성**: 일반적으로 3개의 마스터와 3개의 레플리카, 총 6개의 노드로 구성하는 것이 프로덕션 환경의 최소 권장 사항입니다.

### 2) 가십 프로토콜 (Gossip Protocol)

* **분산 아키텍처**: 중앙 코디네이터 없이 모든 노드가 서로의 상태를 알고 있는 완전 분산 구조입니다.
* **상태 공유**: 각 노드는 무작위로 선택한 다른 노드들에게 주기적으로 핑(Ping) 메시지를 보내 자신의 상태와 자신이 알고 있는 다른 노드의 정보(슬롯 매핑, 노드 상태 등)를 교환합니다.
* **장애 판정**: 여러 노드가 특정 노드의 응답 없음에 동의하면 장애 조치(Failover)가 시작됩니다.

### 3) 클러스터 버스 (Cluster Bus)

* **전용 포트**: 클라이언트 통신 포트(기본 6379)에 **10000을 더한 포트**(예: 16379)를 노드 간 통신용으로 사용합니다.
* **바이너리 프로토콜**: 사람은 읽기 어렵지만 서버 간 통신에 최적화된 빠르고 효율적인 바이너리 프로토콜을 사용하여 정보를 교환합니다. 방화벽 설정 시 이 포트를 반드시 개방해야 합니다.

## 3. 해시 슬롯(Hash Slot)과 샤딩 매커니즘

### 1) 해시 슬롯 개념

* Redis 클러스터는 데이터를 저장하는 공간을 **총 16,384개(0 ~ 16383)**의 해시 슬롯으로 나눕니다.
* 모든 키는 반드시 하나의 슬롯에 할당되며, 마스터 노드들이 이 슬롯들을 나누어서 담당합니다. (예: 마스터 3개 시 각각 약 5,461개씩 할당)

### 2) 키 할당 로직

* **해시 함수**: `CRC16` 알고리즘을 사용합니다.
* **계산식**: `CRC16(key) % 16384` 결과값이 해당 키가 속할 슬롯 번호가 됩니다. 동일한 키는 항상 동일한 슬롯에 할당됩니다.

### 3) 해시태그 (Hashtag) 기법

* **용도**: 여러 개의 키를 강제로 동일한 슬롯(노드)에 배치해야 할 때 사용합니다 (다중키 명령어, 트랜잭션 등).
* **방법**: 키 이름 중 중괄호 `{}` 안에 들어있는 문자열만 해싱에 사용합니다. 예: `{user100}:name`과 `{user100}:age`는 모두 `user100`만 해싱되어 같은 슬롯에 저장됩니다.
* **주의**: 특정 노드에만 데이터가 몰리는 불균형(Skews) 현상이 발생할 수 있으므로 주의해서 사용해야 합니다.

## 4. 데이터 리디렉션과 샤딩 동작

클라이언트는 클러스터 내 어떤 노드에든 요청을 보낼 수 있지만, 해당 노드가 키를 가지고 있지 않으면 리디렉션을 수행합니다.

* **MOVED 리디렉션**: 요청한 키가 다른 노드의 슬롯에 있을 때 발생합니다. 응답에는 올바른 노드의 주소가 포함됩니다.
* **클라이언트 캐싱**: 클라이언트 라이브러리는 `MOVED` 응답을 받으면 해당 슬롯의 위치 정보를 로컬에 캐시하여 다음 요청부터는 올바른 노드로 직접 요청합니다.
* **Cross Slot 에러**: `MGET`, `MSET` 같은 다중키 명령 사용 시 키들이 서로 다른 슬롯에 있으면 에러가 발생합니다. 이 경우 해시태그를 사용해야 합니다.

## 5. 클러스터 재구성 (Reconfiguration)

서비스 중단 없이 노드를 추가/제거하거나 데이터를 재분배하는 과정입니다.

### 1) 노드 추가 프로세스

1. **조인 (Join)**: 새 노드가 기존 노드와 핸드쉐이크(`MEET` 명령)를 통해 클러스터 멤버로 참여합니다.
2. **슬롯 할당 계획**: 16,384개 슬롯을 노드 수로 나누어 각 노드가 가질 목표 슬롯 수를 계산합니다.
3. **마이그레이션 (Migration)**: 기존 노드의 슬롯과 키를 새 노드로 이동시킵니다.

### 2) 슬롯 마이그레이션 5단계

1. **상태 표시**: 소스 노드는 `MIGRATING`, 타겟 노드는 `IMPORTING` 상태로 설정합니다.
2. **키 복사**: `MIGRATE` 명령을 사용하여 키를 하나씩 점진적/원자적으로 복사하고 소스에서 삭제합니다.
3. **중간 요청 처리**: 마이그레이션 중 요청이 오면, 키가 소스에 있으면 직접 처리하고 이미 이동했다면 **ASK 리디렉션**을 보냅니다.
4. **소유권 변경**: 모든 키 이동 완료 후 `SETSLOT` 명령으로 공식 소유권을 타겟 노드로 변경합니다.
5. **전파**: 가십 프로토콜을 통해 변경된 슬롯 맵핑 정보를 전체 클러스터에 알립니다.

## 6. 리디렉션 상세 비교: MOVED vs ASK

* **MOVED**: 슬롯이 완전히 이동했음을 의미합니다. 클라이언트는 이 정보를 **캐시**하여 향후 모든 요청을 새 노드로 보냅니다. (정상 상황)
* **ASK**: 슬롯이 현재 **이동 중**임을 의미하는 임시 리디렉션입니다. 클라이언트는 정보를 캐시하지 않으며, 타겟 노드에 `ASKING` 명령을 먼저 보낸 후 실제 명령을 수행합니다.

## 7. 리밸런싱 (Rebalancing) 전략

노드 간 부하 균형이 깨졌을 때 슬롯을 재분배하는 작업입니다.

* **원인**: 무분별한 해시태그 사용, 데이터 크기의 불균등, 노드 추가/제거 후 수동 조정 필요성 등.
* **최적화**: 목표 슬롯 개수를 설정한 뒤 현재 개수와의 차이만큼만 슬롯을 이동시킵니다. 슬롯 이동은 네트워크와 CPU 자원을 소모하므로 **최소 이동량**을 계산하여 진행하는 것이 핵심입니다.
