# Redis의 기초와 인프라 아키텍처 상세

## 1. Redis 개요 (Remote Dictionary Server)

Redis는 오픈 소스 기반의 **In-Memory 데이터 구조 저장소**로, 백엔드 시스템에서 핵심적인 역할을 수행합니다 [1, 2].

* **정의**: 'Remote Dictionary Server'의 약자로, 외부에서도 접근 가능한 Key-Value 구조의 서버입니다 [2].
* **기본 특징**:
  * 대표적인 **Key-Value 기반 NoSQL** 데이터베이스로 분류됩니다 [1, 3].
  * 모든 데이터를 RAM에 저장하여 밀리초(ms) 단위의 매우 빠른 응답 속도를 보장합니다 [1, 4].
  * **단일 스레드(Single Thread)** 기반으로 동작하여 원자적(Atomic) 연산을 보장하고 동시성 문제를 해결합니다 [5].
  * **다양한 자료구조 지원**: String, List, Set, Hash, Sorted Set 등을 기본적으로 제공합니다 [1, 4].

## 2. 인메모리(In-Memory) 저장소의 기술적 특성

데이터를 디스크가 아닌 메모리에 적재하여 처리함으로써 하드웨어 레벨에서 압도적인 성능 차이를 만듭니다 [5, 6].

### 하드웨어별 접근 속도(Latency) 비교 [5]

| 하드웨어 항목 | 접근 속도 (Latency) | 특징 |
| :--- | :--- | :--- |
| **HDD (하드디스크)** | 5 ~ 10ms | 물리적인 헤드 이동(Seek Time) 발생 |
| **SSD** | 0.1 ~ 0.2ms | 디스크 기반 저장소 |
| **RAM (메모리)** | **나노초(ns) 단위** | 디스크 I/O 및 OS 파일 시스템을 거치지 않음 |

### 장점과 단점 [6-8]

* **장점**: 0.1~1ms 수준의 극도로 빠른 속도, 낮은 지연 시간(Low Latency), 단일 인스턴스 기준 초당 10만 건 이상의 높은 처리량을 제공합니다. 트래픽 폭주 시 DB 커넥션 고갈을 막는 **완충 역할**을 합니다.
* **단점**: 메모리 기반이므로 전원 종료 시 데이터가 사라지는 **휘발성**이 있습니다 (영속성 옵션으로 보완 가능). 또한 RAM 용량 제한이 있으며 비용이 비싸므로 'Hot Data' 위주로 저장해야 합니다.

## 3. 타 데이터베이스와의 비교 [9-13]

| 비교 항목 | Redis | 관계형 DB (MySQL 등) | MongoDB | Memcached |
| :--- | :--- | :--- | :--- | :--- |
| **저장 위치** | 메모리 | 디스크 | 디스크 + 메모리 캐시 | 메모리 |
| **데이터 모델** | Key-Value (다양한 구조) | 테이블 (스키마 기반) | Document (BSON) | String 전용 |
| **속도** | 매우 빠름 (0.1~1ms) | 상대적 느림 (10~100ms) | 빠름 | 매우 빠름 |
| **쿼리 능력** | 단순 Key 조회 | 복잡한 SQL/Join | 풍부한 쿼리/집계(Aggregation) | 단순 조회 |
| **영속성** | 지원 (RDB/AOF) | 지원 (강력함) | 지원 | 미지원 |
| **스레드** | **단일 스레드** | 멀티 스레드 | 멀티 스레드 | **멀티 스레드** |
| **주 목적** | 캐시/메시징/공유 저장소 | 메인 데이터 저장 | 대용량/유연한 데이터 저장 | 단순 캐싱 전용 |

## 4. Redis의 주요 활용 사례 [4, 14]

1. **성능 병목 해소**: 복잡한 쿼리 결과를 저장하는 Cache 레이어로 활용하여 응답 시간을 단축하고 DB 부하를 줄입니다.
2. **확장성 향상 (세션 저장소)**: 여러 서버가 동일한 사용자 세션 정보를 공유할 수 있게 합니다.
3. **실시간 처리**: 초당 수천 건의 업데이트가 필요한 실시간 순위표(Leaderboard), 카운터, 알림 시스템에 적합합니다.
4. **분산 시스템 제어**: 메시지 큐 활용 및 분산 환경에서의 동시성 제어를 위한 **분산 락(Distributed Lock)** 기능을 제공합니다.

## 5. 인프라 아키텍처 및 구성 환경 [15, 16]

### 1) 단일 인스턴스 (Single Instance) [16, 17]

* 하나의 Redis 서버가 모든 읽기/쓰기를 처리하는 가장 기본적인 형태입니다.
* 설정이 간단하나, 장애 발생 시 서비스가 중단되는 **단일 장애점(SPOF)** 문제가 있습니다.

### 2) 마스터-레플리카 (Master-Replica Replication) [18-20]

* **Master 노드**: 쓰기(Write) 명령을 처리하며, 변경 이력을 레플리카로 전송합니다.
* **Replica 노드**: 읽기 전용(Read-only)으로 동작하며 백업 역할을 수행합니다.
* **동기화 과정**: 레플리카가 처음 마스터에 연결하면 마스터는 **RDB 스냅샷**을 생성하여 전송하고, 이후의 쓰기 명령은 실시간으로 전달합니다.

### 3) 센티널 (Sentinel) - 고가용성(HA) [21-24]

장애 발생 시 자동으로 대처하기 위해 독립적인 센티널 프로세스를 실행합니다.

* **장애 판정**:
  * **S-Down (주관적 다운)**: 개별 센티널이 특정 시간 동안 응답을 받지 못할 때 표시합니다.
  * **O-Down (객관적 다운)**: 설정된 정족수(**Quorum**) 이상의 센티널이 합의하면 실제 장애로 판단하고 페일오버를 시작합니다.
* **자동 페일오버**: 마스터 장애 시 레플리카 중 하나를 새로운 마스터로 승격시키고 클라이언트에 정보를 알립니다. 신뢰성을 위해 최소 3개 이상의 구성을 권장합니다.

### 4) 클러스터 (Cluster) - 샤딩(Sharding) [25-28]

데이터를 여러 노드에 분산 저장하여 메모리 한계와 성능을 확장합니다.

* **해시 슬롯 (Hash Slot)**: 총 **16,384개**의 슬롯을 마스터 노드들이 나누어 관리합니다. 키를 해싱하여 해당 슬롯을 담당하는 노드에 저장합니다.
* **Gossip 프로토콜**: 노드들이 서로 주기적으로 통신하며 마스터 상태, 슬롯 정보 등을 공유합니다.
* **제약 사항 (Cross Slot)**: 여러 키를 한 번에 처리(MGET 등)할 때 키들이 서로 다른 슬롯에 있으면 에러가 발생합니다. 이를 해결하기 위해 중괄호 `{}`를 사용하는 **Hashtag** 기법으로 같은 슬롯에 데이터를 모을 수 있습니다.
