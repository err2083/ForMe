# Redis String 타입의 이해와 활용

## 1. Redis String 타입 개요

Redis에서 가장 기본적인 데이터 타입이며, 실제 서비스 운영 시 사용 비중의 80~90%를 차지할 만큼 중요합니다 [1-3].

* **기본 정의**: 단순한 문자열을 넘어 현실적으로는 **바이트 배열(Byte Array)**로 구현되어 있습니다 [2].
* **키(Key)의 특성**: Redis의 모든 키는 기본적으로 String 타입입니다 [3].
* **Binary Safe (바이너리 세이프)**:
  * C언어 문자열과 달리 **Null Byte(\0)**를 문자열의 끝으로 인식하지 않고 데이터 중간에 포함할 수 있습니다 [3, 4].
  * 이로 인해 텍스트뿐만 아니라 이미지(JPG), PDF, 압축 파일, 직렬화된 객체(JSON, Protocol Buffers) 등 모든 이진 데이터를 저장할 수 있습니다 [1, 2, 4].
* **크기 제한**:
  * 단일 키당 최대 **512MB**까지 저장 가능합니다 [1, 4].
  * 이는 문자 기준 약 5억 3천만 자, 150바이트 JSON 객체 기준 약 350만 개를 하나의 키에 저장할 수 있는 크기입니다 [4, 5].
  * **권장 사항**: 네트워크 대역폭 소모와 처리 지연을 방지하기 위해 수 바이트에서 킬로바이트(KB) 단위의 작은 값으로 쪼개서 관리하는 것이 효율적입니다 [1, 5, 6].

## 2. 내부 구조: SDS (Simple Dynamic String)

Redis는 성능 최적화를 위해 C언어의 기본 문자열 대신 SDS라는 자체 구조를 사용합니다 [6, 7].

* **SDS 구성 요소**:
  * `len`: 문자열의 실제 길이 (조회 시 O(1) 시간 복잡도) [7, 8].
  * `free`: 할당된 메모리 중 사용 가능한 여유 공간 [7, 8].
  * `buf`: 실제 문자 데이터가 저장되는 공간 [7, 8].
* **SDS의 장점**:
  * **길이 조회 최적화**: 끝까지 순회할 필요 없이 바로 길이를 확인합니다 [7, 8].
  * **버퍼 오버플로우 방지**: `len`과 `free`를 통해 메모리 경계를 안전하게 관리합니다 [7, 8].
  * **효율적인 Append**: 여유 공간이 있다면 메모리 재할당 없이 즉시 데이터를 추가합니다 [7, 8].

## 3. 3가지 인코딩 방식 및 자동 전환

데이터의 형태와 크기에 따라 메모리 효율을 극대화하기 위해 인코딩 방식을 자동으로 선택합니다 [6, 7, 9].

| 방식 | 설명 | 특징 |
| :--- | :--- | :--- |
| **int** | 정수값 저장 (-2^63 ~ 2^63-1 범위) | 메모리 절약 (~24바이트), 산술 연산 최적화 [7, 9] |
| **embstr** | 44바이트 이하의 짧은 문자열 | 객체 헤더와 SDS를 연속된 메모리에 배치, 캐시 히트율 높음, 읽기 전용 [7, 9, 10] |
| **raw** | 44바이트를 초과하는 일반 문자열 | 헤더와 데이터가 별도 메모리 영역에 관리됨, 수정 및 확장 유연 [7, 10, 11] |

* **인코딩 전환**: 숫자에 문자를 추가(`APPEND`)하면 `int`에서 `raw`로, 읽기 전용인 `embstr`을 수정하려고 시도하면 `raw`로 자동 전환됩니다 [7, 10].

## 4. 핵심 명령어 및 활용

### 1) 기본 저장 및 조회

* **SET**: 키와 값을 저장합니다. 이미 키가 존재하면 기존 값을 덮어쓰며 항상 성공합니다 [12-14].
* **GET**: 키에 저장된 값을 조회합니다. 존재하지 않으면 `nil`을 반환하며, String 타입이 아닌 경우 에러가 발생합니다 [12, 15, 16].

### 2) 카운터 연산 (원자적 연산)

모든 연산은 **원자적(Atomic)**으로 실행되어 **Race Condition(경쟁 상태)**을 방지합니다 [16-18].

* **INCR / DECR**: 값을 1씩 증가 또는 감소시킵니다 [16, 17, 19].
* **INCRBY / DECRBY**: 지정한 N만큼 값을 증가 또는 감소시킵니다 [17, 20].
* **특징**: 키가 없으면 자동으로 0으로 초기화한 후 연산을 수행합니다 [19].

### 3) 다중 작업 (성능 최적화)

네트워크 **RTT(Round Trip Time)**를 줄여 성능을 비약적으로 향상시킵니다 [20-22].

* **MSET**: 여러 키-값 쌍을 한 번에 저장합니다. 원자적으로 실행되어 모든 값이 저장되거나 아예 안 됩니다 [20-22].
* **MGET**: 여러 키의 값을 한 번에 조회합니다 [21, 23].
* **주의**: Redis는 단일 스레드이므로 한 번에 너무 많은 양(수만 개 이상)을 처리하면 다른 요청이 대기하게 되므로 10~100개 단위로 나누어 사용하는 것이 좋습니다 [23].

### 4) 조건부 저장 및 만료 옵션

* **SETEX (Set with Expire)**: 저장과 동시에 만료 시간(TTL)을 설정합니다 (캐싱에 활용) [24, 25].
* **SETNX (Set if Not eXists)**: 키가 존재하지 않을 때만 저장합니다. 중복 방지 및 **분산 락(Distributed Lock)** 구현에 필수적입니다 [24, 26].
* **기타 옵션**: `EX`(초 단위), `PX`(밀리초 단위), `NX`(없을 때만), `XX`(있을 때만) 옵션을 `SET` 명령어 하나로 통합하여 사용할 수 있습니다 [24, 25, 27].

## 5. 키 네이밍 규칙 (Best Practice)

* **계층 구조**: 콜론(`:`)을 구분자로 사용하여 데이터의 계층을 나타냅니다 (예: `user:1000:name`) [14].
* **금지 사항**: 키 이름에 공백을 사용하지 않으며, 언더바(`_`)나 대시(`-`)를 권장합니다 [14, 15].
* **효율성**: 너무 긴 키 이름은 메모리 낭비를 초래하므로 적절한 길이를 유지해야 합니다 [15].
