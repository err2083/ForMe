# Redis 파이프라이닝(Pipelining)의 원리와 성능 최적화

## 1. 파이프라이닝(Pipelining)의 정의

파이프라이닝은 Redis에만 국한된 개념이 아니라 다양한 서비스에서 성능 향상을 위해 사용되는 일반적인 기술입니다.

* **핵심 개념**: 여러 개의 Redis 명령어를 하나의 요청에 담아 한 번에 전송하고, 그 응답들을 일괄적으로 수신하는 방식입니다.
* **연속성**: 응답을 기다리지 않고 명령어를 연속적으로 보내는 것이 가장 큰 특징입니다.
* **일반 방식과의 비교**:
  * **일반 실행**: 명령 1 전송 -> 응답 1 대기 -> 명령 2 전송 -> 응답 2 대기하는 순차적 방식입니다.
  * **파이프라이닝**: 명령 1, 2, N을 일괄 전송한 후, 응답 1, 2, N을 일괄 수신합니다.

## 2. RTT(Round Trip Time)와 네트워크 최적화

파이프라이닝을 사용하는 근본적인 이유는 명령어 실행 시간보다 네트워크 통신 비용이 훨씬 크기 때문입니다.

* **RTT의 정의**: 패킷이 목적지(서버)까지 갔다가 다시 출발지(클라이언트)로 돌아오는 데 걸리는 왕복 시간입니다.
* **네트워크 지연 요소**:
  * 물리적 거리에 따른 전파 지연.
  * 라우터와 스위치를 거치는 과정에서의 전송 지연.
  * 장비 내에서의 처리 지연.
* **성능 병목 현상**: Redis의 인메모리 명령 실행 시간은 마이크로초(μs) 단위로 매우 짧지만, RTT는 밀리초(ms) 단위인 경우가 많습니다.
  * 예: 명령 실행 시간이 0.1ms이고 RTT가 10ms라면, 전체 시간의 99% 이상을 네트워크 대기에 소비하게 됩니다.
* **파이프라이닝의 가치**: 100개의 명령을 실행할 때 일반 방식은 RTT가 100번 발생하여 1초 이상 소요될 수 있지만, 파이프라이닝은 단 1번의 RTT로 모든 작업을 완료하여 처리 시간을 극적으로 단축합니다.

## 3. 파이프라이닝의 동작 원리

### 1) 클라이언트 측 동작

* **버퍼 모드 전환**: 파이프라이닝을 시작하면 클라이언트는 명령을 즉시 네트워크로 전송하지 않고 내부 버퍼에 쌓습니다.
* **일괄 전송 (Flush)**: 버퍼에 명령이 충분히 쌓이거나 명시적으로 플러시를 호출할 때 모든 명령을 TCP 소켓을 통해 한 번에 전송합니다.
* **결과 파싱**: 응답이 일괄 도착하면 버퍼에서 하나씩 읽어 파싱한 뒤 각 명령의 결과를 순서대로 반환합니다.

### 2) Redis 서버 측 동작

* **표준 처리**: Redis 서버는 파이프라이닝을 위한 특별한 로직을 갖지 않으며 일반 명령과 동일하게 처리합니다.
* **이벤트 루프**: 소켓에 연속으로 도착한 명령어들을 순차적으로 실행합니다.
* **출력 버퍼링**: 실행 결과를 서버 내부의 클라이언트 출력 버퍼에 쌓아두었다가 한 번에 배치 전송합니다.

### 3) TCP 수준의 효율성

* TCP 소켓의 송수신 버퍼를 활용하여 작은 데이터들을 모아서 전송함으로써 네트워크상의 패킷 수를 줄이고 오버헤드를 감소시킵니다.

## 4. 파이프라이닝 vs 트랜잭션 상세 비교

| 비교 항목 | 파이프라이닝 (Pipelining) | 트랜잭션 (Transaction) |
| :--- | :--- | :--- |
| **주요 목적** | **성능 최적화** (RTT 감소) | **정합성 보장** (원자성 및 격리성) |
| **실행 시점** | 서버 도착 즉시 순차 실행 | `MULTI`로 큐잉 후 `EXEC` 호출 시 일괄 실행 |
| **격리성** | **보장 안 됨** (타 명령 끼어들기 가능) | **보장됨** (실행 중 타 명령 개입 불가) |
| **응답 방식** | 즉각적 응답 생성 후 버퍼링 전송 | `QUEUED` 응답 후 `EXEC` 시 일괄 반환 |
| **명령 성격** | 주로 서로 독립적인 명령어들 | 서로 연관된 명령어들의 묶음 |

## 5. 실무적 관점 및 학습 철학

* **운영 환경의 한계**: 파이프라이닝은 동시성 제어가 불가능하고 실행 중간 상태가 외부에 노출될 수 있다는 단점이 있습니다. 따라서 데이터의 엄격한 정합성보다는 처리 속도가 최우선인 경우에 선택적으로 도입해야 합니다.
* **학습 방향 (80/20 법칙)**: Redis 지식의 80~90%를 빠르게 습득하여 전체적인 시야를 넓히는 것이 중요합니다. 나머지 사소한 1~3%의 차이에 과도한 시간을 쏟기보다, 직접 구현해보고 문제 상황에 따라 스스로 판단할 수 있는 근거를 만드는 것이 진정한 개발 실력 향상의 길입니다.
