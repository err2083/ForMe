# [심화 분석 학습 노트] 스프링 배치: REST API와 JobOperator를 통한 원격 제어 아키텍처

## 1장. JobLauncher 원격 제어: 네트워크를 통한 죽음의 선고

배치 작업이 서버 내부에서만 자발적으로 일어나는 단계는 지났습니다. 진정한 시스템 지배자는 원격지에서 REST API를 통해 즉시 배치 작업을 트리거할 수 있어야 합니다.

### 1. 작전 환경 구성: 자동 실행 병기 무력화

REST API로 Job을 실행하려면, 먼저 애플리케이션 시작 시 `JobLauncherApplicationRunner`가 자동으로 모든 Job을 처형하는 것을 막아야 합니다.

```yaml
spring:
  batch:
    job:
      enabled: false # JobLauncherApplicationRunner 빈 등록 방지
```

* **시스템 인텔리전스:** 이 프로퍼티를 `false`로 설정하면 `BatchAutoConfiguration` 내의 `@ConditionalOnProperty` 조건이 충족되지 않아 자동 실행기가 빈으로 등록되지 않습니다. 이제 우리가 원하는 시점에만 API 호출로 Job을 구동할 수 있습니다.

### 2. JobLauncher 기반 REST 컨트롤러 구현

원격에서 전달된 'Job 이름'을 실제 처형 대상 객체로 변환하여 실행하는 핵심 로직입니다.

```java
@RestController
@RequestMapping("/api/jobs")
public class JobLauncherController {
    @Autowired
    private JobLauncher jobLauncher;
    
    @Autowired
    private JobRegistry jobRegistry;
    
    @Autowired
    private JobExplorer jobExplorer;

    @PostMapping("/{jobName}/start")
    public ResponseEntity<String> launchJob(@PathVariable String jobName) throws Exception {
        // 1. JobRegistry에서 이름으로 Job 객체 조회
        Job job = jobRegistry.getJob(jobName);
        
        // 2. JobExplorer를 활용해 이전 이력을 기반으로 새 파라미터 생성
        JobParameters jobParameters = new JobParametersBuilder(jobExplorer)
                .getNextJobParameters(job)
                .toJobParameters();
        
        // 3. JobLauncher의 run() 메서드를 호출해 실행 집행
        JobExecution jobExecution = jobLauncher.run(job, jobParameters);
        
        return ResponseEntity.ok("Job launched with ID: " + jobExecution.getId());
    }
}
```

* **JobRegistry:** 모든 Job 빈을 이름으로 관리하는 중앙 저장소입니다.
* **JobExplorer:** 메타데이터 저장소를 탐색하여 마지막 실행 이력을 가져오는 탐색기입니다.
* **인과관계:** 커맨드라인에서 수행되던 파라미터 변환 및 실행 로직이 웹 요청 스레드 내에서 재현됩니다.

### 3. 비동기 처형 엔진: @BatchTaskExecutor의 투입

기본적으로 `JobLauncher`는 `SyncTaskExecutor`를 사용하여 동기 방식으로 동작합니다. 하지만 웹 환경에서 배치 완료까지 스레드를 블로킹하는 것은 자살 행위입니다.

```java
@Configuration
public class BatchCustomConfiguration {
    @Bean
    @BatchTaskExecutor
    public TaskExecutor taskExecutor() {
        // 비동기 실행 엔진 주입. 요청 스레드는 즉시 응답하고 배치는 별도 스레드에서 수행됨.
        SimpleAsyncTaskExecutor executor = new SimpleAsyncTaskExecutor();
        return executor;
    }
}
```

* **@BatchTaskExecutor:** 여러 개의 `TaskExecutor` 빈이 존재할 때, 스프링 부트가 "이것이 배치 전용 엔진이다"라고 인식하게 만드는 한정자(Qualifier)입니다.

## 2장. JobOperator 장악: 시스템 제어의 완전한 사령부

단순히 실행하는 것을 넘어, 실행 중인 작업을 '중지'시키고 '재시작'하는 정교한 운영 능력이 필요합니다. `JobOperator`는 이를 위한 고수준 추상화 인터페이스입니다.

### 1. JobOperator의 무기 창고

`JobOperator`는 `JobLauncher`, `JobExplorer`, `JobRepository`, `JobRegistry`를 모두 내부에 품고 운영 기능을 통합 제공합니다.

* **start(jobName, parameters):** 새로운 인스턴스 시작.
* **stop(executionId):** 실행 중인 작업을 `STOPPING` 상태로 전파.
* **restart(executionId):** 중단되거나 실패한 지점부터 부활.
* **abandon(executionId):** 더 이상 실행 불가능한 작업을 버림.

### 2. JobOperator 기반 통합 컨트롤러 구현

```java
@RestController
@RequestMapping("/api/jobs")
public class JobOperatorController {
    @Autowired
    private JobOperator jobOperator;

    @PostMapping("/{jobName}/start")
    public ResponseEntity<String> launchJob(@PathVariable String jobName) throws Exception {
        Properties jobParameters = new Properties();
        // JobOperator는 Incrementer 처리를 자동으로 안 하므로 수동으로 타임스탬프 추가
        jobParameters.setProperty("run.timestamp", String.valueOf(System.currentTimeMillis()));
        
        Long executionId = jobOperator.start(jobName, jobParameters);
        return ResponseEntity.ok("Job launched with ID: " + executionId);
    }

    @PostMapping("/stop/{executionId}")
    public ResponseEntity<String> stopJob(@PathVariable Long executionId) throws Exception {
        // 즉시 중단이 아닌 '중단 요청'임에 주의
        boolean result = jobOperator.stop(executionId);
        return ResponseEntity.ok("Stop request for job execution " + executionId + " successful: " + result);
    }

    @PostMapping("/restart/{executionId}")
    public ResponseEntity<String> restartJob(@PathVariable Long executionId) throws Exception {
        // 이전 실행의 ExecutionContext를 복원하여 재시작
        Long newExecutionId = jobOperator.restart(executionId);
        return ResponseEntity.ok("Job restarted with new execution ID: " + newExecutionId);
    }
}
```

## 3장. 실전 시뮬레이션: 중지 메커니즘과 상태 복원의 심연

우리는 '보안 프로토콜(SecurityProtocolJob)' 시나리오를 통해 시스템이 어떻게 중단 명령을 감지하고, 재시작 시 어떻게 중단된 지점을 찾아내는지 그 밑바닥을 파헤칩니다.

### 1. '가능한 빨리, 그러나 안전하게': 중지(Stop)의 내부 흐름

`JobOperator.stop()`을 호출해도 배치가 즉시 멈추지 않는 이유는 **데이터 정합성** 때문입니다.

1. **STOPPING 상태 기록:** `JobOperator.stop()`은 `BATCH_JOB_EXECUTION` 테이블의 상태를 `STOPPING`으로 바꿀 뿐입니다.
2. **중단 감지 (Step):** `TaskletStep`이 청크 처리를 마치고 `JobRepository.update()`를 호출할 때, 내부적으로 `checkForInterruption()`이 가동됩니다.
3. **상태 동기화:** `JobExecutionDao.synchronizeStatus()`를 통해 DB에서 `STOPPING` 상태를 읽어옵니다.
4. **폭발(Exception):** `stepExecution.setTerminateOnly()`가 호출되고, 이후 `ThreadStepInterruptionPolicy.checkInterrupted()`에서 `JobInterruptedException`이 터져 나오며 작업이 안전하게 중단됩니다.

### 2. 죽음에서 부활하는 기술: 재시작(Restart) 메커니즘

중단된 보안 프로토콜을 재시작하면, 시스템은 어떻게 4번째 아이템부터 읽어야 함을 알까요?

* **상태 저장 (update):** `AbstractItemCountingItemStreamItemReader`는 청크 커밋 시마다 `currentItemCount`를 `ExecutionContext`에 각인합니다.

```java
@Override
public void update(ExecutionContext executionContext) throws ItemStreamException {
    // 현재까지 읽은 위치를 DB에 기록
    executionContext.putInt(getExecutionContextKey(READ_COUNT), currentItemCount);
}
```

* **상태 복원 (open):** 재시작 시 `open()` 메서드가 가동되어 DB에서 이 값을 읽어와 점프를 집행합니다.

```java
@Override
public void open(ExecutionContext executionContext) throws ItemStreamException {
    if (executionContext.containsKey(getExecutionContextKey(READ_COUNT))) {
        itemCount = executionContext.getInt(getExecutionContextKey(READ_COUNT));
    }
    jumpToItem(itemCount); // 3개 읽었다면 3번 건너뛰고 4번부터 시작
    currentItemCount = itemCount;
}
```

## [부록] 비동기 죽음의 선고: 메시지 기반 Job 실행 (RabbitMQ)

REST API조차 사치인 대규모 분산 환경에서는 메시지 큐를 통한 이벤트 기반 실행이 정답입니다.

```java
@Bean
public IntegrationFlow jobLaunchFlow(ConnectionFactory connectionFactory) {
    return IntegrationFlow
        .from(Amqp.inboundAdapter(connectionFactory, "job-requests-queue"))
        .channel(rabbitJobRequestChannel())
        .transform(this::createJobLaunchRequest) // JSON을 JobLaunchRequest로 변환
        .handle(jobLaunchingGateway()) // 내부적으로 JobLauncher.run() 호출
        .get();
}
```

* **핵심 철학:** 진입점(REST, CLI, MQ)은 달라도, 결국 종착역은 `JobLauncher`와 `JobParameters`라는 단 하나의 코어 엔진으로 수렴됩니다.

## [개념 검증용 자가 진단 질문]

### 원격 제어 및 중지/재시작 메커니즘 심화 진단

1. `JobLauncher`를 REST API 환경에서 사용할 때 별도의 `TaskExecutor` 설정을 하지 않으면 발생하는 **스레드 가용성 측면의 치명적인 결함**은 무엇이며, 이를 해결하기 위해 `@BatchTaskExecutor` 한정자가 내부적으로 어떻게 `JobLauncher`의 동작 방식을 변경시키는지 논술하시오.
2. `JobOperator.stop()` 명령이 하달되었을 때, 배치 시스템이 왜 즉시 스레드를 중단(Interrupt)시키지 않고 **'청크 경계(Chunk Boundary)'**까지 기다렸다가 작업을 멈추는지 데이터 일관성(Data Consistency) 및 롤백 불가능한 리소스(Kafka 등)의 관점에서 설명하시오.
3. `AbstractItemCountingItemStreamItemReader`를 상속받은 리더가 포함된 Job을 재시작할 때, `ExecutionContext`의 `READ_COUNT` 값과 `jumpToItem()` 메서드가 상호작용하여 **중복 처리를 방지하는 구체적인 내부 메커니즘**을 인과관계에 따라 기술하시오.

<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `JobLauncher`는 기본적으로 `SyncTaskExecutor`를 사용하여 호출한 스레드에서 배치를 직접 실행합니다. REST API 환경에서 이를 그대로 사용하면 배치 작업이 끝날 때까지 HTTP 요청 스레드가 점유되어 응답을 보내지 못하고 타임아웃이 발생하거나 서버 가용성이 급격히 저하됩니다. `@BatchTaskExecutor`를 통해 비동기 기반의 `SimpleAsyncTaskExecutor`를 주입하면, `JobLauncher`는 배치를 별도 스레드에서 수행하고 호출 스레드(Web thread)에게 즉시 `JobExecution` ID를 반환함으로써 클라이언트가 작업 진행 상태를 비동기적으로 추적할 수 있게 합니다.

* **2번 질문 해설:** 스프링 배치는 '데이터 무결성'을 최우선으로 합니다. 만약 아이템 처리 중간에 강제로 멈추면, 일부 데이터는 처리되고 일부는 처리되지 않은 불완전한 상태가 됩니다. 특히 Kafka처럼 롤백이 어려운 외부 시스템에 데이터를 쓰는 경우 중간 중단은 치명적입니다. 따라서 스프링 배치는 `JobOperator.stop()`이 호출되면 메타데이터에 중단 의사만 기록(STOPPING)하고, 현재 진행 중인 청크 트랜잭션이 완료되어 커밋되는 시점(청크 경계)에 상태를 대조하여 안전하게 `STOPPED` 상태로 전이시키는 '지연 중단' 방식을 채택합니다.

* **3번 질문 해설:** 청크 단위 작업이 성공할 때마다 `ItemStream.update()`가 호출되어 현재까지 읽은 아이템 번호인 `READ_COUNT`를 `ExecutionContext` 테이블에 영구 저장합니다. 작업 실패 후 재시작 시, `ItemStream.open()` 메서드가 이 저장된 값을 조회합니다. 만약 값이 3이라면 `jumpToItem(3)`을 호출하여 앞선 3개의 데이터를 `read()` 호출 없이 건너뛰고, 실제 비즈니스 로직은 4번째 아이템부터 수행되게 함으로써 이전 성공 지점과 현재 실행 지점 사이의 데이터 중복이나 누락을 원천 차단합니다.
