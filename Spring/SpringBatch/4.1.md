# [심화 분석 학습 노트] 스프링 배치: ItemStream의 정체와 자원 지배 아키텍처

## 1장. 작전의 실체: 드디어 정체를 드러낸 ItemStream

우리는 그동안 `MultiResourceItemReader`, `CompositeItemReader` 등을 다루며 내부적으로 위임 대상들이 `ItemStream`이라는 패턴으로 묶여 있음을 보았습니다. 이것은 단순한 우연이 아닙니다. **ItemStream은 스프링 배치 자원 관리와 상태 추적의 최상위 계급**입니다.

### 1. ItemStream의 정의와 계급 구조

`ItemStream`은 스프링 배치의 핵심 인터페이스로, 대부분의 `ItemReader`와 파일 기반 `ItemWriter`들이 이를 공통적으로 구현합니다.

```java
public interface ItemStream {
    // 자원을 열고 상태를 복원한다.
    default void open(ExecutionContext executionContext) throws ItemStreamException {}
    
    // 현재 실행 상태를 저장한다.
    default void update(ExecutionContext executionContext) throws ItemStreamException {}
    
    // 사용한 자원을 안전하게 닫는다.
    default void close() throws ItemStreamException {}
}
```

### 2. ItemStream의 두 가지 핵심 임무

* **자원 초기화 및 해제:** 파일 핸들러, DB 커넥션 등 시스템 자원의 생명주기 관리.
* **메타데이터 관리 및 상태 추적:** 실행 상태를 기록하여 장애 시 실패 지점부터 재시작할 수 있는 근거 제공.

## 2장. 자원 관리의 철학: 초기화와 해제의 매커니즘

"시스템 자원들이여, 너희의 운명은 우리가 결정한다." 자원 관리는 시스템의 안정성을 결정짓는 바이러스와 같습니다. 누수는 처음엔 미미하나 결국 전체 시스템을 파괴합니다.

### 1. 적절한 초기화 (open)

데이터를 읽기 전 파일을 열거나 DB 커넥션을 맺는 작업입니다. `FlatFileItemReader`의 내부를 보면 이를 명확히 알 수 있습니다.

```java
// FlatFileItemReader.doOpen() 내부 로직
if (!resource.exists()) {
    if (strict) {
        throw new IllegalStateException("Input resource must exist (reader is in 'strict' mode): " + resource);
    }
    logger.warn("Input resource does not exist " + resource.getDescription());
    return;
}
// 데이터를 읽어들일 BufferedReader 준비
reader = bufferedReaderFactory.create(resource, encoding);
```

* **시스템 인텔리전스:** `doOpen()`은 부모 클래스인 `AbstractItemCountingItemStreamItemReader`의 `open()` 메서드에 의해 호출됩니다.

### 2. 적절한 해제 (close)

작업 완료 후 자원을 반납하지 않으면 파일 핸들 누수나 DB 커넥션 풀 고갈이 발생합니다.

```java
// FlatFileItemReader.doClose()
@Override
protected void doClose() throws Exception {
    reader.close(); // BufferedReader를 닫아 시스템 자원 반납
}

// JdbcCursorItemReader의 자원 해제 (AbstractCursorItemReader.doClose)
protected void doClose() throws Exception {
    JdbcUtils.closeResultSet(this.rs); // ResultSet 닫기
    cleanupOnClose(con); // 커서 close
    JdbcUtils.closeConnection(this.con); // DB 커넥션 닫기
}
```

## 3장. 메타데이터의 마법: 상태 추적과 재시작 전략

실패는 또 다른 시작일 뿐입니다. 스프링 배치는 **ExecutionContext**를 통해 매 트랜잭션마다 상태를 기록하고, 재시작 시 이를 복원합니다.

### 1. 상태 복원 (open)

`open(ExecutionContext)` 메서드는 스텝 시작 시 호출되어 이전 실패 시점의 정보를 가져옵니다.

```java
// AbstractItemCountingItemStreamItemReader.open()
public void open(ExecutionContext executionContext) throws ItemStreamException {
    // 이전 실행에서 읽었던 아이템 개수 복원
    if (executionContext.containsKey(getExecutionContextKey(READ_COUNT))) {
        itemCount = executionContext.getInt(getExecutionContextKey(READ_COUNT));
    }
    // 중단된 위치로 이동
    if (itemCount > 0 && itemCount < maxItemCount) {
        jumpToItem(itemCount);
    }
    currentItemCount = itemCount;
}
```

### 2. 점프 전략 (jumpToItem)의 차이

각 Reader는 자신의 물리적 특성에 맞는 방식으로 재시작 지점을 찾아갑니다.

* **FlatFileItemReader:** 랜덤 액세스가 불가능하므로 `readLine()`을 반복 호출하여 순차적으로 이동합니다.
* **JdbcCursorItemReader:** 드라이버가 지원한다면 `rs.absolute(itemIndex)`로 즉시 이동하고, 아니면 `rs.next()`로 한 줄씩 이동합니다.
* **JdbcPagingItemReader:** `page = itemIndex / pageSize` 수식으로 즉시 계산하여 재시작 지점을 찾아내는 가장 효율적인 방식을 취합니다.

### 3. 상태 저장 (update)

매 트랜잭션 커밋 직전에 호출되어 `currentItemCount` 등을 기록합니다. 롤백 시에는 호출되지 않아 잘못된 상태 저장을 방지합니다.

## 4장. 위임과 프록시의 함정: 시스템 안정성의 경계

### 1. 위임 구조와 5.2.2의 수정사항

`CompositeItemReader` 등은 위임 대상들의 `open/update/close`를 순차 호출(Bypass)합니다.

* **치명적 결함 주의:** 5.2.1 이전 버전에서는 위임 대상 중 하나가 `close()` 중 예외를 던지면 나머지 자원들이 정리되지 않는 자원 누수 버그가 있었습니다. 5.2.2 버전부터는 예외를 수집하여 모든 자원을 닫은 후 마지막에 던지도록 수정되었습니다.

### 2. @StepScope와 JDK Dynamic Proxy의 폭발

가장 위험한 함정은 **인터페이스 반환 타입**과 **프록시**의 조합입니다.

```java
// 치명적인 실수 예시
@Bean 
@StepScope 
public ItemReader<SuspiciousDevice> cafeSquadReader() { 
    return new MongoCursorItemReader<>(); // 구체 클래스 대신 인터페이스 반환
}
```

* **원인:** 반환 타입을 인터페이스로 지정하면 스프링은 **JDK Dynamic Proxy**를 생성합니다. 이 프록시는 선언된 `ItemReader` 인터페이스만 구현하며, 원본 클래스가 가진 `ItemStream` 인터페이스는 누락시킵니다.
* **결과:** 스텝 빌더가 `ItemStream` 구현 여부를 검사할 때 프록시 객체는 이를 구현하지 않은 것으로 판단되어 `open()`이 호출되지 않습니다. 결국 `cursor`가 `null`인 상태에서 `read()`가 실행되어 `NullPointerException`이 발생합니다.
* **교훈:** **무기를 장전(open)하지 않고 방아쇠를 당기면 자기 머리에 총알이 박힙니다.** 반드시 구체 클래스를 반환 타입으로 지정하십시오.

## [개념 검증용 자가 진단 질문]

### ItemStream 아키텍처 및 자원 관리 메커니즘 진단

1. `JdbcCursorItemReader`와 `JdbcPagingItemReader`가 실패 후 재시작될 때, 각각 `jumpToItem()` 메서드 내부에서 재시작 지점으로 이동하기 위해 취하는 기술적 메커니즘의 결정적 차이와 그로 인한 성능적 인과관계를 논술하시오.
2. `@StepScope`를 사용하는 `ItemReader` 빈의 반환 타입을 인터페이스(`ItemReader<T>`)로 설정했을 때, 왜 `NullPointerException`이 발생하며 배치가 비정상 종료되는지 스프링의 프록시 생성 방식과 스텝 빌더의 '자동 등록 메커니즘' 관점에서 설명하시오.
3. `CompositeItemReader`나 `CompositeItemWriter`와 같은 위임 구조에서 5.2.2 버전 이전의 `close()` 메서드가 가졌던 아키텍처적 결함은 무엇이며, 이것이 대규모 배치 운영 환경에서 시스템 자원에 어떤 치명적인 영향을 미치는지 서술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `JdbcCursorItemReader`는 기본적으로 `ResultSet.TYPE_FORWARD_ONLY`를 사용하므로, 재시작 시 `moveCursorToRow()`를 통해 중단 지점까지 `rs.next()`를 반복 호출하며 한 줄씩 이동해야 합니다. 반면 `JdbcPagingItemReader`는 인덱스를 페이지 사이즈로 나누는 단순 산술 계산(`itemIndex / pageSize`)만으로 즉시 필요한 페이지 번호를 도출해낼 수 있습니다. 따라서 데이터가 뒤쪽에서 실패했을 경우, 페이징 리더는 즉각적인 점프가 가능하지만 커서 리더는 앞선 데이터를 모두 건너뛰어야 하므로 처리량에 비례해 재시작 성능이 저하되는 차이가 발생합니다.

* **2번 질문 해설:** `@StepScope`가 적용된 빈의 반환 타입이 인터페이스면 스프링은 JDK Dynamic Proxy를 생성하여 해당 인터페이스만 구현한 객체를 만듭니다. 이 과정에서 원본 객체(`MongoCursorItemReader` 등)가 구현했던 `ItemStream` 인터페이스 정보가 유실됩니다. 스텝 빌더는 `build()` 시점에 리더가 `ItemStream`의 구현체인지 확인하여 자동으로 관리 대상에 등록하는데, 프록시 객체는 `ItemStream`이 아니라고 판단되어 등록에서 누락됩니다. 결국 자원을 여는 `open()` 메서드가 호출되지 않아 내부 커서가 초기화되지 않은(`null`) 상태에서 읽기를 시도하게 되어 예외가 발생합니다.

* **3번 질문 해설:** 5.2.2 이전의 결함은 위임 대상 리스트를 순회하며 `close()`를 호출할 때, 중간에 있는 위임 대상에서 예외가 발생하면 즉시 전체 루프가 중단되는 구조였습니다. 이로 인해 리스트 후순위에 있는 위임 대상들은 `close()` 기회를 영원히 박실하게 됩니다. 대규모 환경에서는 닫히지 않은 파일 핸들이나 DB 커넥션이 누적되어 시스템의 가용 자원이 고갈(Resource Leak)되며, 결국 전체 애플리케이션이나 DB 서버의 마비를 초래하는 연쇄 장애의 원인이 됩니다.

</details>
