# [심화 분석 학습 노트] 스프링 배치: JSON 데이터 처형과 중괄호의 네크로맨시

## 1장. JSONL(JSON Lines) 처형: 독립된 레코드의 파싱 전략

실전에서는 표준 JSON 배열보다 각 라인마다 독립적인 JSON 객체가 기록된 **JSONL(또는 JSON Lines)** 형식을 더 자주 마주하게 됩니다. 이는 로그 수집이나 스트리밍 데이터 처리에서 매우 유리한 방식입니다.

### 1. JSONL 데이터의 구조적 특징

```json
{"command":"destroy","cpu":99,"status":"memory overflow"}
{"command":"explode","cpu":100,"status":"cpu meltdown"}
{"command":"collapse","cpu":95,"status":"disk burnout"}
```

위와 같이 각 줄은 그 자체로 완결된 하나의 JSON 객체입니다. 스프링 배치는 이를 처리하기 위해 `FlatFileItemReader`와 `ObjectMapper`를 조합하는 전략을 사용합니다.

### 2. 커스텀 LineMapper를 통한 객체 부활

스프링 배치는 기본적으로 `JsonLineMapper`를 제공하여 JSON을 `Map`으로 변환해주지만, 타입 안정성과 비즈니스 로직의 명확성을 위해 특정 도메인 객체(SystemDeath)로 직접 매핑하는 **커스텀 LineMapper**를 구현하는 것이 권장됩니다.

#### [소스코드 및 상세 분석]

```java
@Configuration
public class SystemDeathJobConfig {
    private final JobRepository jobRepository;
    private final PlatformTransactionManager transactionManager;
    private final ObjectMapper objectMapper; // JSON 파싱을 담당하는 Jackson 엔진 주입

    // ... 생략 ...

    @Bean
    @StepScope
    public FlatFileItemReader<SystemDeath> systemDeathReader(
        @Value("#{jobParameters['inputFile']}") String inputFile) {
        return new FlatFileItemReaderBuilder<SystemDeath>()
            .name("systemDeathReader")
            .resource(new FileSystemResource(inputFile)) // 대상 파일 리소스 지정
            // 핵심 설정: lineMapper() 메서드를 통해 한 줄의 문자열을 객체로 변환하는 로직 정의
            .lineMapper((line, lineNumber) -> objectMapper.readValue(line, SystemDeath.class))
            .build();
    }
}
```

* **objectMapper.readValue(line, SystemDeath.class):** `FlatFileItemReader`가 파일을 한 줄씩 읽어올 때마다 람다식이 호출됩니다. 주입받은 `ObjectMapper`가 텍스트(line)를 분석하여 `SystemDeath` 인스턴스를 생성합니다.
* **유연성:** 이 구조는 범용 도구에 의존하지 않고 우리만의 '처형 프로토콜'에 맞는 객체를 정확히 생성해낼 수 있게 합니다.

## 2장. 위장술의 파괴: 멀티라인 JSON과 레코드 경계 감지

데이터가 보기 좋게 포맷팅되어(Pretty-print) 하나의 JSON 객체가 여러 줄에 걸쳐 분산되어 있는 경우, 단순한 라인 단위 읽기는 실패합니다. 이때 시스템은 **레코드의 끝이 어디인지** 판별할 수 있는 지능이 필요합니다.

### 1. RecordSeparatorPolicy: 경계의 수호자

`RecordSeparatorPolicy`는 텍스트 파일 기반 입력 소스에서 하나의 논리적 레코드가 끝나는 지점을 결정하는 전략 인터페이스입니다.

```java
public interface RecordSeparatorPolicy {
    boolean isEndOfRecord(String line); // 해당 줄이 레코드의 끝인지 판단
    String postProcess(String record);  // 레코드가 완성된 후 추가 처리
    String preProcess(String record);   // 레코드를 시작하기 전 사전 처리
}
```

### 2. JsonRecordSeparatorPolicy를 통한 중괄호 추적

우리는 `SimpleRecordSeparatorPolicy`를 확장한 `JsonRecordSeparatorPolicy`를 사용하여 여러 줄의 JSON을 정복합니다.

#### [소스코드 및 상세 분석]

```java
@Bean
@StepScope
public FlatFileItemReader<SystemDeath> systemDeathReader(
    @Value("#{jobParameters['inputFile']}") String inputFile) {
    return new FlatFileItemReaderBuilder<SystemDeath>()
        .name("systemDeathReader")
        .resource(new FileSystemResource(inputFile))
        .lineMapper((line, lineNumber) -> objectMapper.readValue(line, SystemDeath.class))
        // 핵심 무기: JSON 경계 감지기 장착
        .recordSeparatorPolicy(new JsonRecordSeparatorPolicy()) 
        .build();
}
```

* **isEndOfRecord() 내부 메커니즘:** 이 정책은 파일의 각 라인을 읽으며 여는 중괄호(`{`)와 닫는 중괄호(`}`)의 개수를 카운팅합니다.
    1. 열고 닫는 중괄호의 총 개수가 일치하고,
    2. 현재 줄이 닫는 중괄호(`}`)로 끝날 경우,
    프레임워크는 이를 하나의 완결된 레코드로 인식하여 `LineMapper`에게 전달합니다.

## 3장. JSON 배열 전용 무기: JsonItemReader & JsonFileItemWriter

데이터가 `[` 로 시작하여 `]` 로 끝나는 거대한 배열 형태인 경우, 스프링 배치는 `JsonItemReader`라는 특수 무기를 제공합니다.

### 1. JsonItemReader: 배열 파싱의 엔진

파일뿐만 아니라 URL 등 다양한 소스에서 JSON 배열을 읽어들일 수 있습니다.

#### [소스코드 및 상세 분석]

```java
@Bean
@StepScope
public JsonItemReader<SystemDeath> systemDeathReader(
    @Value("#{jobParameters['inputFile']}") String inputFile) {
    return new JsonItemReaderBuilder<SystemDeath>()
        .name("systemDeathReader")
        // 파싱 핵심 엔진 설정: Jackson을 사용하여 SystemDeath 객체로 변환
        .jsonObjectReader(new JacksonJsonObjectReader<>(SystemDeath.class))
        .resource(new FileSystemResource(inputFile)) // 파일 리소스 주입
        .build();
}
```

* **JsonObjectReader:** 실제 파싱을 담당하는 엔진입니다. `JacksonJsonObjectReader`와 `GsonJsonObjectReader` 중 선택할 수 있습니다.
* **특징:** `FlatFileItemReader`와 달리 내부적으로 JSON 구조 자체를 이해하므로 배열 형태의 데이터를 스트리밍 방식으로 효율적으로 읽어옵니다.

### 2. JsonFileItemWriter: 객체를 배열로 감금하기

객체를 JSON 배열 형식으로 마샬링(Marshalling)하여 파일에 기록합니다.

#### [소스코드 및 상세 분석]

```java
@Bean
@StepScope
public JsonFileItemWriter<DeathNote> deathNoteJsonWriter(
    @Value("#{jobParameters['outputDir']}") String outputDir) {
    return new JsonFileItemWriterBuilder<DeathNote>()
        // 객체를 JSON 문자열로 변환하는 마샬러 설정
        .jsonObjectMarshaller(new JacksonJsonObjectMarshaller<>())
        .resource(new FileSystemResource(outputDir + "/death_notes.json")) // 저장 경로
        .name("logEntryJsonWriter")
        .build();
}
```

* **JsonObjectMarshaller:** `JacksonJsonObjectMarshaller`를 사용하여 `DeathNote` 객체를 JSON 문자열로 변환합니다.
* **출력 형태:** 쓰기 작업이 완료되면 전체 데이터는 `[...]` 배열 형태 내에 개별 객체로 정렬되어 저장됩니다.

## [개념 검증용 자가 진단 질문]

### JSON 배치 처리 아키텍처 심화 진단

1. 단일 라인 JSON(JSONL) 처리를 위해 `FlatFileItemReader`를 사용할 때와 비교하여, 멀티라인으로 구성된 Pretty-print JSON을 처리할 때 `RecordSeparatorPolicy`가 수행하는 내부적인 '상태 유지(Stateful)' 메커니즘은 무엇이며, 이것이 `LineMapper` 호출 시점에 어떤 영향을 미치는가?
2. `JsonItemReader`를 사용할 때 `JsonObjectReader`로 `JacksonJsonObjectReader`를 선택했다면, 이 컴포넌트가 대용량 JSON 배열 파일을 읽을 때 메모리 효율을 극대화하기 위해 내부적으로 어떤 읽기 방식을 취하는지 논술하시오.
3. `JsonFileItemWriter`가 실행을 마치고 생성한 결과물은 항상 `[`와 `]`로 감싸진 배열 형태가 된다. 만약 실무 요구사항이 '배열 형태가 아닌 줄바꿈 기반의 JSONL 형식'으로 파일을 생성하는 것이라면, `JsonFileItemWriter`를 사용하는 대신 어떤 기존 컴포넌트들을 어떻게 조합하여 전략을 수정해야 하는지 그 근거를 제시하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `RecordSeparatorPolicy`는 여러 줄을 읽는 동안 현재 레코드가 완료되었는지에 대한 '진행 상태'를 관리합니다. 특히 `JsonRecordSeparatorPolicy`는 중괄호의 쌍이 맞는지 누적해서 카운팅합니다. 이 정책이 `isEndOfRecord()`에서 `true`를 반환하기 전까지 `FlatFileItemReader`는 각 라인을 내부 버퍼에 계속 합치기만 하며 `LineMapper`를 호출하지 않습니다. 정책에 의해 레코드의 끝이 확인된 순간, 지금까지 모인 '멀티라인 문자열'이 단 한 번의 호출로 `LineMapper`에 전달되어 객체로 변환됩니다.

* **2번 질문 해설:** `JsonItemReader`와 그 내부의 `JacksonJsonObjectReader`는 파일 전체를 한꺼번에 메모리에 올리는 `DOM` 방식이 아니라, 스트림을 통해 토큰 단위로 데이터를 읽어 들이는 `Streaming API` 방식을 취합니다. 배열의 시작(`[`)을 인식한 후 각 객체 노드에 진입할 때마다 해당 객체만큼만 파싱하여 `ItemReader.read()`로 반환합니다. 이를 통해 수 기가바이트(GB)에 달하는 거대한 JSON 배열 파일도 적은 메모리 점유율로 안전하게 처리할 수 있습니다.

* **3번 질문 해설:** `JsonFileItemWriter`는 태생적으로 전체 데이터를 하나의 JSON 배열로 묶는 것에 특화되어 있습니다. 줄바꿈 기반의 JSONL 형식을 생성하려면 `FlatFileItemWriter`를 사용해야 합니다. 구체적으로는 `FlatFileItemWriter`의 `LineAggregator`를 커스텀 구현하여, 내부에서 `ObjectMapper`를 호출해 객체를 JSON 문자열로 변환(serialize)한 뒤 반환하도록 설정해야 합니다. 이렇게 하면 `FlatFileItemWriter`가 각 객체마다 변환된 문자열 뒤에 시스템 줄바꿈 문자를 붙여 기록하므로 원하는 JSONL 형식을 얻을 수 있습니다.

</details>
