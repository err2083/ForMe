# [심화 분석 학습 노트] 스프링 배치: Run Squad 장악과 시스템 자동화의 심연

## 1장. 명령 전달 경로의 해킹: JobLauncherApplicationRunner

우리가 커맨드라인에서 입력하는 단순한 문자열은 **JobLauncherApplicationRunner**라는 컴포넌트를 통해 배치 엔진의 언어로 번역됩니다. 이 과정은 시스템의 입구이자 모든 명령 체인의 시작점입니다.

### 1. 커맨드라인 문자열의 객체화 공정

우리가 `./gradlew bootRun --args='--spring.batch.job.name=brutalizedSystemJob chaos=true,java.lang.Boolean'`와 같이 명령을 내리면, 시스템은 내부적으로 다음과 같은 경로를 거칩니다.

#### [JobLauncherApplicationRunner.run() 분석]

```java
@Override
public void run(ApplicationArguments args) throws Exception {
    // nonOptionArgs(chaos=true... 등)를 배열로 추출
    String[] jobArguments = args.getNonOptionArgs().toArray(new String);
    run(jobArguments); 
}

public void run(String... args) throws JobExecutionException {
    logger.info("Running default command line with: " + Arrays.asList(args));
    // "="을 구분자로 하여 문자열을 Properties 객체로 분할
    launchJobFromProperties(StringUtils.splitArrayElementsIntoProperties(args, "="));
}
```

* **분석:** `StringUtils.splitArrayElementsIntoProperties`는 `chaos=true,java.lang.Boolean`을 `{ "chaos": "true,java.lang.Boolean" }` 형태의 **Properties 객체**로 파싱합니다.

### 2. JobParametersConverter: 타입 부활의 기술

문자열 프로퍼티는 **DefaultJobParametersConverter**를 통해 실제 자바 타입과 식별(Identifying) 속성을 가진 `JobParameters`로 부활합니다.

```java
// DefaultJobParametersConverter.java 생성자
public DefaultJobParametersConverter() {
    DefaultConversionService conversionService = new DefaultConversionService();
    // 다양한 날짜/시간 타입 변환기 등록
    conversionService.addConverter(new DateToStringConverter());
    conversionService.addConverter(new StringToDateConverter());
    conversionService.addConverter(new LocalDateToStringConverter());
    conversionService.addConverter(new StringToLocalDateConverter());
    // ... 중략 ...
    this.conversionService = conversionService;
}
```

* **decode() 메커니즘:** 입력된 `true,java.lang.Boolean` 문자열은 `decode()` 메서드 내에서 쉼표(,)를 기준으로 값, 타입, 식별 여부를 분리하여 **JobParameter** 객체로 생성됩니다.

### 3. 로컬 Job vs 등록된 Job의 처형 경로

`JobLauncherApplicationRunner`는 두 가지 경로로 Job을 찾아 실행합니다.

* **executeLocalJobs():** 애플리케이션 컨텍스트에 `@Bean`으로 등록된 Job들을 직접 순회하며 실행합니다. `@Autowired`로 주입된 `Collection<Job>`을 사용합니다.
* **executeRegisteredJobs():** `JobRegistry`에 등록된 Job을 찾습니다. 이는 동적으로 생성되거나 외부에서 관리되는 Job을 실행할 때 유용합니다.

#### [파라미터 증분과 재시작 로직 (getNextJobParameters)]

```java
private JobParameters getNextJobParameters(Job job, JobParameters jobParameters) {
    // 이미 존재하는 JobInstance라면 이전 파라미터와 병합(merge)하여 재시작 준비
    if (this.jobRepository != null && this.jobRepository.isJobInstanceExists(job.getName(), jobParameters)) {
        return getNextJobParametersForExisting(job, jobParameters);
    }
    
    // JobParametersIncrementer(예: RunIdIncrementer)가 있다면 파라미터 자동 증가
    if (job.getJobParametersIncrementer() == null) {
        return jobParameters;
    }
    
    // JobExplorer를 통해 마지막 실행 이력을 조회하여 run.id 등을 증가시킴
    JobParameters nextParameters = new JobParametersBuilder(jobParameters, this.jobExplorer)
        .getNextJobParameters(job)
        .toJobParameters();
    return merge(nextParameters, jobParameters);
}
```

* **핵심 원리:** `RunIdIncrementer`는 이전 실행의 `run.id`를 조회해 1을 더함으로써, 매번 새로운 `JobInstance`를 생성해 '성공한 Job은 재실행 불가'라는 제약을 우회합니다.

## 2장. 시스템 자동 구성 감시: BatchAutoConfiguration과 심장부

스프링 부트가 배치 환경을 자동으로 구축하는 마법은 **BatchAutoConfiguration**이라는 중앙 통제실에서 일어납니다.

### 1. 자동 구성의 트리거와 조건

```java
@Bean
@ConditionalOnMissingBean
@ConditionalOnProperty(prefix = "spring.batch.job", name = "enabled", havingValue = "true", matchIfMissing = true)
public JobLauncherApplicationRunner jobLauncherApplicationRunner(...) {
    // spring.batch.job.enabled가 true(기본값)일 때만 Runner 생성
    // ...
}
```

* **BatchProperties의 역할:** 커맨드라인의 `--spring.batch.job.name` 값은 `@ConfigurationProperties`에 의해 `BatchProperties` 객체에 바인딩되어 `Runner`에게 전달됩니다.

### 2. DefaultBatchConfiguration: 배치 인프라의 설계도

모든 코어 컴포넌트(`JobRepository`, `JobLauncher` 등)는 **DefaultBatchConfiguration**에서 생성됩니다.

```java
@Configuration(proxyBeanMethods = false)
@Import(ScopeConfiguration.class)
public class DefaultBatchConfiguration implements ApplicationContextAware {
    @Bean
    public JobRepository jobRepository() throws BatchConfigurationException {
        // JobRepository 생성 및 설정
    }
    // ... JobLauncher, JobExplorer, JobRegistry 등 등록
}
```

* **커스터마이징 전략:** 이 클래스의 메서드들은 `protected`로 선언되어 있어, 이를 상속받아 필요한 메서드(예: `getTaskExecutor()`, `getTablePrefix()`)만 오버라이드하면 배치 코어 시스템을 입맛대로 개조할 수 있습니다.

### 3. @EnableBatchProcessing의 정체와 충돌

`@EnableBatchProcessing`은 `BatchRegistrar`를 통해 빈을 등록하는 **원시적 침투 방식**입니다.

* **치명적 결함:** 만약 `@EnableBatchProcessing`을 사용하거나 `DefaultBatchConfiguration`을 직접 상속하면, 스프링 부트의 **BatchAutoConfiguration이 비활성화**됩니다. 이 경우 `JobLauncherApplicationRunner` 등 부트가 제공하는 자동 실행 기능이 사라지므로 주의가 필요합니다.

## 3장. 커맨드라인 종결과 배치 스코프의 비밀

### 1. ExitCode 장악: 시스템 최후의 시그니처

배치 작업의 성공/실패를 외부 시스템(젠킨스 등)에 알리려면 정확한 종료 코드가 필요합니다.

```java
// KillBatchSystemApplication.java
public static void main(String[] args) {
    // SpringApplication.exit를 통해 Job 결과를 시스템 종료 코드로 변환
    System.exit(SpringApplication.exit(SpringApplication.run(KillBatchSystemApplication.class, args)));
}
```

* **JobExecutionExitCodeGenerator:** 기본적으로 `BatchStatus`의 **Ordinal 값**을 종료 코드로 반환합니다(FAILED = 5).
* **커스텀 종료 코드:** `ExitCodeGenerator`를 직접 구현하고 `SimpleJvmExitCodeMapper`를 사용하면 `SKULL_FRACTURE` 같은 커스텀 상태를 `3`번 종료 코드로 매핑할 수 있습니다.

### 2. 배치 스코프(Late-Binding)의 진짜 무기: StepContext

`@StepScope` 빈에서 사용하는 SpEL(`#{jobParameters['...']}`)의 실체는 **StepContext**입니다.

#### [StepScope.getContext()와 SpEL 실행 경로]

```java
private StepContext getContext() {
    // StepSynchronizationManager에서 현재 스레드에 바인딩된 컨텍스트 추출
    StepContext context = StepSynchronizationManager.getContext();
    if (context == null) {
        throw new IllegalStateException("No context holder available for step scope");
    }
    return context;
}
```

* **SpEL의 진실:** `#{jobParameters}` 표현식은 내부적으로 `stepContext.getJobParameters()` 메서드를 호출합니다.
* **런타임 무기고:** `StepContext`는 파라미터뿐만 아니라 `stepName`, `jobName`, `jobInstanceId` 등 모든 런타임 정보를 담고 있는 최종 무기고 역할을 수행합니다.

## [개념 검증용 자가 진단 질문]

### 배치 실행 진입점 및 자동 구성 메커니즘 심화 진단

1. `JobLauncherApplicationRunner`가 Job을 실행하기 전 호출하는 `getNextJobParameters()` 메서드에서, 왜 `JobParametersIncrementer`를 적용하기 전에 `isJobInstanceExists()`를 통해 기존 인스턴스 존재 여부를 먼저 확인하는가? 재시작(Restart)과 증분(Increment) 시나리오에서의 동작 차이를 중심으로 논술하시오.
2. 스프링 부트 환경에서 `@EnableBatchProcessing`을 선언하는 순간, 왜 커맨드라인으로 전달한 `--spring.batch.job.name` 파라미터가 무시되거나 배치 작업이 자동 실행되지 않는 현상이 발생하는지 `BatchAutoConfiguration`의 `@Conditional` 조건과 연계하여 설명하시오.
3. `@StepScope`가 적용된 빈 내부에서 `@Value("#{jobParameters['key']}")`를 통해 값을 주입받을 때, `StepSynchronizationManager`와 `StepContext`가 내부적으로 어떻게 상호작용하여 싱글톤인 빈 팩토리 내에서 런타임에 동적인 값을 찾아내는지 그 프로세스를 기술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `getNextJobParameters()`는 작업의 성격에 따라 다른 전략을 취합니다. 만약 동일 파라미터로 이미 실행된 이력이 있다면 이는 '실패한 작업의 재시작' 시나리오이므로, 파라미터를 새로 생성하는 대신 기존 파라미터를 병합하여 중단된 지점부터 다시 실행할 수 있게 합니다. 반면, 기존 인스턴스가 없거나 성공한 경우라면 `JobParametersIncrementer`를 가동하여 `run.id` 등을 증가시킴으로써 '새로운 실행(New Instance)'으로 인식되게 합니다. 즉, **상태 복구(Restart)가 증분(Increment)보다 우선순위**를 갖도록 설계된 메커니즘입니다.

* **2번 질문 해설:** `BatchAutoConfiguration` 클래스 선언부에는 `@ConditionalOnMissingBean(annotation = EnableBatchProcessing.class)` 조건이 걸려 있습니다. 사용자가 `@EnableBatchProcessing`을 선언하면 이 조건에 의해 부트의 자동 구성 클래스 자체가 로드되지 않습니다. 이로 인해 `BatchAutoConfiguration`이 생성하던 `JobLauncherApplicationRunner` 빈이 등록되지 않게 되고, 결과적으로 커맨드라인 인자를 해석하고 Job을 트리거하는 부트의 기본 메커니즘이 완전히 제거되기 때문입니다.

* **3번 질문 해설:** 빈에 `@StepScope`가 선언되면 스프링은 실제 빈 대신 CGLIB 프록시를 생성합니다. 빈의 메서드가 호출되는 시점에 프록시는 `StepScope.get()`을 호출하고, 이는 다시 `StepSynchronizationManager.getContext()`를 호출합니다. 이때 `StepSynchronizationManager`는 현재 실행 중인 스텝 스레드의 `ThreadLocal`에서 `StepContext`를 꺼내옵니다. SpEL 엔진은 이 `StepContext` 객체에 정의된 `getJobParameters()` 등의 메서드를 리플렉션으로 호출하여 값을 추출한 뒤 프록시의 타겟 빈에 주입하는 **지연 바인딩(Late-Binding)**을 완성합니다.

</details>
