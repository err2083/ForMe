# [심화 분석 학습 노트] 스프링 배치: 멀티스레드 스텝과 분산 폭격 아키텍처

## 1장. 멀티스레드 스텝의 심장: TaskExecutorRepeatTemplate

멀티스레드 스텝은 하나의 `StepExecution`을 여러 스레드가 동시에 실행하여 처리 속도를 극대화하는 전략입니다. 단순히 스레드만 늘리는 것이 아니라, 프레임워크 심층부에서 엔진 자체가 교체되는 과정입니다.

### 1. 멀티스레드 스텝 구성 및 빌더 분석

스텝 빌더에서 `taskExecutor()`를 호출하는 순간, 시스템은 병렬 처리 모드로 진화합니다.

```java
@Bean
public Step threatAnalysisStep() {
    return new StepBuilder("threatAnalysisStep", jobRepository)
        .<HumanThreatData, TargetPriorityResult>chunk(100) // 100개 단위 청크 처리
        .reader(humanThreatDataReader())
        .processor(threatAnalysisProcessor())
        .writer(targetListWriter())
        // 💀 전투력 증강 설정 - 코어 엔진 과부하 개시! 💀
        // 시스템의 모든 혈관에 아드레날린(스레드)을 강제로 주입한다!
        .taskExecutor(taskExecutor()) 
        .throttleLimit(5) // 동시에 참여할 스레드 수 제한
        .build();
}

@Bean
public TaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(5);         // 기본 전투 유닛 수
    executor.setMaxPoolSize(5);          // 최대 전투 유닛 수
    executor.setWaitForTasksToCompleteOnShutdown(true); // 모든 작전 완료 대기
    executor.setAwaitTerminationSeconds(10);             // 종료 대기 시간
    executor.setThreadNamePrefix("T-800-");             // 스레드 식별 코드
    executor.setAllowCoreThreadTimeOut(true);           // 유휴 유휴 종료 허용
    executor.setKeepAliveSeconds(30);                    // 유휴 상태 유지 시간
    return executor;
}
```

* **taskExecutor(taskExecutor()):** 이 설정이 들어가는 순간 `AbstractTaskletStepBuilder`는 내부 엔진을 교체합니다.
* **throttleLimit(5):** 청크 처리에 동시에 참여할 수 있는 스레드의 상한선을 정합니다.

### 2. 내부 엔진의 교체 메커니즘

`AbstractTaskletStepBuilder.build()` 메서드를 해부하면 엔진이 어떻게 바뀌는지 명확히 드러납니다.

```java
// AbstractTaskletStepBuilder.java 내부 로직
if (taskExecutor != null) {
    // 💀 멀티 스레드 스텝을 위한 특별한 구현체 투입 💀
    TaskExecutorRepeatTemplate repeatTemplate = new TaskExecutorRepeatTemplate();
    repeatTemplate.setTaskExecutor(taskExecutor);
    repeatTemplate.setThrottleLimit(throttleLimit);
    stepOperations = repeatTemplate; // 표준 RepeatTemplate 대신 확장 버전 사용
}
step.setStepOperations(stepOperations);
```

* **TaskExecutorRepeatTemplate:** 여러 스레드에서 작업을 병렬로 실행하는 능력을 가진 확장 템플릿입니다. 이것이 `Tasklet.execute()` 메서드를 반복 실행하는 엔진이 됩니다.

### 3. 실행 흐름과 스레드 바인딩 (ThreadLocal)

멀티스레드 환경에서도 각 워커 스레드는 자신만의 `StepExecution`을 가져야 합니다. 이를 위해 `StepSynchronizationManager`가 투입됩니다.

```java
// StepContextRepeatCallback.doInIteration() 내부
public RepeatStatus doInIteration(RepeatContext context) throws Exception {
    // 각 워커 스레드마다 자신의 ThreadLocal 공간에 StepExecution을 바인딩
    StepContext stepContext = StepSynchronizationManager.register(stepExecution);
}
```

* **인과관계:** `TaskExecutorRepeatTemplate`이 각 워커 스레드에서 `doInIteration()`을 독립적으로 실행하므로, 모든 스레드에서 **Step 스코프**가 올바르게 작동할 수 있게 됩니다.

## 2장. 스레드 안전성(Thread Safety): 전장의 지뢰 탐지

멀티스레드 스텝은 강력하지만 '공유 상태 오염'이라는 지뢰가 깔려 있습니다. 데이터 처리 안전성과 실행 상태 안전성이라는 두 가지 관점을 철저히 분리해서 정복해야 합니다.

### 1. 데이터 처리 안전성: Reader와 Writer의 실체

대부분의 `ItemWriter`는 안전하지만, `ItemReader`는 공유 변수 때문에 위험한 경우가 많습니다.

#### [위험 지역 1: FlatFileItemReader]

```java
// FlatFileItemReader.java 내부의 레이스 컨디션 유발 지점
private int lineCount = 0; // 여러 스레드가 공유하는 상태 변수

private String readLine() {
    // ... 중략 ...
    line = this.reader.readLine();
    if (line == null) {
        return null;
    }
    this.lineCount++; // 💀 여러 스레드가 동시에 접근하여 데이터 누락/중복 발생!
    return line;
}
```

#### [위험 지역 2: JpaCursorItemReader]

```java
// JpaCursorItemReader.doRead()
protected T doRead() {
    // 자바 Iterator는 스레드 안전하지 않음. hasNext와 next 사이에 다른 스레드 침투 가능
    return this.iterator.hasNext() ? this.iterator.next() : null;
}
```

#### [안전 지역: JdbcPagingItemReader]

```java
// AbstractPagingItemReader.doRead()
private final Lock lock = new ReentrantLock();

@Nullable
@Override
protected T doRead() throws Exception {
    // 💀 ReentrantLock을 통해 공유 변수(page, current) 보호 💀
    this.lock.lock();
    try {
        // 데이터 읽기 및 인덱스 증가 로직
    } finally {
        this.lock.unlock();
    }
}
```

### 2. 동기화 데코레이터: SynchronizedItemStreamReader

스레드 안전하지 않은 리더를 위해 스프링 배치는 **동기화 갑옷**을 제공합니다.

```java
// SynchronizedItemStreamReader.java
public class SynchronizedItemStreamReader<T> implements ItemStreamReader<T>, InitializingBean {
    private ItemStreamReader<T> delegate; // 위임할 원본 리더
    private final Lock lock = new ReentrantLock();

    @Override
    public T read() throws Exception {
        this.lock.lock(); // 💀 줄을 서시오! 💀
        try {
            return this.delegate.read(); // 원본 리더의 read() 호출을 보호
        } finally {
            this.lock.unlock();
        }
    }
    // open, update, close 등도 delegate에 위임하여 상태 관리 유지
}
```

## 3장. 실행 상태 안전성: 재시작의 퇴로를 끊어라

멀티스레드 스텝에서 가장 치명적인 충돌은 **상태 관리(ExecutionContext)**에서 발생합니다.

### 1. 상태 저장의 모순 (saveState=false)

`ItemStream.update()`는 현재 처리 위치를 기록하려 하지만, 여러 스레드가 동시에 기록하면 메타데이터가 엉망이 됩니다.

```java
// AbstractFileItemWriter.update() 내부
@Override
public void update(ExecutionContext executionContext) {
    // 💀 saveState == true일 때만 기록 💀
    if (saveState) {
        try {
            // 현재 파일 포지션을 기록하려 함. 여러 스레드가 호출하면 지옥이 열림
            executionContext.putLong(getExecutionContextKey(RESTART_DATA_NAME), state.position());
        } catch (IOException e) {
            throw new ItemStreamException("ItemStream does not return current position properly", e);
        }
    }
}
```

* **해결책:** **`.saveState(false)`**를 설정하여 재시작 기능을 수동으로 거세해야 합니다. 이것은 멀티스레드 스텝을 사용하는 모든 `ItemStream` 구현체의 숙명입니다.

### 2. [주의] JsonFileItemWriter의 반전

`FlatFileItemWriter`는 데이터 쓰기 자체는 로컬 `StringBuilder`를 써서 안전하지만, `JsonFileItemWriter`는 다릅니다.

```java
// JsonFileItemWriter.doWrite()의 함정
@Override
public String doWrite(Chunk<? extends T> items) {
    StringBuilder lines = new StringBuilder();
    Iterator<? extends T> iterator = items.iterator();
    // 💀 state.getLinesWritten() 이라는 공유 상태를 참조하여 콤마(,) 여부 결정 💀
    // 여러 스레드가 동시에 0을 읽으면 [{"item1"}{"item2"}] 처럼 콤마 없는 깨진 JSON 탄생!
}
```

* **전술:** `JsonFileItemWriter`는 반드시 `SynchronizedItemStreamWriter`로 감싸야 합니다.

### 3. throttleLimit의 진실과 덫

많은 병사들이 놓치는 성능의 병목 지점입니다.

* **동작 방식:** `TaskExecutorRepeatTemplate`은 `TaskExecutor`가 아무리 많아도 내부적으로 `throttleLimit`만큼만 스레드를 가동합니다.
* **기본값의 저주:** 설정하지 않으면 **기본값 4**가 적용됩니다. 20개의 스레드를 할당해도 `throttleLimit`을 안 쓰면 4개만 일하게 됩니다.
* **주의:** 5.0부터 `deprecated` 되었으나, 6.0 이전까지는 여전히 이 값을 `maxPoolSize`와 일치시켜야 제 성능이 나옵니다.

## [개념 검증용 자가 진단 질문]

### 멀티스레드 스텝 아키텍처 및 스레드 안전성 진단

1. `AbstractTaskletStepBuilder`가 `taskExecutor`가 설정되었을 때 사용하는 `TaskExecutorRepeatTemplate`은 일반 `RepeatTemplate`과 비교하여 **Step의 실행 흐름(doExecute) 중 어느 지점부터 멀티스레딩을 적용**하며, 이 과정에서 `StepSynchronizationManager`가 수행하는 핵심 역할은 무엇인가?
2. `FlatFileItemReader`와 `JdbcPagingItemReader`는 모두 멀티스레드 환경에서 데이터 읽기를 시도하지만, 한쪽은 "Not Thread-safe"하고 다른 쪽은 "Thread-safe"하다. 이 **차이를 결정짓는 내부 필드 관리 방식(공유 변수 vs Lock)**을 인과관계에 따라 기술하시오.
3. 멀티스레드 스텝에서 파일 기반 Writer(예: `FlatFileItemWriter`)를 사용할 때, 데이터 쓰기 자체는 안전함에도 불구하고 왜 반드시 `.saveState(false)`를 설정해야 하는가? 만약 이를 `true`로 방치했을 때 `ItemStream.update()` 과정에서 발생할 수 있는 **메타데이터 정합성 결함**을 논술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `TaskExecutorRepeatTemplate`은 `TaskletStep.doExecute()` 내의 `stepOperations.iterate()` 호출 시점부터 작동합니다. 즉, Job의 메인 스레드는 여기까지 단독으로 오고, 청크 처리 단위(read-process-write 사이클)부터 워커 스레드들에게 작업을 배분합니다. 이때 각 워커 스레드는 독립적인 공간을 가지므로, `StepSynchronizationManager.register()`를 호출하여 공통된 `StepExecution` 객체를 각 스레드의 `ThreadLocal`에 바인딩해줍니다. 이 장치가 있어야만 각 스레드 내에서 생성되는 빈들이 `@StepScope`의 혜택을 온전히 누릴 수 있습니다.

* **2번 질문 해설:** `FlatFileItemReader`는 `lineCount`라는 공유 변수를 아무런 보호 장치 없이 증가(`lineCount++`)시킵니다. 여러 스레드가 동시에 이 값을 읽고 수정하면 레이스 컨디션이 발생해 데이터를 중복으로 읽거나 건너뛰게 됩니다. 반면 `JdbcPagingItemReader`(정확히는 그 부모인 `AbstractPagingItemReader`)는 `page`와 `current` 인덱스를 제어하는 `doRead()` 메서드 전체를 `ReentrantLock`으로 감싸고 있습니다. 따라서 스레드들이 줄을 서서 순차적으로 페이지 인덱스를 확보하게 되므로 데이터 정합성이 유지됩니다.

* **3번 질문 해설:** `FlatFileItemWriter`는 데이터를 쓸 때 각 스레드가 자신만의 로컬 `StringBuilder`를 사용하므로 쓰기 자체는 안전합니다. 하지만 `ItemStream`으로서 수행하는 `update()` 메서드는 `ExecutionContext`라는 공통 저장소에 현재의 '물리적 포지션'을 기록하려 합니다. 멀티스레드 환경에서는 여러 청크가 동시에 완료될 수 있는데, 이때 각 스레드가 서로 다른 포지션 값을 동일한 키값으로 덮어쓰게 됩니다. 결과적으로 DB에는 마지막에 운 좋게 기록된 스레드의 위치만 남게 되어, 재시작 시 어떤 데이터가 실제로 처리되었는지 알 수 없는 '메타데이터 오염' 상태가 됩니다. 이를 방지하기 위해 상태 저장 기능을 강제로 꺼야 합니다.
