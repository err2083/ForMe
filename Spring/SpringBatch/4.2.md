# [심화 분석 학습 노트] 스프링 배치: ItemProcessor - 데이터 지배와 숙청의 아키텍처

## 1장. ItemProcessor의 본질과 필터링/검증 전략

ItemProcessor는 읽어온 데이터를 가공하여 쓰기 단계로 넘기기 전, 우리의 고유한 비즈니스 규칙을 적용하는 영역입니다. 스프링 배치가 제공하는 기본 구현체에 의존하기보다 개발자가 직접 로직을 정의하는 경우가 많으므로, 그 동작 원리를 완벽히 이해하는 것이 필수적입니다.

### 1. 첫 번째 처단 방식: null 반환을 통한 데이터 필터링

`ItemProcessor.process()` 메서드가 `null`을 반환하는 행위는 단순한 빈 값 처리가 아닌 **'처형(Execution)'**입니다. `null`이 반환되는 순간 해당 아이템은 다음 단계인 ItemWriter로 전달되는 Chunk에서 영구히 제외됩니다.

* **내부 동작 프로세스:**
    1. **ItemReader:** 지정된 청크 사이즈만큼 `read()`를 호출하여 **input Chunk**를 생성합니다.
    2. **ItemProcessor:** 생성된 input Chunk의 각 아이템에 대해 `process()`를 호출합니다.
    3. **필터링:** `process()`가 `null`을 반환한 아이템은 최종 **output Chunk**에서 제외됩니다.
    4. **ItemWriter:** 필터링된 결과물인, input Chunk보다 크기가 작아진 output Chunk를 전달받아 기록을 수행합니다.

#### [실전 코드 분석: 시스템 파괴 명령어 감시자]

```java
@Slf4j
public class ExecutionerProcessor implements ItemProcessor<Command, Command> {
    @Override
    public Command process(Command command) {
        // 시스템 파괴 명령어 실행자는 처단 (null 반환)
        if (command.getCommandText().contains("rm -rf /") || command.getCommandText().contains("kill -9")) {
            log.info("☠️ {}의 {} -> 시스템 파괴자 처단 완료. 기록에서 말살.", command.getUserId(), command.getCommandText());
            return null; // 필터링 집행
        }
        
        // sudo 권한 오남용 확인 (예시 확장)
        if (command.isSudoUsed() && !"admin".equals(command.getUserId())) {
            return null;
        }

        return command; // 생존한 데이터만 반환
    }
}
```

* **인과관계:** `null` 반환 시 청크 내의 아이템 개수가 줄어듭니다. 예를 들어 리더가 4개를 읽었으나 프로세서에서 2개가 `null`을 반환하면, 라이터는 오직 2개의 아이템만 받게 됩니다.

### 2. ValidatingItemProcessor: 검증과 실패의 이중주

데이터가 유효하지 않을 때 단순히 필터링할 것인지, 아니면 전체 작업을 중단시킬 것인지는 배치 설계의 핵심 결정 사항입니다. 이를 위해 `ValidatingItemProcessor`가 활용됩니다.

#### [Validator 구현 및 설정 분석]

```java
// 1. 전용 Validator 구현 (Spring Batch 전용 인터페이스 사용)
public class CommandValidator implements Validator<Command> {
    @Override
    public void validate(Command command) throws ValidationException {
        if (command.getCommandText().contains("rm -rf /") || command.getCommandText().contains("kill -9")) {
            throw new ValidationException(
                "☠️ " + command.getUserId() + "의 " + command.getCommandText() + " → 시스템 파괴 명령어 감지. 처단.");
        }
    }
}

// 2. ValidatingItemProcessor 빈 설정
@Bean
public ItemProcessor<Command, Command> commandProcessor() {
    ValidatingItemProcessor<Command> processor = new ValidatingItemProcessor<>(new CommandValidator());
    
    // setFilter 설정의 차이 (중요)
    processor.setFilter(true);  // true: ValidationException 발생 시 null 반환 (필터링)
    // processor.setFilter(false); // false: 기본값, 예외를 상위로 전파하여 Step/Job 즉시 실패 처리
    
    return processor;
}
```

* **setFilter(true):** 자비로운 처단입니다. 유효하지 않은 데이터만 솎아내고 배치는 계속 진행됩니다.
* **setFilter(false):** 자비 없는 처단입니다. `ValidationException`이 발생하는 즉시 상위 스텝으로 전파되어 전체 Job이 실패합니다. 시스템에 잘못된 데이터가 유입되는 것을 원천 차단해야 하는 엄격한 상황에서 사용합니다.
* **호환성:** 기존에 구현된 `org.springframework.validation.Validator`(Spring 표준)가 있다면 `SpringValidator` 어댑터를 통해 그대로 활용할 수 있습니다.

## 2장. 데이터의 변환과 보강: 형태를 바꾸고 힘을 더하라

ItemProcessor의 가장 강력한 무기는 '창조적 파괴'입니다. 단순히 거르는 것을 넘어, 데이터를 새로운 형태로 가공하거나 외부 정보를 덧붙이는 역할을 수행합니다.

### 1. 세 번째 처단 방식: 데이터 변환 (Transformation)

ItemProcessor 인터페이스의 제네릭 타입 `<I, O>`는 데이터 흐름의 정체성을 정의합니다.

* **I (Input):** ItemReader가 넘겨준 원본 타입.
* **O (Output):** ItemWriter가 요구하는 가공된 타입.

#### [시스템 로그 분석 리포트 변환 코드]

```java
@Slf4j
public class CommandAnalyzer implements ItemProcessor<SystemLog, CommandReport> {
    @Override
    public CommandReport process(SystemLog systemLog) {
        // 원본 타입 SystemLog를 새로운 타입 CommandReport로 변환
        CommandReport report = new CommandReport();
        report.setExecutorId(systemLog.getUserId());
        report.setTimestamp(systemLog.getExecutedAt());
        
        // 원본 명령어를 분석하여 위험 등급(severity) 부여
        if (systemLog.getRawCommand().contains("sudo")) {
            report.setAction("권한 상승 시도");
            report.setSeverity("HIGH");
        } else {
            report.setAction("일반 명령어 실행");
            report.setSeverity("LOW");
        }
        
        return report; // 변환된 객체 반환
    }
}
```

### 2. 네 번째 처단 방식: 데이터 보강 (Data Enrichment)

원본 데이터에 외부 API나 DB 조회를 통해 부가 정보를 추가하는 과정입니다. "변환은 형태를 바꾸는 것, 보강은 힘을 더하는 것"입니다.

#### [성능의 함정과 외부 시스템 통신 최적화]

ItemProcessor에서 아이템 건당 외부 API를 호출하면 **N+1 문제**로 인해 성능이 기하급수적으로 하락합니다. (100만 건 기준 약 27.7시간 소요 가능) 이를 해결하기 위해 `ItemWriteListener`를 무기로 사용합니다.

```java
@Slf4j
@RequiredArgsConstructor
public class SystemLogEnrichListener implements ItemWriteListener<SystemLog> {
    private final ObservabilityApiClient observabilityApiClient;

    @Override
    public void beforeWrite(Chunk<? extends SystemLog> items) {
        // 개별 호출이 아닌, Chunk(묶음) 단위로 API를 호출하여 성능 극대화
        // 100만 번 호출을 1만 번(청크 100 기준)으로 단축 (약 16분 소요로 개선)
        List<SystemLog> logList = (List<SystemLog>) items.getItems();
        Map<Long, ServerInfo> serverInfoMap = observabilityApiClient.bulkGetServerInfo(logList);
        
        for (SystemLog log : logList) {
            log.setServerName(serverInfoMap.get(log.getUserId()).getName());
        }
    }
}
```

* **기술적 통찰:** 이 작업을 `ItemWriter` 내부에서 수행하지 않는 이유는 **단일 책임 원칙(SRP)** 때문입니다. Writer는 쓰기에만 집중해야 하며, 데이터 보강과 같은 읽기/가공 성격의 작업은 리스너나 프로세서의 영역에서 처리하는 것이 올바른 아키텍처입니다.

## 3장. 고급 전술: 위임과 조합을 통한 프로세서 체인

여러 개의 가공 절차가 필요할 때, 하나의 거대한 ItemProcessor를 만드는 대신 작은 프로세서들을 조합하여 관리합니다.

### 1. CompositeItemProcessor: 순차적 처형 체인

여러 위임 대상 ItemProcessor를 리스트로 묶어 순차 실행합니다. 이전 프로세서의 출력(O)이 다음 프로세서의 입력(I)으로 전달되는 **타입 연속성**이 생명입니다.

```java
// CompositeItemProcessor 구성 (빌더 방식)
@Bean
public CompositeItemProcessor<SystemLog, CommandReport> compositeProcessor() {
    return new CompositeItemProcessorBuilder<SystemLog, CommandReport>()
        .delegates(Arrays.asList(
            new SystemLogFilterProcessor(), // 1단계: 유효성 필터링
            new CommandAnalyzer()           // 2단계: 리포트 타입 변환
        ))
        .build();
}
```

### 2. ClassifierCompositeItemProcessor: 라우팅 기반 처형

`Classifier`를 사용하여 아이템의 상태나 조건에 따라 서로 다른 ItemProcessor로 라우팅합니다.

```java
public class LogClassifier implements Classifier<SystemLog, ItemProcessor<?, ?>> {
    private final ItemProcessor<SystemLog, CommandReport> adminProcessor;
    private final ItemProcessor<SystemLog, CommandReport> userProcessor;

    @Override
    public ItemProcessor<SystemLog, CommandReport> classify(SystemLog classifiable) {
        // 사용자 등급에 따라 처리 로직 분기
        return classifiable.getUserId() < 1000 ? adminProcessor : userProcessor;
    }
}
```

* **주의사항:** `Classifier`가 반환하는 모든 ItemProcessor의 입출력 타입은 `ClassifierCompositeItemProcessor`에 정의된 타입과 반드시 일치해야 합니다.

## [개념 검증용 자가 진단 질문]

### ItemProcessor 메커니즘 및 성능 최적화 진단

1. `ItemProcessor`에서 `null`을 반환할 때와 `ValidationException`을 던질 때(기본 설정), 스프링 배치 프레임워크 내부에서 발생하는 **Step 상태 변화와 Chunk의 최종 목적지**의 차이점을 논술하시오.
2. 대용량 데이터를 처리하는 과정에서 `ItemProcessor` 내부에 외부 API 호출을 통한 데이터 보강 로직을 작성했을 때 발생하는 **성능 저하의 기술적 원인**을 분석하고, 이를 `ItemWriteListener`의 `beforeWrite`를 활용해 해결할 수 있는 아키텍처적 근거를 설명하시오.
3. `CompositeItemProcessor`를 사용하여 다단계 가공 프로세스를 구축할 때, 각 위임 프로세서(Delegates) 간의 **제네릭 타입(I, O) 불일치**가 발생하면 시스템에 어떤 영향을 주는지 스프링 배치의 데이터 흐름 관점에서 기술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `null`을 반환하면 프레임워크는 이를 의도된 '필터링'으로 간주합니다. 해당 아이템만 output Chunk에서 제외될 뿐, Step은 정상적으로 계속 진행되어 나머지 아이템들이 ItemWriter로 전달됩니다. 반면 `ValidationException`을 던지면(setFilter가 false인 경우), 이는 비정상적 '장애'로 간주됩니다. 예외가 상위로 전파되는 순간 현재 진행 중인 Chunk 처리는 즉시 중단되며, 별도의 Skip/Retry 설정이 없다면 Step과 Job 전체가 `FAILED` 상태로 종료되고 Writer는 실행되지 않습니다.

* **2번 질문 해설:** `ItemProcessor.process()`는 청크 내 아이템 개수만큼 개별적으로 호출되는 단위성 작업입니다. 따라서 100만 건 처리 시 100만 번의 네트워크 I/O(N+1 문제)가 발생하여 극심한 지연을 초래합니다. `ItemWriteListener.beforeWrite()`는 이미 프로세싱이 끝난 아이템들을 **Chunk 전체(List)**로 전달받습니다. 이 시점에 Bulk API를 호출하면 통신 횟수를 `전체 아이템 / 청크 사이즈`로 획기적으로 줄일 수 있습니다. 이는 '가공은 건당 수행하되, 외부 리소스 보강은 묶음으로 수행'하는 성능 최적화 전략입니다.

* **3번 질문 해설:** `CompositeItemProcessor`는 체인 구조입니다. 첫 번째 프로세서가 `A`를 받아 `B`를 반환하면, 두 번째 프로세서는 반드시 `B`를 입력으로 받아야 합니다. 만약 타입이 일치하지 않으면 런타임에 `ClassCastException`이 발생하며 데이터 흐름이 단절됩니다. 이는 스프링 배치의 청크 프로세싱 루프 내에서 이전 단계의 결과값이 다음 단계의 파라미터로 직접 전달되는 파이프라인 구조를 가지기 때문에 발생하는 강력한 제약 사항입니다.
