# [구조화 심화 분석 학습 노트] 스프링 배치: 시스템 종결의 아키텍처

## 1장. 배치 처리의 본질: 자동화된 데이터 처형 시스템

배치(Batch) 처리는 단순히 '나중에 실행하는 작업'이 아닙니다. 그것은 정해진 시간에, 대량의 데이터를, 인간의 개입 없이, 완벽하게 일괄 처리하는 **자율 제어 시스템**입니다.

### 1. 배치 처리의 핵심 정의와 동작 메커니즘

배치는 MMORPG의 자동 사냥 시스템과 같습니다. 개발자가 잠든 사이에도 시스템은 사전에 정의된 로직에 따라 데이터를 수집(몬스터 처치)하고 가공하여 결과(전리품)를 산출합니다.

* **대용량 데이터 처리:** 웹 애플리케이션이 단일 ID 조회에 집중할 때, 배치는 수천, 수만 건의 데이터를 단 한 번의 실행으로 종결시킵니다.
* **정기적 및 자동화:** 시스템의 냉혹한 스케줄링에 따라 매일, 매주, 매달 정해진 시간에 인간의 개입 없이 실행됩니다.
* **유한성과 예측 가능성:** 처리해야 할 대상(오늘의 거래, 이번 달 매출 등)이 명확히 정해져 있으며, 시스템의 리소스(CPU, 메모리, SSD) 한계 내에서 부하를 견딜 수 있도록 신중하게 설계되어야 합니다.

### 2. 배치 시스템의 생존 조건

시스템의 한계를 넘어서는 데이터 처리는 곧 '시스템의 죽음'을 의미합니다. 따라서 배치는 **예측 가능한 부하 처리**를 목표로 설계되어야 하며, 이를 통해 시스템 다운 없이 안정적인 완결성을 보장합니다.

## 2장. 실전 전술: 데이터를 정복하는 6가지 처형 시나리오

실무에서 배치는 단순한 작업 그 이상의 전략적 가치를 지닙니다. 각 시나리오는 시스템의 신뢰성과 성능을 유지하기 위한 구체적인 작전입니다.

### 1. 핵심 운영 배치 시나리오

* **일일 정산 배치 (Daily Settlement):** 시스템의 가장 기본이 되는 작업으로, 자정마다 하루의 거래 내역과 활동 로그를 집계하여 데이터의 최종 상태를 확정합니다.
* **대용량 데이터 마이그레이션:** 시스템 업그레이드나 저장소 변경 시 수행되는 고난도 작전입니다. 수천만 건의 데이터를 안전하게 이동시키며, 실패 시 시스템 마비를 초래할 수 있는 만큼 정밀한 설계가 요구됩니다.
* **리포트 생성 배치 (Report Generation):** 의사결정에 필요한 데이터를 가공하여 이메일이나 알림으로 전달합니다. 정보의 가치를 창출하는 마지막 단계입니다.

### 2. 시스템 최적화 및 안정화 시나리오

* **데이터 정제 배치 (Data Cleansing):** 중복 데이터 제거 및 형식 통일을 통해 데이터 품질을 개선합니다. 이는 시스템의 신뢰성과 검색 성능을 높이는 기반이 됩니다.
* **백업 배치 (Regular Backup):** 장애 발생 시 서비스를 빠르게 정상화하기 위한 안전망입니다. 데이터 손실에 대비하는 시스템의 보험과 같습니다.
* **데이터 통합 배치 (Data Integration):** 흩어진 데이터 소스(CRM, ERP, 외부 API)를 수집하여 일관된 상태로 만듭니다. 이는 고객 행동 패턴 분석 및 중앙 데이터 웨어하우스 구축의 핵심입니다.

## 3장. 스프링 배치 아키텍처: 웹과의 결전 및 핵심 컴포넌트 분석

웹 애플리케이션이 최전방의 접점이라면, 배치는 후방의 보급과 정리를 담당합니다. 이 두 세계의 차이를 이해하는 것이 아키텍처 이해의 시작입니다.

### 1. 웹(Web) vs 배치(Batch) 기술 비교

| 기준 | 웹 애플리케이션 | 배치 애플리케이션 |
| :--- | :--- | :--- |
| **실행 방식** | 사용자 요청 기반 (실시간) | 스케줄 기반 (비대화식) |
| **응답 속도** | 즉각적 피드백 필수 | **정확성과 완결성** 우선 |
| **데이터 처리량** | 단일 요청 단위 | **대량 데이터** (수백, 수천만 건) |
| **오류 처리** | 즉시 에러 응답 반환 | **재시도 및 복구** 중심 |
| **리소스 사용** | 상시 실행 (대기 상태 유지) | 필요 시에만 실행 후 종료 |

### 2. 스프링 배치의 핵심 아키텍처 컴포넌트

스프링 배치는 배치 처리의 일반적인 패턴을 추상화하여 제공합니다.

* **Job & Step:**
  * **Job:** 하나의 완전한 배치 처리 작업 단위입니다.
  * **Step:** Job을 구성하는 세부 실행 단계입니다. 하나의 Job은 비즈니스 로직에 따라 여러 Step으로 쪼개질 수 있습니다.
* **ItemReader, ItemProcessor, ItemWriter:**
  * **ItemReader:** 다양한 데이터 소스(CSV, JSON, DB, NoSQL, Kafka 등)에서 데이터를 읽어옵니다.
  * **ItemProcessor:** 읽어온 데이터를 비즈니스 로직에 맞게 가공합니다.
  * **ItemWriter:** 가공된 데이터를 최종 저장소에 저장합니다.

### 3. 견고한 실행 제어 및 확장성 기법

스프링 배치는 대규모 처리를 위한 강력한 무기들을 탑재하고 있습니다.

* **Chunk 지향 처리:** 대량 데이터를 지정된 크기(Chunk)로 나누어 처리함으로써 메모리 사용을 최적화하고, 중간에 실패하더라도 해당 Chunk부터 다시 시작할 수 있게 합니다.
* **체크포인트와 재시작:** 긴 작업 중 안전 지점을 설정하여, 오류 발생 시 처음이 아닌 **마지막 성공 지점**부터 재개할 수 있는 강력한 복구 기능을 제공합니다.
* **확장성 (Scalability):** 멀티스레드 Step, 병렬 Step, 그리고 여러 서버로 작업을 분산하는 기법을 통해 엔터프라이즈급 데이터 처리를 수행합니다.
* **유연한 데이터 소스 지원:** `FlatFileItemReader`(파일), `JdbcCursorItemReader`(DB), `JsonItemReader`(JSON) 등 이미 구현된 컴포넌트들을 설정만으로 교체하여 사용할 수 있습니다.

## [개념 검증용 자가 진단 질문]

### 배치 아키텍처 및 메커니즘 심화 진단

1. 웹 애플리케이션은 오류 발생 시 즉시 에러 코드를 반환하고 종료하는 방식을 취하지만, 스프링 배치는 왜 '재시도(Retry)'와 '체크포인트(Checkpoint)' 기반의 복구 메커니즘을 핵심 아키텍처로 채택하고 있습니까? 대용량 데이터 처리의 특성과 연계하여 설명하시오.
2. 스프링 배치의 'Chunk 지향 처리' 방식이 대용량 데이터 환경에서 JVM 메모리 관리와 트랜잭션의 안정성에 어떤 기술적 이점을 제공하며, 만약 이 구조를 사용하지 않고 전체 데이터를 한 번에 처리(Tasklet 방식 등)할 때 발생할 수 있는 위험 요소는 무엇입니까?
3. `ItemReader`와 `ItemWriter`가 인터페이스로 추상화되어 있고 스프링 배치가 다양한 구현체를 제공하는 구조적 이유는 무엇입니까? 데이터 소스가 파일(CSV)에서 데이터베이스(RDB)로 변경되는 상황을 가정하여, 이 구조가 시스템의 유지보수성과 유연성에 미치는 영향을 논술하시오.

* **1번 질문 해설:** 웹 애플리케이션은 사용자와의 실시간 상호작용이 핵심이므로 실패 시 즉시 알리는 것이 중요합니다. 그러나 배치는 수천만 건의 데이터를 처리하며 실행 시간이 매우 깁니다. 만약 90% 완료 시점에서 오류가 났을 때 처음부터 다시 시작해야 한다면 막대한 시간과 리소스 낭비가 발생합니다. 따라서 스프링 배치는 마지막 성공 지점을 메타데이터로 기록하는 **체크포인트** 기능을 통해 실패한 지점부터 재개하는 **완결성**과 **신뢰성**을 보장하기 위해 이 구조를 채택합니다.

* **2번 질문 해설:** Chunk 지향 처리는 데이터를 일정한 덩어리로 나누어 **'읽기-가공-쓰기'** 주기를 반복합니다. 이는 한꺼번에 수백만 건을 메모리에 올릴 필요가 없게 하여 **OutOfMemoryError를 방지**합니다. 또한, 각 Chunk 단위로 트랜잭션을 커밋함으로써, 오류 발생 시 전체 작업을 롤백하는 대신 해당 Chunk만 처리되지 않은 상태로 남겨 효율적인 트랜잭션 관리가 가능하게 합니다. 전체 처리를 시도할 경우, 단 하나의 데이터 오류로 수 시간의 작업이 모두 무효화되고 DB 락(Lock) 점유 시간이 길어지는 치명적인 리스크가 발생합니다.

* **3번 질문 해설:** 이는 **의존성 주입(DI)**과 **전략 패턴**을 활용한 설계입니다. 비즈니스 로직(ItemProcessor)은 데이터가 어디서 오고(Reader) 어디로 가는지(Writer) 알 필요가 없습니다. 소스가 파일에서 DB로 변경되더라도 `FlatFileItemReader`를 `JdbcCursorItemReader`로 설정만 교체하면 되므로, 핵심 로직의 변경 없이 데이터 소스 확장이 가능합니다. 이는 기술적 변화에 코드가 독립적으로 살아남게 하는 **유지보수성**의 극치입니다.
