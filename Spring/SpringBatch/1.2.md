# [심화 분석 학습 노트] 스프링 배치: 잡 파라미터와 스코프의 지배 구조

## 1장. 배치의 운명을 결정하는 통제 변수: JobParameters

배치에서 파라미터가 없는 작업은 목적지 없이 떠도는 좀비 프로세스나 다름없습니다. **JobParameters**는 배치가 어떤 데이터를, 어떤 조건으로 처리할지 결정하는 핵심 메커니즘입니다.

### 1. 프로퍼티(-D)와 JobParameters의 결정적 차이

단순히 값을 전달하는 용도라면 시스템 프로퍼티로도 충분해 보이겠지만, 스프링 배치는 그 이상의 가치를 위해 JobParameters를 사용합니다.

* **입력값의 동적 변경:** 프로퍼티는 애플리케이션 시작 시 한 번 주입되는 정적인 값입니다. 반면 JobParameters는 실행 시점마다 다른 값을 주입할 수 있어, 동일한 Job을 다양한 조건으로 유연하게 실행할 수 있게 합니다.
* **메타데이터의 영속성:** 스프링 배치는 JobParameters의 모든 값을 `JobRepository`를 통해 DB에 기록합니다. 이는 **Job 인스턴스를 식별**하고, 실패한 작업을 정확한 지점부터 **재시작**하는 근거가 됩니다. 프로퍼티는 기록되지 않으므로 운영 및 이력 관리가 불가능합니다.

### 2. 커맨드라인을 통한 파라미터 지배 (표기법 분석)

실무에서는 스케줄러를 통한 실행이 잦으므로 커맨드라인 표기법 숙지가 필수입니다.

* **기본 형식:** `parameterName=parameterValue,parameterType,identificationFlag`
* **구성 요소:**
  * `parameterName`: Job 내에서 접근할 Key.
  * `parameterValue`: 실제 값.
  * `parameterType`: 데이터 타입(FQCN 사용). 생략 시 `String`.
  * `identificationFlag`: `true`(기본값)일 경우 JobInstance 식별에 사용됨.

#### [기본 타입 파라미터 실전 코드 분석]

```java
@Slf4j
@Configuration
public class SystemTerminatorConfig {
    // Job 구성: Step과 JobRepository를 주입받아 배치 작업의 최상위 단위를 정의함
    @Bean
    public Job processTerminatorJob(JobRepository jobRepository, Step terminationStep) {
        return new JobBuilder("processTerminatorJob", jobRepository)
            .start(terminationStep)
            .build();
    }

    // Step 구성: Tasklet과 트랜잭션 매니저를 통해 구체적인 실행 단위를 정의함
    @Bean
    public Step terminationStep(JobRepository jobRepository, PlatformTransactionManager transactionManager, Tasklet terminatorTasklet) {
        return new StepBuilder("terminationStep", jobRepository)
            .tasklet(terminatorTasklet, transactionManager)
            .build();
    }

    // Tasklet 구성: @StepScope를 통해 런타임에 파라미터를 동적으로 주입받음
    @Bean
    @StepScope
    public Tasklet terminatorTasklet(
        @Value("#{jobParameters['terminatorId']}") String terminatorId, // String 타입 주입
        @Value("#{jobParameters['targetCount']}") Integer targetCount   // Integer 타입 주입
    ) {
        return (contribution, chunkContext) -> {
            log.info("시스템 종결자 정보:");
            log.info("ID: {}", terminatorId);
            log.info("제거 대상 수: {}", targetCount);
            log.info("⚡ SYSTEM TERMINATOR {} 작전을 개시합니다.", terminatorId);
            log.info("☠️ {}개의 프로세스를 종료합니다.", targetCount);
            return RepeatStatus.FINISHED;
        };
    }
}
```

이 코드는 커맨드라인에서 `terminatorId=KILL-9,java.lang.String targetCount=5,java.lang.Integer`와 같이 전달된 값을 내부적으로 `DefaultJobParametersConverter`를 통해 변환하여 주입받습니다.

### 3. 다양한 데이터 타입과 JSON 표기법의 등장

스프링 배치 5는 `LocalDate`, `LocalDateTime`, `Enum` 등을 기본 지원합니다. 특히 값 안에 쉼표(,)가 포함되어 파라미터 해석이 꼬이는 경우(예: `targets=A,B,java.lang.String`)를 위해 **JSON 기반 표기법**을 제공합니다.

#### [JSON 표기법 및 Enum 처리 분석]

```java
// Enum 정의: 타입 안정성 확보
public enum QuestDifficulty { EASY, NORMAL, HARD, EXTREME }

@Bean
@StepScope
public Tasklet terminatorTasklet(
    @Value("#{jobParameters['questDifficulty']}") QuestDifficulty questDifficulty, // Enum 자동 변환
    @Value("#{jobParameters['infiltrationTargets']}") String infiltrationTargets // JSON 내 쉼표 포함 문자열
) {
    return (contribution, chunkContext) -> {
        log.info("임무 난이도: {}", questDifficulty);
        String[] targets = infiltrationTargets.split(","); // "판교,안산" 문자열 분리
        return RepeatStatus.FINISHED;
    };
}
```

* **실행 명령:** `infiltrationTargets='{"value": "판교,안산", "type": "java.lang.String"}'`
* **내부 동작:** `JsonJobParametersConverter`를 빈으로 등록해야 하며, 이는 `ObjectMapper`를 사용하여 JSON 형식을 해석합니다.

### 4. 파라미터 전달의 내부 흐름 (Internal Flow)

스프링 부트와 결합된 배치는 다음과 같은 정밀한 과정을 거칩니다.

1. **JobLauncherApplicationRunner** 가동: 모든 Job 빈을 주입받고 유효성을 검증합니다.
2. **명령어 해석:** `converter.getJobParameters(properties)`를 호출하여 문자열을 객체화합니다.
3. **Job 실행:** `jobLauncher.run(job, parameters)`를 호출하여 실제 배치를 구동합니다.

## 2장. 배치의 은밀한 비밀: JobScope와 StepScope

스프링 배치는 싱글톤 중심의 스프링 컨테이너 위에서 **지연 바인딩(Late Binding)**과 **동시성 제어**를 위해 특별한 스코프를 운영합니다.

### 1. 스코프의 본질: 생명주기와 프록시

`@JobScope`와 `@StepScope`가 선언된 빈은 애플리케이션 구동 시점에 실제 인스턴스가 생성되지 않습니다. 대신 **프록시(Proxy) 객체**가 자리를 차지하고 있다가, 실제 실행 시점에 `JobParameters`를 바인딩하며 인스턴스를 생성합니다.

* **이점:**
  * 실행 시점에만 결정되는 파라미터를 빈 생성 시 주입 가능.
  * 동일한 Job을 여러 스레드에서 실행하더라도 독립적인 빈 인스턴스를 보장하여 동시성 이슈 해결.
  * 실행 완료 후 즉시 소멸되어 메모리 효율성 극대화.

### 2. 치명적 주의사항: Step 빈에는 선언하지 마라

`Step` 빈 자체에 `@JobScope`나 `@StepScope`를 선언하면 안 됩니다.

* **이유:** 스프링 배치는 스텝이 실행되기 전 메타데이터 관리를 위해 스텝 빈에 접근해야 합니다. 하지만 이때는 아직 스코프가 활성화되지 않은 시점이므로 `ScopeNotActiveException`이 발생하며 시스템이 폭발합니다.
* **해결책:** 파라미터 주입이 필요한 `Tasklet`, `Reader`, `Writer` 등에 스코프를 선언하고, `Step`은 이를 주입받는 구조를 유지해야 합니다.

#### [올바른 구조의 스텝 구성 예시]

```java
@Bean
public Step systemDestructionStep(SystemInfiltrationTasklet tasklet) {
    // Step 빈은 싱글톤으로 유지하되, 내부 구성 요소인 Tasklet에 스코프를 할당함
    return new StepBuilder("systemDestructionStep", jobRepository)
        .tasklet(tasklet, transactionManager)
        .build();
}

@Component
@StepScope // Tasklet 레벨에서 스코프 정의
public class SystemInfiltrationTasklet implements Tasklet {
    private final String targetSystem;

    public SystemInfiltrationTasklet(@Value("#{jobParameters['targetSystem']}") String targetSystem) {
        this.targetSystem = targetSystem;
    }
    // ... execute 구현
}
```

## 3장. 데이터 컨테이너의 지배: ExecutionContext

`JobParameters`가 외부에서 주입되는 불변의 통제 변수라면, **ExecutionContext**는 실행 중에 발생하는 커스텀 데이터를 담는 동적 컨테이너입니다.

### 1. 데이터의 독립성과 공유

* **JobExecutionContext:** Job 전체에서 공유됩니다. `@Value("#{jobExecutionContext['key']}")`로 접근합니다.
* **StepExecutionContext:** 해당 Step 내에서만 유효합니다. `@Value("#{stepExecutionContext['key']}")`로 접근합니다.
* **핵심 원칙:** 하위 수준(Step)의 데이터를 상위 수준(Job)에서 직접 가져올 수 없으며, 서로 다른 Step 간의 데이터 침범을 엄격히 제한하여 데이터 독립성을 보장합니다. 단, Step 간 공유가 필요할 경우 Job 수준의 컨테이너를 전략적으로 활용합니다.

## [개념 검증용 자가 진단 질문]

### 잡 파라미터 및 배치 스코프 아키텍처 심화 진단

1. 왜 스프링 배치는 `JobParameters`를 단순한 애플리케이션 인자로 처리하지 않고 `JobRepository`를 통해 DB에 영속화하는가? 특히 `JobInstance` 식별 및 재시작(Restart) 메커니즘과 연계하여 그 이유를 기술하시오.
2. `@StepScope`가 선언된 빈이 애플리케이션 구동 시점이 아닌 런타임에 주입되는 파라미터를 받을 수 있는 내부적 기술 원리(프록시와 지연 바인딩)를 설명하고, 왜 `Step` 빈 자체에 이 어노테이션을 붙이면 `ScopeNotActiveException`이 발생하는지 그 인과관계를 논술하시오.
3. `ExecutionContext`와 `JobParameters`는 모두 배치 실행 정보를 담는 객체이지만, 데이터의 '변경 가능성'과 '사용 목적' 측면에서 결정적인 차이가 있다. 두 객체의 특징을 비교하고, 실패한 배치를 재시작할 때 `ExecutionContext`가 어떤 역할을 수행하여 중단 지점부터 처리를 가능케 하는지 설명하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** 스프링 배치는 동일한 Job이더라도 전달된 `JobParameters`가 다르면 서로 다른 `JobInstance`로 간주합니다. 이를 DB에 기록함으로써 시스템은 "어떤 파라미터로 실행된 작업이 성공했거나 실패했는지"를 추적할 수 있습니다. 만약 작업이 실패했다면, `JobRepository`에 저장된 파라미터와 실행 상태를 근거로 실패한 해당 `JobInstance`를 찾아내어 중복 실행을 방지하거나 실패 지점부터 다시 실행할 수 있는 논리적 근거를 확보하게 됩니다.

* **2번 질문 해설:** `@StepScope` 빈은 애플리케이션 구동 시 실제 인스턴스가 아닌 CGLIB 프록시 객체로 생성됩니다. 실제 메서드 호출이 일어나는 시점에야 비로소 `JobParameters`를 참조하여 실제 빈을 생성(지연 바인딩)하기 때문에 런타임 파라미터 주입이 가능합니다. 그러나 `Step` 빈 자체에 스코프를 선언하면, 스프링 배치가 스텝을 초기화하기 위해 빈에 접근할 때 아직 해당 스텝의 실행 컨텍스트(스코프)가 활성화되지 않은 상태이므로 프록시가 실제 인스턴스를 생성하지 못해 예외가 발생합니다.

* **3번 질문 해설:** `JobParameters`는 실행 시점에 결정되어 변경되지 않는(Immutable) 통제 변수인 반면, `ExecutionContext`는 실행 도중 발생하는 데이터나 상태를 저장하는 가변적(Mutable) 컨테이너입니다. 배치가 실패한 후 재시작될 때, 스프링 배치는 DB에 저장되어 있던 `ExecutionContext`를 복원합니다. 예를 들어 대용량 처리 중 마지막으로 성공한 인덱스 번호를 여기에 저장해두었다면, 재시작 시 이 값을 복원하여 처음이 아닌 해당 인덱스 다음부터 처리를 이어갈 수 있습니다.

</details>
