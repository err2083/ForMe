# [심화 분석 학습 노트] 스프링 배치: Flow 지배와 상태 머신의 아키텍처

## 1장. Flow의 본질: 순차 처리를 넘어선 지능형 분기

지금까지 우리가 다룬 배치는 `start() -> next()`로 이어지는 단순 순차 실행이었습니다. 하지만 실전 전장에서는 이전 단계의 결과에 따라 완전히 다른 경로를 선택해야 하는 복잡한 상황이 발생합니다.

### 1. 단순 순차 실행의 한계

순차 실행은 정직하지만 유연하지 못합니다.

```java
// 단순 순차 실행 Job? 이걸론 전쟁에서 살아남을 수 없다.
@Bean
public Job lectureScanSimpleJob(JobRepository jobRepository, Step analyzeContentStep, Step publishLectureStep) {
    return new JobBuilder("lectureScanSimpleJob", jobRepository)
        .start(analyzeContentStep)   // 1. 강의 콘텐츠 분석으로 시작
        .next(publishLectureStep)    // 2. 성공하면 강의 게시 처리
        .build();                    // 끝.
}
```

* **분석:** 위 코드는 `analyzeContentStep`이 반드시 성공(`COMPLETED`)해야만 다음 단계로 넘어갑니다. 만약 실패하거나 특정 비즈니스 조건이 발생했을 때 대체 경로를 타는 것이 불가능합니다.

### 2. 작전의 핵심: 상태 머신(State Machine)으로서의 Flow

Flow의 동작 원리는 **"현재 상태에서 특정 조건이 만족되면 다음 상태로 이동하는 시스템"**인 상태 머신과 완벽히 일치합니다.

* **상태(State):** 작업이 머무르는 지점 (StepState, EndState).
* **조건(ExitCode):** 전이를 결정하는 분기 기준 (ExitStatus.exitCode).
* **전이(Transition):** 조건에 따른 이동 명령서.

## 2장. Flow DSL과 전술적 구현: 조건부 분기의 메커니즘

전용 DSL(Domain Specific Language)을 사용하여 배치의 두뇌를 설계하는 방법을 파헤칩니다.

### 1. 기본적인 조건부 분기 Flow 구현

```java
@Bean
public Job lectureScanConditionalJob(JobRepository jobRepository, 
                                     Step analyzeContentStep,      // State 1
                                     Step publishLectureStep,      // State 2
                                     Step summarizeFailureStep) {  // State 3
    return new JobBuilder("lectureScanConditionalJob", jobRepository)
        .start(analyzeContentStep) // 초기 State 설정
            .on("COMPLETED").to(publishLectureStep) // 성공 시 게시 Step으로 전이
        .from(analyzeContentStep) // 기준점을 다시 analyzeContentStep으로 설정
            .on("FAILED").to(summarizeFailureStep) // 실패 시 요약 Step으로 전이
        .end() // Flow 정의 완료
        .build();
}
```

* **.on("EXIT_CODE"):** 이전 단계의 실행 결과(`ExitStatus.exitCode`)를 감시합니다.
* **.to(Step):** 조건이 일치할 때 이동할 다음 지점을 지정합니다.
* **.from(Step):** 하나의 상태에서 여러 갈래의 길을 낼 때, 기준점을 다시 명확히 설정하는 역할을 합니다.

### 2. 커스텀 ExitCode를 통한 정밀 타격

스프링 배치가 기본 제공하는 `COMPLETED`, `FAILED`만으로는 부족합니다. 비즈니스 상황을 대변하는 커스텀 코드가 필요합니다.

#### (1) 태스크릿에서의 커스텀 설정

```java
try {
    // 초상화 출력 작업 💀 ..
} catch (IllegalStateException e) {
    contribution.setExitStatus(new ExitStatus("SKULL_FRACTURE", e.getMessage()));
    throw e;
} catch (ValidationException e) {
    contribution.setExitStatus(new ExitStatus("SYSTEM_BRUTALIZED", e.getMessage()));
    throw e;
} catch (IllegalArgumentException e) {
    contribution.setExitStatus(new ExitStatus("UNKNOWN_CHAOS", e.getMessage()));
    throw e;
}
```

* **분석:** `contribution.setExitStatus()`를 통해 프레임워크에게 현재 상황의 고유 코드를 보고합니다.

#### (2) 청크 지향 처리에서의 침투 (StepExecutionListener)

청크 지향 처리에서는 `StepContribution`에 직접 접근하기 어렵습니다. 따라서 `afterStep` 지점을 공략합니다.

```java
@Component
public class DataDeathCountListener implements StepExecutionListener {
    private static final int CRITICAL_SKIP_THRESHOLD = 10;
    
    @Override
    public ExitStatus afterStep(StepExecution stepExecution) {
        // 스킵 횟수가 임계치를 넘으면 커스텀 ExitStatus 반환
        if (stepExecution.getSkipCount() >= CRITICAL_SKIP_THRESHOLD) {
            return new ExitStatus("DATA_MASSACRE");
        }
        return null; // null 반환 시 기존 ExitStatus 유지
    }
}
```

* **내부 동작:** `AbstractStep.execute()`의 마지막 라인인 `exitStatus = exitStatus.and(...)` 지점에서 리스너의 반환값이 최종 `ExitCode`를 결정하게 됩니다.

## 3장. 심화 아키텍처: 와일드카드, 우선순위, 그리고 결정자(Decider)

### 1. 와일드카드: 패턴 한 방으로 다량의 ExitCode 처형

* **`*` (애스터리스크):** 어떤 문자열(빈 문자열 포함)과도 매칭. 모든 나머지 경우를 처리할 때 유용합니다.
* **`?` (물음표):** 정확히 한 개의 문자와 매칭. 코드 길이가 정해진 패턴에 유리합니다.

### 2. 전이 우선순위(Transition Priority)의 법칙

여러 패턴이 동시에 매칭될 때, 시스템은 가장 구체적인 것을 선택합니다.

* **우선순위 계급:** `정확한 문자열` > `? 포함 패턴` > `* 포함 패턴`.
* **예시:** `ERROR_1` > `ERROR_?` > `ERROR_*`. `?`는 문자 개수를 제한하므로 `*`보다 더 구체적인 것으로 간주됩니다.

### 3. 터미널 상태(Terminal State)의 암시적 규칙

개발자가 `.on().to()`를 정의하지 않은 막다른 지점을 '터미널 상태'라고 부릅니다.

* **암시적 전이:**
    1. 결과가 `COMPLETED`면 -> Job 성공 종료.
    2. 결과가 그 외(`FAILED` 등)면 -> Job 실패 종료.
* **주의:** 한 번이라도 `.on()`을 사용한 상태는 더 이상 터미널 상태가 아니므로, 발생 가능한 모든 `ExitCode`에 대한 경로를 명시해야 합니다. 누락 시 **FlowExecutionException**이 발생합니다.

### 4. JobExecutionDecider: 운명을 결정하는 전용 사령부

Step 내부에 분기 로직을 섞는 것은 '관심사의 분리' 위반입니다. 이를 해결하기 위해 전용 컴포넌트를 투입합니다.

```java
@Component
public static class StudentReviewDecider implements JobExecutionDecider {
    @Override
    public FlowExecutionStatus decide(JobExecution jobExecution, StepExecution stepExecution) {
        int score = (int) stepExecution.getExecutionContext().get("reviewScore");
        if (score >= 10) return new FlowExecutionStatus("EXCELLENT_COURSE");
        if (score >= 5) return new FlowExecutionStatus("AVERAGE_COURSE");
        return new FlowExecutionStatus("NEEDS_IMPROVEMENT");
    }
}
```

* **DecisionState:** `Decider`가 반환하는 `FlowExecutionStatus`를 기반으로 분기가 일어납니다. Step은 오직 데이터 처리에만 집중할 수 있게 됩니다.

### 5. Flow 외부화와 재사용

동일한 Flow 패턴을 여러 Job에서 사용할 때, 이를 빈(Bean)으로 독립시켜 모듈화할 수 있습니다.

```java
// 패턴 2: FlowStep으로 위장시켜 하나의 Step처럼 관리
@Bean
public Step validationStep(JobRepository jobRepository, Flow lectureValidationFlow) {
    return new StepBuilder("validationStep", jobRepository)
        .flow(lectureValidationFlow) // 복잡한 Flow를 하나의 Step으로 포장
        .build();
}
```

### 6. [부록] Spring Batch 5.1의 취약점 (고고학적 유물)

5.1 버전에서는 `DefaultStateTransitionComparator`의 버그로 인해 `*` 문자 개수가 같으면 알파벳 순으로 우선순위를 매기는 오판이 있었습니다. 이를 위해 `*` 대신 `**`를 사용하는 우회 전술이 필요했으나, 5.2 버전에서 수정되었습니다.

## [개념 검증용 자가 진단 질문]

### Flow 아키텍처 및 분기 제어 메커니즘 진단

1. `analyzeLectureStep`에서 `.on("APPROVED").to(successStep)`와 같이 특정 조건에 대한 분기를 설정했을 때, 만약 예상치 못한 `QUALITY_SUBSTANDARD`라는 `ExitCode`가 발생하면 왜 `FlowExecutionException`이 발생하는지 '비터미널 상태(Non-terminal State)'의 전이 규칙 관점에서 논술하시오.
2. `FlowExecutionStatus`와 `ExitStatus`는 서로 다른 객체임에도 불구하고 어떻게 상호작용하여 Job의 최종 `BatchStatus`를 결정하는가? 특히 `EndState.handle()`에서 반환된 값이 `JobFlowExecutor.updateJobExecutionStatus()`를 통해 전파되는 과정을 기술하시오.
3. `StepExecutionListener`를 통해 `ExitCode`를 조작하여 분기하는 방식과 `JobExecutionDecider`를 사용하는 방식의 아키텍처적 차이점은 무엇이며, 왜 복잡한 배치 환경에서는 `JobExecutionDecider`가 '명시성' 측면에서 더 우월한지 설명하시오.

<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** 스프링 배치는 스텝에 `.on()` 메서드를 한 번이라도 사용하는 순간 해당 상태를 '비터미널 상태'로 간주합니다. 터미널 상태는 다음 경로가 없을 때 프레임워크가 암시적 종료 규칙을 적용해주지만, 비터미널 상태는 개발자가 직접 모든 경로를 설계하겠다는 의사 표시로 받아들입니다. 따라서 정의되지 않은 `ExitCode`가 발생하면 시스템은 다음 목적지를 찾지 못해 'Next state not found' 예외를 던지며 자폭하게 됩니다. 이를 방지하려면 와일드카드(`*`)를 사용하여 예외적인 경로를 모두 수용하거나 모든 조건을 명시해야 합니다.

* **2번 질문 해설:** Step이 종료되면 `ExitStatus`의 `exitCode` 문자열을 반환하고, 시스템은 이를 기반으로 `FlowExecutionStatus` 객체를 생성합니다. Flow가 최종적으로 `EndState`에 도달하면 이 객체가 `JobFlowExecutor`에게 전달됩니다. `JobFlowExecutor`는 이 상태값의 이름을 확인하여 `findBatchStatus()` 메서드를 통해 `BatchStatus.values()` 중 일치하는 시작 문자열(예: "COMPLETED", "FAILED")이 있는지 검사합니다. 일치하는 것을 찾으면 Job의 `BatchStatus`를 설정하고, `FlowExecutionStatus`의 이름 그대로를 Job의 최종 `ExitStatus`로 각인시킵니다.

* **3번 질문 해설:** `StepExecutionListener` 방식은 Step 내부(혹은 설정)에 숨겨져 있어 Job 구성 코드만 봐서는 분기가 일어난다는 사실을 알기 어렵습니다(은폐된 로직). 반면 `JobExecutionDecider`는 Flow DSL 상에 `.next(decider)`와 같이 명시적으로 등장하여 "이 지점에서 의사결정이 일어난다"는 것을 선언합니다. 또한 리스너는 Step의 생명주기에 기생하지만, 결정자는 오직 분기만을 위한 전용 컴포넌트이므로 비즈니스 로직과 흐름 제어 로직을 완벽히 분리(관심사의 분리)할 수 있게 해줍니다.

</details>
