# [심화 분석 학습 노트] 스프링 배치: 데이터 숙청과 내결함성 지배 아키텍처

## 1장. ItemProcessor - 읽기와 쓰기 사이, 데이터 숙청의 집행

ItemProcessor는 Reader와 Writer 사이에서 우리의 비즈니스 로직이 실질적으로 집행되는 전장입니다. 여기서 데이터는 변환되거나, 보강되거나, 혹은 영구히 말살됩니다.

### 1. 데이터 필터링: null 반환을 통한 처형

`ItemProcessor.process()` 메서드가 `null`을 반환하는 것은 단순한 로직이 아닌 **데이터의 말살**을 의미합니다.

* **내부 동작 메커니즘:**
    1. **ItemReader:** 지정된 청크 사이즈만큼 `read()`를 호출하여 **input Chunk**를 생성합니다.
    2. **ItemProcessor:** input Chunk의 각 아이템에 대해 `process()`를 호출합니다.
    3. **숙청(Filtering):** `process()`가 `null`을 반환하면 해당 아이템은 **output Chunk**에서 즉시 제외됩니다.
    4. **ItemWriter:** 필터링되어 크기가 작아진 output Chunk를 전달받아 기록을 수행합니다.

#### [실전 코드: 시스템 파괴 명령어 감시자]

```java
@Slf4j
public class ExecutionerProcessor implements ItemProcessor<Command, Command> {
    @Override
    public Command process(Command command) {
        // 시스템 파괴 명령어(rm -rf / 또는 kill -9) 감지 시 null 반환하여 기록에서 말살
        if (command.getCommandText().contains("rm -rf /") || command.getCommandText().contains("kill -9")) {
            log.info("☠️ {}의 {} -> 시스템 파괴자 처단 완료. 기록에서 말살.", command.getUserId(), command.getCommandText());
            return null; // 처형 집행
        }
        return command; // 생존한 데이터만 반환
    }
}
```

### 2. 데이터 검증: ValidatingItemProcessor

데이터가 유효하지 않을 때 단순히 거를 것인지, 아니면 전체 작업을 중단시킬 것인지는 `setFilter()` 설정에 달려 있습니다.

#### [Validator 구현 및 프로세서 설정]

```java
// Spring Batch 전용 Validator 인터페이스 구현
public class CommandValidator implements Validator<Command> {
    @Override
    public void validate(Command command) throws ValidationException {
        if (command.getCommandText().contains("rm -rf /") || command.getCommandText().contains("kill -9")) {
            throw new ValidationException(
                "☠️ " + command.getUserId() + "의 " + command.getCommandText() + " → 시스템 파괴 명령어 감지. 처단.");
        }
    }
}

// 빈 설정 및 필터 전략 결정
@Bean
public ItemProcessor<Command, Command> commandProcessor() {
    ValidatingItemProcessor<Command> processor = new ValidatingItemProcessor<>(new CommandValidator());
    
    // true: ValidationException 발생 시 null 반환 (필터링)
    // false: 예외를 상위로 전파하여 Step/Job 즉시 실패 처리 (기본값)
    processor.setFilter(true); 
    
    return processor;
}
```

### 3. 데이터 변환(Transformation) vs 보강(Enrichment)

* **변환:** 형태를 바꾸는 것. `ItemProcessor<I, O>` 제네릭을 통해 입력 타입 `I`를 출력 타입 `O`로 변경합니다.
* **보강:** 힘을 더하는 것. 외부 소스(API, DB)에서 추가 정보를 가져와 데이터에 주입합니다.

#### [성능 최적화: N+1 문제와 리스너 활용]

ItemProcessor에서 건당 외부 API를 호출하면 100만 건 기준 27.7시간이 소요될 수 있습니다. 이를 해결하기 위해 `ItemWriteListener`를 사용합니다.

```java
@Slf4j
@RequiredArgsConstructor
public class SystemLogEnrichListener implements ItemWriteListener<SystemLog> {
    private final ObservabilityApiClient observabilityApiClient;

    @Override
    public void beforeWrite(Chunk<? extends SystemLog> items) {
        // Chunk 전체를 입력으로 받아 벌크 API 호출로 통신 횟수를 1/100로 단축
        List<SystemLog> logList = (List<SystemLog>) items.getItems();
        // 외부 API 호출 후 데이터 보강 수행
    }
}
```

## 2장. 재시도(Retry) - 반항하는 예외를 제압하는 법

배치 실행 중 일시적인 네트워크 오류나 타임아웃은 '자비'를 베풀어 다시 시도하면 해결될 가능성이 높습니다. 스프링 배치는 `RetryTemplate`을 통해 이 고문을 집행합니다.

### 1. RetryTemplate의 내부 메커니즘

내결함성 모드(`faultTolerant()`)가 활성화되면, ItemProcessor와 ItemWriter의 호출 로직은 `RetryTemplate.execute()` 내부의 `retryCallback`으로 패키징됩니다.

* **canRetry():** `RetryPolicy`를 통해 재시도 가능 여부를 심판합니다.
* **retryCallback:** 실제 비즈니스 로직이 담긴 콜백입니다. 최초 실행과 재시도 모두 이곳에서 수행됩니다.
* **recoveryCallback:** 재시도 한계를 초과했을 때 호출되는 최후의 수단입니다. 기본적으로 예외를 던지거나 건너뛰기(Skip) 로직으로 연결됩니다.

### 2. 재시도 설정의 정석

```java
@Bean
public Step terminationRetryStep() {
    return new StepBuilder("terminationRetryStep", jobRepository)
        .<Scream, Scream>chunk(3, transactionManager)
        .reader(terminationRetryReader())
        .processor(terminationRetryProcessor())
        .writer(terminationRetryWriter())
        .faultTolerant() // 내결함성 기능 활성화
        .retry(TerminationFailedException.class) // 재시도 대상 예외 지정
        .retryLimit(3) // 총 시도 횟수 (최초 1회 + 재시도 2회)
        .backOffPolicy(new ExponentialBackOffPolicy() {{ // 백오프 설정
            setInitialInterval(1000L); // 1초 대기 후 점진적 증가
            setMultiplier(2.0);
        }})
        .build();
}
```

### 3. 컴포넌트별 재시도 동작의 차이

* **ItemProcessor:** **아이템 단위**로 재시도 횟수를 관리합니다. 실패한 아이템만 타겟팅하여 고문을 반복합니다.
* **ItemWriter:** **청크 단위**로 재시도 횟수를 관리합니다. 쓰기는 묶음으로 일어나므로, 특정 아이템이 실패해도 청크 전체의 시도 횟수가 카운팅됩니다.
* **ItemReader:** **재시도 대상이 아닙니다.** 메시지 큐(RabbitMQ 등)처럼 한 번 읽으면 데이터가 사라지는 'Mutable'한 데이터 소스까지 고려했기 때문입니다.

#### [processorNonTransactional()의 중요성]

재시도 시 이미 성공한 아이템들이 반복 실행되는 것을 막으려면 캐시를 활용해야 합니다.

```java
.processorNonTransactional() // ItemProcessor 결과를 캐싱하여 중복 처리 방지
```

이 설정은 트랜잭션 롤백을 막는 것이 아니라, **"동일 입력에 대해 항상 동일 결과(멱등성)"**를 전제로 이전 처리 결과를 재사용하게 하여 성능을 최적화합니다.

## 3장. 건너뛰기(Skip)와 롤백 사면령(noRollback)

재시도로도 해결되지 않는 영구적인 오류 데이터는 시스템의 전체 생존을 위해 '무시'해야 합니다.

### 1. 스캔 모드(Scan Mode): ItemWriter의 스킵 전략

ItemWriter는 청크 단위로 동작하므로 어떤 놈이 범인인지 즉시 알 수 없습니다. 예외 발생 시 시스템은 **스캔 모드**로 돌입합니다.

* **스캔 모드 프로세스:**
    1. **롤백:** 현재 청크 트랜잭션을 취소합니다.
    2. **개별 처리:** 청크 내 모든 아이템을 **Processor -> Writer -> 즉시 커밋** 순으로 하나씩 실행합니다.
    3. **검거:** 개별 실행 중 예외가 발생한 놈을 `SkipPolicy`에 따라 건너뜁니다.
    4. **복귀:** 모든 개별 처리가 끝나면 다시 일반 청크 모드로 돌아갑니다.

### 2. SkipListener와 비직관적 호출 순서

생존자(Skip된 아이템)를 추적하기 위해 리스너를 장착하지만, 호출 순서가 독특합니다.

* **순서:** `onSkipInProcess()` → `onSkipInWrite()` → `onSkipInRead()`.
* **특징:** 재시도 발생 즉시 호출되는 RetryListener와 달리, SkipListener는 **청크 쓰기 완료 후 일괄적으로 호출**됩니다.

### 3. noRollback(): 롤백 사면령

모든 예외가 롤백될 만큼 치명적인 것은 아닙니다. `noRollback()`은 특정 예외에 대해 트랜잭션을 되돌리지 말라고 명령합니다.

* **ItemReader:** 건너뛰기가 우선이며, 건너뛰기 불가능한 경우 롤백 없이 무시합니다. (통계는 남지 않음)
* **ItemProcessor:** **"롤백하지 말고 건너뛰어라"**는 의미입니다. 따라서 `noRollback`에 지정한 예외는 **반드시 `skip()` 설정에도 포함**되어야 합니다. 그렇지 않으면 `NonSkippableProcessException`과 함께 시스템이 폭발합니다.
* **ItemWriter:** 스캔 모드 중 `noRollback` 예외가 터지면 그냥 없던 일 취급하고 다음 아이템으로 넘어갑니다.

## [개념 검증용 자가 진단 질문]

### 배치 내결함성 및 실행 메커니즘 심화 진단

1. `ItemProcessor`에서 발생한 예외로 인해 재시도가 일어날 때, 왜 `processorNonTransactional()` 설정을 권장하는가? 이 설정이 활성화되었을 때와 비활성화되었을 때, `ItemProcessor` 내부의 비즈니스 로직(예: 외부 API 호출)의 실행 횟수가 어떻게 달라지는지 인과관계를 설명하시오.
2. `ItemWriter`에서 예외가 발생하여 '스캔 모드(Scan Mode)'가 활성화되었을 때, 1,000개의 아이템을 가진 청크에서 500번째 아이템이 실패한다면 DB 커밋은 총 몇 번 발생하는가? 또한, 이 방식이 일반적인 청크 처리 방식에 비해 성능상 치명적인 오버헤드를 갖게 되는 아키텍처적 이유를 논술하시오.
3. `ItemProcessor`에서 특정 예외를 `noRollback()`으로 설정했음에도 불구하고, 해당 예외를 `skip()` 설정에 포함하지 않았을 때 `NonSkippableProcessException`이 발생하는 기술적 이유는 무엇인가? 스프링 배치가 '롤백 면제'와 '건너뛰기' 사이에서 요구하는 논리적 일관성의 관점에서 설명하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `ItemProcessor`에서 재시도가 발생하면 청크의 처음부터 다시 처리가 시작됩니다. 이때 비활성화 상태라면 이미 성공했던 앞선 아이템들도 다시 `process()`를 호출하게 되어 불필요한 중복 실행(예: API 중복 호출)이 발생합니다. 반면 `processorNonTransactional()`을 설정하면 스프링 배치는 `ItemProcessor`의 결과물을 캐싱합니다. 재시도 시 캐시된 결과가 있다면 로직을 타지 않고 즉시 결과를 반환하므로, 실패한 아이템에 대해서만 로직이 재실행되어 자원 낭비를 차단합니다.

* **2번 질문 해설:** 스캔 모드에서는 청크 내의 아이템들을 하나씩 개별적으로 처리하며 매 아이템마다 즉시 커밋을 수행합니다. 따라서 1,000개의 아이템이 있다면 실패 여부와 상관없이 총 1,000번의 개별 트랜잭션과 커밋이 발생합니다. 이는 단 한 번의 커밋으로 끝날 청크 처리를 수천 번의 네트워크 I/O와 DB 부하로 치환하는 것이며, 결과적으로 배치 처리 속도를 기하급수적으로 저하시키는 '성능 폭탄'이 됩니다.

* **3번 질문 해설:** `ItemProcessor`의 `noRollback`은 "이 예외가 발생해도 현재 트랜잭션을 유지한 채 해당 아이템만 처리 대상에서 제외(건너뛰기)하라"는 의미를 내포합니다. 즉, 롤백을 하지 않는다는 것은 작업을 계속 진행하겠다는 뜻인데, 정작 해당 아이템을 건너뛸 수 있는 설정(`skip`)이 없다면 시스템은 이 아이템을 어떻게 처리해야 할지 결정할 수 없는 논리적 모순에 빠집니다. 따라서 스프링 배치는 안정성을 위해 "롤백하지 않는 예외는 반드시 건너뛰기 가능해야 한다"는 제약을 강제하는 것입니다.
