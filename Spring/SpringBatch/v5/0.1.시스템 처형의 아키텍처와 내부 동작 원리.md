# [심화 분석 학습 노트] 스프링 배치: 시스템 처형의 아키텍처와 내부 동작 원리

## 1장. 배치 처리의 근간: Job과 Step의 지배 구조

배치 시스템을 지배하기 위해 가장 먼저 정복해야 할 개념은 **Job**과 **Step**입니다. 이들은 시스템의 계층 구조를 형성하는 가장 핵심적인 용어입니다.

### 1. Job: 배치 처리의 최상위 집행 단위

Job은 하나의 완전한 배치 작업 그 자체를 의미합니다. 우리가 현실에서 마주하는 거대한 작업들이 모두 Job이라는 이름으로 추상화됩니다.

* **일일 매출 집계:** 매일 심야에 수행되는 데이터 취합 작업.
* **휴면 회원 정리:** 매주 일요일마다 처리되는 회원 상태 변경 작업.
* **정기 결제:** 매월 1일에 실행되는 금융 트랜잭션 작업.
* **대용량 데이터 이관:** 필요 시점에 대규모 데이터를 다른 저장소로 옮기는 작업.

### 2. Step: Job을 구성하는 실전 실행 단위

하나의 Job은 논리적으로 연결된 하나 이상의 Step으로 구성됩니다. 각 Step은 독립적인 실행 단위이며, 특정 비즈니스 로직을 완수합니다. 예를 들어 '일일 매출 집계' Job은 다음과 같은 Step들로 세분화됩니다.

1. **매출 집계 Step:** 주문 데이터를 읽고(Read), 결제 완료 건을 필터링하며(Process), 결과를 저장(Write)합니다.
2. **알림 발송 Step:** 집계된 요약 정보를 생성하여 관리자에게 전달합니다.
3. **캐시 갱신 Step:** 집계된 데이터로 시스템의 캐시 정보를 최신화합니다.

**핵심 아키텍처 포인트:** Job이 성공적으로 완료되기 위해서는 내부의 모든 Step이 사전에 정의된 순서에 따라 정상적으로 완료되어야 합니다.

## 2장. 시스템 해부: 프레임워크 제공 영역과 개발자 제어 영역

스프링 배치는 거대한 뼈대를 제공하며, 개발자는 그 위에 비즈니스라는 살을 붙입니다. 시스템을 이해하기 위해서는 누가 무엇을 담당하는지 명확히 구분해야 합니다.

### 1. Spring Batch 프레임워크가 제공하는 핵심 컴포넌트

프레임워크는 배치 작업의 실행, 관리, 복구를 위한 인프라를 완벽하게 제공합니다.

* **JobLauncher:** 배치 작업 실행의 시작점입니다. Job을 실행하고 실행에 필요한 파라미터를 전달하는 중추적인 역할을 합니다.
* **JobRepository:** 배치 처리의 모든 기록을 담는 **메타데이터 저장소**입니다. Job과 Step의 시작/종료 시간, 실행 상태, 결과 등을 기록하며, 실패 시 재실행을 위한 근거 데이터로 활용됩니다.
* **ExecutionContext:** 실행 중인 상태 정보를 key-value 형태로 저장하는 객체입니다. Job과 Step 간의 데이터 공유나, 시스템 중단 후 재시작 시 상태 복원에 필수적입니다.
* **데이터 처리 컴포넌트 구현체:**
  * **ItemReader:** JdbcCursorItemReader, JpaPagingItemReader 등 다양한 소스에서 데이터를 읽어오는 도구.
  * **ItemWriter:** JdbcBatchItemWriter, JpaItemWriter 등을 통해 가공된 데이터를 저장소에 기록하는 도구.

### 2. 개발자가 제어하는 영역: 전략적 구성

개발자는 `@Configuration`을 통해 시스템의 흐름을 정의하고 컴포넌트들을 조합합니다.

#### [Job 및 Step 구성 소스코드 분석]

```java
@Bean
public Job dataTerminationJob(Step terminateStep) {
    // JobBuilder를 사용하여 "dataTerminationJob"이라는 이름의 Job 생성
    // jobRepository는 배치의 메타데이터 관리를 위해 필수적으로 전달됨
    return new JobBuilder("dataTerminationJob", jobRepository)
        .start(terminateStep) // Job의 시작점으로 terminateStep을 지정
        .build();
}

@Bean
public Step terminateStep(ItemReader<String> itemReader, ItemWriter<String> itemWriter) {
    // StepBuilder를 사용하여 "terminateStep"이라는 이름의 Step 생성
    return new StepBuilder("terminateStep", jobRepository)
        .<String, String>chunk(10, transactionManager) // 10개 단위의 청크 지향 처리 설정 및 트랜잭션 관리자 지정
        .reader(itemReader) // 데이터 읽기 컴포넌트 설정
        .writer(itemWriter) // 데이터 쓰기 컴포넌트 설정
        .build();
}
```

* **내부 동작 메커니즘:** 개발자는 Spring의 DI(의존성 주입)를 활용하여 Reader와 Writer를 조합합니다. 프레임워크는 이 설정을 바탕으로 트랜잭션 경계를 설정하고 데이터를 청크 단위로 흘려보냅니다.
* **단순 작업 처리(Tasklet):** 모든 작업이 읽기-처리-쓰기 구조는 아닙니다. 파일 복사나 알림 발송 같은 단순 작업은 `Tasklet`을 직접 구현하여 처리하며, 이는 개발자의 핵심 제어 영역입니다.
* **커스텀 컴포넌트:** 제공되지 않는 데이터 소스(과거의 MongoDB나 Redis 등)를 다룰 때는 `ItemReader`나 `ItemWriter`를 직접 구현해야 하며, 비즈니스 로직의 정수인 `ItemProcessor`는 항상 개발자가 직접 작성해야 합니다.

## 3장. 실전 침투: HelloDeath Batch를 통한 시스템 장악

우리는 동일한 배치 Job을 **원시적 침투(Raw Spring Batch)**와 **전략적 침투(Spring Boot)**라는 두 가지 무기로 정복합니다.

### 1. 원시적 침투: 순수 Spring Batch의 파괴력

Spring Boot의 자동 설정 없이 모든 인프라를 수동으로 구축하는 방식입니다. 이를 통해 프레임워크의 내부 요구사항을 명확히 이해할 수 있습니다.

#### [인프라 구축: BatchConfig]

```java
@Configuration
public class BatchConfig extends DefaultBatchConfiguration {
    // DefaultBatchConfiguration 상속: 스프링 배치 5부터 도입된 클래스로
    // JobRepository, JobLauncher 등을 자동 구성하는 기반을 제공함

    @Bean
    public DataSource dataSource() {
        // H2 인메모리 DB 설정 및 스프링 배치 필수 메타데이터 스키마 로드
        return new EmbeddedDatabaseBuilder()
            .setType(EmbeddedDatabaseType.H2)
            .addScript("org/springframework/batch/core/schema-h2.sql")
            .build();
    }

    @Bean
    public PlatformTransactionManager transactionManager() {
        // 배치의 모든 작업은 트랜잭션 내에서 처리되어야 하므로 관리자 등록 필수
        return new DataSourceTransactionManager(dataSource());
    }
}
```

#### [비즈니스 로직: SystemTerminationConfig (Tasklet 기반)]

```java
@Bean
public Step defeatProcessStep() {
    return new StepBuilder("defeatProcessStep", jobRepository)
        .tasklet((contribution, chunkContext) -> {
            int terminated = processesKilled.incrementAndGet(); // 프로세스 카운트 증가
            System.out.println("좀비 프로세스 처형 완료! (" + terminated + "/" + TERMINATION_TARGET + ")");
            
            // RepeatStatus 제어: 조건이 만족될 때까지 Step을 반복 실행함
            if (terminated < TERMINATION_TARGET) {
                return RepeatStatus.CONTINUABLE; // 계속 실행
            } else {
                return RepeatStatus.FINISHED; // 종료
            }
        }, transactionManager) // 트랜잭션 관리자 주입
        .build();
}
```

### 2. 전략적 침투: Spring Boot의 자동화된 무기

Spring Boot는 복잡한 인프라 설정을 감추고 개발자가 비즈니스에만 집중하게 합니다.

* **자동 주입의 마법:** `DefaultBatchConfiguration`이나 `DataSource` 설정을 직접 할 필요가 없습니다. `spring-boot-starter-batch`가 이들을 자동으로 구성하여 `JobRepository`와 `PlatformTransactionManager`를 빈으로 등록해줍니다.
* **진입점의 변화:** 순수 방식이 `CommandLineJobRunner`를 사용하는 반면, 부트 방식은 `KillBatchSystemApplication`이 진입점이 되며 `JobLauncherApplicationRunner`가 내부적으로 Job을 실행합니다.

#### [실무형 메인 클래스 구성]

```java
@SpringBootApplication
public class KillBatchSystemApplication {
    public static void main(String[] args) {
        // SpringApplication.exit()와 System.exit()의 조합
        // 배치의 성공/실패 여부를 외부 시스템(스케줄러 등)에 Exit Code로 전달하기 위한 필수 구조
        System.exit(SpringApplication.exit(SpringApplication.run(KillBatchSystemApplication.class, args)));
    }
}
```

## [개념 검증용 자가 진단 질문]

### 배치 아키텍처 및 실행 메커니즘 심화 진단

1. 스프링 배치에서 `JobRepository`가 단순히 로그를 저장하는 용도를 넘어, 배치 시스템의 '신뢰성과 재시작(Restartability)' 관점에서 어떤 핵심적인 역할을 수행하는지 그 메커니즘을 설명하시오.
2. `defeatProcessStep` 코드 예시에서 사용된 `RepeatStatus.CONTINUABLE`과 `RepeatStatus.FINISHED`의 차이는 무엇이며, 이것이 `Tasklet`의 생명주기와 트랜잭션 범위에 어떤 영향을 미치는지 논술하시오.
3. 원시적 침투(Pure Spring Batch) 방식에서 사용된 `CommandLineJobRunner`와 전략적 침투(Spring Boot)의 `JobLauncherApplicationRunner`는 각각 어떤 방식으로 실행할 Job을 식별하고 구동하는지, 실행 파라미터 전달 방식을 중심으로 비교 분석하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `JobRepository`는 배치 실행의 모든 메타데이터(상태, 시간, 결과 등)를 DB에 영구 저장합니다. 만약 배치 작업이 중간에 실패했을 때, 시스템은 `JobRepository`를 조회하여 이전에 어디까지 성공했는지(ExecutionContext 포함) 확인합니다. 이를 통해 실패한 지점부터 다시 시작할 수 있는 근거를 제공하며, 중복 실행 방지 및 데이터 정합성을 유지하는 핵심 제어판 역할을 수행합니다.

* **2번 질문 해설:** `RepeatStatus.CONTINUABLE`은 현재의 `Tasklet` 로직을 종료하지 않고 다시 실행하라는 신호를 보냅니다. 반면 `FINISHED`는 해당 Step이 완료되었음을 의미합니다. 중요한 점은 `CONTINUABLE`을 반환하더라도 각 실행은 별도의 트랜잭션 범위 내에서 처리되거나 단일 트랜잭션의 반복 단위가 될 수 있으며, 이는 대량 데이터를 한 번에 처리하지 않고 메모리 효율성을 유지하면서 루프를 돌 수 있게 해주는 제어 장치입니다.

* **3번 질문 해설:** `CommandLineJobRunner`는 실행 시 설정 클래스의 FQCN(Full Qualified Class Name)과 Job 이름을 인자로 직접 받습니다(예: `SystemTerminationConfig jobName`). 반면, Spring Boot의 `JobLauncherApplicationRunner`는 애플리케이션 시작 시 컨테이너 내의 모든 Job을 실행하려 시도하거나, `--spring.batch.job.name` 프로퍼티에 명시된 특정 Job만을 필터링하여 실행합니다. 부트 방식은 자동 설정을 통해 인프라 구축 단계를 생략하고 프로퍼티 기반의 제어를 제공한다는 점이 큰 차이입니다.
