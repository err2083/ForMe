# [심화 분석 학습 노트] 스프링 배치 리스너: 실행과 죽음의 모든 순간을 지켜보는 자

## 1장. 리스너의 본질과 계층 구조: 감시망의 형성

리스너는 배치 처리의 주요 생명주기(Lifecycle)를 관찰하고 각 시점에 필요한 동작을 정의하는 강력한 도구입니다. 이는 단순히 로그를 남기는 것을 넘어, 에러 처리, 리소스 초기화, 동적 데이터 주입 등 시스템의 흐름을 바꾸는 개입 장치로 작동합니다.

### 1. 실행 단위별 리스너 명세

#### (1) JobExecutionListener: 전체 작전의 통제

Job 실행의 시작과 종료 시점에 호출되는 리스너입니다.

```java
public interface JobExecutionListener {
    // Job 실행 직전에 호출. 리소스 준비 및 초기화 수행
    default void beforeJob(JobExecution jobExecution) {
    }
    // Job 실행 후에 호출. 결과 통보 및 리소스 정리 수행
    default void afterJob(JobExecution jobExecution) {
    }
}
```

* **내부 동작:** `afterJob()`은 Job의 성공/실패 여부와 관계없이 **무조건 호출**됩니다. 이는 실패한 Job의 상태를 강제로 변경하거나, 장애 알림을 보내는 안전장치로서의 역할을 수행함을 의미합니다.

#### (2) StepExecutionListener: 각 단계의 감시

Step의 시작과 종료 시점에 호출됩니다.

```java
public interface StepExecutionListener extends StepListener {
    // Step 시작 시점(StepScope 활성화 직후)에 호출
    default void beforeStep(StepExecution stepExecution) {
    }
    // Step 종료 시점에 호출. 성공/실패와 무관하게 호출됨
    default void afterStep(StepExecution stepExecution) {
    }
}
```

#### (3) ChunkListener: 처리 덩어리의 감시

청크 단위 처리가 시작/완료되거나 에러가 발생했을 때 호출됩니다.

```java
public interface ChunkListener extends StepListener {
    // 청크 처리 시작 전 호출
    default void beforeChunk(ChunkContext context) {
    }
}
```

* **트랜잭션과의 관계:** `afterChunk()`는 트랜잭션이 **커밋된 후**에 호출됩니다. 반면 `afterChunkError()`는 트랜잭션이 **롤백된 이후**에 호출됩니다.
* **Tasklet에서의 동작:** 청크 지향뿐만 아니라 Tasklet 지향 Step에서도 동작합니다. 특히 `RepeatStatus.CONTINUABLE`을 사용하여 Tasklet이 반복될 때, `StepExecutionListener`는 전체 시작/끝에 한 번만 호출되지만 `ChunkListener`는 **매 반복(execute 호출)마다 호출**되어 세밀한 추적이 가능합니다.

#### (4) Item 단위 리스너: 개별 데이터의 식별

아이템의 읽기, 처리, 쓰기 시점에 개입합니다.

```java
// ItemReadListener.java
public interface ItemReadListener<T> extends StepListener {
    default void beforeRead() { }
    default void afterRead(T item) { } // ItemReader.read()가 null을 반환할 때는 호출되지 않음
    default void onReadError(Exception ex) { }
}

// ItemProcessListener.java
public interface ItemProcessListener<T, S> extends StepListener {
    default void beforeProcess(T item) { }
    default void afterProcess(T item, @Nullable S result) { } // process()가 null을 반환(필터링)해도 호출됨
    default void onProcessError(T item, Exception e) { }
}

// ItemWriteListener.java
public interface ItemWriteListener<S> extends StepListener {
    default void beforeWrite(Chunk<? extends S> items) { }
    default void afterWrite(Chunk<? extends S> items) { } // 트랜잭션 커밋 전, ChunkListener.afterChunk() 전에 호출
    default void onWriteError(Exception exception, Chunk<? extends S> items) { }
}
```

## 2장. 리스너 구현 전략: 인터페이스와 어노테이션의 이중주

개발자는 명시적인 인터페이스 구현과 간결한 어노테이션 방식 중 작전 상황에 맞는 무기를 선택할 수 있습니다.

### 1. 인터페이스 구현 방식 (명시적 통제)

메서드가 모두 `default`로 정의되어 있어 필요한 시점만 골라 오버라이드할 수 있습니다.

```java
@Slf4j
@Component
public class BigBrotherJobExecutionListener implements JobExecutionListener {
    @Override
    public void beforeJob(JobExecution jobExecution) {
        log.info("시스템 감시 시작. 모든 작업을 내 통제 하에 둔다.");
    }
}

// Job에 등록
@Bean
public Job systemMonitoringJob(JobRepository jobRepository, Step monitoringStep) {
    return new JobBuilder("systemMonitoringJob", jobRepository)
        .listener(new BigBrotherJobExecutionListener()) // JobBuilder의 listener() 메서드 사용
        .start(monitoringStep)
        .build();
}
```

### 2. 어노테이션 기반 구현 방식 (간결한 침투)

인터페이스 구현 없이 일반 메서드에 어노테이션을 붙여 리스너로 승격시킵니다.

```java
@Slf4j
@Component
public class ServerRoomInfiltrationListener {
    @BeforeJob
    public void infiltrateServerRoom(JobExecution jobExecution) {
        log.info("판교 서버실 침투 시작. 보안 시스템 무력화 진행중.");
    }
}

@Slf4j
@Component
public class ServerRackControlListener {
    @BeforeStep
    public void accessServerRack(StepExecution stepExecution) {
        log.info("서버랙 접근 시작. 콘센트를 찾는 중.");
    }
}
```

* **주의사항:** `@AfterStep`이 선언된 메서드는 반드시 `ExitStatus`를 반환해야 합니다. 이는 해당 어노테이션이 매핑되는 인터페이스 메서드 명세와 일치해야 하기 때문입니다.
* **다양한 어노테이션:** `@AfterChunk`, `@OnProcessError`, 심지어 건너뛰기 이벤트를 감지하는 `@OnSkipInRead` 등 모든 인터페이스 메서드에 대응하는 어노테이션이 존재합니다.

## 3장. 전략적 활용: 시스템 완전 장악과 데이터 공유

리스너는 단순히 보는 것을 넘어, `ExecutionContext`와 결합하여 데이터를 동적으로 생성하고 전달하는 핵심 통로가 됩니다.

### 1. JobExecutionListener를 이용한 동적 데이터 주입

`JobParameters`는 불변(Immutable)이므로 실행 중 생기는 동적 데이터는 `ExecutionContext`를 통해 전달해야 합니다.

```java
@Slf4j
@Component
public class InfiltrationPlanListener implements JobExecutionListener {
    @Override
    public void beforeJob(JobExecution jobExecution) {
        // 동적 데이터 생성 후 Job 수준의 ExecutionContext에 저장
        Map<String, Object> plan = Map.of("targetSystem", "판교 서버실", "requiredTools", "메모리 시퍼너");
        jobExecution.getExecutionContext().put("infiltrationPlan", plan);
    }
}

// Step에서 활용 (람다 Tasklet 예시)
@Bean
public Step reconStep(JobRepository jobRepository, PlatformTransactionManager transactionManager) {
    return new StepBuilder("reconStep", jobRepository)
        .tasklet((contribution, chunkContext) -> {
            // chunkContext를 통해 상위 Job의 ExecutionContext에 접근
            Map<String, Object> infiltrationPlan = (Map<String, Object>) chunkContext.getStepContext()
                .getJobExecutionContext()
                .get("infiltrationPlan");
            log.info("침투 준비 단계: {}", infiltrationPlan.get("targetSystem"));
            return RepeatStatus.FINISHED;
        }, transactionManager)
        .build();
}
```

### 2. ExecutionContextPromotionListener: 스텝 간 데이터 승격

Step 수준의 데이터는 기본적으로 공유되지 않습니다. 이를 상위 Job 수준으로 올리는 과정을 **'승격(Promote)'**이라고 하며, 스프링 배치는 전용 리스너를 제공합니다.

```java
@Bean
public ExecutionContextPromotionListener promotionListener() {
    ExecutionContextPromotionListener listener = new ExecutionContextPromotionListener();
    listener.setKeys(new String[]{"targetSystem"}); // 승격할 Key 지정
    return listener;
}

// 스캔 Step에서 데이터 저장 및 리스너 등록
// .tasklet 내부에서 stepContext.put("targetSystem", target) 수행 후
// .listener(promotionListener()) 등록 시 자동 승격됨
```

### 3. 리스너와 Batch Scope의 통합

리스너에 `@JobScope`를 적용하면 `JobParameters`를 직접 주입받아 로직에 활용할 수 있습니다.

```java
@Bean
@JobScope
public JobExecutionListener systemTerminationListener(
    @Value("#{jobParameters['terminationType']}") String terminationType
) {
    return new JobExecutionListener() {
        @Override
        public void beforeJob(JobExecution jobExecution) {
            log.info("시스템 제거 시작! 제거 방식: {}", terminationType);
        }
    };
}
```

### 4. 성능 최적화 및 모범 사례

1. **예외 처리의 이중성:** `before` 계열에서 예외 발생 시 Job/Step은 실패합니다. 하지만 `afterJob()`, `afterStep()` 내의 예외는 프레임워크가 무시하며 전체 결과에 영향을 주지 않습니다.
2. **아이템 단위 리스너 경계:** `afterRead`, `afterProcess`는 매 아이템마다 실행됩니다. 여기서 외부 API 호출이나 무거운 로직을 수행하는 것은 **시스템 마비**를 자초하는 행위입니다. 리스너는 가볍게 유지해야 합니다.
3. **단일 책임 원칙:** 리스너는 '감시와 통제'만 수행하십시오. 비즈니스 로직을 리스너에 섞는 순간 유지보수는 불가능해집니다.

## [리스너 아키텍처 및 데이터 흐름 자가 진단]

1. `RepeatStatus.CONTINUABLE`을 반환하여 여러 번 반복되는 Tasklet Step이 있다고 가정할 때, `StepExecutionListener`와 `ChunkListener`의 호출 횟수 차이는 어떻게 발생하며, 이것이 개별 반복 단위의 모니터링에 어떤 영향을 주는가?
2. `ItemReadListener.afterRead()`와 `ItemProcessListener.afterProcess()`는 각각 Reader와 Processor가 `null`을 반환했을 때 다르게 반응한다. 이러한 내부 동작의 차이가 발생하는 기술적 이유와, 이를 활용하여 '필터링된 데이터'에 대한 통계를 수집하는 전략을 설명하시오.
3. `ExecutionContextPromotionListener`를 사용하여 데이터를 승격시킬 때, 이 리스너가 내부적으로 상속받은 리스너 인터페이스는 무엇이며, 승격 작업이 정확히 어느 시점(어떤 메서드 안에서)에 일어나는지 그 메커니즘을 논술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `StepExecutionListener`는 Step의 전체 생명주기에 매핑되므로 Tasklet이 몇 번 반복되든 시작과 끝에 단 한 번씩만 호출됩니다. 반면 `ChunkListener`는 스프링 배치 내부 구조상 Tasklet의 `execute()` 호출 전후를 감싸고 있습니다. 따라서 `CONTINUABLE`에 의해 `execute()`가 10번 호출된다면 `ChunkListener`도 10번 호출됩니다. 이를 통해 각 반복 회차별 처리 시간이나 상태를 세밀하게 추적할 수 있습니다.

* **2번 질문 해설:** `afterRead()`는 `ItemReader.read()`가 데이터를 성공적으로 가져왔을 때만 의미가 있으므로 `null`(데이터 고갈) 시에는 호출되지 않습니다. 하지만 `afterProcess()`는 `null` 반환이 '필터링'이라는 의도된 비즈니스 결과물이기 때문에 호출됩니다. 따라서 `afterProcess()`에서 첫 번째 인자인 input item은 존재하지만 두 번째 인자인 result가 `null`인 경우를 카운팅하면, 전체 데이터 중 몇 개가 필터링되었는지 정확한 통계를 리스너 레벨에서 산출할 수 있습니다.

* **3번 질문 해설:** `ExecutionContextPromotionListener`는 `StepExecutionListener`를 구현한 객체입니다. 승격 작업은 Step의 비즈니스 로직이 모두 끝난 후인 `afterStep()` 메서드 내에서 실행됩니다. 이 리스너는 Step의 `ExecutionContext`에서 사용자가 `setKeys()`로 지정한 키값들을 찾아내어, 해당 Step을 소유한 `JobExecution`의 `ExecutionContext`로 복사(put)해 넣는 방식으로 데이터를 상위 컨테이너로 끌어올립니다.

</details>
