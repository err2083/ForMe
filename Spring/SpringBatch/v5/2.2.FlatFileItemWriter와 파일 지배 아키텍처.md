# [심화 분석 학습 노트] 스프링 배치: FlatFileItemWriter와 파일 지배 아키텍처

## 1장. 파일의 지배자: FlatFileItemWriter의 본질과 구조

지금까지 `FlatFileItemReader`로 파일의 형식을 따랐다면, 이제는 우리가 형식을 결정할 차례입니다. `FlatFileItemWriter`는 도메인 객체를 문자열로 변환하여 파일에 써내려가는 **파일의 지배자**입니다.

### 1. 객체에서 문자열로: 변환의 2단계 메커니즘

도메인 객체를 파일의 한 줄로 기록하기 위해서는 두 가지 핵심 컴포넌트가 유기적으로 상호작용해야 합니다.

* **1단계: FieldExtractor (필드 추출):** 도메인 객체에서 필요한 필드(값)들을 뽑아내어 `Object[]` 형태로 반환하는 역할을 수행합니다.
* **2단계: LineAggregator (문자열 결합):** 추출된 필드값들을 하나의 문자열로 결합합니다. CSV처럼 구분자로 이을 수도 있고, 특정 포맷에 맞게 정렬할 수도 있습니다.

### 2. FieldExtractor의 두 얼굴: Bean vs Record

스프링 배치는 객체의 타입에 따라 최적의 추출기를 자동으로 선택합니다.

* **BeanWrapperFieldExtractor:** 일반적인 Java Bean 객체용입니다. 프로퍼티 이름을 기반으로 `getter` 메서드를 호출하여 값을 가져옵니다.
* **RecordFieldExtractor:** Java Record 객체용입니다. 레코드 컴포넌트의 `accessor` 메서드를 호출하여 값을 추출합니다.

### 3. LineAggregator: 결합의 전략가

`LineAggregator`는 `FieldExtractor`를 내부 부품으로 품고 있는 상위 컴포넌트입니다.

```java
public interface LineAggregator<T> {
    String aggregate(T item);
}
```

* **DelimitedLineAggregator:** 필드들을 특정 구분자(콤마 등)로 연결합니다.
* **FormatterLineAggregator:** `String.format()` 스타일의 포맷 문자열을 사용하여 정교하게 데이터를 배치합니다.
* **내부 동작:** `FlatFileItemWriter`는 `LineAggregator`만 바라봅니다. 객체를 문자열로 바꾸는 모든 책임은 `LineAggregator`에게 있으며, 그 내부에서 `FieldExtractor`가 필드를 뽑아주는 구조입니다.

## 2장. 실전 작전 1: 구분자 형식(CSV)의 파일 쓰기와 기술적 모순

이제 실제 코드를 통해 `DelimitedLineAggregator`와 그 설정을 파헤쳐 보겠습니다.

### 1. 청크 지향 스텝 구성

```java
@Bean
public Step deathNoteWriteStep(
    JobRepository jobRepository,
    PlatformTransactionManager transactionManager,
    ListItemReader<DeathNote> deathNoteListReader,
    FlatFileItemWriter<DeathNote> deathNoteWriter
) {
    return new StepBuilder("deathNoteWriteStep", jobRepository)
        .<DeathNote, DeathNote>chunk(10, transactionManager) // 10개 단위로 처리
        .reader(deathNoteListReader)
        .writer(deathNoteWriter)
        .build();
}
```

* **설명:** `<DeathNote, DeathNote>` 제네릭 설정을 통해 Reader에서 읽은 객체가 그대로 Writer로 전달됨을 명시합니다. 10개 단위로 청크가 형성되어 Writer에 전달됩니다.

### 2. FlatFileItemWriter 구성과 빌더 분석

```java
@Bean
@StepScope
public FlatFileItemWriter<DeathNote> deathNoteWriter(
    @Value("#{jobParameters['outputDir']}") String outputDir) {
    return new FlatFileItemWriterBuilder<DeathNote>()
        .name("deathNoteWriter") // Writer의 식별자 설정
        .resource(new FileSystemResource(outputDir + "/death_notes.csv")) // 출력 경로 설정
        .delimited() // DelimitedLineAggregator 사용 선언
        .delimiter(",") // 구분자를 콤마로 지정
        .sourceType(DeathNote.class) // 객체 타입 정보를 전달하여 FieldExtractor 자동 선택 유도
        .names("victimId", "victimName", "executionDate", "causeOfDeath") // 추출할 필드명 지정
        .headerCallback(writer -> writer.write("처형ID,피해자명,처형일자,사인")) // 파일 상단 헤더 작성
        .build();
}
```

* **delimited():** 호출 시 `DelimitedLineAggregator`가 생성됩니다.
* **sourceType():** 이 메서드에 클래스 정보를 주면 배치가 자동으로 `BeanWrapperFieldExtractor` 혹은 `RecordFieldExtractor`를 선택합니다.
* **names():** `FieldExtractor`가 호출할 `getter` 혹은 `accessor`의 대상 이름을 정합니다. 지정한 순서대로 파일에 기록됩니다.

### 3. [중요] 시스템 인텔리전스: Record 사용 시의 버그와 우회책

`FlatFileItemWriterBuilder`에는 치명적인 모순이 존재합니다. `sourceType()`에 Record 클래스를 전달하면 `RecordFieldExtractor`가 사용되는데, 현재 빌더는 내부적으로 `names()` 설정을 `RecordFieldExtractor`에 전달하지 못하는 버그가 있습니다.

* **현상:** `names()`를 호출해도 `RecordFieldExtractor`는 무조건 전체 필드를 추출하려 합니다.
* **우회 전략 (직접 주입):** 일부 필드만 선택해서 쓰고 싶다면 빌더의 자동 구성 대신 `fieldExtractor()` 메서드로 직접 생성한 객체를 꽂아넣어야 합니다.

```java
// 커스텀 FieldExtractor 직접 생성
public RecordFieldExtractor<DeathNote> fieldExtractor() {
    RecordFieldExtractor<DeathNote> fieldExtractor = new RecordFieldExtractor<>(DeathNote.class);
    fieldExtractor.setNames("victimId", "executionDate", "causeOfDeath"); // 특정 필드만 선택
    return fieldExtractor;
}

// Writer 설정에서 직접 전달
.fieldExtractor(fieldExtractor()) // sourceType()과 names()는 이제 무시되므로 제거 가능
```

## 3장. 실전 작전 2: 커스텀 포맷과 대용량 분할 처리

### 1. FormatterLineAggregator를 이용한 보고서 생성

```java
@Bean
@StepScope
public FlatFileItemWriter<DeathNote> deathNoteWriter(
    @Value("#{jobParameters['outputDir']}") String outputDir) {
    return new FlatFileItemWriterBuilder<DeathNote>()
        .name("deathNoteWriter")
        .resource(new FileSystemResource(outputDir + "/death_note_report.txt"))
        .formatted() // FormatterLineAggregator 사용 선언
        .format("처형 ID: %s | 처형일자: %s | 피해자: %s | 사인: %s") // 포맷 정의
        .sourceType(DeathNote.class)
        .names("victimId", "executionDate", "victimName", "causeOfDeath")
        .headerCallback(writer -> writer.write("================= 처형 기록부 ================="))
        .footerCallback(writer -> writer.write("================= 처형 완료 =================="))
        .build();
}
```

* **formatted():** `fixedLength()`와 유사한 개념이지만 훨씬 강력한 `FormatterLineAggregator`를 활성화합니다.
* **format():** `String.format()` 문법을 그대로 사용하여 텍스트 기반의 유연한 보고서 형식을 만듭니다.

### 2. 파일 처리의 세부 옵션 (생존 전략)

* **shouldDeleteIfExists(boolean):** 기본값 `true`. 기존 파일이 있으면 삭제하고 새로 만듭니다. `false`일 때 파일이 존재하면 예외가 발생합니다.
* **append(boolean):** 기본값 `false`. `true`로 설정하면 기존 파일 끝에 데이터를 덧붙입니다. 이때 `shouldDeleteIfExists`는 자동으로 `false`가 됩니다.
* **shouldDeleteIfEmpty(boolean):** 데이터가 하나도 없을 때 빈 파일을 삭제할지 결정합니다.

### 3. 트랜잭션과 OS 캐시 제어

* **transactional():** 기본값 `true`. 청크 처리가 완료되어 커밋되는 시점에만 파일에 실제로 씁니다. 중간에 롤백되면 파일에는 아무것도 남지 않아 데이터 무결성을 보장합니다.
* **forceSync():** 기본값 `false`. `true` 설정 시 OS 캐시를 거치지 않고 즉시 디스크로 데이터를 밀어넣습니다. 성능은 떨어지지만 시스템 중단 시 데이터 유실 위험을 최소화합니다.

### 4. 대용량 분할: MultiResourceItemWriter

1,000만 건의 데이터를 하나의 파일에 담는 것은 자살 행위입니다. 이때 **MultiResourceItemWriter**를 사용하여 파일을 쪼개야 합니다.

```java
@Bean
@StepScope
public MultiResourceItemWriter<DeathNote> multiResourceItemWriter(
    @Value("#{jobParameters['outputDir']}") String outputDir) {
    return new MultiResourceItemWriterBuilder<DeathNote>()
        .name("multiDeathNoteWriter")
        .resource(new FileSystemResource(outputDir + "/death_note"))
        .itemCountLimitPerResource(10) // 10개마다 새 파일 생성
        .delegate(delegateItemWriter()) // 실제 쓰기를 담당할 Writer 위임
        .resourceSuffixCreator(index -> String.format("_%03d.txt", index)) // 접미사 규칙
        .build();
}
```

* **위임자(Delegator) vs 위임 대상(Delegate):** `MultiResourceItemWriter`는 파일을 언제 새로 만들지 관리만 하고, 실제 데이터 쓰기는 내부의 `FlatFileItemWriter`에게 시킵니다.

### 5. 종합 사례: 분산 서버 로그 처형 작전

여러 서버의 로그를 수집하여 JSON 형식으로 변환하는 고급 공정입니다.

* **SystemCommandTasklet:** `mkdir`, `scp` 등의 시스템 명령어를 배치 흐름에 통합합니다.
* **MultiResourceItemReader:** 여러 서버에서 가져온 로그 파일들을 한 번에 읽어들입니다.
* **JSON 출력 기술:** `FlatFileItemWriter`에 커스텀 `LineAggregator`를 주입하여 `ObjectMapper`로 객체를 JSON 문자열로 변환합니다.

## [개념 검증용 자가 진단 질문]

### 파일 쓰기 아키텍처 및 내부 메커니즘 심화 진단

1. `FlatFileItemWriter`가 데이터를 파일에 기록할 때 직접 `FieldExtractor`를 호출하지 않고 반드시 `LineAggregator`를 거쳐야만 하는 구조적 이유는 무엇이며, 이 구조가 주는 유연성(Flexibility)의 핵심은 무엇인가?
2. Java 17의 `Record` 타입을 사용할 때 `FlatFileItemWriterBuilder`의 자동 구성 방식(`sourceType`, `names`)이 가진 한계점은 무엇이며, 특정 필드만 선택하여 파일에 기록해야 하는 실무 상황에서 이를 어떻게 기술적으로 해결해야 하는지 상세히 서술하시오.
3. `FlatFileItemWriter`의 `transactional(true)` 설정이 활성화되어 있을 때, 대량 데이터 처리 중 예외가 발생하여 트랜잭션이 롤백되는 상황에서의 '내부 버퍼' 동작 방식과 실제 파일 시스템의 상태 변화를 인과관계에 따라 설명하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `FlatFileItemWriter`는 '어떤 데이터를 쓸 것인가'보다 '어떻게 문자열로 결합할 것인가'에 집중합니다. `LineAggregator`는 객체를 한 줄의 문자열로 만드는 책임을 캡슐화하고 있으며, `FieldExtractor`는 단지 그 과정에서 데이터를 공급하는 부품(Component)일 뿐입니다. 이 구조 덕분에 개발자는 `FieldExtractor`를 교체하지 않고도 동일한 데이터를 CSV, 고정 길이, 혹은 커스텀 보고서 포맷 등 다양한 출력 형태로 자유롭게 변환할 수 있는 유연성을 얻게 됩니다.

* **2번 질문 해설:** 현재 `FlatFileItemWriterBuilder`는 `sourceType()`에 Record를 지정하면 자동으로 `RecordFieldExtractor`를 생성하지만, 빌더에 설정된 `names()` 목록을 추출기에 전달하지 않는 버그가 있습니다. 이로 인해 필드 선별이 불가능해집니다. 이를 해결하려면 빌더의 자동 구성을 포기하고, `fieldExtractor()` 메서드를 통해 직접 `RecordFieldExtractor` 인스턴스를 생성한 뒤 `setNames()`로 대상 필드를 명시하고 이를 빌더에 주입(Direct Injection)하는 방식을 사용해야 합니다.

* **3번 질문 해설:** `transactional(true)`일 때 `FlatFileItemWriter`는 데이터를 `write()` 호출 즉시 디스크에 쓰지 않고 메모리 내부 버퍼에 격리합니다. 청크의 모든 프로세스가 성공하여 트랜잭션이 커밋되기 직전인 `beforeCommit()` 시점에 비로소 버퍼의 데이터를 파일에 일괄 기록합니다. 따라서 중간에 예외가 발생하여 롤백되면, 버퍼 내의 데이터는 폐기되고 실제 파일 시스템에는 아무런 기록도 남지 않게 되어 데이터 정합성이 완벽히 보호됩니다.
