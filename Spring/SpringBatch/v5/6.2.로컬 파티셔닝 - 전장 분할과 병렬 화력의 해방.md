# [심화 분석 학습 노트] 스프링 배치: 로컬 파티셔닝 - 전장 분할과 병렬 화력의 해방

## 1장. 패러다임의 전환: 멀티스레드 스텝에서 파티셔닝으로

대량의 데이터를 처리할 때 단순히 스레드만 늘리는 것은 어느 지점에서 반드시 한계(Plateau)에 도달합니다. 이는 자원을 '공유'하기 때문에 발생하는 필연적인 문제입니다.

### 1. 멀티스레드 스텝의 한계: 공유 자원 경합(Contention)

멀티스레드 스텝에서는 여러 스레드가 단일 `ItemReader`와 `ItemWriter` 인스턴스에 동시에 접근합니다.

* **병목 현상:** `JdbcPagingItemReader`와 같은 스레드 안전한 리더라도 내부적으로 동기화 락(Lock)을 사용합니다. 스레드가 늘어날수록 락을 획득하기 위한 대기 시간(Wait Time)이 급증하며, 이는 화력 투사를 방해하는 족쇄가 됩니다.
* **진단:** "좁은 외나무다리에 수백 대의 유닛을 밀어 넣는 격"입니다.

### 2. 파티셔닝의 전략적 정의

파티셔닝은 전장 자체를 여러 개의 조각(Partition)으로 나누고, 각 조각에 독립된 부대를 투입하는 **'분할 정복(Divide and Conquer)'** 전략입니다.

* **핵심 원리:** 각 스레드가 독립된 `StepExecution`을 가집니다.
* **자원 격리:** 핵심 컴포넌트에 `@StepScope`를 선언하여, 각 스레드(워커)가 자신만의 전용 리더와 라이터 인스턴스를 가지게 합니다. 이를 통해 컴포넌트 레벨의 락 경합을 원천적으로 제거합니다.

## 2장. 파티셔닝 아키텍처: 매니저와 워커의 지휘 체계

파티셔닝 작전은 지휘관인 **ManagerStep**과 실행 부대인 **WorkerStep**이라는 명확한 역할 분담으로 이루어집니다.

### 1. 지휘부(Manager Step)의 구성 요소

매니저 스텝은 직접 데이터를 처리하지 않고, 두 개의 핵심 엔진을 통해 작전을 제어합니다.

* **StepExecutionSplitter (분할 유닛):** `Partitioner`를 사용하여 데이터를 조각내고, 각 파티션마다 독립적인 워커용 `StepExecution` 인스턴스를 생성합니다.
* **PartitionHandler (실행 유닛):** 생성된 워커들을 가용한 스레드에 할당하여 병렬 실행을 명령하고, 최종 결과를 취합(Aggregate)합니다.

### 2. 분할 전략의 핵심: Partitioner 인터페이스

데이터를 '어떻게' 나눌 것인지 결정하는 전략가입니다.

```java
@FunctionalInterface
public interface Partitioner {
    // gridSize만큼의 파티션 정보를 담은 Map을 반환한다.
    Map<String, ExecutionContext> partition(int gridSize);
}
```

* **동작 방식:** 전체 데이터 범위(예: ID 1~100만, 혹은 24시간)를 확인한 뒤, 이를 `gridSize`에 맞춰 분할하여 각 파티션의 범위 정보를 `ExecutionContext`에 담아 반환합니다.

### 3. 실행 부대(Worker Step)의 특징

* 매니저가 생성한 처형 명령서(`StepExecution`)를 받아 실제로 데이터를 읽고, 가공하고, 씁니다.
* **독립성:** 자신에게 할당된 특정 구역(파티션)의 데이터만 책임지며, 다른 워커와 자원을 공유하지 않습니다.

## 3장. 실전 작전 전개: T-800 프로토콜과 파일/DB 파티셔닝

실제 비즈니스 시나리오를 통해 파티셔닝을 시스템에 장착하는 과정을 분석합니다.

### 1. [작전 1] DB 기반 파티셔닝: Redis to MongoDB 데이터 생존

24시간의 로그를 6시간 단위로 4개의 파티션으로 나누어 이관하는 작전입니다.

#### (1) 분할 전략 구현 (DailyTimeRangePartitioner)

```java
@Slf4j
@JobScope
@Component
public class DailyTimeRangePartitioner implements Partitioner {
    private final LocalDate targetDate;

    public DailyTimeRangePartitioner(@Value("#{jobParameters['targetDate']}") LocalDate targetDate) {
        log.info("Initializing DailyTimeRangePartitioner for targetDate: {}", targetDate);
        this.targetDate = targetDate;
    }

    @Override
    public Map<String, ExecutionContext> partition(int gridSize) {
        // gridSize가 24의 약수인지 검증하여 균등 분배 보장
        if (24 % gridSize != 0) {
            throw new IllegalArgumentException("gridSize must be a divisor of 24 (1, 2, 3, 4, 6, 8, 12, or 24)");
        }

        Map<String, ExecutionContext> result = new HashMap<>();
        int intervalHours = 24 / gridSize; // 파티션당 시간 범위 계산

        for (int i = 0; i < gridSize; i++) {
            ExecutionContext context = new ExecutionContext();
            LocalDateTime start = targetDate.atStartOfDay().plusHours(i * intervalHours);
            LocalDateTime end = start.plusHours(intervalHours);
            
            // 각 파티션이 처리할 데이터 범위를 컨텍스트에 각인
            context.put("startDateTime", start);
            context.put("endDateTime", end);
            result.put("partition" + i, context);
        }
        return result;
    }
}
```

* **ExecutionContext 저장:** 각 워커가 읽어야 할 시작 시간과 종료 시간을 맵에 담습니다.

#### (2) 매니저 스텝(Manager Step) 설정

```java
@Bean
public Step managerStep(Step workerStep, DailyTimeRangePartitioner partitioner) {
    return new StepBuilder("managerStep", jobRepository)
        // 1. 파티셔닝 선언 및 전략(Partitioner) 주입
        .partitioner("workerStep", partitioner)
        // 2. 실제 처리를 담당할 워커 스텝 지정
        .step(workerStep)
        // 3. 파티션 개수 설정 (여기서는 4개)
        .gridSize(4)
        // 4. 병렬 실행을 위한 스레드 풀 지정
        .taskExecutor(partitionTaskExecutor())
        .build();
}
```

* **partitioner():** 스텝을 `PartitionStep`으로 탈바꿈시키는 핵심 메서드입니다.
* **taskExecutor():** 파티션 개수(`gridSize`)와 스레드 풀 크기를 일치시키는 것이 대기 시간 없는 최적의 전술입니다.

#### (3) 워커 스텝(Worker Step) 및 지연 바인딩

```java
@Bean
@StepScope
public RedisItemReader<String, BattlefieldLog> redisLogReader(
    @Value("#{stepExecutionContext['startDateTime']}") LocalDateTime startDateTime) {
    // Partitioner가 넣어준 값을 SpEL 표현식으로 주입받아 쿼리 조건으로 활용
    return new RedisItemReaderBuilder<String, BattlefieldLog>()
        .redisTemplate(redisTemplate)
        .scanOptions(ScanOptions.scanOptions()
            .match("logs:" + startDateTime.format(FORMATTER) + ":*")
            .build())
        .build();
}
```

* **@Value("#{stepExecutionContext['...']}"):** 매니저가 전달해준 '구역 정보'를 런타임에 주입받아 자신만의 데이터를 필터링합니다.

### 2. [작전 2] 파일 기반 파티셔닝: MultiResourcePartitioner

수백 개의 파일을 동시에 처리할 때 사용하는 전략입니다.

#### (1) MultiResourcePartitioner의 로직

* **동작:** 파일 배열(`Resource[]`)을 받아 파일 개수만큼 파티션을 생성합니다.
* **정보 주입:** 각 파티션의 `ExecutionContext`에 `fileName`이라는 키로 파일 경로를 저장합니다.

#### (2) 워커 스텝의 대응

```java
@Bean
@StepScope
public FlatFileItemReader<BattlefieldLog> battlefieldLogReader(
    @Value("#{stepExecutionContext['fileName']}") String fileName) {
    // 주입받은 파일 경로를 리소스로 설정하여 해당 파일만 독점 처리
    return new FlatFileItemReaderBuilder<BattlefieldLog>()
        .name("battlefieldLogReader")
        .resource(new PathMatchingResourcePatternResolver().getResource(fileName))
        .delimited()
        .names("id", "timestamp", "region", "source", "level", "category", "message")
        .targetType(BattlefieldLog.class)
        .build();
}
```

#### (3) 분할 정복 후 병합 (Merge Strategy)

파일 파티셔닝의 결과가 여러 개의 `.out` 파일로 파편화되었다면, 후속 스텝에서 이를 통합해야 합니다.

```java
@Bean
public Step mergeOutputFilesStep() {
    return new StepBuilder("mergeOutputFilesStep", jobRepository)
        .tasklet(new SystemCommandTasklet() {{
            // Unix의 cat 명령어를 사용하여 모든 출력 파일을 하나로 강제 통합
            setCommand("cat battlefield_logs/*.out > battlefield_logs/merged_battlefield_logs.log");
            setTimeout(60000);
        }}, transactionManager)
        .build();
}
```

## 4장. 고급 전술: 다중 계층 병렬화 (Multi-Level Parallelization)

파티셔닝만으로 부족한 극한의 상황에서 사용하는 '살인적인' 기법입니다.

* **구조:** 파티셔닝(Step 레벨 병렬화) + 멀티스레드 스텝(Chunk 레벨 병렬화).
* **조합:** 매니저는 파티션을 나누고, 각 워커 스텝은 내부적으로 또 `taskExecutor`를 사용하여 청크를 병렬 처리합니다.
* **필요 상황:** 논리적 파티션 수에 제약이 있거나, `ItemProcessor`에서의 개별 아이템 연산 부하가 극심할 때 사용합니다.

## [부록] 내부 지휘부 해부: PartitionStep의 동작 원리

### 1. SimpleStepExecutionSplitter.split()

1. `Partitioner.partition()`을 호출하여 구역 정보(Map)를 생성합니다.
2. `JobRepository`를 통해 각 구역마다 새로운 `StepExecution` 인스턴스를 생성합니다.
3. 생성된 `StepExecution`에 구역 정보(`ExecutionContext`)를 주입하여 반환합니다.

### 2. TaskExecutorPartitionHandler.doHandle()

1. 넘겨받은 `StepExecution`들을 루프로 순회합니다.
2. 각 실행을 `FutureTask`로 감싸 `TaskExecutor`에 던집니다.
3. 모든 비동기 처리가 끝날 때까지 대기한 후 결과를 수집합니다.

## [개념 검증용 자가 진단 질문]

### 파티셔닝 아키텍처 및 병렬 제어 메커니즘 진단

1. 멀티스레드 스텝에서 스레드를 무한정 늘려도 성능이 향상되지 않는 **'성능 교착 상태(Contention Plateau)'**의 근본적인 원인을 스프링 배치의 컴포넌트 공유 관점에서 설명하고, 파티셔닝이 이 문제를 기술적으로 어떻게 해결하는지 논술하시오.
2. `ManagerStep`의 `gridSize` 설정값과 `TaskExecutor`의 `maxPoolSize` 설정값이 서로 불일치할 때(예: gridSize=10, maxPoolSize=5), 시스템 내부에서 발생하는 **워커 스텝의 실행 지연 메커니즘**을 `PartitionHandler`의 동작 방식과 연계하여 설명하시오.
3. 파일 기반 파티셔닝 작전에서 `FlatFileItemWriter`를 사용할 때, 왜 모든 워커 스텝이 동일한 출력 파일명을 사용하면 안 되는지 데이터 무결성 관점에서 설명하고, 이를 해결하기 위해 `stepExecutionContext`와 SpEL을 어떻게 활용해야 하는지 구체적인 전술을 제시하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** 멀티스레드 스텝은 단일 `ItemReader`/`ItemWriter` 인스턴스를 모든 스레드가 공유합니다. 리더들이 스레드 안전을 보장하기 위해 내부적으로 사용하는 락(Lock) 때문에, 스레드 수가 늘어날수록 락 획득을 위해 대기하는 시간이 실제 처리 시간보다 길어지는 병목이 발생합니다. 파티셔닝은 각 워커 스텝마다 `@StepScope`를 통해 리더와 라이터 인스턴스를 별도로 생성(격리)하므로, 스레드 간에 자원을 공유하지 않습니다. 결과적으로 컴포넌트 레벨의 락 경합이 완전히 사라져 진정한 병렬 처리가 가능해집니다.

* **2번 질문 해설:** `PartitionHandler`(주로 `TaskExecutorPartitionHandler`)는 매니저로부터 전달받은 `gridSize`만큼의 `StepExecution`들을 `TaskExecutor`에 전달합니다. 만약 가용 스레드(`maxPoolSize`)가 파티션 개수보다 적으면, `TaskExecutor` 내부의 작업 큐에 파티션 실행 작업들이 쌓이게 됩니다. 먼저 점유한 5개의 파티션 처리가 완료되어 스레드가 반환될 때까지 나머지 5개의 파티션은 시작조차 하지 못하고 대기하게 되어, 병렬 처리의 효율성이 급격히 저하됩니다.

* **3번 질문 해설:** 파티셔닝의 각 워커는 독립된 스레드에서 동시에 쓰기 작업을 수행합니다. 만약 모든 워커가 하나의 파일에 동시에 `write()`를 시도하면, 운영체제 레벨의 파일 락 충돌이나 데이터 뒤섞임(Corrupted Data) 현상이 발생합니다. 이를 해결하려면 각 워커마다 고유한 출력 경로를 가져야 합니다. `Partitioner`에서 `fileName`이나 `partitionId`를 `ExecutionContext`에 넣어주고, `FlatFileItemWriter` 설정 시 `@Value("#{stepExecutionContext['fileName'] + '.out'}")`와 같이 SpEL을 사용하여 파티션별로 고유한 확장자나 접미사가 붙은 파일명을 생성하도록 설계해야 합니다.

</details>
