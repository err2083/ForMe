# [구조화 심화 분석 학습 노트] 스프링 배치: 스텝 유형의 바이너리 초이스와 내부 메커니즘

## 1장. 바이너리 초이스 - 스프링 배치의 두 가지 스텝 유형

배치 처리에는 두 개의 얼굴이 존재합니다. Job은 전체적인 흐름을 정의하고, Step은 그 흐름을 구성하는 개별적인 실행 단위입니다. 모든 배치가 데이터를 읽고, 가공하고, 쓰는 정형화된 패턴을 따르는 것은 아닙니다. 이러한 차이로 인해 스프링 배치의 스텝은 크게 두 가지 모델로 나뉩니다.

* **청크 지향 처리 (Chunk-Oriented Processing):** 대량의 데이터를 일정 덩어리로 나누어 처리하는 모델입니다.
* **태스크릿 지향 처리 (Tasklet-Oriented Processing):** 단일 비즈니스 로직을 수행하는 단순한 작업 모델입니다.

이 두 가지 중 무엇을 선택하느냐에 따라 시스템의 리소스 활용 방식과 복구 전략이 완전히 달라집니다.

## 2장. 태스크릿(Tasklet) 지향 처리: 단순하지만 치명적인 암살자

태스크릿 모델은 비교적 복잡하지 않은 **단순한 작업**을 실행할 때 사용되는 가장 기본적인 스텝 구현 방식입니다.

### 1. 태스크릿의 활용 시나리오

일반적인 ETL(Extract, Transform, Load) 작업과 달리, 다음과 같은 유틸성 작업에 최적화되어 있습니다.

* 매일 새벽 불필요한 로그 파일 제거
* 오래된 파일의 아카이브(압축 및 이동)
* 사용자에게 단순 알림 발송 (이메일, 메시지 등)
* 외부 API를 호출하고 결과를 로깅하거나 저장하는 단발성 작업

### 2. Tasklet 인터페이스와 내부 동작

개발자는 스프링 배치가 제공하는 `Tasklet` 인터페이스를 구현하여 원하는 로직을 정의합니다.

```java
@FunctionalInterface
public interface Tasklet {
    @Nullable
    RepeatStatus execute(StepContribution contribution, ChunkContext chunkContext) throws Exception;
}
```

* **execute():** 실제 비즈니스 로직이 들어가는 메서드입니다.
* **StepContribution / ChunkContext:** 현재 스텝의 상태 정보를 담고 있으며, 실행 이력을 제어하는 데 사용됩니다.

### 3. RepeatStatus: 트랜잭션의 경계와 반복 제어

태스크릿의 핵심은 반환값인 `RepeatStatus`에 있습니다.

* **RepeatStatus.FINISHED:** 해당 스텝이 완료되었음을 알립니다. 성공/실패 여부와 상관없이 반복을 종료하고 다음 단계로 진행합니다.
* **RepeatStatus.CONTINUABLE:** 아직 작업이 남았음을 신호합니다. `execute()` 메서드가 다시 호출되며, 스텝 종료가 보류됩니다.

#### [기술적 심화: 왜 반복문 대신 RepeatStatus인가?]

스프링 배치는 `execute()` 호출 한 번을 하나의 **트랜잭션 단위**로 관리합니다.

* **내부 반복문 사용 시:** 100만 건을 한 번의 `execute()` 내에서 처리하다가 80만 건째 실패하면, 이전의 79만 9,999건이 모두 롤백됩니다.
* **RepeatStatus.CONTINUABLE 사용 시:** 매 호출마다 트랜잭션이 커밋됩니다. 만약 80만 건째 실패하더라도, 이미 처리된 79만 건은 안전하게 DB에 반영된 상태로 남습니다. 즉, **거대한 트랜잭션을 작은 트랜잭션들로 쪼개어 안전성을 확보**하는 메커니즘입니다.

### 4. 태스크릿 스텝의 구성과 트랜잭션 관리

실제 스텝을 정의할 때는 `StepBuilder`를 사용합니다.

```java
@Bean
public Step zombieCleanupStep() {
    return new StepBuilder("zombieCleanupStep", jobRepository) // 스텝 이름과 메타데이터 저장소 설정
        .tasklet(zombieProcessCleanupTasklet(), transactionManager) // 실행할 Tasklet과 트랜잭션 관리자 주입
        .build();
}
```

* **tasklet():** 이 메서드를 호출하는 순간 해당 스텝은 태스크릿 지향 모델로 확정됩니다.
* **PlatformTransactionManager:** DB 작업을 포함할 경우 필수입니다. 하지만 파일 삭제처럼 DB 트랜잭션이 불필요한 경우 `ResourcelessTransactionManager`를 사용하여 오버헤드를 줄일 수 있습니다.

```java
@Bean
public Step zombieCleanupStep() {
    return new StepBuilder("zombieCleanupStep", jobRepository)
        .tasklet(zombieProcessCleanupTasklet(), new ResourcelessTransactionManager()) // No-op 방식의 관리자
        .build();
}
```

### 5. 실무 활용 및 구현 팁

* **복잡한 로직:** 별도의 클래스를 만들어 `Tasklet`을 구현합니다.
* **단순 로직:** 람다 표현식을 사용하여 가독성을 높입니다.

```java
@Bean
public Step deleteOldRecordsStep() {
    return new StepBuilder("deleteOldRecordsStep", jobRepository)
        .tasklet((contribution, chunkContext) -> {
            // JdbcTemplate 등을 활용한 직접적인 DB 조작
            int deleted = jdbcTemplate.update("DELETE FROM logs WHERE created < NOW() - INTERVAL 7 DAY");
            log.info("🗑️ {}개의 오래된 레코드가 삭제되었습니다.", deleted);
            return RepeatStatus.FINISHED; // 단발성 작업 종료
        }, transactionManager)
        .build();
}
```

## 3장. 청크(Chunk) 지향 처리: 대량 데이터의 최종 병기

대부분의 배치 작업은 '읽기 → 처리 → 쓰기'의 패턴을 따르며, 이를 효율적으로 구조화한 것이 바로 청크 지향 처리입니다.

### 1. 청크(Chunk)의 본질: 왜 덩어리로 나누는가?

청크는 데이터를 일정 단위로 쪼갠 묶음입니다. 100만 건을 한 번에 처리하지 않고 100개 또는 1,000개 단위로 나눕니다.

* **메모리 보호:** 100만 건을 한꺼번에 JVM 메모리에 올리면 `OutOfMemoryError`로 시스템이 폭발합니다. 청크 단위로 처리하면 메모리에는 오직 해당 청크 사이즈만큼의 데이터만 존재하게 됩니다.
* **가벼운 트랜잭션:** 트랜잭션 범위가 청크 단위로 설정됩니다. 실패 시 해당 청크만 롤백하면 되므로 복구가 매우 빠르고 효율적입니다.

### 2. 청크 지향 처리의 3대장 (ItemReader, ItemProcessor, ItemWriter)

* **ItemReader (수혈관):** 데이터 소스(파일, DB 등)에서 데이터를 **하나씩** 읽어옵니다.

    ```java
    public interface ItemReader<T> {
        T read() throws Exception; // 데이터가 없으면 null 반환 (종료 신호)
    }
    ```

* **ItemProcessor (가공/필터링):** 읽어온 아이템을 **하나씩** 가공하거나 변환합니다. `null`을 반환하면 해당 아이템은 Writer로 전달되지 않고 필터링됩니다.

    ```java
    public interface ItemProcessor<I, O> {
        O process(I item) throws Exception;
    }
    ```

* **ItemWriter (최종 집행자):** 가공된 아이템들을 **청크 단위(List/Chunk)**로 묶어서 한 번에 저장합니다.

    ```java
    public interface ItemWriter<T> {
        void write(Chunk<? extends T> chunk) throws Exception;
    }
    ```

### 3. 청크 지향 스텝의 조립과 데이터 흐름

```java
@Bean
public Step processStep(JobRepository jobRepository, PlatformTransactionManager transactionManager) {
    return new StepBuilder("processStep", jobRepository)
        .<CustomerDetail, CustomerSummary>chunk(10, transactionManager) // 10개 단위 청크 및 제네릭 타입 정의
        .reader(itemReader())       // Reader 구현체 등록
        .processor(itemProcessor()) // Processor 구현체 등록
        .writer(itemWriter())      // Writer 구현체 등록
        .build();
}
```

* **제네릭 정의:** `<I, O>`에서 `I`는 Reader가 반환할 타입, `O`는 Processor가 반환하여 Writer가 받을 타입을 명시합니다.
* **동작 프로세스 (청크 사이즈 10인 경우):**
    1. **Reader:** `read()`를 10번 호출하여 10개의 아이템을 수집.
    2. **Processor:** 수집된 10개의 아이템에 대해 각각 `process()`를 호출(총 10번).
    3. **Writer:** 변환된 10개의 아이템이 담긴 **Chunk 객체 하나**를 `write()`에 전달하여 일괄 저장.

### 4. 반복의 종료 조건: null의 미학

스프링 배치는 언제 멈춰야 할지를 `ItemReader.read()`의 반환값으로 판단합니다.

* `read()`가 실제 데이터를 반환하는 동안은 청크 단위 반복이 계속됩니다.
* `read()`가 **null**을 반환하는 순간, 프레임워크는 "더 이상 읽을 데이터가 없다"고 판단하고 현재 청크의 쓰기 작업까지 마친 후 스텝을 종료합니다.

### 5. 청크 사이즈의 트레이드오프

* **큰 사이즈:** I/O 횟수가 줄어들어 속도가 빠르지만, 메모리 사용량이 늘어나고 실패 시 롤백 비용이 큽니다.
* **작은 사이즈:** 리소스 사용이 안정적이고 실패 복구가 빠르지만, 잦은 트랜잭션 커밋과 I/O 발생으로 전체 수행 시간은 길어집니다.

## [개념 검증용 자가 진단 질문]

### 배치 처리 모델 및 트랜잭션 아키텍처 진단

1. `Tasklet`을 구현할 때 로직 내부에서 `while`문을 사용하여 수만 건의 데이터를 처리하는 것보다, `RepeatStatus.CONTINUABLE`을 반환하며 여러 번 호출되도록 설계하는 것이 아키텍처 관점에서 왜 더 안전한지 트랜잭션의 성격과 연계하여 논술하시오.
2. 청크 지향 처리 과정에서 `ItemProcessor`가 `null`을 반환했을 때, `ItemWriter`에 전달되는 데이터 덩어리(Chunk)에는 어떤 변화가 생기며, 이것이 필터링 메커니즘으로서 어떻게 동작하는지 내부 흐름을 설명하시오.
3. 총 105개의 데이터가 존재하고 청크 사이즈가 10인 스텝이 실행될 때, `ItemReader.read()`가 총 몇 번 호출되는지 계산하고, 마지막 11번째 청크 반복에서 발생하는 구체적인 제어 흐름(종료 인식 과정)을 서술하시오.

* **1번 질문 해설:** 스프링 배치는 `Tasklet.execute()`가 한 번 실행될 때마다 하나의 트랜잭션을 시작하고 종료(커밋)합니다. `while`문을 내부에 작성하면 수만 건의 작업이 하나의 거대한 트랜잭션으로 묶이게 되어, 중간에 단 하나의 오류만 발생해도 전체가 롤백되는 리스크가 있습니다. 반면 `CONTINUABLE`을 사용하면 각 호출 단위로 트랜잭션이 커밋되므로, 실패 전까지 처리된 데이터는 안전하게 보존되어 부분적 성공을 보장할 수 있습니다.

* **2번 질문 해설:** `ItemProcessor.process()` 메서드가 `null`을 반환하면, 스프링 배치는 해당 아이템을 유효하지 않거나 처리 대상이 아닌 것으로 간주합니다. 따라서 해당 아이템은 쓰기 대상 목록에서 즉시 제외됩니다. 결과적으로 `ItemWriter.write()`에 전달되는 `Chunk` 객체에는 `null`을 반환한 아이템을 제외한 나머지 아이템들만 담기게 되며, 이를 통해 별도의 복잡한 조건문 없이도 데이터 필터링을 완수할 수 있습니다.

* **3번 질문 해설:** `ItemReader.read()`는 총 **106번** 호출됩니다. 처음 10개씩 10번의 반복(100번 호출) 후, 마지막 11번째 반복에서 남은 5개를 읽기 위해 5번 호출됩니다. 그 후 다음 호출(106번째)에서 비로소 `null`을 반환하게 됩니다. 이때 스프링 배치는 `null`을 확인하는 순간 "데이터 소스 고갈"로 인식하여 마지막 5개에 대한 Processor/Writer 작업을 수행한 뒤 전체 스텝을 종료합니다.
