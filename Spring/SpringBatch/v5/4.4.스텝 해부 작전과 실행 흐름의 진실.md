# [심화 분석 학습 노트] 스프링 배치: 스텝 해부 작전과 실행 흐름의 진실

## 1장. 스텝 해부학 I: 구성 요소와 빌더의 설계 원리

스텝의 내부 구조를 파헤치려면 먼저 그 뼈대를 만드는 빌더(Builder)부터 해부해야 합니다. 우리가 흔히 사용하는 API들이 내부적으로 어떤 객체를 생성하고 연결하는지 확인하는 과정입니다.

### 1. SimpleStepBuilder: 청크 지향 처리의 전문 설계자

우리는 보통 `StepBuilder.chunk()`를 호출하여 청크 지향 스텝을 구성합니다. 이 메서드는 내부적으로 `SimpleStepBuilder`를 반환하며, 이것이 청크 처리의 전용 빌더입니다.

```java
// StepBuilder.java
public <I, O> SimpleStepBuilder<I, O> chunk(int chunkSize, PlatformTransactionManager transactionManager) {
    return new SimpleStepBuilder<I, O>(this).transactionManager(transactionManager).chunk(chunkSize);
}
```

* **chunk():** `SimpleStepBuilder` 인스턴스를 생성하고 트랜잭션 매니저와 청크 사이즈를 설정합니다.
* **기술적 실체:** 태스크릿 지향 처리 시에는 `TaskletStepBuilder`가, 내결함성 기능(Retry/Skip) 적용 시에는 `FaultTolerantStepBuilder`가 사용되지만, 이들 모두의 종착역은 결국 **TaskletStep**입니다.

### 2. 리스너 자동 등록 메커니즘 (Listener Registration)

스프링 배치는 개발자가 등록한 리스너가 여러 인터페이스를 동시에 구현했을 때 이를 지능적으로 감지하여 중복 등록을 방지하고 적절한 카테고리에 할당합니다.

```java
// SimpleStepBuilder.java
protected void registerStepListenerAsItemListener() {
    for (StepExecutionListener stepExecutionListener : properties.getStepExecutionListeners()) {
        checkAndAddItemListener(stepExecutionListener);
    }
    for (ChunkListener chunkListener : this.chunkListeners) {
        checkAndAddItemListener(chunkListener);
    }
}

private void checkAndAddItemListener(StepListener stepListener) {
    if (stepListener instanceof ItemReadListener) {
        listener((ItemReadListener<I>) stepListener);
    }
    if (stepListener instanceof ItemProcessListener) {
        listener((ItemProcessListener<I, O>) stepListener);
    }
    if (stepListener instanceof ItemWriteListener) {
        listener((ItemWriteListener<O>) stepListener);
    }
}
```

* **작전 팁:** 만약 하나의 객체가 `StepExecutionListener`와 `ItemReadListener`를 모두 구현했다면, 빌더에서 `StepExecutionListener` 타입으로만 등록해도 `checkAndAddItemListener`가 내부적으로 `ItemReadListener`임을 감지하여 적절히 등록해줍니다. 따라서 **가장 범위가 넓은 상위 리스너 타입으로 등록**하는 것이 코드를 간결하게 유지하는 비결입니다.

### 3. 자원 관리와 리스너 자동화 (registerAsStreamsAndListeners)

`ItemReader`, `ItemProcessor`, `ItemWriter`가 `ItemStream`이나 `StepListener`를 구현하고 있다면, 빌더는 이를 자동으로 인식하여 등록합니다.

```java
// SimpleStepBuilder.java
protected void registerAsStreamsAndListeners(ItemReader<? extends I> itemReader, 
                                             ItemProcessor<? super I, ? extends O> itemProcessor, 
                                             ItemWriter<? super O> itemWriter) {
    // 내부적으로 각 객체가 ItemStream이나 StepListener를 구현했는지 확인 후 자동 등록
}
```

* **자동화의 실체:** 개발자가 직접 `.stream()` 메서드를 호출하지 않아도 리소스 초기화(`open()`)와 상태 저장(`update()`)이 정상적으로 수행되는 이유가 바로 이 코드에 있습니다.

## 2장. 스텝 해부학 II: TaskletStep의 심장과 반복 엔진

놀라운 진실은 **'태스크릿 지향'과 '청크 지향' 모두 실체는 TaskletStep**이라는 점입니다. 청크 지향 처리는 단지 `ChunkOrientedTasklet`이라는 특수한 태스크릿 구현체를 사용하는 `TaskletStep`일 뿐입니다.

### 1. AbstractTaskletStepBuilder의 조립 공정

`TaskletStep`이 생성되고 핵심 부품들이 장착되는 과정입니다.

```java
// AbstractTaskletStepBuilder.java
public TaskletStep build() {
    registerStepListenerAsChunkListener(); // StepExecutionListener가 ChunkListener 구현 시 자동 등록
    TaskletStep step = new TaskletStep(getName()); // 스텝 인스턴스 생성
    super.enhance(step); // 이름, JobRepository, StepExecutionListener 설정

    step.setChunkListeners(chunkListeners.toArray(new ChunkListener));
    step.setTransactionManager(transactionManager);
    step.setTransactionAttribute(transactionAttribute);

    // 심장 이식: stepOperations (RepeatTemplate)
    if (stepOperations == null) {
        stepOperations = new RepeatTemplate();
    }
    
    step.setTasklet(createTasklet()); // ChunkOrientedTasklet 또는 사용자 Tasklet 설정
    step.setStepOperations(stepOperations);
    step.setStreams(streams.toArray(new ItemStream));

    return step;
}
```

* **stepOperations:** 청크 단위 작업(read/process/write 사이클)을 반복하는 **스텝의 심장**입니다. 기본적으로 `DefaultResultCompletionPolicy`를 사용하여 `RepeatStatus.FINISHED`가 반환될 때까지 반복합니다.

### 2. ChunkOrientedTasklet의 생성과 구성

청크 지향 처리의 실제 엔진인 `ChunkOrientedTasklet`은 `ChunkProvider`와 `ChunkProcessor`라는 두 날개로 움직입니다.

```java
// SimpleStepBuilder.java
@Override
protected Tasklet createTasklet() {
    RepeatOperations repeatOperations = createChunkOperations(); // chunkOperations (읽기 반복 엔진)
    SimpleChunkProvider<I> chunkProvider = new SimpleChunkProvider<>(getReader(), repeatOperations);
    SimpleChunkProcessor<I, O> chunkProcessor = new SimpleChunkProcessor<>(getProcessor(), getWriter());
    
    // 리스너 배분
    chunkProvider.setListeners(new ArrayList<>(itemListeners));
    chunkProcessor.setListeners(new ArrayList<>(itemListeners));
    
    ChunkOrientedTasklet<I> tasklet = new ChunkOrientedTasklet<>(chunkProvider, chunkProcessor);
    tasklet.setBuffering(!readerTransactionalQueue);
    return tasklet;
}
```

* **chunkOperations:** 아이템 읽기(`ItemReader.read()`)를 반복하는 엔진입니다. `SimpleCompletionPolicy`를 사용하여 `chunkSize`에 도달할 때까지 반복합니다.
* **ChunkProvider:** 데이터를 읽어 청크를 구성하는 역할.
* **ChunkProcessor:** 구성된 청크를 가공하고 저장하는 역할.

## 3장. 스텝 실행 흐름: 핸들러에서 라이터까지의 대장정

구성이 완료된 스텝이 실제로 어떻게 실행되는지 그 호출 스택을 추적해 보겠습니다.

### 1. SimpleStepHandler: 전투의 지휘관

Job은 `SimpleStepHandler`를 통해 스텝 실행을 명령합니다.

```java
// SimpleStepHandler.java
@Override
public StepExecution handleStep(Step step, JobExecution execution) throws ... {
    // StepExecution과 ExecutionContext 생성
    StepExecution stepExecution = execution.createStepExecution(step.getName()); 
    // Step의 execute() 실행
    step.execute(stepExecution); 
}
```

* **치명적 결함의 근거:** 위 코드에서 `step.getName()`을 호출할 때는 아직 스텝이 실행 전이므로 `StepScope`가 비활성 상태입니다. 따라서 **Step 빈 자체에 @StepScope를 선언하면 이 시점에 예외가 발생**하여 시스템이 붕괴됩니다.

### 2. AbstractStep과 TaskletStep: 트랜잭션의 보호막

스텝 실행이 시작되면 `StepScope`가 활성화되고 트랜잭션 경계가 설정됩니다.

```java
// AbstractStep.java
public final void execute(StepExecution stepExecution) {
    doExecutionRegistration(stepExecution); // StepScope 활성화 (ThreadLocal 바인딩)
    listener.beforeStep(stepExecution);
    open(stepExecution.getExecutionContext()); // ItemStream.open() 호출

    doExecute(stepExecution); // TaskletStep에서 구현한 핵심 로직 실행
    
    doExecutionRelease(); // StepScope 비활성화 및 정리
}
```

### 3. 청크 반복의 실체 (TaskletStep.doExecute)

`stepOperations`가 돌면서 트랜잭션 템플릿 안에서 태스크릿을 집행합니다.

```java
// TaskletStep.java 내부 (의사코드 및 핵심 로직)
stepOperations.iterate(new StepContextRepeatCallback(stepExecution) {
    public RepeatStatus doInChunkContext(...) {
        return new TransactionTemplate(transactionManager).execute(status -> {
            // 청크 단위 트랜잭션 시작
            StepContribution contribution = stepExecution.createStepContribution();
            chunkListener.beforeChunk(chunkContext);
            
            // ChunkOrientedTasklet 실행
            RepeatStatus result = tasklet.execute(contribution, chunkContext);
            
            // 통계 반영 (읽기/쓰기/스킵 건수 누적)
            stepExecution.apply(contribution); 
            return result;
        });
    }
});
```

### 4. 청크 처리의 최종 단계: Provider와 Processor

`ChunkOrientedTasklet` 내부에서는 읽기와 처리가 다음과 같이 분리되어 실행됩니다.

* **SimpleChunkProvider.provide():** `chunkOperations`를 돌리며 `itemReader.read()`를 호출합니다. `chunkSize`를 채우거나 리더가 `null`을 반환하면 멈춥니다.
* **SimpleChunkProcessor.process():** `transform()`을 통해 청크 내 아이템을 하나씩 `itemProcessor.process()`하고, 마지막에 `itemWriter.write()`를 호출하여 청크 전체를 일괄 저장합니다.

```java
// SimpleChunkProcessor.java
protected Chunk<O> transform(StepContribution contribution, Chunk<I> inputs) {
    Chunk<O> outputs = new Chunk<>();
    for (I item : inputs) {
        O output = doProcess(item); // ItemProcessor 호출
        if (output != null) outputs.add(output);
    }
    return outputs;
}

protected void write(StepContribution contribution, Chunk<I> inputs, Chunk<O> outputs) {
    doWrite(outputs); // ItemWriter.write() 호출
    contribution.incrementWriteCount(outputs.size());
}
```

## [개념 검증용 자가 진단 질문]

### 스텝 아키텍처 및 내부 메커니즘 심화 진단

1. `SimpleStepBuilder`의 `build()` 과정 중 `registerStepListenerAsItemListener()` 메서드가 수행하는 역할과, 이를 통해 개발자가 얻을 수 있는 설계상의 이점(특히 다중 인터페이스 구현 시)을 설명하시오.
2. `TaskletStep` 내부에서 `stepOperations`와 `chunkOperations`라는 두 가지 `RepeatTemplate`이 존재한다. 이 두 엔진은 각각 무엇을 반복하며, 각각의 반복이 종료되는 결정적 조건(CompletionPolicy 관점)은 무엇인지 기술하시오.
3. `SimpleStepHandler`가 스텝의 `execute()`를 호출하기 전에 `step.getName()`을 먼저 호출하는 과정이, 왜 스텝 빈(Step Bean)에 `@StepScope`를 설정했을 때 `ScopeNotActiveException`을 유발하는 치명적인 원인이 되는지 내부 바인딩 메커니즘과 연계하여 논술하시오.

* **1번 질문 해설:** 이 메서드는 빌더에 등록된 `StepExecutionListener`나 `ChunkListener`가 `ItemReadListener` 등 하위 리스너 인터페이스를 동시에 구현했는지 전수 조사합니다. 만약 구현했다면 자동으로 해당 아이템 리스너로도 등록해줍니다. 이 설계 덕분에 개발자는 하나의 클래스에 여러 리스너를 구현하더라도 빌더에 단 한 번만(`listener()` 메서드 한 번 호출) 등록하면 되며, 이는 객체의 응집도를 높이고 설정 코드를 획기적으로 줄여주는 이점을 제공합니다.

* **2번 질문 해설:** `stepOperations`는 **청크 단위 작업(Tasklet 실행)**을 반복하며, `RepeatStatus.FINISHED`가 반환될 때까지 계속됩니다. 반면 `chunkOperations`는 **아이템 읽기(ItemReader.read())**를 반복합니다. `chunkOperations`의 종료 조건은 두 가지인데, 하나는 읽은 아이템 수가 설정된 `chunkSize`에 도달했을 때(`SimpleCompletionPolicy`)이고, 다른 하나는 리더가 `null`을 반환하여 더 이상 읽을 데이터가 없을 때(`DefaultResultCompletionPolicy`의 FINISHED 인식)입니다.

* **3번 질문 해설:** 스텝의 실제 `execute()`가 실행되어야 내부의 `doExecutionRegistration()`이 호출되면서 비로소 `ThreadLocal`에 `StepExecution`이 바인딩되고 `StepScope`가 활성화됩니다. 그러나 `SimpleStepHandler`는 스텝을 실행하기 위한 `StepExecution` 메타데이터를 생성하기 위해 **스텝 실행 직전**에 `step.getName()`을 호출합니다. 이때 스텝 빈이 `@StepScope` 프록시라면 이름을 가져오기 위해 실제 빈 생성을 시도하지만, 아직 바인딩 전이라 스코프가 활성 상태가 아니므로 예외가 발생하는 것입니다.
