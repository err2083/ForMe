# [심화 분석 학습 노트] 스프링 배치: RDB 정복 아키텍처와 내부 동작 원리

## 1장. 배치의 생존 전략: 대용량 데이터 처리의 철학

웹 애플리케이션의 세계와 배치의 세계는 물리 법칙 자체가 다릅니다. 웹이 단일 레코드의 빠른 응답을 중시한다면, 배치는 **수천만 건의 데이터를 메모리 고갈 없이 안전하게 완수**하는 것에 목숨을 겁니다.

* **웹의 방식:** `SELECT * FROM users WHERE id = 123;` (가볍고 빠름)
* **배치의 현실:** `SELECT * FROM users WHERE created_at >= '2024-01-01';` (수백만 건의 로드)
* **경고:** 대량 데이터를 한 번에 조회하면 JVM은 `OutOfMemoryError`라는 묘비명을 남기고 사망합니다. 이를 방지하기 위해 스프링 배치는 **분할 처리(Chunk Processing)** 전략을 사용하며, 그 중심에 두 가지 읽기 무기인 **커서(Cursor)**와 **페이징(Paging)**이 존재합니다.

## 2장. JDBC ItemReader & Writer: 데이터베이스를 굴복시키는 법

### 1. JdbcCursorItemReader: 스트리밍으로 데이터를 제압하라

데이터베이스와 커넥션을 유지한 채 `ResultSet.next()`를 통해 데이터를 한 줄씩 읽어오는 방식입니다.

#### [핵심 구성 요소]

* **DataSource:** DB 연결 관리 (HikariCP 등).

* **RowMapper:** `ResultSet`의 원시 데이터를 Java 객체로 변환.
* **PreparedStatement:** 쿼리 실행 및 결과 조회.
* **PreparedStatementSetter:** 동적 쿼리 파라미터 바인딩.

#### [소스코드 및 상세 분석]

```java
@Bean
public JdbcCursorItemReader<Victim> terminatedVictimReader() {
    return new JdbcCursorItemReaderBuilder<Victim>()
        .name("terminatedVictimReader") // ItemReader의 이름을 설정. ExecutionContext 내에서 구분자로 사용됨.
        .dataSource(dataSource) // 주입받은 DB 연결 정보를 설정.
        .sql("SELECT * FROM victims WHERE status = ? AND terminated_at <= ?") // 실행할 쿼리 정의. ?는 플레이스홀더.
        .queryArguments(List.of("TERMINATED", LocalDateTime.now())) // ? 위치에 들어갈 파라미터 주입. 내부적으로 ArgumentPreparedStatementSetter 작동.
        .beanRowMapper(Victim.class) // BeanPropertyRowMapper를 사용하여 DB 컬럼(snake_case)을 객체 필드(camelCase)로 자동 매핑.
        .build();
}
```

* **동작 원리:** `open()` 시점에 쿼리를 실행하여 커서를 열고, `read()` 호출 시마다 커서를 이동하며 데이터를 가져옵니다.
* **성능 최적화 (Fetch Size):** 실제로 매번 네트워크 통신을 하는 것이 아니라, JDBC 드라이버가 `fetchSize`만큼 데이터를 미리 버퍼링합니다.
  * **MySQL 주의:** `useCursorFetch=true` 설정이 없으면 전체 결과를 한 번에 메모리에 올리려 하므로 반드시 접속 속성에 추가해야 합니다.
* **커서 연속성:** Step의 트랜잭션과 별개의 커넥션을 사용하여 트랜잭션 커밋 여부와 상관없이 데이터를 끝까지 읽습니다.
* **ORDER BY의 필수성:** 실패 후 재시작 시 `jumpToItem()`이 정확한 위치로 이동하려면 정렬 순서가 보장되어야 합니다.

### 2. JdbcPagingItemReader: 페이지 단위로 처형하라

데이터를 일정 크기(Page)로 잘라 읽으며, 각 페이지마다 새로운 쿼리를 실행하여 DB 커넥션 점유를 최소화합니다.

#### [핵심 메커니즘: Keyset 페이징]

* **Offset 방식의 문제:** `OFFSET 10000`은 앞의 1만 개를 읽고 버리므로 뒤로 갈수록 느려집니다.

* **Keyset 방식:** `WHERE id > 1000 LIMIT 10`처럼 이전 페이지의 마지막 키값을 기준으로 조회하여 성능을 일정하게 유지합니다.

#### [소스코드 및 상세 분석]

```java
@Bean
public JdbcPagingItemReader<Victim> terminatedVictimReader() {
    return new JdbcPagingItemReaderBuilder<Victim>()
        .name("terminatedVictimReader")
        .dataSource(dataSource)
        .pageSize(5) // 한 번의 쿼리로 가져올 데이터 양. chunkSize와 일치시키는 것이 관리상 유리.
        .selectClause("SELECT id, name, process_id, terminated_at, status") // 조회할 필드 정의.
        .fromClause("FROM victims") // 조회할 테이블 정의.
        .whereClause("WHERE status = :status AND terminated_at <= :terminatedAt") // Named Parameter 사용. 가독성 향상.
        .sortKeys(Map.of("id", Order.ASCENDING)) // Keyset 페이징의 핵심. 정렬 키는 반드시 유니크해야 함(PK 권장).
        .parameterValues(Map.of(
            "status", "TERMINATED",
            "terminatedAt", LocalDateTime.now()
        )) // Named Parameter에 실제 값 바인딩.
        .beanRowMapper(Victim.class)
        .build();
}
```

* **PagingQueryProvider:** DB 타입(MySQL, Oracle 등)에 맞는 최적의 페이징 쿼리를 자동으로 생성합니다. `firstPageSql`과 `remainingPagesSql`을 내부적으로 구분하여 관리합니다.

### 3. JdbcBatchItemWriter: 효율적인 쓰기의 정수

개별 `INSERT`가 아닌 `batchUpdate`를 활용하여 수백 건의 데이터를 한 번의 네트워크 패킷으로 묶어 전송합니다.

#### [소스코드 및 상세 분석]

```java
@Bean
public JdbcBatchItemWriter<HackedOrder> orderStatusWriter() {
    return new JdbcBatchItemWriterBuilder<HackedOrder>()
        .dataSource(dataSource)
        .sql("UPDATE orders SET status = :status WHERE id = :id") // 실행할 DML 정의.
        .beanMapped() // 객체의 필드명을 네임드 파라미터에 자동으로 매핑.
        .assertUpdates(true) // 업데이트된 행이 0개일 경우 예외를 던져 데이터 무결성 보장.
        .build();
}
```

* **최적화 팁:** MySQL/PostgreSQL 사용 시 `rewriteBatchedStatements=true` 설정을 추가하면 드라이버 레벨에서 더욱 효율적인 Multi-Value INSERT로 변환됩니다.

## 3장. JPA ItemReader & ItemWriter: 엔티티의 영속성 제어

JPA를 사용하는 순간 **정권 교체**가 일어납니다. `DataSourceTransactionManager` 대신 `JpaTransactionManager`가 투입되어 엔티티의 생명주기를 관리합니다.

### 1. JpaCursorItemReader: 엔티티 스트리밍

`EntityManager.getResultStream()`을 사용하여 엔티티를 하나씩 읽어옵니다.

#### [소스코드 및 상세 분석]

```java
@Bean
@StepScope
public JpaCursorItemReader<Post> postBlockReader(
    @Value("#{jobParameters['startDateTime']}") LocalDateTime startDateTime,
    @Value("#{jobParameters['endDateTime']}") LocalDateTime endDateTime
) {
    return new JpaCursorItemReaderBuilder<Post>()
        .name("postBlockReader")
        .entityManagerFactory(entityManagerFactory) // JPA 엔진 주입.
        .queryString("""
            SELECT p FROM Post p JOIN FETCH p.reports r 
            WHERE r.reportedAt >= :startDateTime AND r.reportedAt < :endDateTime
        """) // JPQL 정의. Fetch Join으로 N+1 문제 사전 차단.
        .parameterValues(Map.of(
            "startDateTime", startDateTime,
            "endDateTime", endDateTime
        ))
        .build();
}
```

* **주의:** Hibernate 외의 JPA 구현체는 `getResultStream()`이 내부적으로 `getResultList()`를 호출하여 전체 데이터를 메모리에 올릴 수 있으므로 구현체 확인이 필수입니다.

### 2. JpaPagingItemReader: 성능의 늪과 트랜잭션의 배신

가장 많이 쓰이지만 가장 위험한 Reader입니다. **Offset 기반 페이징**을 수행하기 때문입니다.

#### [문제점 분석: @BatchSize 무력화]

* `JpaPagingItemReader`는 페이지를 읽을 때마다 독자적으로 트랜잭션을 시작하고 커밋합니다 (`transacted(true)`가 기본값).

* 이로 인해 `ItemProcessor`에서 지연 로딩된 엔티티에 접근할 때 이미 트랜잭션이 종료되어 `@BatchSize` 최적화가 동작하지 않고 N+1 문제가 발생합니다.

#### [해결책: transacted(false) 설정]

```java
@Bean
public JpaPagingItemReader<Post> postBlockReader() {
    return new JpaPagingItemReaderBuilder<Post>()
        .name("postBlockReader")
        .entityManagerFactory(entityManagerFactory)
        .queryString("SELECT p FROM Post p ORDER BY p.id ASC")
        .pageSize(5)
        .transacted(false) // Reader 자체의 불필요한 트랜잭션 종료 방지. @BatchSize 활성화.
        .build();
}
```

* **필수 수정:** 트랜잭션이 없으면 `detach()`가 발생하므로, 지연 로딩을 쓰지 말고 `EAGER` 로딩이나 `Fetch Join`을 전략적으로 사용해야 합니다.

### 3. JpaItemWriter: 영속성 컨텍스트 감금

엔티티를 DB로 흘려보내는 마지막 관문입니다.

#### [소스코드 및 상세 분석]

```java
@Bean
public JpaItemWriter<BlockedPost> postBlockWriter() {
    return new JpaItemWriterBuilder<BlockedPost>()
        .entityManagerFactory(entityManagerFactory)
        .usePersist(true) // 새로운 데이터는 persist() 호출. false일 경우 merge() 호출.
        .build();
}
```

* **ID 전략의 결정적 차이:**
  * **IDENTITY:** DB에 넣어야 ID를 알 수 있으므로 하이버네이트의 쓰기 지연(Batch Update)이 무력화됩니다.
  * **SEQUENCE:** ID를 미리 가져올 수 있어 Batch Update가 가능합니다. 배치 처리 성능을 위해서는 **SEQUENCE 전략과 충분한 allocationSize**가 필수입니다.

## [개념 검증용 자가 진단 질문]

### 관계형 데이터베이스 배치 아키텍처 심화 진단

1. `JdbcCursorItemReader`를 사용할 때 MySQL 환경에서 `useCursorFetch=true` 설정이 누락되었을 경우 발생하는 내부적인 현상과, 그것이 배치 시스템의 안정성에 끼치는 치명적인 영향은 무엇인가?
2. `JpaPagingItemReader`에서 `transacted` 설정값이 기본값(`true`)일 때, 왜 `ItemProcessor`에서 연관 엔티티를 조회할 때 `@BatchSize`가 의도대로 동작하지 않는지 트랜잭션 생명주기 관점에서 설명하시오.
3. 배치 처리 성능을 극대화하기 위해 JPA의 ID 생성 전략으로 `IDENTITY`보다 `SEQUENCE`를 권장하는 기술적인 이유를 하이버네이트의 '쓰기 지연(Write-behind)' 메커니즘과 연계하여 논술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** MySQL 드라이버는 기본적으로 쿼리 결과 전체를 한 번에 메모리로 로딩하려는 성질이 있습니다. `useCursorFetch=true`가 없으면 `JdbcCursorItemReader`가 한 행씩 읽으려 해도 드라이버가 이미 수백만 건의 데이터를 JVM 힙 메모리에 적재해 버립니다. 이는 즉각적인 `OutOfMemoryError`를 유발하며, 배치가 스트리밍 방식으로 동작한다는 설계를 무력화시켜 시스템을 중단시킵니다.

* **2번 질문 해설:** `JpaPagingItemReader`의 `doReadPage()`는 `transacted`가 `true`일 때 페이지 조회 직전에 트랜잭션을 시작하고, `getResultList()` 호출 직후 즉시 커밋 및 종료합니다. 반면 `@BatchSize`나 지연 로딩은 동일한 트랜잭션 범위 내에서 연관 관계에 접근할 때 작동합니다. Reader에서 이미 트랜잭션이 닫혔기 때문에, 이후 단계인 `ItemProcessor`에서 연관 엔티티에 접근하면 영속성 컨텍스트가 닫혀 있거나 트랜잭션이 없어 일괄 조회가 불가능해지고 N+1 문제가 개별적으로 발생하게 됩니다.

* **3번 질문 해설:** 하이버네이트는 성능 향상을 위해 엔티티 변경 사항을 모았다가 트랜잭션 커밋 시점에 배치 업데이트(`executeBatch`)를 실행합니다. 이를 위해서는 엔티티가 영속화될 때 식별자(ID)를 미리 알고 있어야 합니다. `SEQUENCE` 전략은 DB 시퀀스에서 ID를 미리 발급받아 메모리에 올릴 수 있어 쓰기 지연이 가능하지만, `IDENTITY` 전략은 실제 `INSERT`를 실행해야만 DB로부터 ID를 확인할 수 있습니다. 따라서 하이버네이트는 `persist()` 호출 즉시 `INSERT` 쿼리를 날려야만 하고, 결과적으로 여러 건의 `INSERT`를 하나로 묶는 배치 업데이트 기능을 사용할 수 없게 되어 성능이 저하됩니다.

</details>
