# [심화 분석 학습 노트] 스프링 배치: 테스트 무결성 검증과 정밀 타격 아키텍처

## 1장. 작전 준비: 배치 테스트 환경의 기초 공사

배치 시스템은 일반적인 웹 애플리케이션과 달리 프레임워크가 관리하는 메타데이터(JobRepository)와 강력하게 결합되어 있습니다. 따라서 테스트 역시 이 인프라를 고려한 특수한 무장이 필요합니다.

### 1. 무기 장착 (의존성 추가)

먼저 `build.gradle`에 배치 테스트 전용 라이브러리를 추가해야 합니다.

```java
dependencies { 
    // 💀 추가하라: 기본 Spring Boot Test - 이건 기본 중의 기본 💀 
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
    // 💀 핵심 무기: Spring Batch Test 유틸리티 💀
    testImplementation 'org.springframework.batch:spring-batch-test'
}
```

* **spring-batch-test:** `JobLauncherTestUtils`, `MetaDataInstanceFactory` 등 배치 전용 테스트 도구들을 포함하고 있는 핵심 라이브러리입니다.

### 2. 테스트 전용 H2 환경 및 자동 실행 봉쇄

운영 DB를 오염시키지 않기 위해 메모리 DB인 H2를 구성하고, 테스트 시 Job이 멋대로 실행되는 것을 막아야 합니다.

**[application-test.yml]**

```yaml
spring: 
  datasource: 
    url: jdbc:h2:mem:testdb;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE 
    driver-class-name: org.h2.Driver 
    username: sa 
    password: 
  sql: 
    init: 
      mode: always 
      schema-locations: classpath:schema.sql 
  batch: 
    jdbc: 
      initialize-schema: always 
    job: 
      enabled: false # 💀 핵심: JobLauncherApplicationRunner 자동 실행 방지 💀
```

* **spring.batch.job.enabled: false:** 이 설정은 매우 중요합니다. `JobLauncherApplicationRunner`는 이 값이 `true`(기본값)일 때 컨텍스트 로드와 동시에 모든 Job을 실행하려 합니다. 테스트에서는 우리가 원하는 시점에 직접 Job을 실행해야 하므로 이를 반드시 꺼두어야 합니다.

## 2장. E2E 테스트: Job과 Step의 통합 처형 작전

E2E(End-to-End) 테스트는 배치 잡 전체 또는 개별 스텝을 실제와 유사한 환경에서 실행하며 컴포넌트 간의 연동을 검증합니다.

### 1. @SpringBatchTest와 JobLauncherTestUtils

배치 테스트의 심장부입니다. 이 어노테이션은 배치 테스트에 필요한 유틸리티들을 자동으로 빈으로 등록해줍니다.

```java
@SpringBatchTest // 💀 배치 테스트 핵심 유틸리티 자동 구성 💀
@SpringBootTest 
@ActiveProfiles("test") 
class InFearLearnStudentsBrainWashJobTest { 
    @Autowired 
    private JobLauncherTestUtils jobLauncherTestUtils; // 💀 Job/Step 실행 사령부 💀

    @PostConstruct 
    public void configureJobLauncherTestUtils() throws Exception { 
        // 💀 여러 Job 빈이 존재할 경우 테스트 대상 Job을 명시적으로 지정 💀
        jobLauncherTestUtils.setJob(inFearLearnStudentsBrainWashJob); 
    }
}
```

* **JobLauncherTestUtils:** `launchJob()`, `launchStep()` 메서드를 통해 실행을 제어합니다.
* **setJob():** 컨텍스트에 Job 빈이 여러 개일 경우 `BatchTestContextBeanPostProcessor`가 어떤 Job을 주입할지 결정하지 못하므로, 위와 같이 명시적으로 타겟을 지정해야 합니다.

### 2. 고유 파라미터 생성 (JobInstance 중복 회피)

동일한 파라미터로 Job을 다시 실행하면 `JobInstanceAlreadyCompleteException`이 발생합니다. 이를 피하기 위해 유니크한 파라미터가 필요합니다.

```java
// 💀 getUniqueJobParametersBuilder(): 테스트용 고유 JobParameters 생성 💀
JobParameters jobParameters = jobLauncherTestUtils.getUniqueJobParametersBuilder() 
    .addString("filePath", tempDir.toString()) 
    .toJobParameters();

// 💀 launchJob(): 실제 운영 환경과 동일한 방식으로 Job 실행 💀
JobExecution jobExecution = jobLauncherTestUtils.launchJob(jobParameters);
```

* **getUniqueJobParametersBuilder():** 내부적으로 `random`이라는 키에 랜덤 Long 값을 담아 매 실행마다 새로운 `JobInstance`가 생성되도록 보장합니다.

### 3. 개별 Step 테스트와 임시 Job (TestJob)

복잡한 워크플로우에서 특정 스텝만 떼어내어 검증하고 싶을 때 `launchStep()`을 사용합니다.

```java
@Test 
@DisplayName("💀 세뇌 Step 실행 후 출력 파일 및 컨텍스트 검증") 
void shouldExecuteBrainwashStepAndVerifyOutput() throws IOException { 
    // ... Given 설정 ...
    // 💀 launchStep(): 특정 스텝만 독립적으로 실행 💀
    JobExecution jobExecution = jobLauncherTestUtils.launchStep("inFearLearnStudentsBrainWashStep", jobParameters); 
    
    // 💀 결과 검증 💀
    assertThat(jobExecution.getStatus()).isEqualTo(BatchStatus.COMPLETED);
}
```

* **TestJob의 실체:** `launchStep()` 호출 시 로그에 `TestJob`이라는 이름이 등장합니다. 이는 `JobLauncherTestUtils`가 내부적으로 `StepRunner`를 사용하여 해당 스텝만을 실행하기 위해 생성하는 **임시 래퍼(Wrapper) Job**입니다.

## 3장. 단위 테스트: @StepScope 지뢰밭 돌파와 정밀 타격

단위 테스트는 `ItemReader`, `ItemProcessor`, `ItemWriter` 등의 개별 부품을 고립시켜 검증합니다. 이때 가장 큰 난관은 실행 시점에 생성되는 `@StepScope` 빈의 주입 문제입니다.

### 1. 배치 스코프의 활성화 메커니즘

`@SpringBatchTest`는 `StepScopeTestExecutionListener`를 등록하여 테스트 메서드 실행 전후로 배치 스코프를 강제로 활성화합니다.

```java
// 내부적으로 StepSynchronizationManager를 활용 💀
@Override 
public void beforeTestMethod(TestContext testContext) { 
    // 테스트 전 StepScope 활성화 로직
}
```

* **동작 원리:** 테스트 환경에서도 `StepSynchronizationManager`에 `StepExecution`을 바인딩하여, `@Value("#{jobParameters[...]}")`와 같은 지연 바인딩 표현식이 정상 작동하게 만듭니다.

### 2. MetaDataInstanceFactory: 가짜 메타데이터 생성기

테스트를 위해 DB에 실제로 `JobInstance` 등을 생성하는 것은 번거롭습니다. 이때 메타데이터 객체를 즉석에서 제조하는 공장을 사용합니다.

```java
// 💀 MetaDataInstanceFactory를 활용한 커스텀 StepExecution 생성 💀
public StepExecution getStepExecution() { 
    JobParameters jobParameters = new JobParametersBuilder()
        .addString("filePath", "/test/path")
        .toJobParameters();
    
    // 💀 복잡한 계층 구조(Instance -> Execution -> StepExecution)를 한 번에 생성 💀
    return MetaDataInstanceFactory.createStepExecution(jobParameters); 
}
```

* **getStepExecution() 탐지:** `StepScopeTestExecutionListener`는 테스트 클래스 내에 이 이름의 메서드가 있다면 자동으로 호출하여 반환된 `StepExecution`을 컨텍스트에 바인딩합니다.

### 3. StepScopeTestUtils: 유연한 동적 스코프 제어

`getStepExecution()` 오버라이드 방식은 테스트 인스턴스 초기화 시점에 호출되므로 메서드별로 다른 파라미터를 쓰기 어렵습니다. 이를 해결하기 위해 `StepScopeTestUtils`를 투입합니다.

```java
@Test 
@DisplayName("💀 ItemWriter 단위 테스트 - 세뇌 대상 파일 출력 검증") 
void shouldWriteBrainwashedVictimsToFileCorrectly() throws Exception { 
    // Given
    JobParameters jobParameters = new JobParametersBuilder()
        .addString("filePath", tempDir.toString())
        .toJobParameters();
    StepExecution stepExecution = MetaDataInstanceFactory.createStepExecution(jobParameters);

    // 💀 StepScopeTestUtils.doInStepScope(): 명시적으로 스코프를 열고 닫음 💀
    StepScopeTestUtils.doInStepScope(stepExecution, () -> { 
        brainwashedVictimWriter.open(new ExecutionContext()); 
        brainwashedVictimWriter.write(new Chunk<>(brainwashedVictims)); 
        brainwashedVictimWriter.close(); 
        return null; 
    });
}
```

* **doInStepScope():** `Callable`을 인자로 받아, 해당 로직이 실행되는 동안에만 지정한 `StepExecution` 기반의 배치 스코프를 활성화합니다. 이를 통해 각 테스트 메서드마다 독립적이고 동적인 파라미터 주입이 가능해집니다.
* **open/close 수동 호출:** 단위 테스트 시에는 스텝이 자동으로 리소스를 열고 닫아주지 않으므로, `ItemStream` 구현체인 경우 직접 호출해야 파일 핸들이 정상적으로 생성되고 정리됩니다.

## 4장. 고급 리스너 전략: 상태 승격과 실행 순서의 비밀

여러 리더/프로세서/라이터의 상태를 다음 스텝으로 넘기기 위해 `CompositeStepExecutionListener`와 `ExecutionContextPromotionListener`를 조합할 때 주의할 점이 있습니다.

```java
@Bean 
public CompositeStepExecutionListener compositeStepExecutionListener(...) { 
    CompositeStepExecutionListener composite = new CompositeStepExecutionListener(); 
    composite.setListeners(new StepExecutionListener[]{ 
        executionContextPromotionListener, // 💀 1번으로 등록
        brainwashStatisticsListener       // 💀 2번으로 등록
    });
    return composite;
}
```

* **역순 실행의 법칙:** `CompositeStepExecutionListener`는 `beforeStep()`은 등록 순서대로, **`afterStep()`은 등록 순서의 역순**으로 호출합니다.
* **인과관계:** `brainwashStatisticsListener`가 통계를 계산하여 컨텍스트에 넣은 후, `executionContextPromotionListener`가 이를 JobContext로 승격시켜야 합니다. 따라서 `afterStep()`에서 후자가 나중에 실행되도록 등록 순서를 역순(전자를 뒤에)으로 배치해야 작전이 성공합니다.

## [개념 검증용 자가 진단 질문]

### 배치 테스트 아키텍처 및 생명주기 제어 진단

1. `JobLauncherTestUtils`를 사용하여 전체 Job이 아닌 `launchStep()`을 호출했을 때, 실행 로그에 실제 Job 이름이 아닌 `TestJob`이라는 정체불명의 이름이 나타나는 내부적인 이유는 무엇이며, 이 과정에서 `StepRunner`는 어떤 역할을 수행하는가?
2. `@StepScope`가 적용된 빈을 단위 테스트할 때, 단순한 `@Autowired` 주입만으로는 `ScopeNotActiveException`을 피할 수 없는 아키텍처적 근거를 제시하고, `StepScopeTestExecutionListener`가 `StepSynchronizationManager`와 상호작용하여 이 문제를 어떻게 해결하는지 상세히 기술하시오.
3. 테스트 클래스에서 `getStepExecution()` 메서드를 오버라이드하여 커스텀 `StepExecution`을 제공하는 방식과 비교하여, `StepScopeTestUtils.doInStepScope()`를 사용했을 때 얻을 수 있는 **'테스트 유연성'**의 핵심은 무엇이며, 특히 JUnit의 `@TempDir` 초기화 시점과 연계하여 그 필요성을 논술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** 스프링 배치에서 스텝은 단독으로 실행될 수 없으며 반드시 부모 Job의 컨텍스트 내에서 실행되어야 합니다. `launchStep()` 호출 시 `JobLauncherTestUtils`는 내부의 `StepRunner`를 가동합니다. `StepRunner`는 해당 스텝만을 실행하기 위해 런타임에 `SimpleJob` 인스턴스를 즉석에서 생성하고 이름을 `TestJob`으로 명명합니다. 이 임시 Job에 타겟 스텝을 설정하고 실행함으로써, 스텝이 필요로 하는 `JobExecution` 및 `ExecutionContext` 인프라를 제공하는 일종의 '가상 시뮬레이션 환경'을 구축하는 것입니다.

* **2번 질문 해설:** `@StepScope` 빈은 프록시 객체로 생성되며, 실제 빈 인스턴스는 `ThreadLocal`에 유효한 `StepContext`가 바인딩되어 있을 때만 생성됩니다. 일반 테스트 코드는 이 바인딩 과정이 없으므로 프록시가 실제 객체에 접근하려 할 때 예외가 발생합니다. `StepScopeTestExecutionListener`는 테스트 메서드 실행 전(`beforeTestMethod`)에 `StepSynchronizationManager.register(stepExecution)`를 호출하여 인위적으로 `StepExecution`을 현재 스레드에 바인딩합니다. 이로 인해 테스트 스레드 내에서도 배치 스코프가 활성화되어 `@Value` 등을 통한 파라미터 주입이 정상적으로 이루어지게 됩니다.

* **3번 질문 해설:** `getStepExecution()` 방식은 `StepScopeTestExecutionListener`에 의해 테스트 인스턴스가 준비되는 시점(BeforeClass/BeforeAll 단계와 유사)에 호출됩니다. 이 시점에는 JUnit의 `@TempDir` 필드가 아직 주입되거나 초기화되지 않았을 가능성이 높아, 파일 경로 등을 파라미터로 넘기기 위해 부가적인 임시 디렉토리 생성 로직이 필요해집니다. 반면 `StepScopeTestUtils.doInStepScope()`는 실제 `@Test` 메서드 내부에서 실행되므로, 이미 초기화가 완료된 `@TempDir`에 자유롭게 접근할 수 있습니다. 또한 메서드마다 각기 다른 `JobParameters`를 동적으로 생성하여 주입할 수 있어, 다양한 시나리오를 하나의 테스트 클래스에서 검증하기에 훨씬 유리한 구조를 제공합니다.
