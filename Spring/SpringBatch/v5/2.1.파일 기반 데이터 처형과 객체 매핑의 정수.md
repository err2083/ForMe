# [심화 분석 학습 노트] 스프링 배치: 파일 기반 데이터 처형과 객체 매핑의 정수

## 1장. 배치의 근간: 청크 지향 처리와 파일 처리의 지옥문

배치 처리의 본질은 단순하지만, 대량의 데이터를 다루는 순간 시스템의 생존을 건 전투가 시작됩니다. 스프링 배치는 이를 **청크 지향 처리(Chunk-Oriented Processing)**라는 강력한 전략으로 돌파합니다.

### 1. 청크 지향 처리(COP): 대량 데이터의 분쇄 작전

데이터를 청크(Chunk)라는 단위로 잘게 썰어 처리하는 방식입니다. 이는 메모리 최적화와 트랜잭션 관리의 핵심입니다.

* **ItemReader (첫 번째 처형자):** 데이터를 하나씩 읽어들입니다. 청크 크기만큼 이 과정이 반복됩니다.
* **ItemProcessor (두 번째 처형자):** 읽어들인 데이터를 가공하거나, 부적격한 데이터를 걸러냅니다.
* **ItemWriter (마지막 처형자):** 청크 단위로 모인 데이터를 한 번에 저장소에 기록합니다.

### 2. 파일 처리의 실체: 지옥의 문

개발자가 직접 `File I/O`를 구현하려 할 때 마주하는 지옥의 시나리오는 다음과 같습니다.

* **자원 관리 실패:** 파일 핸들러 유출로 인한 시스템 붕괴.
* **메모리 고갈:** 대용량 파일을 한 번에 읽으려다 발생하는 `OutOfMemoryError`.
* **예외 처리:** 복잡한 `try-catch` 지옥과 데이터 정합성 파괴.
* **동시성 이슈:** 멀티스레드 환경에서의 데드락과 데이터 레이스.
* **트랜잭션 부재:** 중간 실패 시 수동 롤백의 고통.

**FlatFileItemReader**는 이 모든 저주에서 개발자를 해방시키고 비즈니스 로직에만 집중하게 만드는 구원자입니다.

## 2장. 기술의 해부: FlatFileItemReader와 매핑 메커니즘

**플랫 파일(Flat File)**은 행과 열로 구성된 단순한 텍스트 파일(CSV, TSV 등)입니다. **FlatFileItemReader**는 이 파일들을 한 줄씩 읽어 도메인 객체로 변환합니다.

### 1. LineMapper: 변환의 중추

`LineMapper`는 파일의 한 줄(텍스트)을 객체로 변신시킵니다. 내부 동작의 핵심인 `DefaultLineMapper`는 **2단계 작전**을 수행합니다.

#### [DefaultLineMapper의 내부 로직 분석]

```java
// DefaultLineMapper.java 내부의 핵심 로직 (원문 기반 재구성)
@Override
public T mapLine(String line, int lineNumber) throws Exception {
    FieldSet fieldSet = tokenizer.tokenize(line);  // 1단계: 토큰화 작전
    return fieldSetMapper.mapFieldSet(fieldSet);  // 2단계: 객체 매핑 작전
}
```

* **1단계: LineTokenizer:** 문자열을 의미 있는 필드(Token)로 분리하고 `FieldSet` 객체에 담습니다.
  * `DelimitedLineTokenizer`: 구분자(쉼표 등) 기준 분리.
  * `FixedLengthTokenizer`: 고정 길이 기준 분리.
* **2단계: FieldSetMapper:** `FieldSet`에 담긴 데이터를 실제 자바 객체로 변환합니다.
  * `BeanWrapperFieldSetMapper`: 자바 빈 규약(Setter 메서드)을 사용하여 데이터를 주입합니다.

### 2. FieldSet: 데이터의 컨테이너

`FieldSet`은 단순한 배열이 아닙니다. 토큰화된 데이터(`tokens`)와 각 토큰이 매핑될 프로퍼티 이름 목록(`names`)을 동시에 관리하여 데이터의 정체성을 보장합니다.

## 3장. 실전 전술: 다양한 파일 형식의 정복과 커스터마이징

### 1. 구분자 기반 파일(CSV) 읽기 전략

쉼표 등으로 구분된 파일을 처리하는 방식입니다.

#### [소스코드 및 상세 분석]

```java
@Bean
@StepScope
public FlatFileItemReader<SystemFailure> systemFailureItemReader(
    @Value("#{jobParameters['inputFile']}") String inputFile) {
    return new FlatFileItemReaderBuilder<SystemFailure>()
        .name("systemFailureItemReader") // 리더 식별자. 실행 상태 추적에 사용됨
        .resource(new FileSystemResource(inputFile)) // 처리 대상 파일 경로 설정
        .delimited() // DelimitedLineTokenizer 사용 선언
        .delimiter(",") // 데이터 구분 문자를 쉼표로 지정
        .names("errorId", "errorDateTime", "severity", "processId", "errorMessage") // FieldSet에 부여할 필드명
        .targetType(SystemFailure.class) // 변환할 실제 클래스 타입 지정
        .linesToSkip(1) // 헤더 라인(컬럼명) 건너뛰기
        .strict(true) // 파일 존재 여부 및 데이터 구조를 엄격하게 검증
        .build();
}
```

* **핵심 포인트:** `targetType()`은 런타임에 클래스 정보를 제공하기 위해 필수입니다. 제네릭 타입은 컴파일 시점에 사라지기 때문에(Type Erasure), `BeanWrapperFieldSetMapper`가 새 인스턴스를 생성하려면 실제 클래스 정보가 필요합니다.

### 2. 고정 길이(Fixed Length) 파일 읽기 전략

구분자 없이 정해진 위치와 길이로 데이터를 식별하는 방식입니다.

#### [소스코드 및 상세 분석]

```java
@Bean
@StepScope
public FlatFileItemReader<SystemFailure> systemFailureItemReader(
    @Value("#{jobParameters['inputFile']}") String inputFile) {
    return new FlatFileItemReaderBuilder<SystemFailure>()
        .name("systemFailureItemReader")
        .resource(new FileSystemResource(inputFile))
        .fixedLength() // FixedLengthTokenizer 사용 선언
        .columns(new Range[]{ // 각 필드의 시작과 끝 위치(Range) 정의
            new Range(1, 8),     // errorId
            new Range(9, 29),    // errorDateTime
            new Range(30, 39),   // severity
            new Range(40, 45),   // processId
            new Range(46, 66)    // errorMessage
        })
        .names("errorId", "errorDateTime", "severity", "processId", "errorMessage")
        .targetType(SystemFailure.class)
        .build();
}
```

* **공백 처리:** 고정 길이 파일의 자리 맞춤용 공백은 `DefaultFieldSet`이 내부적으로 `trim()`을 수행하여 깔끔하게 제거합니다.

### 3. 기술적 심화: 커스텀 타입 변환과 복잡한 로그 파싱

#### (1) LocalDateTime 커스텀 변환

`BeanWrapperFieldSetMapper`가 처리하지 못하는 복잡한 타입은 `customEditors()`를 통해 해결합니다.

```java
.customEditors(Map.of(LocalDateTime.class, dateTimeEditor()))

private PropertyEditor dateTimeEditor() {
    return new PropertyEditorSupport() {
        @Override
        public void setAsText(String text) {
            DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
            setValue(LocalDateTime.parse(text, formatter));
        }
    };
}
```

#### (2) 정규식 기반 파싱 (RegexLineTokenizer)

구조가 복잡한 로그 파일은 정규식으로 필드를 추출합니다.

```java
tokenizer.setRegex("\\[\\w+\\]\\[Thread-(\\d+)\\]\\[CPU: \\d+%\\] (.+)");
// ... 빌더 설정 중
.lineTokenizer(tokenizer)
.fieldSetMapper(fieldSet -> new LogEntry(fieldSet.readString(0), fieldSet.readString(1)))
```

#### (3) 다중 형식 혼재 (PatternMatchingCompositeLineMapper)

한 파일에 여러 종류의 라인 패턴(ERROR, ABORT 등)이 섞여 있을 때 사용합니다. 패턴에 따라 다른 `Tokenizer`와 `Mapper`를 유동적으로 적용하는 전략가 역할을 수행합니다.

### 4. 확장 전략: Java 17 Record와 다중 파일 처리

* **Record 지원:** `targetType()`에 자바 `record`를 전달하면 `RecordFieldSetMapper`가 활성화됩니다. 이는 Setter가 아닌 모든 필드를 받는 생성자를 통해 데이터를 주입합니다.
* **MultiResourceItemReader:** 여러 파일을 순차적으로 읽는 리더입니다.

    ```java
    @Bean
    @StepScope
    public MultiResourceItemReader<SystemFailure> multiSystemFailureItemReader(
        @Value("#{jobParameters['inputFilePath']}") String inputFilePath) {
        return new MultiResourceItemReaderBuilder<SystemFailure>()
            .name("multiSystemFailureItemReader")
            .resources(resources) // 대상 Resource 목록 배열
            .delegate(systemFailureFileReader()) // 실제 읽기를 수행할 위임 리더
            .build();
    }
    ```

  * **동작 방식:** 위임받은 `FlatFileItemReader`에서 `resource()` 설정을 제거하고, `MultiResourceItemReader`가 현재 처리 중인 파일을 동적으로 주입하며 순차 처리를 집행합니다.

## [개념 검증용 자가 진단 질문]

### 파일 기반 배치 아키텍처 및 매핑 메커니즘 진단

1. `FlatFileItemReaderBuilder` 설정 시 제네릭 타입으로 대상 객체(예: `SystemFailure`)를 명시했음에도 불구하고, 왜 반드시 `.targetType(SystemFailure.class)`를 추가로 호출해야 하는지 자바 언어의 특성과 연계하여 기술적인 이유를 설명하시오.
2. `FixedLengthTokenizer`를 사용하는 고정 길이 파일 처리 상황에서 `strict(true)` 설정이 데이터 무결성 보장을 위해 수행하는 구체적인 검증 메커니즘은 무엇인가? (단순 파일 존재 여부 제외)
3. `MultiResourceItemReader`가 동작할 때, 내부에 설정된 위임 리더(`delegate`)의 설정에서 왜 `.resource()` 호출을 의도적으로 누락시켜야 하는지, 그리고 이들 사이에서 리소스 주입의 주도권이 어떻게 이동하는지 논술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** 자바는 컴파일 시점에만 제네릭 정보를 유지하고 런타임에는 이를 제거하는 **타입 소거(Type Erasure)** 특성을 가지고 있습니다. 스프링 배치의 `BeanWrapperFieldSetMapper`는 런타임에 새로운 객체 인스턴스를 생성(new)해야 하는데, 제네릭 정보가 사라진 상태에서는 어떤 클래스를 생성해야 할지 알 수 없습니다. 따라서 런타임에 실제 클래스 메타데이터를 제공하기 위해 `targetType()`을 통해 클래스 정보를 명시적으로 전달해야만 리플렉션을 통한 객체 생성이 가능해집니다.

* **2번 질문 해설:** 고정 길이 파일에서의 `strict(true)` 모드는 라인 길이의 엄격한 일치 여부를 검증합니다. 구체적으로, 파일에서 읽어 들인 텍스트 한 줄의 실제 길이를 계산하여, 우리가 `Range` 배열로 지정한 마지막 필드의 끝 위치와 정확히 일치하는지 확인합니다. 만약 데이터가 잘렸거나 정해진 규격보다 길 경우 예외를 발생시켜 잘못된 데이터가 시스템에 유입되는 것을 원천 차단하는 핵심 메커니즘으로 작동합니다.

* **3번 질문 해설:** `MultiResourceItemReader`는 여러 파일을 관리하며 순차적으로 처형하는 '관리자' 역할을 수행합니다. 실제 데이터 파싱은 위임 리더(`delegate`)가 수행하지만, **"어떤 파일을 읽을 것인가"**에 대한 결정권은 관리자인 `MultiResourceItemReader`에게 있습니다. 만약 위임 리더에 특정 리소스를 하드코딩해버리면 관리자가 주는 파일을 읽지 못하게 됩니다. 따라서 위임 리더는 리소스 설정을 비워두고, 관리자가 순차적으로 주입해 주는 리소스를 받아 처리하는 수동적인 구조를 가져야만 다중 파일 처리가 가능해집니다.

</details>
