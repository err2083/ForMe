# [심화 분석 학습 노트] 스프링 배치: Job Squad와 메타데이터 지배 구조

## 1장. 배치 처리의 생존 전략: 추적성과 재시작의 철학

현실의 배치 시스템은 완벽하지 않으며 네트워크 장애, DB 오류 등 수많은 변수로 인해 중단될 수 있습니다. 특히 대용량 데이터 처리 중 99% 지점에서 실패했을 때, 처음부터 다시 시작하는 것은 막대한 리소스 낭비이며 데이터 정합성 문제를 야기합니다.

### 1. 원시적 처리의 한계와 요구사항

실패 지점을 기록하지 않는 단순 루프 방식은 재시작 시 어디서부터 다시 처리해야 할지 알 수 없습니다.

```java
// 이런 원시적 코드는 중간에 실패하면 복구가 불가능하다.
for (int i = 0; i < 1000000; i++) {
    processTransaction(transactions.get(i)); // 999,000번째에서 실패 시 대재앙 💀
}
```

안정적인 배치 시스템을 위해서는 **작업 추적**, **상태 저장**, **실패 지점부터의 재시작** 기능이 필수적입니다. 스프링 배치는 이를 위해 강력한 **메타데이터 관리 시스템**을 제공합니다.

## 2장. 스프링 배치 핵심 도메인: 시스템의 정체성과 상태

### 1. JobInstance (논리적 실행 단위)

Job이 '무엇을 할지' 정의하는 설계도라면, JobInstance는 '언제, 어떤 데이터로 실행되는지'를 구체화한 실체입니다.

* **식별 기준:** Job 이름과 **JobParameters**의 조합이 하나의 고유한 JobInstance를 결정합니다.

```java
// JobBuilder를 통한 고유 이름 부여
new JobBuilder("monthlySalesSettlementJob", jobRepository) 
```

이 이름은 시스템 내에서 특정 Job을 식별하는 키가 되며, 커맨드라인 인자를 통해 실행 대상을 결정합니다.

### 2. JobParameters (신원 확인의 열쇠)

JobParameters는 단순한 매개변수가 아니라 JobInstance의 정체성을 결정하는 핵심 요소입니다.

* **identifying 속성:** 기본값은 `true`이며, 이 값이 `true`인 파라미터들만 JobInstance 식별에 사용됩니다.

```bash
# 커맨드라인 실행 예시 (identifying 설정 포함)
./gradlew bootRun --args='--spring.batch.job.name=brutalizedSystemJob chaos=true,java.lang.Boolean verbose=true,java.lang.String,false'
```

위 예시에서 `verbose`는 `false`로 설정되어 JobInstance 식별에서 제외되므로, 옵션 변경만으로는 새로운 인스턴스가 생성되지 않습니다.

### 3. JobExecution (실제 실행 시도)

JobInstance가 논리적 단위라면, JobExecution은 그 인스턴스의 **실제 실행 시도(또는 이력)**를 나타냅니다. 하나의 JobInstance가 실패하여 여러 번 재시도될 때, 각 시도는 별개의 JobExecution으로 기록됩니다.

* **BatchStatus:** 실행 상태를 나타내는 Enum으로 `COMPLETED`, `FAILED`, `STARTED` 등이 존재합니다.

### 4. JobInstance 재실행 제한과 우회 기술

스프링 배치는 성공(`COMPLETED`)한 JobInstance의 재실행을 엄격히 제한하여 중복 처리를 방지합니다. 이를 우회하여 동일 파라미터로 반복 실행하려면 `JobParametersIncrementer`를 사용합니다.

```java
@Bean
public Job brutalizedSystemJob() {
    return new JobBuilder("brutalizedSystemJob", jobRepository)
        .incrementer(new RunIdIncrementer()) // run.id를 자동으로 증가시켜 시스템을 속임 💀
        .start(brutalizedSystemStep())
        .build();
}
```

`RunIdIncrementer`는 내부적으로 `run.id` 파라미터를 추가하여 매번 새로운 JobInstance가 생성되도록 유도합니다.

## 3장. 메타데이터 저장소와 실행 흐름의 심연

### 1. 물리적 테이블 구조 분석

스프링 배치는 6개의 핵심 테이블을 통해 상태를 감시합니다.

* **BATCH_JOB_INSTANCE:** `JOB_NAME`과 `JOB_KEY`(파라미터 해시값)의 유니크 제약조건을 통해 중복 인스턴스를 차단합니다.
* **BATCH_JOB_EXECUTION:** 실행 시간, 상태, 종료 코드를 기록합니다. `VERSION` 컬럼을 통한 **낙관적 락(Optimistic Lock)**으로 동시성 제어를 수행합니다.

```sql
-- 낙관적 락의 내부 작동 원리
UPDATE BATCH_JOB_INSTANCE SET VERSION = VERSION + 1 
WHERE JOB_INSTANCE_ID = ? AND VERSION = ? -- 버전이 다르면 업데이트 실패 💀
```

* **BATCH_STEP_EXECUTION:** 각 스텝의 읽기/쓰기/롤백 카운트를 기록하며, 부모 `JOB_EXECUTION_ID`를 참조합니다.
* **ExecutionContext (CONTEXT 테이블):** 비즈니스 상태를 JSON/직렬화 형태로 저장합니다.

### 2. Job Squad 실행 메커니즘 (JobLauncher & JobRepository)

실행의 시작점인 `JobLauncher`와 데이터 관리를 담당하는 `JobRepository`의 상호작용을 해부합니다.

#### [JobExecution 생성 로직: SimpleJobRepository.createJobExecution]

```java
public JobExecution createJobExecution(String jobName, JobParameters jobParameters) {
    JobInstance jobInstance = jobInstanceDao.getJobInstance(jobName, jobParameters);
    
    if (jobInstance != null) {
        // 재시작 상황: 이미 성공한 이력이 있는지 검사
        if (!identifyingJobParameters.isEmpty() && (status == BatchStatus.COMPLETED || status == BatchStatus.ABANDONED)) {
            throw new JobInstanceAlreadyCompleteException("이미 완료된 작업입니다. 파라미터를 변경하십시오.");
        }
        // 이전의 ExecutionContext를 복구하여 연결
        executionContext = ecDao.getExecutionContext(jobExecutionDao.getLastJobExecution(jobInstance));
    } else {
        // 첫 실행: 새로운 인스턴스와 컨텍스트 생성
        jobInstance = jobInstanceDao.createJobInstance(jobName, jobParameters);
        executionContext = new ExecutionContext();
    }
    // ... JobExecution 생성 및 즉시 DB 저장
}
```

이 메서드는 재시작 시 이전 상태를 복원하고, 성공한 작업의 중복 실행을 차단하는 제어판 역할을 합니다.

#### [Job 실행과 스코프 활성화: AbstractJob.execute]

실질적인 작업 실행 단계에서는 **JobScope**의 활성화가 핵심입니다.

```java
public final void execute(JobExecution execution) {
    // JobScope 활성화: ThreadLocal에 JobContext 바인딩
    JobSynchronizationManager.register(execution); 
    
    try {
        updateStatus(execution, BatchStatus.STARTED); // DB 상태 업데이트
        listener.beforeJob(execution); // 리스너 호출
        doExecute(execution); // SimpleJob에서 Step 체인 실행
    } finally {
        execution.setEndTime(LocalDateTime.now());
        JobSynchronizationManager.release(); // JobScope 비활성화 및 정리 💀
    }
}
```

`JobSynchronizationManager.register()`가 호출되는 이 시점부터 프록시 객체가 실제 빈에 접근할 수 있게 됩니다.

### 3. Step Squad와 상태 전파

`StepExecution`은 실제 Step의 실행 시도를 나타내며, `SimpleStepHandler`가 그 생명주기를 관리합니다.

* **상태 전파 원리:** `JobExecution`의 최종 상태는 마지막으로 실행된 `StepExecution`의 상태에 의해 결정됩니다. 만약 중간 스텝이 실패하면 이후 스텝은 실행되지 않으며 Job은 `FAILED`가 됩니다.
* **재시작 시 Step 동작:** 이미 `COMPLETED`된 스텝은 기본적으로 건너뛰며, 실패한 스텝부터 새로운 `StepExecution`을 생성하여 실행을 재개합니다.

## [개념 검증용 자가 진단 질문]

### Job 및 메타데이터 아키텍처 심화 진단

1. `JobParameters`의 `identifying` 속성이 `false`로 설정된 파라미터가 포함된 경우, 스프링 배치가 `JobInstanceAlreadyCompleteException`을 판별하는 내부 로직에 어떤 변화가 생기며, 이것이 실제 운영 환경에서 동일 작업의 반복 실행에 어떤 영향을 주는지 기술하시오.
2. `SimpleJobRepository.createJobExecution()` 메서드 내부에서 재시작(Restart) 상황일 때 이전 실행의 `ExecutionContext`를 조회하여 새로 생성되는 `JobExecution`에 설정해주는 과정이, 스프링 배치의 '실패 지점부터 재시작'이라는 메커니즘을 기술적으로 어떻게 완성시키는지 논술하시오.
3. `BATCH_JOB_EXECUTION` 테이블의 `VERSION` 컬럼을 활용한 낙관적 락(Optimistic Lock) 메커니즘이 배치 애플리케이션의 동시성 제어 측면에서 왜 중요한지, 만약 이 장치가 없다면 다중 서버(분산 환경)에서 동일한 `JobExecution`을 조작할 때 발생할 수 있는 데이터 정합성 결함을 설명하시오.

<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** 스프링 배치는 `Job Name`과 `identifying=true`인 `JobParameters`를 해싱하여 `JOB_KEY`를 생성하고 이를 통해 `JobInstance`를 식별합니다. `identifying`이 `false`인 파라미터는 `JOB_KEY` 생성에서 제외됩니다. 따라서 식별 파라미터가 동일하다면 비식별 파라미터(예: 로깅 레벨 등)가 아무리 바뀌어도 동일한 `JobInstance`로 인식됩니다. 이 경우 이전 실행이 `COMPLETED` 상태라면 `JobInstanceAlreadyCompleteException`이 발생하여 재실행이 차단됩니다. 반대로 모든 파라미터가 `non-identifying`이거나 파라미터가 없다면, 현행 버전에서는 상태 검사를 건너뛰어 반복 실행이 가능해지는 구조적 특이점이 존재합니다.

* **2번 질문 해설:** `createJobExecution()`은 재시작 시 메타데이터 저장소에서 가장 최근의 `JobExecution`을 찾아 그에 속한 `ExecutionContext`를 로드합니다. 이 컨텍스트에는 이전 실행의 `ItemStream.update()`를 통해 저장된 '마지막 처리 지점(예: 읽은 아이템 개수)' 정보가 담겨 있습니다. 이 데이터를 새 `JobExecution`에 주입함으로써, 이후 실행되는 스텝의 `ItemStream.open()` 단계에서 해당 정보를 복원하여 처음이 아닌 실패했던 지점부터 데이터를 읽어 들일 수 있는 논리적 연속성을 확보하게 됩니다.

* **3번 질문 해설:** 배치가 분산 환경이나 멀티 스레드 환경에서 실행될 때, 여러 프로세스가 동시에 동일한 `JobExecution`의 상태(예: 종료 시간, 상태 값)를 업데이트하려고 시도할 수 있습니다. `VERSION` 컬럼은 업데이트 시점에 내가 읽었던 버전과 DB의 현재 버전이 일치하는지 `WHERE` 조건으로 확인합니다. 만약 다른 프로세스가 먼저 업데이트하여 버전이 올라갔다면 현재 프로세스의 업데이트는 실패하고 `OptimisticLockingFailureException`이 발생합니다. 이 장치가 없다면 'Lost Update' 문제가 발생하여 최종 작업 상태가 부정확하게 기록되거나, 이미 종료된 작업의 상태를 덮어쓰는 등 메타데이터의 신뢰성이 파괴됩니다.
