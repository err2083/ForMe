# [심화 분석 학습 노트] 스프링 배치: NoSQL 정복과 리포지토리 추상화 아키텍처

## 1장. MongoDB 작전: 도큐먼트 스트리밍과 벌크 처형

MongoDB는 스키마의 제약 없이 JSON 형태의 BSON 도큐먼트를 다루는 무기입니다. 스프링 배치는 이를 처리하기 위해 커서와 페이징이라는 두 가지 상반된 전략을 제공합니다.

### 1. MongoCursorItemReader: 스트리밍으로 데이터를 해방하라

Spring Batch 5.1에서 도입된 신병기로, MongoDB의 커서를 활용해 데이터를 순차적으로 읽어옵니다. 전체를 메모리에 올리지 않고 필요한 만큼만 가져오는 것이 핵심입니다.

* **내부 동작 메커니즘:**
  * **초기 침투:** 첫 쿼리 실행 시 101개의 도큐먼트 선발대를 파견합니다.
  * **후속 공격:** 첫 물량을 처리한 후 `batchSize` 설정에 따라 다음 병력을 요청합니다.
  * **16MB의 한계:** 단일 배치의 최대 크기는 BSON 제한인 16MB를 넘을 수 없습니다.
  * **내부 버퍼링:** `read()` 호출 시 먼저 내부 버퍼를 확인하고, 비어있을 때만 서버에 요청하여 네트워크 비효율을 차단합니다.

#### [실전 코드 분석: 해커 패턴 추적 작전]

```java
@Bean
@StepScope
public MongoCursorItemReader<SecurityLog> securityLogReader(
    @Value("#{jobParameters['searchDate']}") LocalDate searchDate
) {
    // 조회 기간 설정을 위한 날짜 변환 로직
    Date startOfDay = Date.from(searchDate.atStartOfDay(ZoneId.systemDefault()).toInstant());
    Date endOfDay = Date.from(searchDate.plusDays(1).atStartOfDay(ZoneId.systemDefault()).toInstant());

    return new MongoCursorItemReaderBuilder<SecurityLog>()
        .name("securityLogReader") // ItemReader 식별자 설정
        .template(mongoTemplate) // MongoDB 통신의 핵심 엔진 주입
        .collection("security_logs") // 대상 컬렉션 명시 (도메인 클래스에 @Document가 있다면 생략 가능)
        .jsonQuery("{ 'label': 'PENDING_ANALYSIS', 'timestamp': { $gte: ?0, $lt: ?1 } }") // JSON 쿼리 정의 및 플레이스홀더(?0, ?1) 사용
        .parameterValues(List.of(startOfDay, endOfDay)) // 쿼리 파라미터 바인딩
        .sorts(Map.of("timestamp", Sort.Direction.ASCENDING)) // 시간순 분석을 위한 정렬
        .targetType(SecurityLog.class) // 도큐먼트를 변환할 자바 클래스 타입 지정
        .batchSize(10) // 한 번에 가져올 도큐먼트 수 최적화
        .build();
}
```

### 2. Query 객체를 이용한 타입 세이프 작전

문자열 기반의 `jsonQuery` 대신 Spring Data의 `Query` 객체를 직접 주입할 수 있습니다. 이는 자동 완성 지원과 타입 안정성을 제공합니다.

* **기술적 모순(빌더의 버그):** `Query` 객체 내부에 이미 정렬 조건을 넣었더라도, 빌더의 검증 로직 때문에 `sorts()` 메서드를 중복으로 호출해야 하는 이슈가 존재합니다(Spring Batch 5.2 기준).

### 3. MongoPagingItemReader: 성능의 함정과 오프셋 지옥

이 리더는 MongoDB의 `skip()`과 `limit()`을 사용합니다. 하지만 샤딩 환경에서는 치명적입니다.

* **성능 추락의 원인:** `mongos`가 각 샤드의 데이터를 모두 모아 다시 정렬한 뒤 `skip`을 적용하므로, 데이터가 뒤로 갈수록 오버헤드가 기하급수적으로 증가합니다.
* **페이징 지옥:** 배치 처리 중 조건 필드(예: label)를 수정하면, `skip`에 의해 미처리 데이터가 허공으로 사라지는 현상이 발생합니다. 따라서 **커서 기반 방식** 사용이 강력히 권장됩니다.

### 4. MongoItemWriter: 도큐먼트라는 각인을 새기다

`bulkWrite`를 통해 여러 쓰기 연산을 한 방에 처리하여 네트워크 부하를 최소화합니다.

* **Mode의 선택:** `INSERT`(추가), `UPSERT`(수정/추가), `REMOVE`(삭제) 중 선택합니다.
* **복합키의 해방(primaryKeys):** Spring Batch 5.2.3부터 도입된 `primaryKeys()` 메서드를 통해 `_id`가 아닌 비즈니스 키(예: attackerId + timestamp)로 대상을 특정할 수 있습니다.
* **트랜잭션 관리:** `MongoTransactionManager`는 MongoDB 4.0+ 및 레플리카 셋 환경에서만 지원됩니다. 지원되지 않는 환경에서는 커밋 직전까지 쓰기를 지연시키는 **버퍼링 전략**을 사용하여 불완전한 기록을 방지합니다.

## 2장. Redis 작전: 인메모리 스캔과 휘발성 심판

Redis는 디스크의 속박을 벗어난 인메모리 혁명입니다. Spring Batch 5.1부터 전용 리더와 라이터를 지원합니다.

### 1. RedisItemReader: Key들을 스캔하여 처형하라

Redis의 `SCAN` 명령을 사용하는 커서 기반 리더입니다.

* **내부 동작 3단계:**
    1. **초기화:** `open()` 시점에 `SCAN`으로 첫 키 목록 확보.
    2. **개별 타격:** `read()` 호출 시마다 키에 대해 `GET` 호출(아이템당 1회 통신).
    3. **재침투:** 키를 모두 소비하면 추가 `SCAN` 실행.
* **치명적 약점:**
  * **네트워크 폭격:** 아이템당 RTT(왕복 시간)가 발생하여 대량 처리 시 부하가 큽니다.
  * **재시작 불가:** `SCAN`은 순서를 보장하지 않으므로 실패 지점 복구가 불가능합니다. 따라서 빌더에 이름(`name`) 설정조차 없습니다.

#### [실전 코드 분석: 공격 로그 수집]

```java
@Bean
@StepScope
public RedisItemReader<String, AttackLog> attackLogReader() {
    // JSON 직렬화를 위한 ObjectMapper 설정 (JavaTimeModule 포함)
    ObjectMapper objectMapper = new ObjectMapper();
    objectMapper.registerModule(new JavaTimeModule());

    // RedisTemplate 직접 생성: Jackson2JsonRedisSerializer 장착을 위함
    RedisTemplate<String, AttackLog> template = new RedisTemplate<>();
    template.setConnectionFactory(redisConnectionFactory);
    template.setDefaultSerializer(new Jackson2JsonRedisSerializer<>(objectMapper, AttackLog.class));
    template.afterPropertiesSet();

    return new RedisItemReaderBuilder<String, AttackLog>()
        .redisTemplate(template) // 전용 템플릿 주입
        .scanOptions(ScanOptions.scanOptions()
            .match("attack:*") // 조회할 키 패턴 지정
            .count(10) // 한 번의 SCAN으로 가져올 키 개수 힌트 제공
            .build())
        .build();
}
```

### 2. RedisItemWriter: 휘발성 심판대에 세우다

`itemKeyMapper`를 통해 객체에서 키를 추출하여 `SET` 또는 `DEL` 명령을 수행합니다.

* **구성 요소:** `redisTemplate`과 키 추출을 위한 `itemKeyMapper`가 필수입니다.
* **삭제 모드:** `delete(true)` 설정 시 `DEL` 명령을 통해 흔적을 지웁니다.

## 3장. Spring Data Repository 작전: 추상화의 힘

특정 저장소의 기술에 종속되지 않고 일관된 방식으로 처리할 때 사용합니다.

### 1. RepositoryItemReader: 리포지토리의 힘을 빌려 탈취하라

`PagingAndSortingRepository`의 페이징 기능을 활용합니다.

```java
public RepositoryItemReader<HackNote> reader() {
    return new RepositoryItemReaderBuilder<HackNote>()
        .name("hackNoteItemReader")
        .repository(hackNoteRepository) // Spring Data Repository 주입
        .methodName("searchNotesByMessageAndSentimentIsNull") // 호출할 메서드명
        .arguments(List.of("PWN")) // 메서드 인자 리스트
        .pageSize(10) // 페이징 크기
        .sorts(Map.of("timestamp", Sort.Direction.DESC)) // 정렬 기준
        .build();
}
```

* **경고:** 이 역시 오프셋 방식 페이징이므로 데이터 규모에 따른 성능 이슈를 고려해야 합니다.

### 2. RepositoryItemWriter: 심판을 기록하다

`CrudRepository`의 저장 기능을 사용하며, 기본적으로 `saveAll`을 통한 벌크 연산을 수행합니다.

* **Type-safe 검증:** `RepositoryMethodReference`를 사용하면 문자열이 아닌 실제 메서드 호출 방식으로 컴파일 시점에 시그니처를 검증할 수 있습니다.

## [개념 검증용 자가 진단 질문]

### NoSQL 배치 아키텍처 및 메커니즘 심화 진단

1. MongoDB 샤딩 환경에서 `MongoPagingItemReader`가 `MongoCursorItemReader`보다 성능 면에서 압도적으로 불리한 아키텍처적 원인은 무엇이며, 특히 `mongos` 레벨에서 발생하는 구체적인 병목 현상을 논술하시오.
2. `RedisItemReader`가 배치 작업의 핵심 기능 중 하나인 '실패 후 재시작(Restartability)'을 지원하지 못하는 이유를 `SCAN` 명령의 특성과 연계하여 설명하고, 이로 인해 빌더 구성에서 생략되는 설정 요소는 무엇인가?
3. `MongoItemWriter`가 `MongoTransactionManager`를 사용할 수 없는 단일 인스턴스 환경에서, 데이터의 원자적 커밋을 흉내 내기 위해 사용하는 '쓰기 지연(Write-behind) 버퍼링' 전략의 내부 동작 방식과 한계를 기술하시오.

<details>
<summary>💡 자가 진단 질문 정답 및 해설 보기</summary>

* **1번 질문 해설:** `MongoPagingItemReader`는 오프셋 기반 페이징을 사용합니다. 샤딩 환경에서 `mongos`는 각 데이터 샤드에 쿼리를 던지고 결과를 모두 취합한 뒤, 전체 데이터에 대해 다시 정렬을 수행하고 나서야 `offset`만큼 건너뛰는 작업을 처리합니다. 샤드 서버에서의 1차 정렬/스킵보다 `mongos` 서버에서 모든 데이터를 모아 수행하는 2차 정렬/스킵 오버헤드가 훨씬 크기 때문에, 페이지가 뒤로 갈수록 성능은 끝없이 추락하게 됩니다.

* **2번 질문 해설:** `RedisItemReader`의 기반인 `SCAN` 명령은 Redis 내부 데이터 구조상 반환되는 데이터의 순서를 보장하지 않습니다. 배치 재시작 시에는 이전에 어디까지 읽었는지에 대한 순차적 식별이 가능해야 실패 지점부터 재개할 수 있는데, 순서가 매번 바뀌면 이것이 불가능합니다. 이러한 구조적 특성 때문에 `RedisItemReader`는 재시작을 지원하지 않으며, 결과적으로 재시작을 위한 식별값인 이름(`name()`) 설정도 필요하지 않아 구성에서 제외됩니다.

* **3번 질문 해설:** `MongoItemWriter`는 트랜잭션 매니저가 없을 때 `FlatFileItemWriter`와 유사한 전략을 취합니다. 청크 내의 아이템들을 즉시 DB에 쓰지 않고 메모리 내부 버퍼에 보관하다가, 트랜잭션이 커밋되기 직전인 `beforeCommit()` 시점에 `bulkWrite` 연산을 한꺼번에 수행합니다. 이를 통해 네트워크 통신을 최소화하고 불완전한 데이터가 중간에 기록되는 것을 방지하려 노력하지만, 이미 실행된 `bulkWrite` 도중 오류가 발생했을 때 이전 기록을 원자적으로 롤백해주는 진정한 트랜잭션의 기능은 수행하지 못한다는 한계가 있습니다.

</details>
