# [심화 분석 학습 노트] 스프링 배치: 통합 작전과 위임 아키텍처의 정수

## 1장. CompositeItemReader: 통합 처형 프로토콜

데이터가 하나의 소스에만 존재한다는 가정은 전장에서의 방심과 같습니다. 데이터는 여러 샤드(Shard)에 흩어져 있거나, 레거시 시스템과 신규 시스템에 파편화되어 존재하기 마련입니다. **CompositeItemReader**는 이러한 분산된 전력을 하나로 묶어 순차적으로 집행하는 통합 사령부 역할을 수행합니다.

### 1. 동작 메커니즘: 연쇄적 데이터 탈취

이 컴포넌트는 내부적으로 여러 개의 `ItemReader`를 리스트 형태로 보유하며, 이를 순서대로 실행합니다.

* **동작 원리:** 현재 활성화된 `ItemReader`가 데이터를 읽어오다가 더 이상 읽을 데이터가 없어 `null`을 반환하는 순간, 사령부는 즉시 다음 `ItemReader`에게 바톤을 넘깁니다.
* **빌더의 부재:** 이 녀석은 전용 빌더를 제공하지 않습니다. 개발자가 직접 인스턴스를 생성하여 위임 대상들을 주입해야 합니다.

### 2. 실전 코드 분석

```java
// 동일한 타입(Customer)을 반환하는 리더들을 리스트로 구성
List<ItemStreamReader<Customer>> readers = List.of(
    shard1ItemReader, // 첫 번째 샤드에서 데이터를 읽는 리더
    shard2ItemReader  // 두 번째 샤드에서 데이터를 읽는 리더
);

// 직접 생성자를 통해 위임 리더들을 주입하여 통합 리더 생성
CompositeItemReader<Customer> compositeReader = new CompositeItemReader<>(readers);
```

* **List.of(shard1ItemReader, shard2ItemReader):** 위임 대상인 리더들을 순서대로 배치합니다. `shard1`이 완전히 소멸(null 반환)되어야 `shard2` 작전이 시작됩니다.
* **new CompositeItemReader<>(readers):** 통합 사령부를 구축합니다. 내부적으로는 현재 읽고 있는 리더의 인덱스를 관리하며 순차 처리를 집행합니다.

### 3. 전략적 가치

* **샤딩 환경 대응:** 별도의 복잡한 샤드 관리 로직 없이, 각 데이터 소스를 바라보는 개별 리더들만 준비하면 단일 스텝에서 모든 샤드를 평정할 수 있습니다.
* **시스템 통합:** 레거시 DB와 신규 NoSQL의 데이터를 하나의 스트림으로 묶어 처리함으로써 시스템 간 장벽을 무너뜨립니다.

## 2장. CompositeItemWriter: 전방위적 데이터 동시 타격

하나의 데이터를 가공하여 여러 곳에 동시에 각인시켜야 할 때, **CompositeItemWriter**는 가장 강력한 다중 타격 무기가 됩니다.

### 1. 동작 메커니즘: 동시 전파(Broadcast)

청크 단위로 들어온 데이터를 등록된 모든 위임 대상 라이터들에게 순차적으로 전달하여 실행합니다.

```java
// 내부 동작의 본질
public void write(Chunk<? extends T> chunk) throws Exception {
    for (ItemWriter<? super T> writer : delegates) {
        writer.write(chunk);
    }
}
```

* **for 루프 진입:** 위임 사령부에 등록된 라이터들을 하나씩 순회합니다.
* **writer.write(chunk):** 동일한 데이터 덩어리(Chunk)를 모든 라이터에게 똑같이 전달합니다. 이를 통해 RDB 저장, NoSQL 이력 관리, 이메일 발송 등의 작업을 연쇄적으로 폭발시킵니다.

### 2. 구성 전략과 기술적 인과관계

구성은 생성자 주입 방식과 빌더 방식 두 가지를 모두 지원합니다.

```java
// 1. 생성자 주입 방식: 가장 직관적이며 간결함
CompositeItemWriter<Hacker> writer = new CompositeItemWriter<>(
    List.of(firstWriter, secondWriter, thirdWriter)
);

// 2. 빌더 방식: 가독성과 확장성을 고려한 구성
CompositeItemWriter<Hacker> writer = new CompositeItemWriterBuilder<Hacker>()
    .delegates(List.of(firstWriter, secondWriter, thirdWriter)) // 위임 대상 리스트 등록
    .build();
```

* **delegates():** 라이터들의 실행 순서를 결정합니다. `firstWriter`의 성공이 보장되어야 다음 라이터로 진행됩니다.

### 3. [심화] 트랜잭션 동기화와 버퍼링의 비밀

왜 `FlatFileItemWriter`나 `MongoItemWriter`는 데이터를 즉시 쓰지 않고 버퍼링을 할까요? `CompositeItemWriter`를 사용하면 그 이유가 명확해집니다.

* **치명적 불일치 상황:** 만약 `FlatFileItemWriter`가 파일에 즉시 기록하고, 그 뒤에 실행된 `JdbcBatchItemWriter`에서 예외가 발생한다면? DB는 롤백되지만 이미 써버린 파일은 되돌릴 수 없습니다.
* **해결책(지연 쓰기):** 이들은 `beforeCommit()` 시점까지 실제 쓰기를 미룹니다. 모든 라이터의 `write()` 작업이 성공적으로 끝나고 트랜잭션이 커밋되기 직전에야 비로소 파일과 NoSQL에 데이터를 각인합니다. 즉, **모든 라이터의 성공을 확인한 뒤 최후의 커밋을 함께 집행**함으로써 데이터 무결성을 보호합니다.

## 3장. ClassifierCompositeItemWriter: 데이터의 운명을 결정하는 심판관

모든 데이터를 똑같이 대하는 것은 비효율적입니다. 데이터의 특성에 따라 처형 방식(Writer)을 달리해야 할 때, **ClassifierCompositeItemWriter**가 투입됩니다.

### 1. 동작 메커니즘: 정밀 타격 분류

청크 내의 각 아이템을 `Classifier`라는 심판대에 올려, 아이템별로 최적화된 처형자를 동적으로 선택합니다.

* **내부 동작 프로세스:**
    1. 청크 진입 후 아이템을 하나씩 꺼냄.
    2. `classify(item)` 호출: 심판관이 아이템을 분석하여 타격할 라이터를 반환.
    3. 아이템들을 각 라이터별 버킷(Bucket)에 담음.
    4. 청크 분류 완료 후, 각 라이터에게 할당된 아이템들만 전달하여 집행.

### 2. 심판관(Classifier) 설계와 코드 분석

로그의 심각도에 따라 라이터를 선택하는 시나리오를 통해 구조를 파헤쳐 보겠습니다.

```java
public static class SystemLogClassifier implements Classifier<SystemLog, ItemWriter<? super SystemLog>> {
    private final ItemWriter<SystemLog> criticalWriter;
    private final ItemWriter<SystemLog> normalWriter;

    // 생성자를 통해 상황별 처형자(라이터)들을 주입받음
    public SystemLogClassifier(ItemWriter<SystemLog> criticalWriter, ItemWriter<SystemLog> normalWriter) {
        this.criticalWriter = criticalWriter;
        this.normalWriter = normalWriter;
    }

    @Override
    public ItemWriter<? super SystemLog> classify(SystemLog classifiable) {
        // CPU 사용량이 90% 이상이거나 메모리 1GB 초과, 혹은 타입이 CRITICAL인 경우 긴급 처리
        if (classifiable.getCpuUsage() >= 90 || classifiable.getMemoryUsage() > 1024*1024*1024 || "CRITICAL".equals(classifiable.getType())) {
            return criticalWriter;
        }
        // 그 외의 정상 상황은 일반 처리
        return normalWriter;
    }
}
```

* **classify() 메서드:** 데이터의 필드 값을 검사하여 런타임에 처리 전략을 변경하는 핵심 지능입니다.
* **타입 기반 분류:** `instanceof`나 `SubclassClassifier`를 사용하여 객체의 클래스 타입에 따라 라이터를 자동 매핑할 수도 있습니다.

### 3. 시스템 구성 및 실행 결과

```java
@Bean
public ClassifierCompositeItemWriter<SystemLog> classifierWriter() {
    ClassifierCompositeItemWriter<SystemLog> writer = new ClassifierCompositeItemWriter<>();
    // 우리만의 분류 규칙을 가진 심판관을 장착
    writer.setClassifier(new SystemLogClassifier(criticalLogWriter(), normalLogWriter()));
    return writer;
}
```

* **setClassifier():** 이 메서드 하나로 시스템의 분기 로직이 결정됩니다.
* **결과 분석:** CPU 95%인 로그는 `CriticalLogWriter`가 가동되어 긴급 알림을 보내고, 정상 로그는 `NormalLogWriter`가 정적 파일에 조용히 기록합니다.

### 4. [기술적 통찰] ItemStream의 정체

통합 작전에서 쓰이는 위임자들의 코드를 보면 공통적으로 `ItemStream`이라는 시그니처가 발견됩니다.

* **공통점:** `MultiResourceItemReader/Writer`, `CompositeItemReader/Writer` 모두 `ItemStream`을 구현하거나 위임 대상으로 삼고 있습니다.
* **이유:** 상태 관리(ExecutionContext) 때문입니다. 여러 개를 묶어서 실행하더라도 실패 시 어디까지 처리했는지 기억해야 재시작이 가능합니다. 이 위임자들은 내부 위임 대상들이 `ItemStream`인지 확인하고, 그렇다면 그들의 상태 관리 메서드(`open`, `update`, `close`)를 대신 호출해주는 **관리자 역할**까지 수행하는 것입니다.

## [개념 검증용 자가 진단 질문]

### 통합 작전 아키텍처 및 위임 메커니즘 심화 진단

1. `CompositeItemWriter`에 `JdbcBatchItemWriter`(DB용)와 `FlatFileItemWriter`(파일용)를 함께 구성하여 사용할 때, 만약 `FlatFileItemWriter`가 버퍼링 전략을 사용하지 않고 `write()` 호출 즉시 파일에 데이터를 쓴다면 어떤 '데이터 정합성 결함'이 발생할 수 있는지 트랜잭션 롤백 시나리오와 연계하여 논술하시오.
2. `ClassifierCompositeItemWriter`가 청크 단위로 데이터를 처리할 때, 내부적으로 아이템들이 각 위임 라이터(`delegate`)에게 전달되는 순서와 방식은 일반적인 `CompositeItemWriter`와 어떻게 다른지 '분류(Classification) 과정'과 '버킷팅(Bucketing)' 관점에서 설명하시오.
3. 스프링 배치는 `ClassifierCompositeItemWriter`는 제공하지만 `ClassifierCompositeItemReader`는 제공하지 않는다. 그렇다면 하나의 파일 내에서 레코드의 패턴(예: Header, Body, Footer)에 따라 서로 다른 도메인 객체로 변환해야 하는 상황일 때, 우리는 어떤 컴포넌트의 어떤 기능을 활용하여 이와 유사한 효과를 낼 수 있는지 기술하시오.

* **1번 질문 해설:** `CompositeItemWriter`는 등록된 순서대로 라이터를 실행합니다. 파일 라이터가 즉시 쓰기를 수행한 후, 다음 순서인 DB 라이터에서 런타임 예외가 발생하면 DB 트랜잭션은 롤백되지만 이미 파일 시스템에 물리적으로 기록된 데이터는 삭제되지 않습니다. 결과적으로 DB에는 존재하지 않는 데이터가 파일에는 존재하는 '치명적 불일치'가 발생합니다. 이를 방지하기 위해 파일 라이터는 버퍼링을 통해 실제 쓰기를 트랜잭션 커밋 직전까지 미뤄서 DB 라이터의 성공 여부와 보조를 맞추는 '트랜잭션 동기화'를 수행해야 합니다.

* **2번 질문 해설:** 일반적인 `CompositeItemWriter`는 모든 아이템이 담긴 청크 전체를 모든 위임 라이터에게 통째로 넘겨 순차 실행합니다. 반면 `ClassifierCompositeItemWriter`는 청크를 받으면 즉시 쓰지 않고, 청크 내의 아이템들을 하나씩 분류기(`Classifier`)에 던집니다. 분류 결과에 따라 아이템들을 각 라이터별 임시 바구니(버킷)에 모읍니다. 청크 내 모든 아이템의 분류가 끝나면, 각 라이터에게 그들에게 할당된 아이템 리스트만을 전달하여 `write()`를 호출합니다. 즉, 아이템 단위의 '선별적 위임'이 핵심 차이점입니다.

* **3번 질문 해설:** `ClassifierCompositeItemReader`는 없지만, 파일 읽기 공정의 `LineMapper` 단계에서 이를 처리할 수 있습니다. 챕터 2에서 배운 `PatternMatchingCompositeLineMapper`가 그 해답입니다. 이 컴포넌트는 파일의 각 라인이 가진 특정 패턴(예: "01"은 헤더, "02"는 데이터)을 분석하여, 각 패턴에 맞는 `LineTokenizer`와 `FieldSetMapper`를 동적으로 선택합니다. 이를 통해 읽기 단계에서부터 데이터의 성격에 따라 서로 다른 객체 타입으로 부활시키는 '패턴 기반 분류'를 완수할 수 있습니다.
