# 쿠버네티스 기

## 1. 컨테이너 격리 기술의 본질과 쿠버네티스 운영 자동화의 내부 작동 원리

쿠버네티스는 단순한 컨테이너 오케스트레이터를 넘어, 복잡한 인프라 관리를 **선언적 자동화(Automation)**로 해결하는 엔진입니다. 본 섹션에서 다루는 핵심 기술의 내부 메커니즘을 심화 분석합니다.

### 1) 커널 레벨의 격리: 컨테이너 가상화의 물리적 토대

컨테이너는 가상 머신(VM)과 달리 호스트 OS의 커널을 공유하면서도, 리눅스 커널의 두 가지 핵심 기술을 통해 독립된 환경을 구축합니다.

* **Namespace (논리적 분리):** 프로세스가 바라보는 시스템 자원(커널 영역)을 격리합니다. 이를 통해 컨테이너는 자신만의 네트워크 스택, 프로세스 트리(PID), 마운트 지점 등을 가진 것처럼 동작합니다 [1].
* **Cgroups (물리적 제한):** 컨테이너가 사용할 수 있는 CPU, 메모리, 네트워크 대역폭 등 하드웨어 자원의 상한선을 강제합니다.
* **존재 이유:** VM은 각 환경마다 게스트 OS를 설치해야 하므로 오버헤드가 크고 무겁지만, 컨테이너는 커널을 공유하여 실행 속도가 빠르고 자원 효율성이 극대화됩니다.

### 2) 인프라 자생력: 오토 힐링(Auto Healing)과 오토 스케일링(Auto Scaling)

쿠버네티스는 관리자가 일일이 개입하지 않아도 클러스터의 상태를 유지하는 '조정 루프(Control Loop)'를 가집니다.

* **오토 힐링 (Auto Healing):** 특정 노드에 장애가 발생하여 그 위의 서비스가 중단되면, 쿠버네티스는 이를 즉시 감지하여 가용 자원이 있는 다른 노드로 파드(Pod)를 자동으로 옮겨 재생성합니다.
* **오토 스케일링 (Auto Scaling):** 서비스 접속량(Traffic)의 변화에 따라 필요한 서버 자원량을 유연하게 조절합니다. 이는 미리 예측하기 어려운 트래픽 변동에 대응하여 비용을 절감하고 서비스 가용성을 보장하기 위함입니다.

### 3) 무중단 배포 전략: Deployment를 통한 버전 관리

새로운 버전의 애플리케이션을 배포할 때, 서비스 중단 없이 업데이트를 수행하는 메커니즘입니다.

* **Rolling Update:** 구 버전의 파드를 하나씩 제거함과 동시에 신 버전의 파드를 하나씩 생성하여 트래픽을 점진적으로 전환합니다.
* **Blue-Green / Canary:** 전체 환경을 교체하거나(Blue-Green), 일부 사용자에게만 신 버전을 노출(Canary)하여 안정성을 검증한 후 전체 배포를 진행하는 방식도 지원합니다.

## 2. 시스템 아키텍처 관점에서의 기술 검증 및 심화 사고 질문

**질문 1:** 보안 측면에서 VM과 컨테이너의 구조적 차이가 실제 장애 발생 시 어떤 연쇄 반응의 차이를 만듭니까? 단순히 "VM이 더 안전하다"는 관점을 넘어 커널 공유 관점에서 설명하십시오.

**질문 2:** 특정 노드가 하드웨어 결함으로 완전히 다운되었을 때, 쿠버네티스의 '오토 힐링' 메커니즘이 해당 노드에서 실행 중이던 파드를 다른 노드로 옮기는 논리적 판단 근거는 무엇입니까?

**질문 3:** 애플리케이션 업데이트 중 `Deployment`가 `Rolling Update`를 수행할 때, 트래픽 유실을 방지하기 위해 서비스(Service) 오브젝트는 내부적으로 어떻게 '구 버전'과 '신 버전'의 파드를 구분하여 트래픽을 제어합니까?

<details>
<summary style="cursor: pointer; color: #007bff;">👉 자가 진단 질문 정답 및 해설 보기</summary>

**정답 1:** VM은 하이퍼바이저를 통해 게스트 OS 레벨에서 완벽히 분리되어 있어 한 VM이 뚫려도 호스트나 다른 VM으로의 전파가 어렵습니다. 반면 컨테이너는 호스트 OS 커널을 공유하므로, 만약 특정 컨테이너가 커널 취약점을 공격해 OS 권한을 획득할 경우 해당 호스트에서 실행 중인 모든 컨테이너가 위험에 노출될 수 있는 연쇄 반응의 위험성이 존재합니다.

**정답 2:** 쿠버네티스 마스터는 노드들의 상태를 주기적으로 체크합니다. 노드가 다운되면 해당 노드의 Kubelet으로부터 응답이 끊기게 되고, 컨트롤러 매니저는 해당 노드에 할당된 파드들이 'Desired State(원하는 상태)'와 일치하지 않음을 감지합니다. 이후 스케줄러가 남은 노드 중 자원 여유가 있는 곳을 찾아 파드를 재생성하도록 명령을 내림으로써 서비스를 유지합니다.

**정답 3:** 서비스 오브젝트는 `Label Selector`를 통해 파드를 관리합니다. 업데이트 과정에서 신규 버전의 파드가 생성되고 'Running' 상태가 되면 서비스의 셀렉터에 부합하는 라벨을 가지게 됩니다. 이때 서비스는 신규 파드의 IP를 자신의 엔드포인트 목록에 추가하고 트래픽을 분산하기 시작하며, 동시에 제거되는 구 버전 파드의 IP는 목록에서 삭제하여 트래픽 유실을 방지합니다.

</details>

## 3. 선언적 명세(Declarative) 기반의 배포 실무 및 연결성 장애 해결 가이드

### [실습 예제] 서비스 연결성이 포함된 파드 배포 매니페스트

강의 필기 노트의 Hello World 시나리오를 바탕으로 작성된 표준 YAML 예제입니다.

```yaml
# 1. 애플리케이션 실행 단위 정의 (Pod)
apiVersion: v1
kind: Pod
metadata:
  name: hello-pod
  labels:
    app: hello          # 서비스가 파드를 찾기 위한 식별자
spec:
  containers:
  - name: hello-container
    image: tmkube/hello # 도커 허브에 저장된 Node.js 앱 이미지 [11, 12]
    ports:
    - containerPort: 8000 # 컨테이너 내부 포트 [11]

---

# 2. 외부 접속을 위한 네트워크 정의 (Service)
apiVersion: v1
kind: Service
metadata:
  name: hello-svc
spec:
  selector:
    app: hello          # hello-pod의 라벨과 매칭되어 연결됨 [9, 12]
  ports:
    - port: 8200        # 서비스로 접근하는 포트 [13]
      targetPort: 8000  # 파드 내부로 전달되는 포트 [10, 13]
  externalIPs:          # 외부에서 접근 가능한 특정 노드 IP 지정 [10, 13]
  - 192.168.0.30
```

### [트러블슈팅] 서비스 연결 실패 및 파드 미접속 장애

* 장애 상황: 서비스를 생성했으나 외부 IP/Port를 통해 접속했을 때 Connection Refused가 발생하거나 애플리케이션 응답이 없음.
* 해결 프로세스:
  * 라벨 매칭 확인: Service의 selector와 Pod의 labels가 정확히 일치하는지 확인합니다.
    * 명령어: kubectl get pods --show-labels 및 kubectl describe svc hello-svc
  * 엔드포인트 활성화 여부: 서비스가 연결할 파드의 IP를 정상적으로 획득했는지 확인합니다.
    * 명령어: kubectl get endpoints hello-svc (목록이 비어 있다면 라벨 불일치임)
  * 컨테이너 내부 상태 확인: 파드는 'Running'이지만 내부 앱이 죽었을 수 있습니다.
    * 명령어: kubectl logs hello-pod를 통해 앱 로그 확인
    * 명령어: kubectl exec -it hello-pod -- ls 명령으로 컨테이너 내부 파일(hello.js) 존재 확인
  * 포트 포워딩 검증: 매니페스트의 targetPort와 컨테이너 앱이 실제 리스닝 중인 포트가 일치하는지 대조합니다
* 긴급 조치: 서비스와 상관없이 파드에 직접 터널링하여 앱 정상 작동 확인
  * kubectl port-forward pod/hello-pod 8000:8000
