실시간으로 생성되고 퍼지고 검색되며 수많은 사용자들이 동시에 타임라인을 조회하는 거대한
서비스가 됩니다.
과연 이러한 거대한 서비스를 어떻게 설계해야 안정적이고 빠르게 동작하게 할 수 있을까요?
이번 강의에서는 확장성 , 가용성 , 성능, 성능, 보안, 시큐리티 등 실제
서비스를 설계할 때 반드시 고려해야 할 요소들을 하나씩 짚어가며 트위터 같은 대규모
시스템을 처음부터 끝까지 설계해 보는 경험을 하게 될 것입니다.
그럼 지금부터 함께 시작해 보겠습니다.
어떤 시스템 디자인 인터뷰든지 가장 먼저 해야 할 일은 시스템의 요구사항을 이해하는
것입니다. 이 단계의 주요한 목표는 두 가지입니다
첫째는 설계를 시작하기 전에 인터뷰어와 여러분이 같은 페이지에 있는지를 확인하는 것입니다
그리고 둘째는 설계에서 지원해야 할 기능들을 명확히 파악하는 것입니다
이 요구사항 분석 단계는 두 가지 범주로 나뉩니다
첫 번째는 기능적 요구사항 즉 Functional Requirements 입니다 기능적인
요구사항은 사용자들이 시스템에서 기대하는 핵심 기능을 의미합니다
예를 들면 트윗을 생성할 수 있는 기능 같은 것이죠
그 다음은 비기능적인 요구사항 즉 Non-functional requirements입니다
이는 시스템의 품질 속성을 뜻하며 보안, 안정성, 성능 등이 이에 해당됩니다
트위터 같은 시스템 서비스를 기준으로 한 기능 요구사항은 다음과 같습니다.
사용자는 계정을 만들고 로그인을 할 수 있어야 합니다.
그리고 사용자는 트윗을 생성, 수정, 삭제할 수 있어야 합니다.
그리고 다른 사용자를 팔로워 할 수 있어야 합니다.
또 사용자는 자신이 팔로워한 사람들의 트윗을 타임라인에서 볼 수 있어야 합니다.
트윗에 좋아요를 누르거나 리트윗을 하거나 댓글을 남길 수 있어야 합니다.
사용자는 트윗을 검색할 수 있어야 하고 또 웹 어플리케이션과 모바일 어플리케이션, iOS나
안드로이드 등을 모두 제공해야 합니다.
또 비기능적 요구사항은 다음과 같습니다.
시스템은 수억 명의 일일 활성 사용자를 지원할 수 있어야 합니다.
따라서 트윗 생성 및 읽기 쓰기 요청을 대량으로 처리할 수 있어야 합니다.
그리고 시스템은 99.99%의 가용성을 보장해야 하며 사용자 데이터의 보안과 프라이버시를
보장해야 합니다.
무엇보다도 매우 낮은 레이턴시를 유지해야 합니다 사용자가 트윗을 로딩하는데 오랜 시간이
걸려서는 안됩니다
화이트보드에서 설계를 할 때 가장 논리적인 방식은 요구사항에 따라서 왼쪽에서 오른쪽으로
진행하는 것입니다
그래서 우리가 가장 먼저 추가할 것은 클라이언트를 왼쪽에 추가하는 것입니다.
왼쪽에 클라이언트를 정의하고 클라이언트에서 보낸 요청은 또 로드밸런서로 들어오는 그런
흐름이 되겠습니다.
로드밸런서는 들어오는 트래픽을 여러 서버에 분산시켜주는 역할을 하며 거의 모든 시스템
디자인 인터뷰에서 스케일러빌리티를 위해서 필수적인 요소입니다.
로드밸런서에 대해서 인터뷰 중 언급할 만한 두 가지 주요 포인트가 있는데요
첫 번째는 라우팅 알고리즘입니다
이건 로드밸런서가 요청을 백엔드 서버로 어떻게 분배할지를 결정합니다
일반적인 알고리즘에서는 라운드로빈 모든 서버에 순차적으로 요청을 분배하는 것 또는 최소
연결, 가장 적은 연결을 가진 서버에 요청을 보내는 것 또는 Client IP
주소를 기반으로 특정 서버에 항상 같은 요청을 보내는 형태 이런 알고리즘을 언급할 수가
있습니다 두 번째는 OSI 모델에서 노드밸런스가 작동하는 계층에 대한 언급입니다
로드밸런스는 다양한 계층에서 동작할 수 있습니다
가장 일반적인 것은 Layer 4, L4 라고 언급하죠.
전송계층에서 동작하는 것입니다.
TCP나 UDP 레벨에서 동작하는 그런 로드밸런스가 있을 수 있고요.
그리고 레이어 7은 어플리케이션 계층에서 동작하는 로드밸런스입니다.
HTTP나 HTTPS 프로토콜을 지원하는 경우가 이에 해당됩니다.
L7 로드밸런서는 요청의 실제 내용을 기반으로 라우팅을 결정을 내릴 수 있기 때문에 훨씬
다양한 트래픽 제어가 가능합니다
예를 들어 URL이나 아니면 헤더 정보를 기반으로 트래픽을 제어할 수 있습니다
그래서 트위터 같은 시스템에서는 라우팅 알고리즘은 라운드 로빈을 사용합니다
그리고 간단하고 공정하며 또 스테이트 리스트 백핸드 서버를 쓸 것이기 때문에 저희에게는 이
방식이 적절하겠습니다.
또 로드 밸런스는 레이어 7에서 동작하게 설정을 할 것입니다.
이렇게 하면 요청 내용을 기반으로 라우팅을 할 수 있고 복잡한 트래픽 관리가 가능해집니다.
그 다음 로드밸런서는 API 게이트웨이로 트래픽을 분배합니다
우리는 마이크로 서비스 아키텍처를 선택할 것이기 때문에 API 게이트웨이를 사용합니다
이건 시스템을 작은 독립적인 서비스로 나누는 방식이며 각 서비스는 시스템 내에 특정 기능
하나만 책임지게 됩니다
API Gateway는 로드밸런서로부터 들어온 요청을 받아서 해당 요청을 적절한 서비스로
전달합니다. 이 세가지 구성요소, 즉 Client, 로드밸런서, API
게이트웨이는 90% 이상의 시스템 디자인 인터뷰에 등장합니다.
그 다음에는 핵심 서비스들을 나열하는 것입니다.
보통 이건 우리가 앞에서 정의했던 기능적 요구사항과 거의 정확하게 일치합니다.
사용자가 트윗을 생성, 수정, 삭제할 수 있어야 합니다.
이를 처리하기 위해서 CRUD 서비스를 추가하겠습니다.
CRUD는 Create, Read, Update, Delete의 약자입니다.
그리고 사용자는 트윗에 좋아요, 리트윗, 댓글을 달 수 있어야 합니다.
좋아요와 리트윗은 기존 트윗에 대한 업데이트이므로 트윗의 CRUD 서비스에서 처리합니다.
그러나 댓글 즉 리플라이 서비스는 별도의 서비스로 분리할 것입니다.
왜냐하면 확장성 때문에 나중에 댓글만 독립적으로 확장해야 할 수 있어야 되기 때문입니다.
그리고 검색 서비스를 추가합니다
또 타임라인 생성을 위한 타임라인 서비스도 추가하겠습니다
마지막으로 계정 생성 및 팔로우 관리 등을 처리할 수 있는 프로파일 서비스를 추가합니다
여기서 소셜 그래프도 같이 관리됩니다
다음 단계는 각 서비스가 필요로 하는 요구사항을 제대로 충족할 수 있는지를 확인하는
것입니다. 우선 트윗 CRUD 서비스부터 시작하겠습니다.
이 서비스는 트윗을 생성, 수정, 삭제할 수 있어야 하고 트윗 좋아요, 리트윗도 지원해야
하며 각 트윗에 대한 메타 데이터, 즉 타임 스탬프, 그리고 사용자 아이디, 미디어 첨부
파일 등 이런 정보들을 처리해야 합니다.
그리고 비기능적 요구사항 측면에서 보면 이 서비스는 매우 높은 처리량, 매우 높은
throughput을 감당할 수 있도록 설계해야 합니다.
왜냐하면 트윗 생성, 좋아요, 리트윗은 트윗에서 가장 자주 발생하는 타스크이기 때문입니다.
특히 읽기, 리드 작업이 그렇습니다.
그럼 트윗을 어떻게 저장해야 할까요
트위터는 실제 맨하튼이라는 자체의 NoSQL 데이터베이스를 사용합니다
우리도 유사하게 NoSQL Document DB를 사용할 텐데 오픈소스 대안인
MongoDB를 선택하겠습니다
MongoDB는 성능 측면에서 효율적이며 읽기 및 쓰기가 빈번하게 발생하는 앱일 경우에
매우 적합합니다
또한 우리는 트윗에 대해서 복잡한 조인이 필요하지 않기 때문에 아주 잘 맞습니다.
사용자가 트윗을 요청할 때 그냥 해당 다큐먼트 하나만 반환하면 됩니다.
트윗은 기본적으로 JSON 형식의 Key Value 페어로 이루어진 문서입니다.
그리고 다음은 트윗 다큐먼트의 예시를 보면 트윗 작성자 트윗 내용
그리고 타임스텝, 해시태그, 멘션 등 메타 데이터 그리고 위치 정보나 미디어, 리트윗,
좋아요 정보들이 포함된 트윗 다큐먼트 정보를 언급할 수 있겠습니다.
그리고 우리는 트윗 다큐먼트 안에 첨부된 미디어에 대한 참조만 저장을 하고 미디어 파일
자체는 별도의 저장소에 저장해야 합니다.
이 경우 우리는 아마존 S3와 같은 오브젝트 스토리지를 사용할 것입니다.
미디어는 거의 대부분의 경우 Blob 스토리지에 저장됩니다.
이들은 구조화되지 않은 대량 데이터를 다루도록 설계되어 있으며 대규모 스케일에서도 빠르게
미디어를 제공할 수 있도록 지원해 줍니다.
따라서 면접에서 미디어 처리가 필요하다면 S3와 같은 블록 스토리지를 쓰는 게 거의
정답이라고 보면 되겠습니다.
이제 쓰기 경로를 다시 한번 확인해 보면 사용자가 포스트 요청을 보냅니다
그때 로드밸런스가 이를 받아서 API Gateway로 전달하고 API Gateway는 이를
트윗의 CRUD 서비스로 전달합니다
트윗 CRUD 서비스는 미디어를 S3로 업로드하고 트윗의 컨텐츠를 Document
Storage, 즉 MongoDB에 저장합니다
업데이트나 수정도 거의 동일한 경로를 따릅니다.
여기에 하나 더 고려할 점이 있는데 우리는 해당 서비스가 너무 많이 쓰기 요청을 보내서
시스템이 마비시켜지는 것을 막아야 합니다.
이를 위해서 쓰기 서비스에 rate limiter를 추가할 수 있겠습니다.
즉, 특정 시간 동안 특정 사용자가 생성할 수 있는 트윗의 개수를 제한하는 것입니다.
이제 읽기 경로를 살펴보도록 하겠습니다
여기서 가장 중요한 것은 바로 속도입니다
이를 위해서 Cache를 읽기 경로에 추가할 것입니다 자주 조회하는 트윗은 Redis나
Memcache 같은 Cache 서버에 저장하여서 빠르게 제공되도록 할 수 있습니다
이렇게 하면 데이터베이스를 매번 조회하지 않아도 됩니다
그리고 전세계의 사용자들에게 컨텐츠를 빠르게 제공하기 위해서 CDN도 추가할 수 있습니다
정적 컨텐츠, 미디어 파일 등을 자주 조회되는 그런 트윗까지 CDN에서 제공하면 사용자의
위치에서 가까운 서버에서 컨텐츠를 제공할 수 있어 레이턴시가 줄어들고 전세계적으로 빠른
사용자 경험을 제공할 수 있습니다
이번엔 Reply CRUD 서비스입니다.
쓰기 측면에서 댓글들을 별도의 Document Storage에 저장할 것입니다.
여기서 근데 여러분들이 왜 트윗 문서 안에 그냥 댓글을 같이 저장하지 않지?
라는 생각을 할 수 있는데 만약에 작은 시스템이라면 그렇게 해도 괜찮을 수 있습니다.
하지만 우리가 댓글을 분리해서 저장하는 몇 가지 중요한 이유가 있습니다
첫 번째는 확장성입니다
인기 있는 트윗은 수천 개의 댓글을 받을 수 있습니다
댓글을 따로 저장하면 트윗 문서가 지나치게 커지지 않게 되고 무엇보다 댓글 서비스만
독립적으로 확장할 수 있습니다
그리고 성능적인 이슈입니다
댓글을 별도로 저장하면 트윗을 읽을 때 모든 댓글을 함께 불러오지 않아도 되므로 읽기
속도가 빨라집니다
실제로 트위터를 사용해 보면 트윗이 먼저 뜨고 그리고 아래로 스크롤을 해야 댓글이
추가적으로 로딩 되는 걸 볼 수 있습니다
댓글에 대해서도 서비스가 남용되는 걸 방지하기 위해서 쓰기 경로와 같이 rate
limiter를 추가할 수 있습니다
이렇게 하면 서비스의 품질을 유지할 수 있습니다
일반적으로 트윗을 읽으면 그 트윗에 대한 댓글이 같이 보이는 길을 기대를 하는데요 그래서
우리는 댓글을 트윗과 묶어서 함께 불러오도록 만들 것입니다
이를 빠르게 하기 위해서 댓글 데이터베이스는 트윗 아이디를 기준으로 인덱싱을 해 두어야
합니다 즉 트윗을 불러올 때 해당 트윗의 아이디를 빠르게 관련된 댓글을 조회할 수 있고
캐시할 수 있게 되어야 됩니다
그 다음은 댓글 문서의 예시입니다
댓글 문서, 댓글 트윗의 문서와 매우 비슷하게 구성되어 있고 특이한 점은 댓글 문서는

우리는 이 필드를 기준으로 해서 인덱스를 만들어야 합니다
이제 우리는 트윗 생성, 수정, 좋아요, 리트윗, 댓글까지 모두 지원할 수 있게 되었습니다
다음은 검색 서비스입니다
트위터 검색은 이 시스템의 다른 컴포넌트들과 마찬가지로 정말 복잡하며 이것만으로도 면접
질문 하나로 나올 정도의 그런 복잡도를 가질 수 있습니다
하지만 우리는 간단한 방식으로 접근을 해서 기본적인 부분만 처리하도록 하겠습니다
검색은 엄청나게 빠르게 이루어져야 합니다
그래서 다큐멘트 DB에서 트윗들을 하나씩 훑어서 보고 일치하는 검색어를 찾는 그런 것은
현실적으로 너무 느리기 때문에 사용할 수 없습니다
대신에 우리는 일렉트릭 서치를 도입해서 별도의 텍스트 기반 인덱스를 구축할 것입니다
우리는 트윗 내용, 사용자 이름, 해시태그에 대해서 역방향 인덱스를 생성합니다
이렇게 하면 특정 트윗 프로필 트렌딩한 주제들을 최소한의 시간으로
지연 시간으로 매우 빠르게 검색할 수 있습니다.
ElectricSearch는 대규모 데이터셋을 낮은 레이턴시와 높은 신뢰도로 처리할 수
있도록 설계된 분산 검색 엔진입니다.
그래서 전체 텍스트 검색, 풀 텍스트 서치에 필요한 어플리케이션에서는 대부분 일렉티드
서치가 굉장히 강력한 선택지입니다.
면접에서도 텍스트 검색이 필요하다 하면 거의 무조건 엘라스틱 서치를 언급하는 것이
좋습니다. 그렇다면 일렉티드 서치는 어떻게 데이터 업데이트를 받을까요?
바로 Change Data Capture 라는 과정을 통해서입니다.
이 프로세스는 트윗 데이터의 변화를 감지하고 추적하여 Elasticsearch 웹서치 인덱스가 최신 상태로
유지될 수 있도록 해줍니다
즉, 트윗 DB의 변경이 생기면 그 변경 사항이 자동으로 Elasticsearch 워치에 전파되는 그런
구조를 갖게 됩니다
다음은 타임라인 서비스입니다
소셜미디어 서비스를 설계하는 인터뷰 즉 예를 들어서 페이스북이나 트위터 설계 문제가 나오면
면접의 대부분의 시간을 바로 이 타임라인 서비스에 보내는 경우가 많이 있습니다
이번에는 조금 단순화해서 우리는 사용자의 타임라인을 친구의 게시물로만 구성하도록 하겠습니다
따라서 추천 시스템 등은 포함하지 않겠습니다
타임라인을 구성하는 가장 단순한 방법은 우리가 흔히 fan out on read 읽을 때
확산하는 방식입니다.
fan out on read 읽을 때 확산하는 방식 사용자가 타임라인을 요청을 하면 그가
팔로우한 계정 목록을 조회하고 그들의 트윗을 모두 가져가서 시간 순으로 정렬한 뒤에
타임라인을 변환하는 방식입니다
하지만 이 방식은 너무 느리고 비싸기 때문에 우리가 요구했던 낮은 레이턴시 시간 요건을
만족하지 못합니다
그래서 대신에 비동계 처리에 힘을 빌려야 합니다
그 전략은 바로 Fan-out on write. 쓸 때 확산하는 것입니다.
누군가 새로운 트윗을 작성하면 그 트윗은 메시지 큐에 들어가게 됩니다.
이 큐는 버퍼 역할을 하며 동시에 여러 개의 트윗이 생성될 때 속작업이 과부하되지 않도록
그렇게 막아주는 역할을 합니다.
전용 워커들이 계속해서 이 큐를 모니터링 해야 합니다
이 워커들은 큐에서 트윗을 하나씩 꺼내서 각 트윗 작성자의 팔로우 목록을 줘야 합니다
그 다음에 각 팔로워는 자신만의 타임라인 캐시를 가지고 있어야 합니다
이건 사용자가 팔로우한 사람들의 최신 트윗을 빠르게 접근할 수 있도록 저장하는 그런 역할을
합니다 그래서 워커들은 새로운 트윗을 받아서 각 팔로워의 타임라인 캐시에 그 트윗을 맨
앞에 추가해주게 됩니다
그 결과 이 팔로워들이 나중에 자신의 타임라인을 조회할 때 그때 새로운 트윗이 타임라인 맨
앞쪽에 보이게 그렇게 해줄 수 있습니다
이러한 Fan-out-on-write 방식은 쓰기 작업이 많아지지만 일기 작업을 엄청나게
빠르게 만들어 줍니다.
왜냐하면 데이터가 미리 준비되어 캐시에 들어가 있기 때문이죠.
이는 앞서 말했던 비기능적인 요구사항 중 낮은 레인턴시 요건을 완벽하게 충족해 줍니다.
그런데 여기에 한 가지 문제가 있습니다.
수백만 팔로워를 가진 사용자 측 예를 들면 연예인들의 팔로워 캐시를 전부 업데이트를 하려면
그 양이 너무 많아져서 시스템을 과부하시킬 수 있습니다
그래서 우리는 이런 사용자를 위해서 하이브리드 접근 방식을 사용합니다
평범한 일반 사용자일 경우에는 Fan-out on-write를 이용하고 이러한 인플루언서,
연예인들의 경우에는 fan out on read 방식으로 전환합니다
즉 유명 연예인이 트윗을 쓸 때에는 아무것도 하지 않는 것이죠
그래서 팔로워가 타임라인을 확인할 때 그때 비로소 해당 트윗을 가져와서 타임라인의 동쪽으로
합치는 그런 방식을 취할 수가 있습니다
이제 남은 건 프로파일 서비스입니다
이 서비스는 사용자의 프로필 생성 및 관리 그리고 팔로워
관리를 담당하는 그런 기능이 되겠습니다
우리는 사용자 데이터를 저장할 때 간단한 SQL 데이터를 사용할 것입니다
여기엔 전통적인 관계형 테이블 구조를 사용할 것이고 예를 들어서 사용자 이름, 이메일,
자기소개 같은 컬럼으로 저장될 것입니다.
이 SQL은 트위터의 사용자 데이터에 아주 적합한 선택이 될 수 있습니다.
왜냐하면 구조화된 속성에 대해서 효율적인 쿼리가 가능합니다.
그리고 ACID 트랜잭션을 통해서 데이터의 무결성을 보장할 수 있기 때문에 또 장점이
되겠습니다 그리고 또한 복잡한 조인 쿼리나 집계 쿼리도 도움이 될 수 있습니다
이제 팔로우 관계에 대해서는 그래프 데이터베이스 그래프 DB를 사용할 수 있습니다
왜냐하면 그래프 DB는 네트워크 구조 특히 소셜 관계를 모니터링 하는데 최적화되어 있는
솔루션이기 때문입니다.
게다가 어플리케이션이 성장하면 이 그래프는 추천 시스템, 새로운 팔로우 추천, 무결성
시스템 등 다양한 고급 기능을 위해 해법을 제공해 줄 수 있습니다.
또한 인증 인과 처리를 위해서 별도의 Auth 서버도 따로 둘 수도 있습니다.
프로파일 서비스는 어스 서비스를 이용해서 인증 처리를 맡게 되고 그리고 인증과 접근 제어는
매우 민감하고 전문적인 작업이기 때문에 이것을 별도로 분리함으로써 좀 더 강력한 보안에
집중되는 그런 보안을 집중할 수 있는 그런 서비스로 운영할 수 있고 그리고 다른 외부
서비스와 연동할 수 있는 유연성을 확보할 수도 있습니다
이렇게 해서 트위터 서비스의 전체적인 시스템 설계가 완성되었습니다.
실제 인터뷰에서는 시스템 설계에 대한 설명을 한 이후에 보안, 모니터링, 테스트에 대한
언급을 반드시 해야 됩니다.
먼저 보안에 대해서 살펴보도록 하겠습니다.
여기에는 4가지 주요 고려사항이 있습니다.
첫 번째는 인증 및 인과 모든 요청이 합법적인 사용자로부터 왔는지 그리고 그 사용자가 해당
작업을 수행할 수 있는 권한이 있는지를 확인해야 됩니다
이건 우리가 앞서 만든 OAuth 서비스가 담당할 수 있습니다
두 번째는 데이터 암호화 입니다
사용자의 데이터는 전송 중에 또 암호화 되고 또 저장 중에도 암호화 되어야 합니다.
HTTPS를 통해 전송 중인 데이터를 암호화하고 대부분의 최신 DB는 저장할 때 저장
데이터베이스를 암호화하는 스위치를 제공하고 있기 때문에 이것도 간단하게 설정을 할 수
있습니다. 그리고 Rate Limiting 입니다.
앞서 트윗이나 댓글에 대한 Rate Limiter를 추가했는데요 여기에 추가로 DDoS
공격도 막아야 될 수 있습니다
이를 위해서 API Gateway 단에서 IP 기준의 Rate Limiting을 추가할 수
있습니다 즉 한 IP에서 특정 시간 동안 보낼 수 있는 요청 수를 제한하는 것이 되겠습니다
그리고 인풋값을 검증하는 것도 필요할 수 있습니다.
클라이언트 측에서 사용자가 입력한 모든 값은 검증 또는 Sanitize 되어야 합니다.
이것은 SQL injection이나 Cross-site scripting 등의 공격을
방지하기 위함입니다.
보통은 클라이언트와 서버 양쪽 모두에서 이 방어를 구축해야 할 수 있습니다.
다음은 모니터링 입니다
여기서도 3가지의 핵심 요소가 있습니다
시스템 상태 확인 즉 시스템 헬스 체크 입니다
트윗 CRUD 서비스나 프로파일 서비스 같은 어떤 컴포넌트들은 장애가 나면 응답이 느려지면
바로 즉시 알 수 있어야 합니다
이를 위해서 우리는 프레메테우스와 그라파나 등을 사용할 수 있습니다
Prometheus는 성능 지표를 수집하고 Grafana는 시각화 대시보드를 제공해 줄
수 있습니다 그리고 로깅이 되겠습니다
로깅은 트윗의 작성, 로그인 모든 사용자의 활동을 로그로 남겨 주도록 할 수 있습니다
그래서 디버깅 및 보안 위협 등을 추적하기에 필수적인 요소가 되겠습니다
우리는 Elasticsearch, Logstash, Kibana 등을 이용해서 이 기능을
구현할 수 있습니다.
Elasticsearch는 로그를 저장하고 인덱싱하는 역할을 하고 그리고 Logstash는
로그에 대한 데이터를 처리하는, 변환 처리하는 그런 역할을 하고 Elasticsearch에서 인덱싱된
데이터를 로그 데이터를 시각화하는 그런 서비스를 맡게 됩니다
모든 서비스 즉 API 게이트웨이 또는 DB 등을 처리하는 이 서비스는 중앙 로그
시스템으로 로그를 보내도록 그렇게 구성을 해야 합니다

Prometheus와 연동되는 Alert Manager가 있을 수 있구요.

마지막으로 테스트에 대해서 간단히 이야기하겠습니다.
로드 테스팅, 부하 테스팅 단계를 거쳐야 될 수 있습니다.
새로운 기능을 도입하기 전에는 시스템이 그 부하를 견딜 수 있는지 확인해야 합니다.
특히 트윗 서비스나 프로파일 서비스 같은 핵심 서비스는 반드시 테스팅해야 합니다.
이를 통해서 병목 현상이나 실패 지점을 미리 찾아낼 수 있습니다.
그리고 자동화 테스트를 적용해야 할 수 있습니다 마이크로 서비스 아키텍처에서는 서비스 간의

코드 변경이 있을 때마다 CI 툴, 예를 들어서 Jenkins나 GitHub Actions
