안녕하세요.
오늘은 CDN과 GSLB에 대해서 알아보도록 하겠습니다.
먼저 CDN은 Content Delivery Network의 약자로 전 세계의 분산된
서버들을 통해 콘텐츠를 사용자에게 신속하게 전달하는 네트워크 기술입니다.
반면에 GSLB는 Global Server Load Balancing의 약자로 여러 서버
간의 트래픽을 서비스의 안정성과 성능을 조정하는 기술입니다.
이 두 가지 기술은 글로벌 사용자들에게 더 빠르고 안정적인 서비스를 제공하기 위해 매우
중요하며 특히 대규모 트래픽을 다루는 서비스나 글로벌한 사용자 기반을 가진 서비스에서
필수적인 역할을 합니다.
CDN과 GSLB 개념, 동작원리 그리고 실제 활용 사례에 대해서 앞으로 알아보도록
하겠습니다. 이번 시스템 디자인을 위한 요구사항은 전세계 건설현장에서 사용할
수 있는 업무 시스템 consist-site.com을 구축하는 것을 가정해 보도록
하겠습니다. 이 시스템은 전세계 건설 현장에서 사용되며 미국에서 개발되고 글로벌 건설
현장에서 운영됩니다.
시스템 아키텍스 측면에서는 모든 건설 현장 사이트에서 동일한 시스템 로직을 사용해야 하며
통일성을 유지해야 합니다.
그리고 각 건설 현장은 독립적인 데이터베이스를 관리하도록 설계해야 합니다.
네트워크는 건설 현장의 특성상 매우 느리거나 단열적인 소프트웨어 연결 문제가 발생될 수
있습니다. 이 시스템은 본사 서버에 연결되지 않아도 원활히 동작할 수 있어야 하며 최대
2초의 레이턴시와 20%까지의 패키지 손실을 허용해야 합니다.
특히 이 시스템에서 처리되는 데이터는 비즈니스의 중요한 데이터로 데이터 무결성을 유지하는
것이 필수적입니다.
인프라 측면에서는 외부 CDN 서비스를 통해 스태틱 콘텐츠를 배포할 수 있으며 각 현장에
로컬 서버를 두어 시스템이 원활하게 운영될 수 있도록 할 계획입니다.
각 현장에서 수집된 통계 데이터는 본사로 보고되도록 설정하여 데이터의 집계 및 분석이
이루어질 수 있도록 해야 합니다.
또한 모든 현장에서는 HTTPS로 consist-site.com 도메인을 사용해야 합니다.
우리가 가장 먼저 고려해야 할 사항은 각 지역별로 데이터가 별도로 관리되는 상황에서 즉 각
Availability Zone마다 데이터가 분리된 상태에서 시스템이 어떤 유형의
트래픽을 필요로 할 것인가 하는 점입니다.
예를 들어 프론트엔드에서 자주 호출되는 스태틱 리소스가 있을 수 있습니다.
https://consistsite.com slash js slash ui.js 같은 자바스크립트
파일이나 https://consistentsite.com slash index.html과 같은
html 파일이 해당됩니다.
그러나 가장 중요한 트래픽은 백엔드 API 호출입니다.
예를 들어 API slash query id equal user1 같은 API 요청이 있을
수 있습니다.
데이터베이스 동기화, 파일 시스템 동기화도 고려해야 합니다.
다음으로 고려할 사항은 이 앱이 AWS에서 개발 및 배포가 되는 경우 스테딕 콘텐츠가
CDN을 통해 배포된다면 어떤 트래픽 흐름이 발생 될 것인가 하는 것입니다.
예를 들어 UI.js와 같은 스테딕 콘텐츠는 CDN을 사용하여 각 지역에서 가장 가까운
서버에서 캐시될 수 있습니다.
이를 통해 사용자는 원본 서버에 직접 접근하지 않고도 빠르게 콘텐츠를 다운로드할 수
있습니다. 예를 들어, 유저 3이 시스템에 협속할 경우에 UI.js 파일은 AG1의 원본
서버가 아닌 CDN 네트워크 내에 데이터 센터 3과 가까운 엣지 서버에서 다운로드 받게
됩니다. 이는 전송 속도를 대폭 향상시키고 원본 서버의 부하를 줄여줍니다.
다음으로 생각해볼 문제는 consistsite.com이 User1, User2, User3에게
동일한 IP 주소로 보일 것인가 입니다.
만약 서로 다른 IP 주소를 보여줘야 된다면 각 유저의 PC에서 참조되는 DNS 서버는
해당 IP 주소가 올바르게 맵핑되어야 합니다.
즉, 동일한 consistsite.com 도메인이더라도 DNS 서버는 User1,
User2, User3에게 각각 다른 IP 주소를 리턴해줘야 합니다.
그리고 비즈니스 시스템이 동일한 로직을 사용하고 있기 때문에 주요한 메타 데이터는 본사에서
생성되고 각 지역 현장과 양방향으로 동기화 되어야 합니다.
이를 통해 모든 지역이 최신의 시스템 설정과 비즈니스 규칙을 공유할 수 있으며 데이터의
관성을 유지할 수 있습니다.
추가로 현장에서 수립된 통계 데이터는 본사로 반방향으로 수집 및 동기화가 되어야 합니다.
이러한 통계 데이터는 본사의 중앙 시스템에서 분석되고 보고서로 활용될 수 있어야 합니다.
이러한 동기화 구조를 통해서 각 지역은 독립적으로 운영되면서 긴밀하게 연결되어 모든
데이터가 일관성 있게 관리되고 활용될 수 있습니다.
다음 사안은 스타렉 컨텐츠는 AWS 클라우드 프론트와 같은 CDN을 통해서 효율적으로
배포될 수 있지만 지역별로 API 서버에서 제공되는 다이내믹 컨텐츠는 어떻게 처리되어야
하나입니다 다이내믹 컨텐츠는 사용자의 요청에 따라서 실시간으로 처리가 되어야 하며 각
지역의 API 서버가 사용자의 위치한 지역에 있는 데이터베이스와 상호작용을 해서
리스폰스합니다.
이를 통해서 사용자는 가장 가까운 API 서버에서 데이터를 요청하고 빠르게 그리고 또
안정적으로 응답을 받을 수 있어야 합니다.
데이터 센터별로 분산되어 있는 다른 IP 주소를 가지고 있는 서버들에게 어떻게 동일한
도메인을 맵핑시킬 수 있을까요?
이제 CDN을 통해서 이것이 가능한 이유에 대해서 살펴보도록 하겠습니다.
여기에는 Anycast Network, godns, BGP, Border Gateway
Protocol 같은 기술이 사용됩니다.
Anycast Network은 전 세계 여러 서버에 동일한 IP 주소를 제공시켜 줄 수
있는 방식입니다.
이때 라우팅 테이블이 사용자의 물리적인 위치에 가장 가까운 서버로 요청이 자동으로 전달되어
주기 때문에 전 세계에서 동일한 IP가 있지만 가까운 쪽의 서버에서 주도를 얻을 수 있게끔
최적화하는 성능을 얻을 수 있게 해줍니다.
BGP는 CDN 및 클라우드 서비스 프로바이더들이 흔히 사용하는 프로토콜입니다.
그래서 AWS 글로벌 액셀러레이트나 클라우드 플레이어 같은 솔루션을 통해서 프로젝트에서
직접 구현이 가능하고 이 프로토콜을 통해서 네트워크 경로를 최적화하고 글로벌 사용자에게
빠르게 그리고 안정적으로 서비스를 제공할 수 있습니다.
GeoDNS는 사용자의 IP 주소를 기반으로 가장 가까운 CDN LTE 서버의 IP 주소를
반환합니다. 이를 통해 사용자는 물리적으로 가까운 서버에 연결되어 빠른 응답 속도를 경험할
수 있고요. 이러한 기술들은 CDN이 전 세계에서 효율적으로 콘텐츠를 제공할 수 있도록
지원하고 사용자는 자신의 위치에서 가장 적합한 서버로 연결되어 빠른 서비스를 경험할 수
있습니다. 이제 GSLB 에 대해 살펴보도록 하겠습니다.
GSLB는 전 세계 여러 지역에 위치한 서버 간의 트래픽을 최적화하기 위해 사용됩니다.
이 기술은 서버의 상태, 부하, 그리고 사용자의 위치와 같은 다양한 요인을 기반으로
트래픽을 효율적으로 라우팅합니다.
이를 통해서 사용자들은 전 세계 어디서든지 최적의 성능을 경험할 수 있습니다.
CDN 또는 대규모 클라우드 프로바이더 글로벌 어플리케이션 회사 같은 그런 곳에서
GSLB는 필수적인 역할을 합니다.
GSLB를 사용하면 각 서버가 안정적으로 동작할 뿐만 아니라 트래픽이 특정 서버에
과부하되지 않도록 분배되어 전반적인 성능을 향상시킬 수 있습니다.
대표적인 솔루션 제공 업체로는 F5 Networks, Citrix, 그리고 AWS의
Route53가 있으며 이러한 기술을 통해 글로벌 환경에서 안정적이고 빠른 서비스 제공이
가능합니다. 이제 DNS를 기반으로 한 GSLB와 Anycast 네트워크의 결합에
대해서 설명드리도록 하겠습니다
GSLB는 DNS를 이용해서 사용자의 요청을 가장 적합한 서버로 라우팅하며 이것이
Anycast IP와 결합되면 Anycast 네트워크가 자동으로 사용자의 위치에 가장
가까운 서버로 트래픽이 전달됩니다
이 과정을 단계별로 살펴보면 첫 번째, 클라이언트가 로컬 DNS 서버에
constsite.com의 IP 주소를 요청합니다.
둘째로, 로컬 DNS 서버는 상위 DNS 서버에 똑같이 constsite.com의 IP
주소를 요청합니다.
세 번째, DNS 서버는 GSLB에 대한 정보를 로컬 DNS 서버에 전달하고 네 번째,
로컬 DNS 서버는 GSLB 서버에 constsite.com의 IP 주소를 또 요청합니다.
다섯 번째, GSLB는 사용자의 위치, 서버의 상태 등을 고려해서 최적의 서버를 선택하고
해당 서버의 IP 주소를 로컬 DNS로 보냅니다.
마지막으로 로컬 DNS 서버는 이 IP 주소를 캐시하고 클라이언트에게 전달합니다.
보시는 것처럼 상위의 DNS 서버는 DNS 쿼리를 GSLB에 위임해야 합니다.
이제 CDN의 동작 과정을 설명드리겠습니다.
CDN 센터에는 엣지 노드의 모니터링과 GSLB를 통해서 전체 시스템의 트래픽 흐름을
관리합니다. CDN 센터 외부에는 다수의 엣지 서버들과 오리진 서버가 포함됩니다.
먼저 CDN의 고객은 어플리케이션을 빌드한 후에 오리진 서버에 배포합니다.
이 과정에서 어플리케이션이 빌드가 되고 그리고 스태틱 콘텐츠가 생성됩니다.
이후에 이 스태틱 콘텐츠들은 전 세계에 분산된 CDN을 통해서 배포됩니다.
CDN과 DNS를 적절히 구성하면 트래픽은 CDN 엣지 서버로 자동으로 라우팅됩니다.
엣지 서버는 캐시된 정적 컨텐츠를 사용자에게 빠르게 제공될 수 있습니다.
만약에 캐시되지 않은 컨텐츠가 요청이 되면 예를 들어서 304.modified가 아닌 그런
컨텐츠가 요청이 되면 CDN은 이를 오리진 서버에서 가져와서 캐시한 후에 사용자에게
제공하게 됩니다.
이번에는 클라이언트 사이드의 전략에 대해서 확인해 보도록 하겠습니다.
클라이언트 측에서 성능을 최적화하고 네트워크 효율성을 높이기 위해 다음과 같은 전략을
사용할 수 있습니다.
먼저 데이터 전송을 최소화하는 것이 중요합니다.
이를 위해서 선택적인 데이터 전송과 데이터 압축을 통해 필요한 데이터만 효율적으로 전송하고
배치를 통해서 여러 요청을 한 번에 처리할 수 있어야 합니다.
그리고 애플리케이션 유형에 따라서 최적화 전략이 달라질 수 있는데 웹 애플리케이션뿐만
아니라 Swift나 안드로이드, 플러터 등을 통해서 네이티브 애플리케이션을 만들거나 그리고
프로그레시브 웹 애플리케이션과 같은 애플리케이션 또 경우에 따라서 선택할 수도 있습니다
배포 단계에서는 CDN을 활용해서 전 세계 사용자들에게 빠른 콘텐츠 제공이 가능하고 또한
어플리케이션 설치와 네트워크 프로토콜 최적화를 통해서 성능을 더욱 향상시킬 수 있습니다
예를 들어서 HTTP 뿐만 아니라 필요에 따라서 gRPC 같은 프로토콜을 이용할 수도
있습니다. 특히 비동기 작업, 어싱크로나이즈드 오퍼레이션을 통해서 사용자의 경험을 개선할
만약에 Synchronized API가 처리되지 않고 펜딩이 될 경우에는 화면이 멈추는
경험을 할 수 있습니다.
따라서 또한 적절한 타임아웃 설정도 필수입니다.
그리고 마지막으로 캐시는 정적 컨텐츠를 로컬에서 저장함으로써 서버의 부하를 줄이고 응답
시간을 단축시키는데 중요한 역할을 합니다
클라이언트 쪽에서는 로컬 스토리지나 세션 스토리지 같은 것을 이용해서 다이나믹 컨텐츠를
캐시하여 재사용할 수도 있습니다
예를 들어 자주 쓰는 카테고리나 아이템 목록을 캐시에서 재사용하는 방식을 쓸 수 있습니다.
이제 서버 사이드에 대해서 말씀드리겠습니다.
그 중에서 네트워크 최적화의 첫 번째 전략은 콘텐츠 전송 네트워크(CDN) 활용입니다.
Anycast DNS를 통해서 트래픽을 여러 위치로 분산시켜 사용자를 가장 가까운 서버로
라우팅할 수 있고 라우팅을 구성하여 패킷이 가장 효율적인 경로로 따라가도록 할 수도
있습니다. 또한 데이터 전송 최소화가 중요한데 클라이언스 사이드와 마찬가지로 선택적 데이터
전송, 데이터 압축, 배치를 활용해서 필요한 데이터만 사용하고 여러 번 요청을 한 번에
처리하는 방식으로 네트워크 부하를 줄일 수 있습니다.
그리고 기동기 처리를 사용하면 긴 작업을 처리하는 동안에 메인 셀들을 차단하지 않도록
하면서 응답성과 사용자의 경험을 향상시킬 수 있습니다.
그리고 마지막으로 모니터링 및 문제 해결을 꼽을 수 있습니다.
특히 동기화 작업을 위한 네트워크 상태를 지속적으로 모니터링 함으로써 잠재적인 문제를
빠르게 감지하고 해결해야 합니다.
이제 데이터베이스 동기화에 대해서 확인해 보겠습니다
데이터베이스 동기화는 커스텀 스크립트를 작성하거나 DB 리플리케이션 방식을 사용할 수도
있습니다 예를 들어 MySQL에서는 소스의 바이너리 로그를 복제하는 방식으로 데이터를
동기화할 수 있고 PostgreSQL과 MongoDB도 각기 다른 복제 방식을 지원합니다
또한 오라클 GoldenGate와 같은 동기화 도구를 사용해서 보다 효율적인 동기화를 구현할 수
있습니다. 특히 OGG의 경우에 DB의 DML이 수행될 때 모든 로그, 즉
Create, Insert, Update, Delete 등의 데이터베이스 수행 로그를
원격지로 전송해서 DB를 동기화시킬 수 있습니다.
MySQL의 레플리케이션도 유사한 기술을 활용한 것으로 DC 간의 레플리케이션을
지원합니다.
데이터는 다음과 같은 유형으로 분류할 수 있습니다 마스터 데이터, 고객 정보, 제품 정보,
건설 정보, 레퍼런스 데이터 아이템 카테고리, 코드, 국가 단위 정보 이벤트
데이터는 유저의 액티비티, 트랜잭션, 그리고 메타 데이터는 리소스 및
프로세스 메타 데이터 등이 될 수 있고 통계 데이터는 프로젝트 진행 상황 이런 것들이
예시가 될 수 있습니다.
이 중에서 레퍼런스 DB와 메타 DB는 본사에서 현장으로 주로 원웨이 싱크가 되어야 되고
그리고 통계 DB는 현장에서 수집되어서 본사로 옮겨야 되는 방식으로 운영되어야 합니다.
이제 파일 동기화를 살펴보겠습니다.
파일 동기화는 여러 도전 과제를 수반하는데 그 중 가장 큰 문제는 레이턴시와
밴드위스입니다.
특히 대용량 파일을 전송할 때에는 상당한 밴드위스를 소비하며 네트워크 속도에 영향을 줄
수가 있습니다.
또한 신뢰성도 중요합니다.
네트워크 장애나 하드웨어 오류가 발생됐을 때 데이터 무별성을 유지하고 데이터 손실을
방지하는 것이 중요합니다.
시스템 요구사항 중에 레이텐시가 최대 2초, 그리고 패킷러스가 최대 20%까지 허용된다는
네트워크 환경에서는 10MB의 파일도 한 번에 보내지 못할 수가 있습니다.
또한 특정 프로세스가 10MB의 파일을 센터로 동기화할 때 밴드위스를 모두 점유해서 다른
유저가 가용할 밴드위스가 없을 수가 있습니다.
따라서 파일 압축하고 분할 전송하고 조합하고 또 실패를 재시도하는 그런 기능들이
필수입니다. 또한 필요하면 센터로 동기화하는 것은 업무시간 이후에 배치 형태로 수행되어야
할 수도 있습니다.
파일 동기화를 구현하기 위한 대표적인 기술로써는 커스터마지 된 도구를 사용하거나 아니면
FTP나 R-SYNC와 같은 전통적인 파일 전송 프로토콜을 사용할 수도 있습니다.
이러한 기술들을 적절히 활용하면 네트워크 성능을 최적화하면서 데이터의 무관성을 보장할 수
있습니다. 예를 들어 R-SYNC의 BW limit 옵션을 통해서 분할 전송이 가능합니다.
따라서 별도의 전송용 어플리케이션을 구현하지 않더라도 전통적인 파일 전송 프로토프를
이용해서 분할 전송과 조합하는 그런 또 재시도하는 것들을 다 수행할 수가 있습니다.
이제 GSLB 아키텍처에 대해 설명 드리겠습니다.
GSLB는 여러 구성 요소로 이루어져 있습니다.
트래픽을 최적화하기 위해 다양한 역할을 수행합니다.
먼저, 룰 서버는 관리자의 정의된 라우팅 규칙을 배포 가능한 형태로 생성합니다.
DNS 에이전트는 DNS 응답 상태를 확인하는 역할을 하고, HTTP 에이전트는 HTTP
응답을 모니터링합니다.
DNS 서버는 최적화된 DNS 정보를 업데이트하고, BGP 프로토콜을 이용해서 이를 외부
DNS 서버로 점포합니다.
이 과정을 단계별로 살펴보면 첫째로 관리자가 라우팅 규칙을 정의합니다.
해당 규칙은 룰 서버에서 배포 가능한 형태로 생성이 됩니다.
그 다음으로 HTTP DNS 체크 에이전트는 룰 서버에서 규칙을 받아오고 이 에이전트들은
규칙에 따라서 타겟 서버의 네트워크 상태를 확인합니다.
그리고 수집된 네트워크 통계는 StatCollectorA에서 C계열 데이터베이스로
기록됩니다.
마지막으로 그 통계를 기반으로 해서 규칙에서 정의된 기준에 따라서 DNS 레코드가
업데이트가 됩니다.
이러한 아키텍처는 트래픽을 효율적으로 라우팅하고 네트워크의 상태에 따라서 최적화된 경로를
제공할 수 있습니다.
최적의 서버로 트래픽을 라우팅하기 위해서 GSLB는 다양한 라우팅 규칙이 있습니다.
첫 번째로 Geolocation Routing입니다.
예를 들어 사용자의 위치가 미국이나 캐나다에 속하면 트래픽을
usconst.site.com으로 라우팅합니다.
두 번째는 핸스체크 라우팅입니다.
서버의 상태를 주기적으로 확인하여 uscourses.site.com 서버가 정상적으로 작동하고
있을 때에만 그 서버로 트래픽을 보냅니다.
세 번째는 부하 기반 라우팅입니다.
서버의 부하가 80% 미만일 때만 해당 서버로 트래픽을 라우팅하여 과부하를 방지합니다.
네 번째는 라운드로빙 방식입니다.
이 방식은 US, KR, EU 등 다양한 서버들 사이의 트래픽을 순차적으로 분배해서 부하를
균등하게 나누는 방식입니다.
다섯 번째는 밴드위스 기반 라우팅입니다.
네트워크 대역법 사용량이 예를 들어서 usconsult.site.com에서 70% 미만일
경우에 그걸 해당 서버로 라우팅하여 대역복을 효율적으로 사용하게 됩니다.
첫 번째는 레이턴스 기반 라우팅입니다.
usfirstsite.com의 레이턴시가 KR 쪽보다 낮으면 US 쪽 서버로 트래픽을
라우팅하여 사용자의 경험을 향상시킵니다.
일곱 번째는 유저 프로파일 기반 라우팅입니다.
유저가 만약에 VIP 고객일 경우에 트래픽을 전용 서버인 vrp.site.com
서버로 라우팅하여 특별한 서비스를 제공할 수 있습니다.
마지막으로 재해복구 DR 라우팅입니다.
usconst.site.com 서버가 다운될 경우에 트래픽을 KR쪽으로 라우팅하여 서비스
중단을 방지하고 보호가용성을 유지할 수 있습니다.
이로써 건설 현장의 시스템 사용자들은 가장 가까운 CDN 엣지 서버를 통해서 스태틱
콘텐츠를 신속하게 다운로드할 수 있어 화면 로딩 시간을 줄이고 사용성을 높일 수 있습니다.
GSLB (Global Server Load Balancing)를 통해서 API 요청이 가장 빠르고 활용성 높은 데이터 센터인 서버로 지정적으로
라우팅할 수 있습니다.
예를 들어 cut-size.com으로 최초 접근하면 CDN을 통해서 로그인 페이지가 바로
로깅됩니다. 또한 사용자 정보를 입력하고 로그인을 수행하면 GSLB를 통해서 최적의 DC
서버의 API 서버로부터 요청 리스폰스를 받게 됩니다.
이상으로 CDN과 GSLB를 이용한 네트워크 및 시스템 최적화 방법에 대해서
살펴보았습니다.
