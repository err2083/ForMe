컨시스턴트 해싱에 대해 알아보도록 하겠습니다.
컨시스턴트 해싱은 해시 함수를 이용해서 요청이나 데이터를 특정 서버에 할당하는 방식입니다.
주요 특징은 서버가 추가되거나 제거될 때 재할당되는 데이터의 양이 최소화된다는 점입니다.
이는 시스템의 확장성과 유지 보수성을 크게 향상시킵니다.
Consistent Hashing은 주로 로드 밸런싱에서 사용되는 개념으로 간단히 말해서
들어온 요청을 어떻게 서버에 효율적으로 분배할 것인가를 고민한 개념입니다.
예를 들어 클라이언트를 구분하는 키값인 Request ID는 IP 주소일 수도 있고 사용자
ID일 수도 있습니다.
중요한 것은 이 Request ID를 해시함수에 통과시켜 나온 값에 따라 어떤 서버로
요청을 보낼지를 결정하는 것입니다.
예를 들어 Request ID가 클라이언트의 IP 주소인 10.20.30.40이라고 가정해
보겠습니다. 이 값을 입력 값으로 사용하는 해시함수가 14920이라는 값을 안 한다고
하겠습니다. 이 값을 총 서버의 수인 4로 나눈 나머지 값이 0이기 때문에 이 요청은
인덱스가 0인 서버 지로가 처리하게 됩니다.
부하를 균등하게 분배하기 위해서 일반적으로 사용하는 방법은 해시 함수를 이용하는 것입니다.
서버에 들어오는 요청, 즉 각 부하를 서버에 할당하기 위해서는 서버마다 식별자인 인덱스를
부여해야 합니다.
이를 위해서 해시값을 서버수로 나눈 나머지 값을 서버의 인덱스로 사용할 수 있습니다.
왼쪽의 테이블은 8개의 문자열을 해시함수로 A 서버에 할당된 예시입니다.
각 키의 해시값을 계산한 후에 그 값을 서버의 수인 4로 나누어 남은 값이 해당 키를
할당할 서버의 인덱스가 됩니다.
오른쪽 테이블은 서버의 수가 3일 경우 같은 방식으로 계산된 서버 인덱스를 보여줍니다.
이번 페이지의 왼쪽 테이블은 각 요청을 키로 나타냈을 때 이 키들이 서버에 어떻게
분배되는지 설명하고 있습니다.
보다시피 Key0부터 Key7까지 8개의 우측의 4대의 서버에 균등하게 각 서버에 2개씩
할당된 것을 볼 수 있습니다.
서버의 인덱스는 해시키를 N으로 나눈 나머지로 계산되고 이렇게 생성된 번호는
S0,S1,S2,S3와 같이 표시됩니다.
하지만 만약 S1 서버가 삭제된다면 S1에 할당된 Key0, Key4는 재분배되어야
합니다. 오른쪽 테이블은 이 경우를 보여주며 S0에 4개, S3에 1개가 할당되어 균등한
분배가 이뤄지지 않은 예시를 보여줍니다.
여기에서 보다시피 한 데이 서버가 삭제되더라도 모든 서버의 키가 재분배되는 비효율성을 볼
수 있습니다. 이 문제를 해결하기 위해 도입된 것이 Consistent
Hashing입니다.
해시 키 재배치, 즉 Rehash 문제를 해결하기 위해 서버와 키의 값의 할당
관계를 효율적으로 표현할 수 있도록 도입된 개념이 Hash Ring입니다.
만약에 샤본 함수를 이용해서 생성된 해시 값을 일렬로 나열하면 0부터 160순까지의 해시
스페이스로 표현할 수 있습니다.
이 해시 스페이스의 처음 값과 마지막 값을 연결하여 원형으로 표시한 것이 해시 링입니다.
이제 이 해시 링 위에 서버와 서버에 들어오는 요청을 배치함으로써 특정 요청이 어떤
서버에서 처리될지 설명할 수 있게 됩니다.
Hash Ring에 S0,S1,S2 서버를 배치하고 그 위에 K0,K1,K2 요청을 배치할
수 있습니다. 각 요청은 시계방향으로 가장 가까운 서버에서 처리됩니다.
예를 들어 K0는 S0, K1은 S1, 그리고 K2는 시계방향으로 가장 가까운 S2
서버에서 처리됩니다.
이제 해시링에서 해시키 재배치, 즉 리해시 문제를 살펴보도록 하겠습니다.
왼쪽의 해시링은 0부터 1까지의 비율로 표시하고 해시값의 범위를 표시한 그 예시입니다.
예를 들어 S0는 0, S1은 0.3, S2는 0.6의 비율을 차지하고 있습니다.
해시링의 정의에 따라서 0에서 0.3까지의 요청 값은 S1 서버에서 0.3에서 0.6까지는
S2 서버에서 그리고 0.6에서 1까지는 S0 서버가 서비스를 차지합니다.
따라서 예를 들어 K5는 0.73의 값을 가지기 때문에 시계 방향으로 가장 가까운 서버인
S0 서버가 이 요청 처리합니다.
그런데 0.8의 값을 가지는 S3 서버가 추가된다면 K5는 이제 S3
서버에서 처리하고 그리고 0.87 값을 가지는 K6 요청은 S0 서버에서 처리됩니다.
S3 서버가 추가되어 있음에도 불구하고 S1과 S2 서버는 rehashing될 필요가 없이
S0 서버만 rehashing되면 됩니다.
또 다른 예로 S2 서버가 제거되면 원래 K3와 K4를 처리하던 S2 서버가 없어지게
됩니다. 이로 인해서 시계방향으로 가장 가까운 서버인 S0가 K3와 K4를 포함한
총 4개의 요청 K3, K4, K5, K6 모두를 처리해야 합니다.
따라서 S2 서버가 삭제되었음에도 S1은 리헤싱될 필요 없이 S0 서버만 리헤싱되면
됩니다. 서버가 추가되거나 삭제될 때 특정 서버에 많은 요청이
집중되어 부하가 커질 수 있다는 문제점이 있습니다.
이 문제를 가상 로드, Virtual Load를 통해서 완화할 수 있습니다.
기존의 물리 서버인 S0, S1, S2 서버에 K1, K2 요청이 다음과 같이 쓰기의
방향으로 가장 가까운 S1 서버가 이 두 요청을 모두 처리하고 있습니다.
이렇게 한쪽 서버에 몰리는 문제를 해결하기 위해서 각 물리 서버 사이에 가상 노드를
균일하게 배치할 수 있습니다.
가상 노드는 물리 서버에 연결된 임의 노드로 예를 들어 S0 서버에는 S0-0, S0-1,
S0-2와 같은 가상 노드를 생성할 수 있습니다.
각 물리 서버별로 생성된 가상 노드 서버들을 균일하게 배치할 수 있습니다.
예를 들어 S0와 S1 사이에 회색, 핑크색, 하늘색 가상 노드를 할당하면 K1 요청은
핑크색 가상 노드가 K2 요청은 회색 가상 로드가 처리하게 되어 S1에 집중되던 부하가
분산됩니다. 가상 로드의 개수를 늘리면 특정 물리 서버의 부하가 집중되는 현상을 더욱 더
균등하게 분산시킬 수 있습니다.
Consistent Hashing은 실제로 널리 사용되는 기술로써 다음과 같은 사례에서
활용되고 있습니다.
아마존의 DynamoDB의 파티셔닝 관련 컴포넌트 아파치 카산드라 클러스터 데이터
파티셔닝 디스코드 채팅 애플리케이션 UMI의 CDN 또한 NGINX의 로드
밸런싱 알고리즘 옵션 중 하나로도 포함되어 있습니다.
Consistent Hashing의 장점은 다음과 같습니다.
첫째, 요청이 서버들에 골고루 분배되어 효율적인 자원 활용이 가능합니다.
둘째, 서버를 추가하거나 제거할 때 옮겨야 하는 데이터의 양이 적어져서 시스템 확장이
용이합니다.
마지막으로 일부 서버에 장애가 발생하더라도 데이터가 다른 서버에 분산되어 있어 시스템의
가용성을 유지할 수도 있습니다.
이렇게 부하의 균등한 분배를 위해서 사용되는 Consistent Hashing에 대해서
알아보았습니다.
감사합니다.
