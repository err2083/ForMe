그래서 이 URL을 통해서만 클라이언트는 비디오를 AWS S3로 업로드할 수 있게 됩니다
사용자의 비디오를 보호하기 위해서 다음의 솔루션을 도입할 수 있습니다
디지털 저작권 관리 DRM 시스템을 도입할 수 있습니다
또 AES 암호화, 즉 비디오를 암호화하고 접근 권한을 설정하는 방식을 적용할 수 있습니다
암호화된 비디오는 재생시에만 복화됩니다.
따라서 허가된 사용자만 암호화된 비디오를 시청할 수 있습니다.
마지막으로 워터 마크를 적용할 수도 있습니다.
CDN은 비쌉니다.
특히 데이터가 클수록 더 비싸집니다.
이러한 비용을 어떻게 낮출 수 있을까요?
인기 비디오만 CDN에 올려서 재생하고 그렇지 않은 비디오는 비디오 서버에서 직접 서비스
할 수 있습니다 또 인기 없는 비디오는 미리 인코딩 해 둘 필요도 없습니다 짧은 비디오라면
요청 받은 시점에 바로 인코딩해서 재생해도 됩니다
어떤 비디오는 특정 지역에서만 인기가 있으므로 해당 지역에서만 가지고 있어도 됩니다
경우에 따라서 CDN을 직접 구축할 수도 있습니다
그리고 인터넷 서비스 제공자 와 직접 제외해서 최적화할 수도 있습니다.
이 모든 최적화는 콘텐츠 인기도, 이용 패턴, 비디오 크기 등 비디오 데이터에 근거한
것입니다. 따라서 최적화를 시도하기 전에 시청 패턴을 분석하는 것은 중요합니다.
안정성을 위해서는 에러 핸들링이 중요합니다.
에러의 종류에 따라서 다양한 처리가 필요합니다.
일반적인 오류의 유형은 다음과 같습니다.
회복 가능 오류 이것은 몇 번 재시도를 하면 보통 해결되는 오류입니다.
하지만 계속해서 실패하고 복구가 어렵다고 판단되면 클라이언트에게 적절한 오류 코드를
반환해야 합니다.
그리고 회복 불가능한 오류는 해당 뷰에 대한 작업을 중단하고 클라이언트에게 바로 적절한
오류 코드를 반환해야 합니다
그래서 시스템 컴포넌트 각각에서 발생하는 오류에 대한 전형적인 해결 방법들을 처리해야
합니다 업로드 오류는 몇 번 재시도해서 업로드를 진행합니다
그리고 비디오 분할 오류는 오래된 버전의 클라이언트가 GOP 경계에 따라서 비디오를
분할하지 못하는 경우라면 전체 비디오를 서버로 바로 전송하고 서버가 해당 비디오를 분할
처리해서 분할 오류를 해결할 수가 있습니다
그리고 트랜스코딩 오류는 재시도를 통해서 해결될 수 있고 전처리 오류는 DAG 그래프를
재생성합니다 DAG 스케줄러 오류는 작업을 다시 스케줄링하고 API
서버는 Stakeless 서버로 개발되어야 합니다
서버에 장애가 발생하더라도 새로운 요청은 다른 API 서버로 우회될 수 있도록 그렇게
구성이 되어야 합니다
그리고 메타 데이터 해시 서버 장애를 회피하기 위해서는 데이터를 다중화해야 합니다
이 경우에 다른 노드에서 데이터를 여전히 가져올 수 있습니다
그래서 장애가 난 택시 서버는 새로운 것으로 교체되어야 합니다
메타데이터 지휘 서버가 방해되는 것을 방지하기 위해서는 프라이머리 서버가 죽었을 때
세컨더리 서버들 가운데서 하나를 프라이머로 선출해서 교체해야 합니다
만약 세컨더리 서버가 죽었다면 다른 세컨더리 서버를 통해서 일기연산을 처리하고 죽은 서버는
새것으로 교체합니다

작업 서버 장애 시에는 다른 서버에서 해당 작업을 재시도할 수 있도록 그렇게 이중화해야

데이터베이스 계층 규모의 확장을 위해서는 다중화, 샤딩에 대한 고민이 필요하고요 그리고
