# 알림 시스템 설계 (Design a Notification System)

1. 시스템 개요 및 요구사항
    - 알림 시스템은 iOS 푸시, 안드로이드 푸시, SMS, 이메일 등 다양한 채널을 통해 사용자에게 정보를 전달하는 서비스입니다
    - 알림 종류: 모바일 푸시(iOS/Android), SMS 메시지, 이메일
    - 요구사항:
        - 연성 실시간(Soft real-time): 약간의 지연은 허용되나 가능한 빠르게 전달되어야 함
        - 지원 단말: iOS, 안드로이드, 랩톱/데스크톱
        - 규모: 하루에 모바일 푸시 1,000만 건, SMS 100만 건, 이메일 500만 건을 처리하는 대용량 시스템
        - 사용자 설정: 사용자가 알림 수신을 거부(Opt-out)하거나 상세 설정할 수 있어야 함
2. 알림 채널별 컴포넌트
    - 각 알림 유형에 따라 연동되는 외부 서비스가 다릅니다.
        - iOS 푸시: 알림 제공자(Provider) → APNS(Apple Push Notification Service) → iOS 단말기
        - 안드로이드 푸시: 알림 제공자 → FCM(Firebase Cloud Messaging) → 안드로이드 단말기
        - SMS: 트릴리오(Twilio), 넥스모(Nexmo) 등의 서드파티 서비스 사용
        - 이메일: SendGrid, Mailchimp, AWS SES 등의 서드파티 서비스 사용
3. 연락처 정보 수집 플로우
    - 알림을 보내기 위해 사용자의 단말 토큰, 전화번호, 이메일 주소 등을 수집하는 과정이 필요합니다
    - 과정: 사용자가 앱 설치 또는 계정 등록 시 API 서버가 정보를 수집하여 데이터베이스에 저장합니다
    - 데이터 모델: 사용자(User) 테이블과 디바이스(Device) 테이블로 구성되며, 한 사용자가 여러 대의 단말을 가질 수 있는 1:N 관계로 설계합니다
4. 개선된 알림 시스템 아키텍처
    - 단일 서버 구조의 병목 현상과 장애 지점(SPOF)을 해결하기 위해 확장성 있는 구조로 설계합니다
    - 알림 서버 (Notification Server):
        - 인증(App Key/Secret) 및 알림 검증(Validation) 수행
        - 데이터베이스나 캐시에서 사용자 및 단말 정보를 조회
        - 알림 메시지를 생성하여 각 채널에 맞는 **메시지 큐(Message Queue)**에 입력
    - 메시지 큐: 컴포넌트 간 의존성을 제거하고, 특정 서비스 장애 시에도 버퍼 역할을 수행하여 시스템 안정성을 높임
    - 작업 서버 (Workers): 메시지 큐에서 알림 이벤트를 꺼내 서드파티 서비스로 전달
5. 상세 설계 및 신뢰성(Reliability) 확보
    - 알림 시스템에서 가장 중요한 것은 데이터 소실 방지와 중복 방지입니다
    - 데이터 소실 방지: 알림 데이터를 데이터베이스(Notification Log)에 보관하고, 전송 실패 시 재시도 메커니즘을 통해 다시 큐에 넣어 발송함
    - 중복 전송 방지: 이벤트 ID를 검사하는 Deduplication(중복 제거) 알고리즘을 적용하여 동일한 알림이 여러 번 발송되는 것을 방지함
    - 알림 템플릿: 메시지의 일관성을 유지하고 작성 효율을 높이기 위해 인자(Parameter)만 조정하여 사용하는 템플릿 도입
    - 전송률 제한 (Rate Limiting): 사용자가 너무 많은 알림을 받아 피로감을 느끼지 않도록 빈도를 조절함
6. 모니터링 및 추적
    - 큐 모니터링: 큐에 쌓인 알림 개수를 확인하여 작업 서버의 성능 상태 파악
    - 이벤트 추적: 알림 확인률, 클릭률, 실제 앱 사용 전환율 등의 메트릭을 수집하여 사용자 행동 분석에 활용
