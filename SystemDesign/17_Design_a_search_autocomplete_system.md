# 검색 질의 자동 완성 시스템 설계

1. 시스템 개요 및 요구사항
    - 사용자가 검색창에 글자를 입력할 때, 입력된 프리픽스(Prefix)로 시작하는 검색어 목록을 실시간으로 추천해 주는 시스템입니다
    - 기능적 요구사항:
        - 입력된 프리픽스로 시작하는 질의 목록 표시
        - 빈도수(Frequency) 등 특정 랭킹에 따른 상위 K개 질의 노출
        - 데이터 업데이트는 일 단위 또는 주 단위 배치로 수행
    - 비기능적 요구사항:
        - 낮은 지연 시간(Latency): 100ms 이하의 빠른 응답 속도
        - 확장성(Scalability): 대용량 트래픽 처리가 가능해야 함
        - 고가용성: 시스템 장애 시에도 서비스가 유지되어야 함
2. 개략적 규모 추정 (Back-of-the-envelope Estimation)
    - 사용자 지표: DAU 1,000만 명, 인당 하루 10개 질의, 질의당 평균 20글자 가정
    - 쿼리 부하: 평균 초당 24,000건(QPS), 최대(Peak) QPS는 약 48,000건 발생
    - 저장 용량: 매일 약 20%의 새로운 질의가 생성된다고 가정할 때, 매일 약 0.8GB의 데이터 추가
3. 핵심 데이터 구조: 트라이 (Trie)
    - 프리픽스 검색을 효율적으로 수행하기 위해 트라이(Trie) 자료구조를 사용합니다
    - 구조: 루트 노드는 빈 문자열이며, 각 노드는 글자 하나를 저장하고 자식 노드에 대한 포인터를 가짐
    - 성능 문제: 단순히 트라이를 탐색하여 모든 자식 노드를 찾아 정렬하면 시간 복잡도가 O(p)+O(c)+O(clogc)가 되어 느려질 수 있음 (p: 프리픽스 길이, c: 자식 노드 개수)
    - 최적화 방안:
        - 프리픽스 길이 제한: p의 값을 일정 수준으로 제한하여 탐색 범위 축소
        - 노드 내 인기 질의 캐싱: 각 노드에 해당 프리픽스로 시작하는 상위 K개의 질의를 미리 정렬하여 저장함으로써 정렬 과정을 생략함
4. 시스템 아키텍처 및 인덱싱 서비스
    - 전체 시스템은 데이터 인덱싱 서비스와 쿼리 서비스로 나뉩니다
    - 인덱싱 파이프라인:
        - 로그 수집 및 집계: 검색 로그를 수집하여 특정 기간(예: 일주일) 단위로 질의 빈도수를 계산
        - 트라이 생성 및 직렬화: 집계된 데이터를 바탕으로 트라이를 만들고 이를 Key-Value 저장소에 적재할 수 있도록 직렬화함
        - 데이터 저장: 프리픽스를 Key로, 상위 K개 질의 목록을 Value로 하여 Cassandra나 DynamoDB 같은 DB에 저장
    - 쿼리 서비스: API 서버는 트라이 DB나 캐시에서 입력된 Key(프리픽스)를 검색하여 즉시 결과를 반환
5. 운영 및 확장 전략
    - 업데이트 전략: 운영 중인 트라이 DB를 직접 수정하는 것은 오버헤드가 크므로, 새로운 버전의 DB를 완전히 새로 만든 후 트래픽을 교체하는 방식이 권장됨
    - 필터 레이어: 부적절한 질의어나 잘못된 단어를 제거하기 위해 API 서버 앞단에 필터 레이어를 구축
    - 저장소 샤딩(Sharding):
        - 알파벳 분포에 따라 특정 문자(예: 'a', 'e')에 트래픽이 몰리는 핫키(Hotspot) 문제가 발생할 수 있음
        - 이를 해결하기 위해 샤드를 나누고, 트래픽이 많은 샤드에는 더 많은 복제본(Replica)을 배치하며, 이를 관리하기 위한 별도의 샤드 맵 매니저를 운영함
