# 처리율 제한 장치 설계 (Design a Rate Limiter)

1. 개요 및 필요성
    - 처리율 제한 장치는 단위 시간당 전송되는 트래픽의 양을 제어하는 애플리케이션입니다
    - 사례: 한 사용자가 초당 포스팅을 2건 이상 못 하게 하거나, 동일 IP에서 하루에 계정 생성을 10개로 제한하는 경우
    - 도입 이유:
        - 보안: DoS(Denial of Service) 공격 방어
        - 비용 절감: 불필요한 트래픽을 방지하여 서버 리소스 및 비용 아낌
2. 설계 요구사항 (Requirements)
    - 정확성: 설정된 제한을 정확하게 지켜야 함
    - 성능: 높은 처리량과 최소한의 지연 시간(Low Latency) 보장
    - 효율성: 메모리를 효율적으로 사용해야 함
    - 가용성: 시스템 장애가 없어야 하며, 분산 환경에서도 잘 작동해야 함
3. 처리율 제한 장치의 위치
    - 클라이언트 측: 구현은 가능하나, IP 주소나 계정 공유 시 제어가 어렵고 변조 가능성이 있어 비권장됨
    - 서버 측: 서버 내부에 라이브러리 형태로 직접 구현
    - API 게이트웨이: 로드 밸런싱과 함께 처리율 제한을 전담하는 마이크로서비스를 두는 방식(최근 추세)
4. 핵심 알고리즘 비교
    |||||
    |---|---|---|---|
    알고리즘|동작 원리|장점|단점
    토큰 버킷 (Token Bucket)|일정한 주기로 토큰을 채우고, 요청 시 토큰을 소진함|구현이 쉽고 메모리 효율적이며 짧은 트래픽, 폭주(Burst) 처리 가능| 버킷 크기와 리필율이라는 2개의 파라미터 튜닝이 까다로움
    새는 양동이 (Leaking Bucket)|FIFO 큐를 사용하여 요청을 일정한 속도로 처리함|큐 크기가 제한되어 메모리 효율적이며, 일정한 처리 속도 유지|트래픽 폭주 시 오래된 요청이 버려질 수 있고 튜닝이 어려움
    고정 윈도우 카운터 (Fixed Window Counter)|고정된 시간 단위로 카운터를 생성해 제한함|메모리 효율이 좋고 직관적임|윈도우 경계에 요청이 몰릴 경우 설정값 이상의 트래픽이 통과될 수 있음
    이동 윈도우 로그 (Sliding Window Log)|각 요청의 타임스탬프를 로그로 기록하여 관리함|경계점 문제가 없어 매우 정확함|모든 요청의 시간을 기록하므로 메모리 소비가 큼
    이동 윈도우 카운터 (Sliding Window Counter)|고정 윈도우와 이동 윈도우 로그의 하이브리드 방식|트래픽 폭주 시에도 계산이 고르게 분산되며 메모리 효율이 좋음|이전 윈도우의 요청이 균등하게 분포되어 있다는 가정이 깨지면 오차 발생 가능

5. 상세 설계 및 구현 요소
    - 데이터 저장소: 카운터 정보는 디스크보다 빠른 Redis와 같은 메모리 기반 캐시 시스템을 사용함 (INCR, GET 명령 활용)
    - 동적 설정 (Rules): 사용자별, 메시지 유형별로 다른 제한 규칙을 UI에서 편집하고 API 게이트웨이에 브로드캐스트하여 실시간 적용 가능
    - 에러 처리: 제한을 초과한 요청에는 HTTP 429(Too Many Requests) 에러를 반환하며, 헤더에 Retry-After 등의 정보를 담아 재시도 시점을 알려줌
    - 메시지 큐 활용: 거절된 요청을 즉시 버리지 않고 나중에 처리하고 싶을 때는 별도의 메시지 큐에 보관할 수 있음
