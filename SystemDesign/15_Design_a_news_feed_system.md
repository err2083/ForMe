# 뉴스 피드 시스템 디자인 (Design a News Feed System)

1. 뉴스 피드의 정의 및 요구사항
    - 정의: 사용자에게 자주 업데이트되는 콘텐츠를 제공하는 데이터 포맷으로, 나의 포스트를 팔로워에게 공유하고 내가 팔로잉하는 사람들의 포스트를 모아보는 서비스입니다
    - 기능적 요구사항:
        - 포스트 게시(Publish) 및 친구의 포스트 보기
        - 최신순 정렬(시간순 나열)
        - 사용자당 최대 친구 수: 5,000명
        - 미디어 파일(이미지, 비디오) 포함 가능
    - 비기능적 요구사항:
        - 일일 활성 사용자(DAU): 1,000만 명
        - 대용량 트래픽 처리를 위한 확장성 및 낮은 지연 시간
2. API 설계
    - 뉴스 피드 시스템은 크게 두 가지 핵심 API로 구성됩니다
    - 피드 발행 API: POST /v1/userId/feed
        - 인자: 포스트 내용(content), 인증 토큰(auth_token)
    - 뉴스 피드 가져오기 API: GET /v1/userId/feed
        - 인자: 인증 토큰(auth_token)
3. 피드 발행(Feed Publishing) 아키텍처
    - 사용자가 글을 쓰면 시스템은 다음과 같은 과정을 거칩니다
    - 웹 서버: 인증 및 **처리율 제한(Rate Limiter)**을 통해 스팸을 방지합니다
    - 포스트 서비스: 새 포스트를 DB와 캐시에 저장합니다
    - 팬아웃(Fan-out) 서비스: 나를 팔로잉하는 사람들의 뉴스 피드에 내 포스트를 전달합니다
    - 알림(Notification) 서비스: 친구들에게 새 글 알림을 보냅니다
4. 팬아웃(Fan-out) 전략
    - 가장 핵심적인 부분으로, 포스트를 팔로워들에게 어떻게 전달할지 결정합니다
    - Push 모델 (Fan-out-on-write): 글을 쓰는 시점에 팔로워의 피드 캐시에 직접 넣어줍니다
        - 장점: 실시간 생성이 가능하고 읽기 속도가 매우 빠름
        - 단점: 팔로워가 많은 유명인(Celebrity)의 경우 수백만 명에게 데이터를 밀어넣는 핫키(Hotkey) 문제가 발생함
    - Pull 모델 (Fan-out-on-read): 사용자가 피드를 읽는 시점에 친구들의 포스트를 가져옵니다
        - 장점: 비활성 사용자를 위해 피드를 미리 만들지 않아도 되어 효율적이며 핫키 문제가 없음
        - 단점: 피드를 읽을 때마다 생성해야 하므로 속도가 느릴 수 있음
    - 하이브리드 모델: 일반 사용자는 Push 방식을 사용하고, 팔로워가 너무 많은 유명인은 Pull 방식을 적용하여 핫키 문제를 해결합니다
5. 상세 설계 및 최적화
    - 친구 목록 관리: 관계 데이터를 저장하고 검색하기에 최적화된 그래프 DB를 사용하여 친구 목록을 가져옵니다
    - 메시지 큐 활용: 팬아웃 워커가 메시지 큐에서 작업을 가져와 뉴스 피드 캐시에 데이터를 입력함으로써 컴포넌트 간 **느슨한 연결(Loose Coupling)**을 유지합니다
    - 계층적 캐시 구조: 뉴스 피드, 콘텐츠(Hot/Normal), 소셜 그래프, 액션(좋아요 등), 카운터 등으로 캐시를 분리하여 효율을 높이고 실패율을 줄입니다
    - 미디어 처리: 사진이나 비디오는 CDN을 통해 사용자에게 빠르게 전달합니다
6. 데이터베이스 및 확장성 고려사항
    - DB 확장: 수직적 확장(Vertical)보다는 대규모 트래픽에 유리한 **수평적 확장(Horizontal Scaling)**과 NoSQL을 고려합니다
    - 복제(Replication): Master-Slave 구조를 통해 쓰기는 Master에서, 읽기는 Slave(Read Replicas)에서 처리하여 부하를 분산합니다
    - 일관성 모델: 금융권처럼 엄격한 데이터가 필요하면 **강한 일관성(Strong Consistency)**을, 그렇지 않다면 **최종 일관성(Eventual Consistency)**을 선택합니다
    - 기타: 무상태(Stateless) 웹 계층 설계로 탄력성을 확보하고, 여러 지역(Multi-region)을 지원하여 장애에 대비합니다
