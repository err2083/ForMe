# 마이크로서비스 설계 (Design Microservices)

1. 마이크로서비스 도입 배경 및 사례
    - 마이크로서비스는 대규모 서비스의 확장성과 독립적인 운영을 위해 실리콘밸리를 필두로 전 세계적으로 널리 사용되는 아키텍처입니다
    - 이커머스 플랫폼 (Amazon, eBay 등): 높은 트래픽 처리, 다중 트랜잭션 프로세스, 결제 게이트웨이 및 외부 서비스 통합 시 각 서비스의 독립성을 존중하기 위해 사용합니다
    - 스트리밍 서비스 (Netflix 등): 대용량 데이터 핸들링, 실시간 프로세싱, 고가용성(High Availability) 만족을 위해 도입합니다
    - 소셜 미디어 플랫폼: 수백만 유저를 대상으로 한 빈번한 A/B 테스팅과 새로운 기능(Feature)의 신속한 업데이트를 위해 필수적입니다
2. 마이크로서비스의 핵심 정의
    - 독립적 운영: 각 서비스는 아키텍처 내에서 핵심 비즈니스 단위를 담당하며, 특정 기능을 독립적으로 수행하고 운영됩니다
3. 시스템 요구사항 및 주요 컴포넌트
    - 마이크로서비스를 성공적으로 구축하기 위해 필요한 핵심 요소들은 다음과 같으며, 많은 부분이 쿠버네티스(Kubernetes) 환경 내에서 구현됩니다
    - 통신 및 트래픽 관리
        - API 게이트웨이 (API Gateways): 클라이언트 요청이 들어오는 단일 진입점(Single point of entry)으로, 요청을 적절한 마이크로서비스로 라우팅합니다
        - 로드 밸런서 (Load Balancer): 트래픽을 여러 서비스 인스턴스에 공정하게 분산하여 특정 인스턴스가 병목 지점이 되거나 장애가 발생하는 것을 방지합니다
        - 서비스 메시 (Service Mesh): 서비스 간 통신 보안(인증)과 트래픽 관리를 담당하며, 코드 변경 없이도 가시성(Observability)을 제공합니다
        - 메시지 브로커 (Message Broker): 서비스 간 비동기 통신과 결합도 완화(Decoupling)를 지원합니다. 대표적으로 Kafka, RabbitMQ 등이 사용됩니다
    - 서비스 관리 및 구성
        - 서비스 레지스트리 및 디스커버리: 서비스 인스턴스의 위치를 추적하고, 서비스들이 동적으로 서로를 찾아 통신할 수 있게 합니다
        - 구성 관리 (Configuration Management): 각 서비스의 설정 데이터를 중앙에서 관리하여 동적인 업데이트와 환경 간 일관성을 유지합니다. 쿠버네티스에서는 ETCD가 이 역할을 수행합니다
        - 데이터 매니지먼트: 각 서비스는 독립적인 데이터베이스를 관리(Decentralized data management)하며, 서비스 특성에 맞는 최적의 DB 기술을 선택할 수 있습니다
    - 운영 및 보안
        - 모니터링 및 로깅: 서비스의 건강 상태와 성능(CPU, 메모리, 트래픽 등)을 추적합니다. Prometheus, Grafana, Splunk 등의 도구가 활용됩니다
        - 분산 추적 (Distributed Tracing): 여러 서비스를 거치는 요청의 경로를 추적하여 성능 병목 지점이나 실패 지점을 파악합니다
        - 보안 (Security): 인증된 서비스와 유저만 접근할 수 있도록 보장합니다. OAuth, JWT, Role Binding 등의 기술이 사용됩니다
        - CI/CD 파이플라인: 코드 변경, 테스트, 업데이트 배포를 자동화하여 신기능과 버그 수정을 신속하게 릴리즈합니다. Jenkins, Spinnaker 등이 대표적입니다
