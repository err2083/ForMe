# 키-밸류 스토어 설계 (Design a Key-Value Store)

1. 키-밸류 스토어 개요
    - 정의: 키와 값의 쌍으로 데이터를 저장하는 비관계형 데이터베이스(NoSQL)입니다
    - 기본 연산: 주로 put(key, value)와 get(key) 명령을 통해 데이터를 쓰고 읽습니다
    - 특징: Redis가 대표적인 예시이며, 키는 일반 텍스트나 암호화된 해시 키(Hashed Key) 형태로 사용할 수 있습니다
2. 확장 가능한 아키텍처로의 여정
    - 싱글 서버 (Single Server): 구현과 관리가 쉽지만 저장 용량에 한계가 있습니다, 데이터 압축이나 자주 사용되는 데이터만 저장하는 최적화가 필요합니다
    - 분산 키-밸류 스토어 (Distributed Key-Value Store): 대규모 데이터를 다루기 위해 여러 서버에 데이터를 분산하며, 이때 CAP 정리를 반드시 고려해야 합니다
3. CAP 정리 (CAP Theorem)
    - 분산 데이터 시스템은 다음 세 가지를 동시에 모두 만족할 수 없으며, 최대 두 가지만 선택 가능합니다
    - Consistency (일관성): 어떤 노드에 접속하더라도 항상 가장 최신의 데이터를 읽어야 합니다
    - Availability (가용성): 일부 노드에 장애가 생겨도 요청에 대해 항상 응답을 받아야 합니다
    - Partition Tolerance (파티션 내결함성): 네트워크 장애로 인해 노드 간 통신에 문제가 생겨도 시스템이 계속 작동해야 합니다
    - 조합 예시:
        - CP 시스템: 가용성을 희생하고 일관성과 파티션 내결함성을 중시합니다 (예: HBase, MongoDB)
        - AP 시스템: 일관성을 희생(최신 데이터가 아닐 수 있음)하고 가용성과 파티션 내결함성을 중시합니다 (예: Cassandra, DynamoDB)
4. 핵심 시스템 컴포넌트
    - 데이터 파티션 및 복제
        - 데이터 파티션: **컨시스턴트 해싱(Consistent Hashing)**을 사용하여 데이터를 노드에 균등하게 분산하고, 노드 추가/삭제 시 데이터 이동을 최소화합니다
        - 데이터 복제: 높은 가용성과 신뢰성을 위해 데이터를 여러 노드에 복제합니다, 해시 링에서 시계 방향으로 만나는 N개의 노드에 저장하며, 장애 대비를 위해 다른 랙이나 리전에 분산 저장하는 것이 일반적입니다
    - 일관성 관리
        - 정족수 합의 (Quorum Consensus): N(복제본 수), W(쓰기 정족수), R(읽기 정족수) 값을 조절하여 일관성과 지연 시간(Latency) 사이의 균형을 맞춥니다
        - 강한 일관성 (Strong Consistency): W + R > N인 경우로, 항상 최신 데이터를 보장합니다 (예: 금융 거래)
        - 결과적 일관성 (Eventual Consistency): 일시적으로 불일치할 수 있으나 결과적으로는 싱크가 맞는 모델입니다 (예: 쇼핑 카트)
    - 장애 감지 및 처리
        - 가십 프로토콜 (Gossip Protocol): 각 노드가 하트비트(Heartbeat) 정보를 무작위 노드와 교환하여 시스템 전체의 장애 노드를 효율적으로 파악합니다
        - 힌티드 핸드오프 (Hinted Handoff): 일시적 장애 발생 시 다른 노드가 대신 쓰기 연산을 수행하고, 장애 노드 복구 후 데이터를 전달하여 일관성을 유지합니다
        - 머클 트리 (Merkle Tree): 데이터 블록의 해시 값을 트리 구조로 관리하여 대규모 데이터 간의 불일치를 빠르고 안전하게 비교하고 복구합니다
5. 데이터 흐름 (Write & Read Path)
    - 쓰기 경로 (Write Path): 클라이언트 요청 → 커밋 로그(Commit log) 기록 → 메모리 캐시 저장 → 일정 크기 도달 시 디스크의 **SSTable(Sorted-String Table)**로 플러시
    - 읽기 경로 (Read Path): 메모리 캐시 확인 → 없으면 **블룸 필터(Bloom Filter)**를 통해 디스크 내 데이터 존재 여부 확인 → SSTable에서 데이터를 찾아 반환
