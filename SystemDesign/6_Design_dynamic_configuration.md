# 동적 구성 시스템 설계 (Design Dynamic Configuration)

1. 동적 구성(Dynamic Configuration)의 정의
    - 동적 구성은 전체 시스템을 다시 시작하거나 재배포할 필요 없이 시스템 튜닝이나 작동 방식을 실시간으로 변경할 수 있게 해주는 기능입니다
    - 주요 유스케이스:
        - 속도 조절기(Rate Limiter)의 유닛당 요청 개수 조정
        - 새로운 기능을 특정 사용자에게만 노출하는 기능 게이트웨이(Feature Gateway)
        - A/B 테스트 및 노출률 조정
2. 설계 요구사항 (Requirements)
    - 데이터 타입: 문자열(String) 속성만 지원한다고 가정
    - 크기: 구성 데이터의 크기는 수십 메가바이트(MB)까지 커질 수 있음
    - 지연 시간: 구성 변경 시 클라이언트에 반영되는 시간은 10초 이하여야 함
    - 확장성: 최대 100만 대의 머신까지 접속 가능해야 함
    - 고가용성: 백엔드 저장소에 장애가 발생하더라도 서비스 자체에는 영향이 없어야 함
3. 설계 모델 비교
    - 풀 모델 (Pull Model)
        - 방식: 키-밸류 저장소에 설정값을 저장하고, 클라이언트가 주기적으로 **폴링(Polling)**하여 변경 사항을 확인합니다
        - 사례: 넷플릭스의 오픈소스인 Archaius가 이 방식을 사용합니다
        - 단점: 저장소 장애 시 최신 값을 가져오지 못하고 에러가 지속되며, 시스템 재시작 시 최신 값을 로드할 수 없는 문제가 발생할 수 있습니다
    - 푸시 모델 (Push Model / Observer Pattern)
        - 방식: 상태가 바뀌었을 때 Subject(설정값)가 등록된 Observer(클라이언트)에게 통보해 주는 방식입니다
        - 핵심 도구: **아파치 주키퍼(Apache ZooKeeper)**가 이 모델을 지원합니다
4. 아파치 주키퍼(Apache ZooKeeper) 활용
    - 주키퍼는 분산 환경의 조율(Coordination)을 담당하는 시스템으로, 동적 구성에 적합한 기능을 제공합니다
    - 구조: 유닉스 파일 시스템과 유사한 계층적 트리 노드 구조를 가집니다
    - Watch 기능: 특정 노드에 Watch를 등록하면 해당 노드의 상태가 변할 때 클라이언트로 즉시 이벤트가 통보됩니다
    - 팁 (신규 속성 처리): 존재하지 않는 노드에는 Watch를 걸 수 없으므로, 부모 노드에 Watch를 설정하여 하위 자식 노드의 추가/삭제/변경 이벤트를 모두 감지합니다
5. 고급 아키텍처: 사이드카(Sidecar) 및 로컬 캐싱
    - 주키퍼 자체의 장애나 인스턴스 재시작 시의 안정성을 확보하기 위해 사용되는 패턴입니다
    - 사이드카 패턴: 메인 응용 프로그램의 구성 요소를 별도의 프로세스나 컨테이너로 분리하여 배포하는 방식입니다
    - 동작 흐름
        - ZooKeeper: 속성 값들을 트리 노드 형태로 저장.
        - Agent (Sidecar): 주키퍼와 싱크를 맞추며 변경 사항 발생 시 TreeCache 기능을 이용해 로컬 디스크에 덤프(파일 저장).
        - Local Cache: 로컬 디스크의 특정 디렉토리에 파일로 저장된 설정값.
        - Main App: 로컬 파일 시스템을 모니터링하다가 파일이 업데이트되면 다시 읽어 메모리에 적재.
    - 장점: 주키퍼가 중단되어도 로컬 캐시를 통해 기존 설정값을 유지하며 서비스가 가능합니다
6. 활용 기술 및 오픈소스
    - ZooKeeper 클라이언트: Apache Curator(TreeCache 기능 제공), Kazoo(Python)
    - 파일 모니터링 모듈: Watchdog(Python), fsnotify(Go)
