# 채팅 시스템 설계 (Design a Chat System)

1. 요구사항 분석 (Requirements)
    - 채팅 시스템은 목적에 따라 설계가 크게 달라지므로 범위를 확정하는 것이 중요합니다
    - 기능적 요구사항
        - 1:1 채팅 및 소규모 그룹 채팅(최대 100명) 지원
        - 사용자 접속 상태 표시(Online Presence)
        - 다중 디바이스 지원 및 동시 로그인 허용
        - 푸시 알림(Push Notification)
    - 비기능적 요구사항:
        - 5,000만 DAU (Daily Active Users) 처리 가능성
        - 메시지 전송 지연 최소화 (Low Latency)
        - 안정성(Reliability): 메시지 손실 방지
2. 통신 프로토콜 선택
    - 발신자(Sender)와 수신자(Receiver) 간의 효율적인 통신 방식이 핵심입니다
    - HTTP (Keep-Alive): 발신자가 메시지를 보낼 때 사용. TCP 핸드셰이크 오버헤드를 줄이기 위해 Keep-Alive 헤더를 활성화합니다
    - 수신 방식의 비교:
        - 폴링(Polling): 클라이언트가 주기적으로 서버에 메시지 유무를 확인. 서버 부하가 크고 비효율적입니다
        - 롱 폴링(Long Polling): 메시지가 올 때까지 연결을 유지. 서버 가용성이 떨어지고 구현이 복잡합니다
        - 웹소켓(Websocket): 양방향(Bi-directional) 및 지속적 연결을 지원하는 가장 일반적인 해결책입니다. 낮은 지연 시간으로 리얼타임 통신에 적합합니다
3. 하이레벨 아키텍처 (High-Level Design)
    - 서비스를 성격에 따라 크게 세 가지 카테고리로 나눕니다
    - 무상태 서비스 (Stateless Services): 로그인, 회원가입, 사용자 프로필 관리 등 일반적인 HTTP 요청/응답 기반 API 서버입니다
    - 상태 유지 서비스 (Stateful Services): 채팅 서버와 접속 상태(Presence) 서버입니다. 클라이언트와의 연결을 유지해야 하므로 확장이 까다롭습니다
    - 제3자 서비스 (Third-party Integration): 앱이 꺼져 있을 때 알림을 보내는 푸시 알림 서비스입니다
4. 스토리지 모델 (Storage Model)
    - 데이터의 특성에 따라 적합한 데이터베이스를 선택합니다
    - 일반 데이터 (User Profile, Setting, Friend List): 안정성이 검증된 **관계형 데이터베이스(RDBMS)**를 사용하며, 확장성을 위해 복제(Replication) 및 샤딩(Sharding)을 적용합니다
    - 채팅 이력 (Chat History):
        - 특성: 방대한 양의 데이터(하루 약 600억 메시지), 최근 대화 위주의 접근 패턴, 검색 및 멘션 기능 필요
        - 선택: 수평적 확장이 용이하고 읽기/쓰기 지연 시간이 낮은 **Key-Value 저장소(NoSQL)**가 유리합니다
    - 메시지 ID 생성: 유니크해야 하며 시간순으로 정렬 가능해야 합니다. 로컬 시퀀스 번호 생성기를 쓰면 그룹 내에서만 유니크함을 보장하면 되므로 구현이 더 간단합니다
5. 상세 설계 (Deep Dive)
    - 서비스 디스커버리 (Service Discovery)
        - 사용자에게 가장 적합한(위치, 용량 등 기준) 채팅 서버를 할당합니다. Apache Zookeeper와 같은 도구를 사용하여 가용 서버 리스트를 관리하고 어사인합니다
    - 메시지 동기화 및 흐름
        - 다중 디바이스 동기화: cur_max_message_id 변수를 사용하여 각 디바이스가 마지막으로 읽은 메시지 이후의 데이터만 가져오도록 설계합니다
        - 소규모 그룹 채팅: 각 멤버마다 별도의 메시지 큐(Inbox)를 두어 관리하면 설계가 단순해집니다. 단, 그룹이 커지면 비용이 많이 발생합니다
    - 접속 상태 관리 (Presence Server)
        - 하트비트(Heartbeat): 네트워크가 불안정한 모바일 환경에서 잦은 연결/끊김으로 인한 오판을 막기 위해, 클라이언트가 주기적으로 하트비트 이벤트를 서버에 보내 온라인 상태를 확인합니다
        - 상태 정보 전송: 친구들이 내 상태를 알 수 있도록 발행/구독(Pub/Sub) 모델을 사용합니다
