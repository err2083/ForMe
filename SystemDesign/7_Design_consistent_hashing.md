# 컨시스턴트 해싱 (Consistent Hashing)

1. 컨시스턴트 해싱의 개요
    - 정의: 해시 함수를 이용해 요청이나 데이터를 특정 서버에 할당하는 방식입니다
    - 목적: 주로 로드 밸런싱에서 들어온 요청을 서버에 효율적으로 분배하기 위해 사용됩니다
    - 핵심 특징: 서버가 추가되거나 삭제될 때 재할당되는 데이터의 양을 최소화하여 시스템의 확장성과 유지 보수성을 높입니다
2. 기존 해싱 방식의 문제점
    - 방식: 보통 서버 인덱스 = hash(key) % N (서버 수) 공식을 사용합니다
    - 문제점 (Rehash 문제): 서버의 수가 변하면(N이 변경되면) 거의 모든 키의 해시 결과값이 변하게 되어, 대규모 데이터 재분배가 발생합니다, 이는 시스템에 큰 부하를 주고 비효율적입니다
3. 컨시스턴트 해싱의 동작 원리
    - 이 방식을 통해 서버와 키의 할당 관계를 효율적으로 관리합니다.
    - 해시 링 (Hash Ring)
        - 해시 함수(예: SHA-1)의 출력 범위를 일렬로 나열한 뒤, 처음과 끝을 연결하여 원형으로 만든 것입니다
        - 이 링 위에 서버와 요청(키)을 각각 배치합니다
    - 서버 할당 방식
        - 각 요청(키)은 해시 링 위에서 시계 방향으로 가장 가까운 서버에 할당됩니다
        - 서버 추가: 서버가 추가되면 새로 추가된 서버와 그 전 서버 사이에 있는 일부 키들만 재배치됩니다, 다른 서버의 키들은 영향을 받지 않습니다
        - 서버 삭제: 서버가 삭제되면 해당 서버가 담당하던 키들만 시계 방향의 다음 서버로 이동합니다
4. 가상 노드 (Virtual Nodes)
    - 문제: 물리 서버만 배치할 경우 서버 간 간격이 불균등하여 특정 서버에 부하가 집중될 수 있습니다
    - 해결책: 각 물리 서버를 가리키는 **가상 노드(vnode)**를 링 위에 균일하게 여러 개 배치합니다
    - 효과: 가상 노드의 개수를 늘릴수록 부하가 더욱 균등하게 분산되어 자원을 효율적으로 활용할 수 있습니다
5. 장점 및 실제 활용 사례
    - 장점:
        - 서버 추가/삭제 시 이동해야 하는 데이터 양이 적어 시스템 확장이 용이합니다
        - 요청이 골고루 분배되어 자원 활용도가 높습니다
        - 일부 서버 장애 시에도 시스템 가용성을 유지하기 쉽습니다
    - 활용 사례:
        - Amazon DynamoDB: 파티셔닝 컴포넌트
        - Apache Cassandra: 클러스터 데이터 파티셔닝

        - Akamai/NGINX: CDN 및 로드 밸런싱 알고리즘 옵션
