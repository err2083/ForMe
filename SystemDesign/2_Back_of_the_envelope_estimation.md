# 개략적인 규모 추정 (Back-of-the-envelope Estimation)

1. 개요 및 필요성
    * 정의: 편지 봉투 뒤에 간단히 메모하듯, 가벼운 대략적인 숫자 예측을 통해 시스템 사이징을 하는 기법입니다
    * 필요성: 시스템 디자인 인터뷰나 실무에서 사용자 수, 트래픽, DB 사이즈 등에 대한 질문에 논리적으로 답변하기 위해 필요합니다
    * 핵심: 정답을 맞히는 것보다 자신만의 수치와 로직을 가지고 합리적인 가정을 세우는 과정이 중요합니다

2. 반드시 알아야 할 시스템 사이징 용어
    * 사용자 관련:
        * Concurrent Users: 특정 시간 내에 시스템을 사용 중인 사용자 수 (시스템 사이징 시 주요 지표)
        * Simultaneous Users: 동일 시점에 동일한 트랜잭션을 발생시키는 사용자 수 (성능 테스트 시 중요)
        * DAU/MAU: 일간/월간 활성 사용자 수
    * 성능 관련:
        * TPS (Transactions Per Second): 초당 트랜잭션 수 (예: 1페이지 로딩 = 1 TPS)
        * QPS (Queries Per Second): 초당 쿼리 수 (예: 1페이지 로딩 시 N개의 쿼리 발생 가능)
        * Throughput: 주어진 시간 내 서버가 처리할 수 있는 작업량 (TPS 또는 Bps 단위)
        * Response Time: 응답 시간 (쾌적한 UX를 위해 보통 200ms 이내 권장)

3. 엔지니어가 기억해야 할 주요 수치
    * 데이터 단위: 1KB (2^10), 1MB (2^20), 1GB (2^30), 1TB (2^40), 1PB (2^50)
    * 레이턴시(Latency) 감각:
        * 메모리는 디스크보다 훨씬 빠릅니다. (L1 캐시: 0.5ns, 메인 메모리: 100ns vs 디스크 탐색: 10ms)
        * 네트워크 전송은 비용이 큽니다. 따라서 데이터를 보내기 전에 **압축(Compress)**하는 것이 효율적입니다
        * 타 리전(지역)으로의 데이터 전송은 상당한 지연을 초래합니다

4. 규모 추정 인터뷰 팁 (Estimation Strategy)
    * 가정 세우기(Assumption): 3억 명의 유저, 50%가 매일 접속 등 계산하기 편한 숫자로 가정을 설정하고 화이트보드에 기록합니다
    * 숫자 단순화: 86,400초(하루)를 100,000초로 반올림하여 계산하는 등 **계산하기 쉬운 수치(Round numbers)**를 선택합니다
    * 단위 명시: 숫자 뒤에 KB, MB, GB 등 단위를 정확히 기재하여 혼동을 방지합니다
    * 5가지 핵심 요소: Load, Storage, Bandwidth, Latency, Resource를 기준으로 시스템을 디자인해 나갑니다

5. 사례 비교 (SNS vs 홈페이지)
    * SNS (트위터 등): 유저 수가 많고 쓰기 비중(Posting)이 높아 막대한 스토리지와 높은 QPS가 필요합니다 (예: 3억 MAU 기준, 5년치 데이터 약 55PB 필요)
    * 홈페이지: 사용자 수와 활동 빈도가 낮아 상대적으로 요구되는 자원이 적고 QPS도 매우 낮게 형성됩니다
