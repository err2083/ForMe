# 분산 환경에서의 유일 ID 생성기 설계

1. 시스템 디자인 프로세스 (4단계)
    - 시스템 디자인은 사용자의 요구사항과 시스템이 구현해야 할 기능을 받아 구현 가능한 블루프린트로 만드는 과정이며, 다음의 4단계를 따릅니다
    1. 사용자 요구사항 분석 및 디자인 스코프 정의
    2. 하이레벨 디자인(High-Level Design) 검토 및 선택
    3. 상세 설계 (Design Deep Dive)
    4. 디자인 리뷰 (Design Review)
2. 요구사항 및 디자인 스코프 (Step 1)
    - 유일 ID는 보안, 액세스 제어, 사용자 세션 유지 및 데이터 무결성 보장을 위해 필수적입니다, 이번 설계의 주요 목표는 다음과 같습니다.
    - 특성: 숫자로 된 유일한 값(Numerical value)이어야 하며, 64비트(8바이트) 크기를 가집니다
    - 정렬: ID가 생성된 시간 순서대로 정렬(Sorting)이 가능해야 합니다
    - 성능: 초당 약 10,000(10k)개의 ID를 생성할 수 있어야 합니다
    - 가용성: 시스템은 절대 중단되지 않는 높은 복원력(Highly Resilient)을 갖춰야 합니다
3. 하이레벨 디자인 비교 (Step 2)
    - 유일 ID 생성을 위해 검토 가능한 네 가지 주요 어프로치는 다음과 같습니다
    |||||
    |---|---|---|---|
    방식|설명|장점|단점
    티켓 서버 (Ticket Server)|중앙의 전용 서버가 ID를 생성해 배분함|구현이 매우 간단함|서버 장애 시 전체 시스템이 마비되는 SPOF(Single Point of Failure) 문제 발생
    멀티 마스터 복제|여러 서버가 홀수/짝수 등 정해진 간격으로 ID를 생성함|가용성이 좋아지며 확장이 가능함|여러 데이터 센터에 걸친 확장(Scaling)이 어렵고, 서버 간 시간 순서 보장이 힘듦
    UUID 서버|유니버설 유니크 아이디를 생성함|서버 추가 시 중복 걱정 없이 쉽게 확장 가능함|16바이트로 길고, 숫자가 아닌 문자열이며, 시간순 정렬이 불가능함
    트위터 스노우플레이크|여러 필드를 조합한 Packed ID 방식을 사용함|모든 요구사항(숫자, 8바이트, 정렬 가능)을 만족하며 확장이 용이함|설계가 상대적으로 정교함
4. 상세 설계: 트위터 스노우플레이크 (Step 3)
    - 가장 적합한 모델인 트위터 스노우플레이크(Twitter Snowflake) 방식은 64비트 ID를 4개의 섹션으로 나누어 구성합니다
    - 타임스탬프 (Timestamp): 가장 앞부분(MSB 쪽)에 위치하여 ID의 시간 순서 정렬을 보장합니다. 단위는 보통 밀리초(ms)를 사용합니다
    - 데이터 센터 ID & 머신 ID: 서비스 규모에 맞게 비트 길이를 할당하여 여러 지역과 서버에서 독립적으로 ID를 생성할 수 있게 합니다
    - 일련번호 (Sequence Number): 동일한 밀리초 내에 여러 요청이 들어올 경우를 대비해 순차적으로 번호를 부여합니다
5. 디자인 리뷰 및 고려사항 (Step 4)
    - 설계 완성 후 동료 엔지니어들과 다음 사항들을 논의하여 품질을 높입니다
    - 시계 동기화 (Clock Synchronization): 여러 서버의 타임스탬프를 일치시키기 위해 NTP(Network Time Protocol) 등을 활용하는 방안을 논의합니다
    - 고가용성 (High Availability): ID 생성 서버의 적정 개수를 설정합니다
    - 비트 길이 할당: 각 필드(타임스탬프, ID, 시퀀스 등)의 구체적인 길이를 정합니다
    - ID 할당 정책: 데이터 센터와 머신 ID를 지역적으로 어떻게 분배할지 결정합니다
